* Decisions

- Fix the root cause (Widgets-free logic living inside a
  Widgets-requiring component, forcing duplicate compilation into a
  test binary) rather than the symptom (defining
  =ORES_QT_API_LIBRARY= for the test target too, which would have
  silenced the specific MSVC warning in minutes). The quick fix was
  explicitly proposed and rejected in favour of tracing the smell.
- New component's header namespace and export macro
  (=ores.qt.headless/=, =ORES_QT_HEADLESS_API=) are deliberately
  distinct from =ores.qt.api='s own (=ores.qt/=, =ORES_QT_API=), even
  though every other =ores.qt.*= plugin shares the =ores.qt/= header
  namespace by convention — because =ores.qt.api= consumes this
  component's headers directly, so both export.hpp headers can end up
  included in the same translation unit, and a shared macro name would
  silently clash (last =#define= wins) instead of failing loudly.

* Out of scope

- Splitting any other =ores.qt.*= sub-component the same way — this
  story only extracted the files that were *already* being
  duplicate-compiled into a test binary (the actual source of the
  bug), not a general Widgets/non-Widgets audit of the rest of the
  codebase.
