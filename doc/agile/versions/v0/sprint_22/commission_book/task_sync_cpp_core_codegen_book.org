* Plan

Regenerated book's C++ core (api/core/sql) from the graph-based
codegen, then moved its Qt layer from =ores.qt/trading= to
=ores.qt/refdata= to match the new codegen output location. Manual UI
testing (login, list, save, rename, history) surfaced several bugs
beyond the mechanical regen, each fixed and, where the root cause was
generic, landed as a reusable codegen capability rather than a
Book-only patch:

- Cross-plugin symbol-lookup crash on login: =BookController= needed
  =ORES_QT_REFDATA_EXPORT= since =TradingPlugin= constructs it but it
  lives in =libores.qt.refdata.so= (built with
  =-fvisibility=hidden=). Landed as a new opt-in codegen capability,
  =has_export_macro=.
- =book_changed_event= field mismatch (=e.ids= vs =e.book_ids=) broke
  the build.
- Book saves silently failed: =has_change_reason_cache= was never
  wired for Book, so =promptChangeReason= had no cache to read from.
- Renaming a book silently no-op'd (version bumped, name unchanged):
  =name= was wrongly modelled as an immutable =is_key= field — that's
  a uniqueness constraint, not an immutability one. Real trading
  systems allow renaming books (GL mapping/id are the durable
  identity, not the display name). Removed =is_key= from =name=;
  surfaced the real UUID =id=, read-only, in the detail dialog instead.
- Fixed two codegen bugs surfaced by giving Book a uuid-typed =is_key=
  field for the first time: the primary key wasn't checked by the
  detail-field uuid-type lookup, and the =is_key= UI->entity sync path
  unconditionally did a =std::string= assignment even for uuid fields
  (wouldn't compile). Also made =is_key= fields statically
  =readOnly="true"= in the generated =.ui=, since not every detail-window
  code path was calling =setReadOnly()=.
- Closing a Book history window closed the whole app: shared
  =HistoryDialogBase='s Close button called =window()->close()=
  directly, which resolves to the main window (not the containing MDI
  subwindow) when embedded. Added =closeRequested()=/=requestClose()=
  mirroring =DetailDialogBase='s existing pattern, wired generically
  into every codegen'd controller's =showHistoryWindow= — fixes this
  for every entity's history dialog, not just Book's.
- PR review (three independent automated passes) caught that the
  initial regeneration had silently dropped all five of Book's insert
  trigger validations (soft FKs on =parent_portfolio_id=/=party_id=/
  =owner_unit_id=; value checks on =ledger_ccy=/=book_status=), because
  =book.org= never had a =* Foreign keys= / =* Insert trigger= section
  for the codegen to draw from. Added the missing sections (mirroring
  =ores.refdata.counterparty.org=), regenerated, and verified the
  restored trigger is semantically identical to the pre-regen SQL (diff
  + full =compass db recreate=).
