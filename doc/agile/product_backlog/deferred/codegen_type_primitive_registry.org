#+title: Introduce type primitive registry to replace ad-hoc type flags in codegen
#+description: Replace the growing set of is_uuid/is_date/is_timestamp/is_enum flags in codegen with a principled type registry mapping type aliases to their SQL, entity, and domain representations.
#+type: capture
#+level: cross
#+filetags: :codegen:architecture:marketdata:refactor:
#+created: 2026-07-01
#+updated: 2026-07-11

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *deferred* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

The codegen currently has a growing set of ad-hoc boolean flags in
=projects/ores.codegen/src/codegen/core.py= that special-case type
conversions between the SQL/entity layer and the domain layer:
=is_uuid=, =is_date=, =is_timestamp=, =is_nullable_string=,
=is_optional_uuid=, =is_optional_timestamp=, and now =is_enum=.  Each
new flag requires edits in three places: =core.py= (detection logic),
the entity header Mustache template (entity struct emission), and the
mapper implementation template (conversion expression).  The flags are
mutually exclusive by construction but nothing enforces that, and each
new type requires knowing the existing flag topology to avoid
conflicts.

The correct fix is a *type primitive registry* — a lightweight version
of the language-agnostic model (LAM) typemap concept from MASD/dogen.
Each entry in the registry maps a short type alias (used in org model
column =:ores_type:= properties) to a full descriptor:

| Field              | Meaning                                                              |
|--------------------+----------------------------------------------------------------------|
| =sql_type=         | Postgres column type (=text=, =uuid=, =timestamptz=, ...)           |
| =domain_cpp_type=  | C++ type in the domain struct (=boost::uuids::uuid=, =asset_class=) |
| =entity_cpp_type=  | C++ type in the entity struct — what sqlgen sees (=std::string=, =db_timestamp=) |
| =domain_includes=  | List of headers to inject into the domain header                     |
| =entity_to_domain= | Expression template for mapper: entity value → domain value          |
| =domain_to_entity= | Expression template for mapper: domain value → entity value          |
| =nullable_wrapper= | How to wrap when =nullable: true= (e.g. =std::optional<{type}>=)    |

Built-in types get hardcoded entries (=string=, =uuid=, =bool=, =int=,
=timestamp=, =date=).  User-defined types — C++ enum classes, value
objects, strong typedefs — get entries in a table in the meta model
org file or a companion YAML file loaded by =core.py=.

In the entity org model, a column currently written as:

#+begin_src org
** asset_class
:PROPERTIES:
:type:     text
:cpp_type: std::string
:nullable: false
:END:
#+end_src

becomes:

#+begin_src org
** asset_class
:PROPERTIES:
:ores_type: ores.marketdata.domain.asset_class
:nullable:  false
:END:
#+end_src

=core.py= looks up =ores.marketdata.domain.asset_class= in the
registry and derives all four template variables (entity type, domain
type, includes, conversion expressions) without any new =is_X= flag.
The Mustache templates collapse to a single generic conversion branch:

#+begin_src mustache
{{#has_conversion}}
    r.{{name}} = {{{entity_to_domain_expr}}};
{{/has_conversion}}
{{^has_conversion}}
    r.{{name}} = v.{{name}};
{{/has_conversion}}
#+end_src

The include association is critical: today the domain header template
has no way to know that emitting =domain::asset_class= requires
=#include "ores.marketdata.api/domain/asset_class.hpp"=.  With the
registry, the includes travel with the type definition and are injected
automatically — no manual =** Domain includes= paste block needed for
registered types.

Migration path: introduce the registry with entries for the new
user-defined types first (enum classes).  Retire the existing
hardcoded =is_uuid=/=is_date=/=is_timestamp= flag logic incrementally
in a follow-up, replacing them with registry lookups.  Both can coexist
during the transition because =core.py= can fall back to flag-based
detection when no =:ores_type:= property is present.

* Why

Every type that has a different SQL representation from its domain
representation currently forces three-file edits and a new boolean
flag.  The pattern broke down visibly when adding =year_month_day= and
=time_point= natural keys (two separate flag pairs, two mapper
branches, hardcoded includes in templates).  Adding C++ enum support
without the registry would be a fourth instance of the same pattern.
The registry makes adding a new mapped type a single-file change (one
row in the registry table) with zero template edits.

The include-association problem is equally important: currently the
entity org model has a =** Domain includes= section that must be kept
manually in sync with whichever =cpp_type= values are used.  If a
column uses =domain::asset_class= but the author forgets to add the
include, the build fails with a cryptic "incomplete type" error rather
than a clear codegen diagnostic.  The registry eliminates this class
of error entirely.
