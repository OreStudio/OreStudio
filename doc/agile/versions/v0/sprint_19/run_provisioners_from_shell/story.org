* Goal

Today a system can only be provisioned through the Qt wizards: the
system provisioner (bootstrap admin, base system, initial tenant), the
tenant provisioner (bundles, data sources, parties) and the party
provisioner (counterparties, reports, activation). This story makes
all three flows runnable from =ores.shell=, covering everything the
wizard pages can do — though not as a 1-1 mapping: the shell is a
different medium, and the command surface should be designed
idiomatically for it (see =* Decisions=). It also delivers a library of
org-mode-authored ores-shell scripts — tangled via an emacs batch
build — including an end-to-end script that reproduces the manual
provisioning sequence: create =super_admin= (password
=Secure-Password-123=), install the base system, provision the tenant
with a =tenant_admin= account, run party provisioning, with all the
logins and logoffs in between. An error-free run ends with a fully
provisioned system.

* Acceptance

- All three provisioner wizards (system, tenant, party) have complete
  shell equivalents: everything each wizard page configures or
  executes is achievable from the shell, with no GUI-only steps left.
  The command surface is designed for the shell medium, not a 1-1
  transliteration of the wizard pages.
- An ores-shell script library exists, authored as org-mode files with
  detailed commentary, tangled to runnable =.ores= scripts by an emacs
  batch build target.
- The end-to-end provisioning script runs against a fresh database
  without errors and produces a fully provisioned system
  (=super_admin=, base system, tenant with =tenant_admin=, party
  setup), including the logins and logoffs between stages.

* Decisions

- /No 1-1 wizard-to-shell mapping./ The shell does not replicate the
  wizard UI. The wizard pages define the required capability coverage
  — every input and backend effect must be reachable — but the command
  surface should be designed for the medium of ores-shell: think about
  what makes the interaction smoother there (sensible defaults, one
  command spanning what the GUI splits across several pages, flags
  instead of sequential prompts, scriptability first). Where the shell
  medium suggests a better interaction shape than the wizard's, prefer
  it.

* Out of scope

- Changes to the Qt wizards themselves; they remain the interactive
  path.
- Interactive, wizard-style guided prompting inside the shell — this
  story delivers scripted/command-line provisioning only.
