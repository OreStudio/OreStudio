* Goal

Regression coverage in =ores.iam.core= proving the generated cache
(from the =nats-event-cache= facet, migrated onto party) actually
works: publishing =ores.refdata.party_changed= for a given tenant
causes the cache to reload that tenant's partition, and
=lookup_party=/=compute_visible_party_ids= reflect the updated data
without any restart. Without this, the original silent-staleness bug
could regress unnoticed the same way it went unnoticed originally —
and since this is now generated code, the test should be written so
it would also protect any future entity onboarded onto the facet.

session):

  against the already-running =ores.refdata.service=/=ores.iam.service=.
- =ores.iam.service.0.log= shows repeated
  =party_cache= "Loaded 82 parties for tenant ..." reloads during
  provisioning (the =ores.refdata.party_changed= notifications firing
  and being picked up), then a subsequent =tenant_admin@barclays_plc=
  login (=auth_handler= "Completed ... auth.login") with no "still
  marked Inactive" warning logged anywhere.
- DB directly confirms =BARCLAYS PLC= =status = 'Active'=, consistent
  with what the login saw via the cache (not a stale read).
- This is the real, minimal reproduction of the original bug's fix:
  IAM up the whole time, activation via the normal message path,
  login reflecting it without a restart.
