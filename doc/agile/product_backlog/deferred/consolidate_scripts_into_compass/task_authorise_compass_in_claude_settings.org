for LLM workflows — without paying a permission prompt per
invocation, on every worktree, without manual approvals accumulating

settings skills=), with =compass bearings= warning when the generated
copies go stale relative to their org sources.

* Notes

** Why this matters

Compass is purpose-built to simplify LLM workflows — orientation
([[id:FE388774-0714-4D93-AEB0-D6221314D4F9][How do I orient a new session?]]), navigation
([[id:BDA8F7FB-5CEC-42D8-BC78-766BBC8D4285][How do I search docs with compass?]],
[[id:DEAC11AE-3753-49FA-994A-98594162A605][How do I index notes for compass?]]), agile authoring
([[id:5B670808-D28B-4818-A2FA-EA5220D35E8D][How do I add a doc with compass?]],
[[id:84C9177E-C737-428A-BDF9-E55FA6C9680C][How do I start a unit of work with compass?]]), session
continuity ([[id:B4C7D2E1-9F3A-4A6B-8E5D-3C1F7B2A9D4E][How do I use the session journal?]],
[[id:F3373CC1-C2B3-4371-A63F-FBEC54297265][How do I know what I was working on?]]), and status
([[id:A7D3C1E5-9F2B-4A8D-B6E4-3C7F1D9A2E5B][How do I get a sprint or story status?]],
[[id:0D378978-7CF1-45F5-896F-63E281CC02CB][How do I see what every worktree is doing?]]). Project
memory instructs the LLM to [[id:C6735DF2-B905-4A85-BAAA-B7F5CBBEC92D][prefer compass over Unix tools]] and to
[[id:7F176058-4FF7-4731-920B-26FE318E338F][always use compass for agile artefacts]]. A tool we mandate on
every session must not pay a permission prompt per invocation.

** Evidence of the current friction

- The checked-in allowlist covers
  =Bash(./projects/ores.compass/compass.sh *)= — but CLAUDE.md's first
  instruction is =./compass.sh bearings=, the *root wrapper*, which is
  not covered. Every session starts with a prompt.
- =settings.local.json= has accumulated 10+ ad-hoc compass entries
  (=bash=-prefixed, env-var-prefixed, subcommand-specific, relative and
  =./=-prefixed variants) — each one a manual approval someone clicked,
  none of them shared across worktrees or contributors.

** Options considered

1. /Blanket allow, checked in/ (CHOSEN): allow both invocation forms
   for all subcommands in [[proj:.claude/settings.json][=.claude/settings.json=]].
2. /Read-only subset/: allow =list/show/search/where/status/journal
   where/bearings/fleet=; keep prompting for =add/task new/journal
   update/env=. Rejected: the friction lands exactly on the
   highest-frequency LLM flows (scaffolding, journaling), and the
   "safety" is illusory — the same session already holds =Edit(**)=,
   =Write(**)= and =git *=, so it can do by hand, unprompted,
   everything the mutating compass commands do.
3. /Per-user settings.local.json/: the status quo. Demonstrably
   produces drift (see evidence above) and helps neither fresh
   worktrees nor other contributors.

** Discovery: the pipeline already existed; the decision evolved

Three rounds of design happened inside this task, worth recording:

1. /First attempt: check in .claude/settings.json./ We found
   =.gitignore= blanket-ignores [[proj:.claude][=.claude/=]], so the "shared" allowlist
   was never in git — every worktree carried a private copy. The
   first fix narrowed the ignore to track =settings.json=. Marco
   overruled: [[proj:.claude][=.claude/=]] must never be checked in; users recreate it
   from org-mode builds.
2. /Second discovery: the literate pipeline already existed./
   [[id:2D83251F-0E24-4B97-B736-08F710316634][Claude Code settings]] is the checked-in, peer-reviewed literate
   source (per-entry rationale, noweb-assembled master block), tangled
   to [[proj:.claude/settings.json][=.claude/settings.json=]] by the =deploy_settings= CMake target —
   exactly the skills model ([[proj:doc/llm/skills][=doc/llm/skills=]] → =deploy_skills= →
   [[proj:.claude/skills][=.claude/skills/=]]). The live settings had silently drifted from
   the org source, which is precisely the failure mode the generated
   approach prevents when regeneration is habitual.
3. /Third round: where does regeneration live in compass?/ First cut
   was =compass env claude=; rejected — Provision is about runtime
   environment (.env, certs, keys). Generating artifacts from
   checked-in sources is the Build pillar, which already aliases
   =site= and =manual= onto deploy targets. Final: =compass build
   settings= and =compass build skills= aliases — zero new machinery,
   and =compass build settings skills= recreates all of [[proj:.claude][=.claude/=]].

** Rationale for the chosen model

- Compass is repo-owned, version-controlled tooling; its mutating
  surface is repo-local files (doc scaffolds, =.journal.org=, =.env=)
  and git branch creation — all reversible, all reviewable in the PR
  that carries them.
- Authorising the entry point implicitly authorises whatever compass
  shells out to (git, gh, psql) /through compass/ — that is the
  point: compass is the trusted boundary, and its subcommands are the
  audited, recipe-documented way to do those operations
  ([[id:28F78E07-834D-4755-BB03-89BD7C17B01E][Compass basic command syntax]]).
- Checked-in settings make the authorisation uniform across all five
  worktrees and any future contributor, and changes to it are
  reviewed like any other code.
