This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:CB7351E9-8E7E-4BA4-9399-4EB22F6192D4][Commission Acme Bank: a realistic holding-group test entity]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Task 2 of the Acme provisioning architecture, widened at the user's
request to include account contact information in the same pass —
folding it in here rather than as a fully separate later task means
one round of end-to-end testing covers both, instead of two.

Two halves:

1. *Codegen*: extend =ores.codegen= to turn the =acme_bank= JSON
   dataset ([[id:9E96F1DB-85DE-415C-AE83-2D8BDD9D7158][Generate Acme Bank source data]]) into populate SQL matching
   hand-written =ores.sql= conventions — DQ artefact rows for LEI
   entities/relationships, business units, portfolios, books, and
   accounts (a new artefact shape), one dataset per legal entity.
2. *Account contact information*: accounts have no name field today
   (only username/email — see the parent story's discussion). Model
   it properly:
   - =ores_iam_account_contact_informations_tbl=, a junction mirroring
     the existing =refdata_party_contact_informations_tbl=/
     =refdata_counterparty_contact_informations_tbl= pattern, extended
     with a real name field.
   - A corresponding DQ artefact table + dataset, generated as part of
     the same JSON→SQL pass (Acme's staff get real names, not just
     usernames).
   - An =ores.qt= notebook contact-information page for accounts,
     matching the existing party/counterparty contact-info pages.

  dataset into populate SQL that reads and follows the same
  conventions as hand-written =ores.sql= files.
- New DQ artefact tables/datasets for LEI entities/relationships,
  business units, portfolios, books, and accounts, one dataset per
  Acme legal entity.
- =ores_iam_account_contact_informations_tbl= exists, mirroring the
  party/counterparty contact-information junction pattern, with a
  real name field.

* Plan

1. Add =ores_iam_account_contact_informations_tbl= via the legacy
   =entity_org= codegen (=ores.codegen.entity=), mirroring
   =ores.refdata.party_contact_information.org= but scoped to
   =account_id= (not =party_id=), with a required =full_name= field.
   Generate SQL, C++ (domain/service/repository/messaging), and Qt
   (model/controller/dialogs) layers. Hand-add the missing
   =ores_iam_accounts_touch_version_fn= (accounts predates the entity
   codegen system). Move the Qt output from the codegen's naively
   derived =ores.qt.iam= into the actual =ores.qt.admin= subcomponent.
   Wire a hand-written =AccountChildEntityTables= (mirroring
   =PartyChildEntityTables=) into =AccountDetailDialog= for the
   Contact Information notebook tab.
2. Add =full_name= to generated accounts (=generate_data.py='s
   =gen_person=/=add_account=) and regenerate the JSON.
3. Write =generate_sql.py=: reads the =acme_bank= JSON files and
   emits populate SQL under =projects/ores.sql/populate/acme/= —
   dataset registration (=acme.lei_entities=, =acme.lei_relationships=,
   and per-company =acme.<company>.{business_units,portfolios,books,
   accounts,account_contact_informations}=) plus per-family artefact
   INSERT scripts, matching the hand-written =ores.sql= conventions
   observed in =testdata_business_units_artefact_populate.sql=. Two
   new DQ staging tables (=dq_accounts_artefact_tbl=,
   =dq_account_contact_informations_artefact_tbl=) plus their
   artefact_type registrations (target/publish-from-dq wiring
   explicitly deferred to the server-side-orchestration follow-up
   task).
4. Verify end-to-end via =compass db recreate=, fixing every schema
   mismatch the run surfaced: =business_units=/=portfolios=/=books=
   artefact tables use uuid-typed self/cross-referential parent/owner
   columns (resolved from the JSON's deterministic per-code uuids),
   not the natural-key codes originally assumed; =portfolios.is_virtual=
   is an integer column, not boolean; =books= has no =owner_unit_id=
   column but does require =functional_currency=/=book_status=; the
   new DQ artefact create scripts and the Acme populate scripts were
   both missing from their respective =\ir= include chains
   (=dq_create.sql=, =populate.sql=); a catalog row and a
   dataset-registration methodology were both required before the
   first dataset upsert would succeed.
5. Full local build (=linux-gcc-debug-ninja=) and =compass build rat=
   to confirm the whole tree still compiles/tests green, fixing two
   codegen-generated defects surfaced by the rebuild: a bogus
   =#include "ores.qt/IamExport.hpp"= (a nonexistent export header —
   no sibling class in =ores.qt.admin= uses per-class export macros at
   all, so removed to match the existing convention) and a field-name
   mismatch (=parent_account_id= vs the domain struct's actual
   =account_id=) in =AccountContactInformationController=.

  generated via the legacy entity codegen) gives accounts a real

  Information" notebook tab on =AccountDetailDialog= matching the

  populate SQL scripts under =projects/ores.sql/populate/acme/=

  wired into =populate.sql= and verified via a full
  =compass db recreate= — all Acme data lands correctly in the DQ
  staging tables, including business-unit/portfolio/book parent-id
  resolution.

  green (all 75 test suites passed, 0 failures), after fixing the two
  codegen defects noted above.
