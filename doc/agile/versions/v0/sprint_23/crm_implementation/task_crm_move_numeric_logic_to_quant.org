This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:B38B3869-02FD-4CC7-99BD-9A77904ACA19][Cross-rates matrix (CRM)]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

The CRM cross-rates matrix UI panel (see
[[id:C685A8BE-0DF9-4D23-8AAB-E9DF9069E482][the panel task]]) currently
computes two things client-side that don't belong there: inverted
rates (1/rate, shown when no direct quote exists for a cell) and
%-change deltas vs. the previous reload. Both are pure numerical
domain logic, not UI concerns, and both currently live in
=CrmCrossRatesMatrixMdiWindow.cpp= with no unit test coverage of their
own.

Move this logic into =ores.analytics.quant= (the CRM's own engine
library, where =rate_engine=/=crm_topology= already live), self-contained
and thoroughly tested there, then expose it through
=ores.marketdata='s NATS protocol so the Qt client (and any other
future consumer -- HTTP, shell, Wt) gets the computed values directly
rather than re-deriving them.

* Plan

** Scope this task needs to resolve, not just implement

- *Where does each piece of logic belong?* Figure out the actual split
  between =ores.analytics.quant= (pure numerical/domain logic:
  inversion, delta computation, convention-aware formatting) and
  =ores.marketdata= (NATS glue: request/response shape, what
  =rate_engine= needs to track to serve "current + previous" in one
  call).
- *Current + previous with delta.* =rate_engine= (or a thin wrapper)
  needs to track each series' previous value so =crm_rate_item= (or a
  new response shape) can carry both the current price and a computed
  delta/% change -- not force every caller to diff two separate polls
  itself, the way the Qt client does today.
- *Inverted flag.* The CRM should support an "inverted" request flag
  (per-pair or per-request) so it returns 1/rate for a pair with no
  direct quote automatically -- not have every caller special-case
  "look up the reverse key and invert" client-side, the way the Qt
  client does today.
- *FX spot convention as input, not a dependency.* Whatever
  convention-aware behaviour is needed (canonical base/quote ordering,
  display precision) should take the relevant FX spot convention
  fields as caller-supplied parameters into the computation -- CRM/
  =ores.analytics.quant= must *not* take an actual dependency on
  whatever component eventually owns a convention entity/type (see
  [[id:76362D7D-44E6-4990-8058-2F9C41BF08DD][the FX market-convention capture]]
  for that separate, not-yet-built piece) -- keeping the numeric core
  decoupled from refdata.
- *Self-contained and thoroughly tested.* All of the above lands in
  =ores.analytics.quant= (or wherever it ends up) with real Catch2
  coverage of the numeric logic itself (inversion, delta computation,
  edge cases: zero rate, first observation with no previous, etc.),
  not just exercised incidentally through the Qt client.

* Acceptance

- Rate inversion (1/rate for a pair with no direct quote) is computed
  server-side (=ores.analytics.quant=), not by the Qt client scanning
  for the reverse key -- =CrmCrossRatesMatrixMdiWindow= no longer
  contains that logic itself, only display wiring.
- %-change delta vs. the previous observation is computed server-side
  and delivered in the response -- the client no longer maintains its
  own =previousRates_= map and diffs itself.
- Both behaviours are covered by real Catch2 tests in
  =ores.analytics.quant= (or wherever the logic lands), independent of
  the Qt client.
- No new dependency from =ores.analytics.quant=/CRM onto a refdata FX
  convention type -- convention-relevant fields (precision,
  base/quote ordering, etc.) are plain caller-supplied parameters.
- The Qt matrix panel's "Show Inverted" toggle keeps working, now
  backed by the server-computed value instead of a client-side lookup.

* Result

Landed the split proposed during planning:

- =ores.analytics.quant= gained =domain::crm_rate_view= plus two new,
  NATS/Qt-free services: =rate_inverter= (stateless -- resolves a
  requested pair against a batch of =derived_rate=s, falling back to
  the reverse pair's inverse when no direct entry exists and guarding
  the zero-rate division case) and =rate_delta_tracker= (a thin
  stateful decorator, mutex-guarded, keyed by the displayed
  (base, quote) pair -- deliberately layered on top of =rate_engine=
  rather than folded into it, so its own immer-based concurrency model
  stays untouched). 16 new Catch2 cases cover both, independent of
  NATS/Qt.
- =ores.marketdata=: =crm_ingest_bridge= now owns one
  =rate_delta_tracker= per (tenant, party, crm_name) alongside its
  =rate_engine=, and exposes =resolved_rates()= combining both new
  quant services. =crm_protocol='s =get_crm_rates_request= gained an
  =inverted= flag; =crm_rate_item= gained =inverted= and
  =delta_pct=. =crm_handler= maps the resolved views straight onto the
  wire.
- =CrmCrossRatesMatrixMdiWindow= no longer computes inversion or
  %-change itself -- =previousRates_= is gone entirely, and the
  "Show Inverted" toggle now just sets the request flag instead of
  post-processing the response.
- Convention-aware formatting (display precision, base/quote canonical
  ordering) was scoped out during planning: neither the original Qt
  code nor the CRM protocol ever implemented it, so there was nothing
  to extract -- filed separately as
  [[id:17CAD54D-0C46-427F-9117-E2B82346BAE5][a follow-up task]] for a
  plain C++ presentation overlay in =ores.marketdata.client=.

All acceptance criteria met; =ores.analytics.quant.tests= (147 cases,
11674 assertions) and =ores.marketdata.service.lib=/=ores.qt.marketdata.lib=
build clean.
