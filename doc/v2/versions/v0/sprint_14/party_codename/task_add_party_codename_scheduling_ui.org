* Goal

Add an =adjective_noun= codename per party (for per-party queue isolation) and the report scheduling UI on top of it.

* Acceptance

- Codename auto-generated by SQL trigger using =ores_utility_generate_whimsical_name_fn()=.
- Used as prefix for per-party PGMQ queues: ={codename}_report_events=.
- Codename exposed in Qt party list (hidden by default) + read-only field in party detail dialog.
- Report definition UI: missing =party_id= on save fixed; event subscriptions for live updates; client-side calculation of 'Next Fire' column from cron expressions; edits trigger change-reason dialog.
- Centralised session-context propagation (actor / tenant / party) through raw SQL execute functions; service-account constants enforced.
- C++23 requires-expression helper =stamp_auth= in =tenant_aware_handler= centralises stamping of =modified_by= / =performed_by= / =tenant_id= / =party_id=.
- MQ queue monitor scoped to current party via =ores_mq_list_party_queues_fn=, =ores_mq_metrics_party_fn=, =ores_mq_metric_samples_fn=.
- MQ metrics scrape job registration made idempotent + party-scoped.
