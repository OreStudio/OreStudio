* Goal

Stop maintaining the ores-shell script library as a parallel set of
org files under =projects/ores.shell/scripts/=. We already document
shell usage as [[id:7474935C-95D1-3034-B833-83DCD9FB3A09][recipes]] (76 under =doc/recipes/shell/=), each with an
=ores-shell= source block — that is the natural single source of
truth. Make the recipes the source and *generate* the runnable =.ores=
library from them by tangling, exactly as the codegen and clang-format
libraries are tangled.

Concretely:

- Express each provisioning flow as a recipe — "How do I provision the
  system with Barclays?" (GLEIF), "...with Acme?" (synthetic), and the
  other variants — instead of the hand-written =bootstrap.org= /
  =provision_all.org=.
- Give the generated scripts descriptive, per-variant names:
  =barclays_system_provision.ores=, =acme_system_provision.ores=, …
- Tangle the recipe =ores-shell= blocks into the script library the Qt
  panel and =compass shell -f= load, so a recipe and its runnable
  script never drift.

** Decisions (2026-06-08)

- *Scope:* /every/ recipe under =doc/recipes/shell/= (76 today) tangles
  to a runnable =.ores= in the script library — not just provisioning.
  The recipe is the single source of truth; its =ores-shell= block is
  the script. The mechanism is generic so a new recipe automatically
  yields a new library script.
- *Provisioning variants to add as recipes (first cut):* Barclays
  (GLEIF), Acme (synthetic), the bootstrap smoke, and the smaller
  party-only / tenant-only flows. These get descriptive script names
  (=<entity>_system_provision.ores=); generic recipes tangle to
  =<recipe-slug>.ores=.
- *Retire* the hand-written =projects/ores.shell/scripts/bootstrap.org=
  and =provision_all.org= — their flows move into recipes and the
  =.ores= become generated artefacts.
- *No drift:* a re-tangle / zero-diff check, mirroring the codegen and
  clang-format tangle libraries.

* Result

The shell script library is now generated from the recipes. A new
=ores-build-recipe-scripts.el= (driven by the existing
=tangle_shell_scripts= CMake target) tangles each =doc/recipes/shell/*.org=
=ores-shell= block into =projects/ores.shell/scripts/library/<category>/<name>.ores=
— output redirected "elsewhere" via =org-babel-tangle-file='s
TARGET-FILE, =LANG-RE= keeping the =sh= runner blocks out. Each script
is grouped into a category sub-folder (from the recipe's category
filetag), self-documents via a leading title/description comment block,
and has the trailing =exit= stripped (kept only so org-babel's REPL
terminates) so running it never closes the shell. The artefacts are
committed and a =shell-script-drift= CI check re-tangles and fails on
any diff.

Provisioning flows now live as recipes (scaffolded with =compass add
recipe=): Barclays (GLEIF), Acme (synthetic), bootstrap smoke, and the
tenant-only / party-only stages, wired into a Provisioning group in the
shell recipe inventory. The hand-written =bootstrap.org= /
=provision_all.org= are retired; the library is packaged (installed to
=share/ores/shell-scripts=, folder structure preserved).

On the Qt side the script panel recurses the category folders into a
tree (descriptions as tooltips read from the script header), and a
right-click offers Open (editor) / Execute (=load <path>= in the
terminal). The shell and editor MDI windows persist their geometry.

Also folded in: dashboard database lifecycle moved off the deprecated
=recreate_env.sh= / =teardown_database.sh= onto =compass db recreate= /
=compass db drop -k=; =compass db recreate= verified working
end-to-end.
