* Acceptance

- Written design decision: facet vs handcrafted widget + generated binding.
- Decision is consistent with the hierarchy-widget precedent (task 2 of this story).
- Rough API shape for the chosen approach is specified.

* Plan

Investigated (1) the existing foreign_keys/list_by paged query already
generated for the 4 child entities, (2) the current hand-maintained
=EntityDetailDialog=/=EntityDetailOperations= rendering of identifier
and contact child tables, (3) the cost of a brand-new Qt facet vs
reusing the =:implements <UUID>= paste-block convention, and (4)
whether the 4 child entities' column shapes are similar enough for a
single reusable widget.

* Notes

Decision: **handcrafted, reusable widget + thin generated binding**
(same shape as the hierarchy-widget decision in task 2), not a new
codegen Qt facet. Evidence:

- The backend fetch-by-parent-id capability is *already* fully
  codegen-generated for all 4 child entities via a =:list_by: true=
  flag on the FK field (=projects/ores.refdata/modeling/ores.refdata.party_identifier.org=
  and siblings) — =list_..._by_<parent>_id=/=count_..._by_<parent>_id=
  service calls, offset/limit pagination. Nothing to build here.
- The current hand-maintained code (=EntityDetailDialog.cpp=,
  =EntityDetailOperations.hpp=) is already almost entirely generic:
  plain =QTableWidget= setup, populate loop, add/edit modal
  construction, and delete wiring are identical between identifier
  and contact cases and between party/counterparty. The =identifier_entry=/
  =contact_entry= DTOs and the =entity_detail_operations= virtual
  interface are already entity-agnostic — only the column list and
  the four data-access calls vary per entity.
- Column shapes: only *two* distinct shapes across all 4 entities —
  =party_identifier=/=counterparty_identifier= are field-for-field
  identical (id_scheme, id_value, description); =party_contact_information=/
  =counterparty_contact_information= are field-for-field identical
  (contact_type, street lines, city, state, country_code, postal_code,
  phone, email, web_page). A single parameterized widget covers both.
- A new facet would require inventing new org-model vocabulary for
  columns/validators the handcrafted DTOs already express, plus
  generating 4 near-identical bespoke widget classes — for no benefit
  over parameterizing one class.
- The detail-dialog codegen templates currently have *no* =:implements=
  paste-block seams at all, so seams need adding regardless of which
  option is chosen — this is not a point in favour of the facet.

Rough API for the reusable widget (e.g. =EmbeddedChildTableWidget=):
constructed with a =columns= list (header label, field accessor,
editor widget kind, validator), a =fetch_page(offset, limit)= callback
bound to the entity's generated list/count calls, =save(row)= and
=remove(id)= callbacks bound to the generated save/delete calls, and
a =parent_id= (settable, for dialog reuse on selection change). The
generated per-entity binding (task 2/3 of this story) instantiates
this widget twice per detail dialog (identifiers, contacts) with the
column defs and callbacks wired to that entity's generated service —
no bespoke widget subclass needed.

Once proven in task 3 against party/counterparty,
=EntityDetailDialog=/=EntityDetailOperations= (~2,300 lines) can be
retired.
