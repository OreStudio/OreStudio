* Goal

Make the "day-scaled default kappa/sigma" claim from the
tick-batch-publishing task actually true. Manual verification of the
published =synthetic.ir_curve_configs.basic= dataset produced
unrealistic (~150-250) published rates despite day-scaled
=kappa=0.5/365=. Suspected cause: at that small a =kappa=, the Vasicek
discount-factor recursion's variance term (proportional to
=sigma^2/kappa^2=) is numerically unstable over the tick counts a 2Y
entry needs (730 daily ticks) — a genuine calibration/numerical-
stability problem, not a mechanical one (the tick-batch pipeline itself
— generation, batching, publish, ingest, persistence — is proven
correct via this same dataset).

Split out of [[id:FC800706-1673-4E48-84C2-22569E836DF5][Seed realistic
IR curve sample data]] because this is a quant/numerics bug in the
existing Vasicek recursion, independent of sourcing new vintage data or
wiring it into the feed.

  calendar day" convention.

  realistic rates end-to-end (verified via the same manual
  verification path that surfaced the bug) for at least a 2Y entry.
