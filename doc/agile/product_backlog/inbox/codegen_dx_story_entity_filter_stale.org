* What

[[id:A083FE26-521D-4046-9E58-AFB901B28809][Story: Codegen developer experience improvements]] (BACKLOG, not
started) lists three gaps, one of which is "No entity-level filter...
There is currently no way to restrict generation to a named subset."
That's no longer true: =compass codegen entity generate <entity>
[--address <addr>]= (=projects/ores.compass/src/compass_codegen_entity.py=)
already does exactly this — targets one entity across every
applicable profile, with =list=/=show=/=diff= companion subcommands.
Whoever built this either didn't know about or close out this story
item, so the story's Goal/description is now stale on point 2 (points
1 and 3 — spurious component-mode ERRORs, and logical entity
groups/scopes — may still be genuinely open, needs review). Compounding
the staleness: =doc/recipes/codegen/how_do_i_regenerate_all_layers_for_a_domain_entity.org=
told readers "There is no single command that targets one entity
across all profiles... Run component-wide and scope the diff
manually," actively steering people to the wrong, noisier workflow.
Already fixed that recipe directly (points at =codegen entity
generate= now); this capture is to re-scope/partially-close the story
itself and check for other docs making the same stale claim.

* Why

Followed the stale recipe's advice while adding book's
=rates_centre_code= field: ran =codegen regenerate --component
refdata-cpp --profile all-cpp= (and =--profile qt=), which touched 300+
files across every entity in the component, requiring extensive manual
discarding — including once accidentally reverting genuine hand-written
changes that happened to live outside a "Book"-named file. All of that
churn would have been avoided by the already-existing =codegen entity
generate book= command.
