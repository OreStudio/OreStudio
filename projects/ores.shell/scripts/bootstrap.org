#+title: bootstrap — development smoke bootstrap
#+description: Bootstrap a fresh development system: initial admin, two tenants, one account per tenant, suggested role grants.
#+property: header-args:ores-shell :tangle bootstrap.ores

This is a [[file:README.org][library]] script; the runnable artefact is the tangled
=bootstrap.ores=, regenerated by the build — do not edit it by hand.

* Purpose

Bootstrap a fresh development system far enough to exercise
multi-tenant behaviour: create the initial admin, two tenants (=acme=
and =example=), one account in each, and print the role-grant
commands each account needs. An error-free run (=load= aborts on the
first failed command) leaves a system two admins can log into.

Note this is the historical smoke script, predating the =provision=
porcelain; the full wizard-equivalent flow lives in the end-to-end
provisioning script.

* Expectations

- All services are up (=compass services start=).
- The shell is started from the environment (=compass shell=), so the
  =.env= variables are present for expansion and the subject prefix and
  TLS context are inherited by =connect=.
- The database is fresh (=compass db recreate -y -k=) so the system
  is in /bootstrap mode/; the =bootstrap= command refuses otherwise.
- Nobody is logged in (the script performs its own login).

* Steps

** Connect

Connect using the environment rather than a hard-coded endpoint:
=connect= expands =$ORES_NATS_URL= from the =.env=-derived environment
(host and port are not known ahead of time), and inherits the subject
prefix and TLS context the shell was started with.

#+begin_src ores-shell
connect $ORES_NATS_URL
#+end_src

** Create the initial admin and log in

=bootstrap= is only available in bootstrap mode and creates the
initial =super_admin= account. The subsequent login establishes the
authenticated session every later command needs.

#+begin_src ores-shell
bootstrap super_admin Secure-Password-123 super_admin@localhost.com
accounts login super_admin Secure-Password-123
#+end_src

** Create the tenants

Two organisation tenants so cross-tenant behaviour is testable:
=tenants add <code> <name> <type> <hostname> <description>=.

#+begin_src ores-shell
tenants add acme "Acme Corp" organisation acme.com "Acme Corporation tenant"
#+end_src

** Create an account in each tenant

Accounts are created with =accounts create <principal> <password>
<totp_secret> <email>=; the =567= placeholder TOTP secret is fine for
development.

#+begin_src ores-shell
accounts create newuser61@acme.com Secure-Password-123 567 newuser61@acme.com
tenants add example "Example Corp" organisation example.com "Example Corporation tenant"
accounts create newuser60@example.com Secure-Password-123 567 newuser60@example.com
#+end_src

Note the interleaving: each account is created right after its tenant
exists; the second =tenants add= sits between the two account
creations, exactly as the original hand-written script did.

** Print suggested role grants

=permissions suggest= prints ready-to-paste =accounts assign-role=
commands for each new account in its tenant — the grants are
deliberately left for the operator to review and apply.

#+begin_src ores-shell
permissions suggest newuser60 example.com
permissions suggest newuser61 acme.com
#+end_src

** Exit

#+begin_src ores-shell
exit
#+end_src
