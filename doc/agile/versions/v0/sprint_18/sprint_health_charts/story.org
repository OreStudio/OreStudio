* Plan

** Metrics

These are the charts, their data sources, and why they matter:

1. *PRs & Commits per Day.* Bar chart — dual axis (PRs left, commits right).
   Source: =git log --merges= for PRs, =git shortlog -sn= for commits.
   Why: PR velocity is the primary signal. Commits/PR ratio reveals whether PRs are small (good) or bloated (bad).

2. *Cumulative Stories Done.* Line chart — days on X, cumulative count on Y.
   Source: =git log= over =doc/agile/versions/v0/sprint_N/=, grep for changes from STARTED→DONE in story state.
   Why: Burndown equivalent. Shows whether the sprint is on track, ahead, or behind.

3. *PR Cycle Time.* Bar chart — one bar per PR, height = hours from open to merge.
   Source: =gh pr view --json createdAt,mergedAt= for each PR merged during the sprint.
   Why: Long cycle times indicate review bottlenecks. Spikes deserve investigation.

4. *Daily Line Churn.* Stacked bar chart — added (green) vs deleted (red) per day.
   Source: =git diff --stat origin/main@{sprint.start}..HEAD --per-day= (approximated by commit-date bucketing).
   Why: Churn direction tells you whether the team is building (more adds) or refactoring (balanced adds/deletes). Zero days signal blockers.

** Data Pipeline

#+begin_src text
git log + gh API  ──→  sprint_metrics.py  ──→  CSV files  ──→  gnuplot  ──→  PNGs  ──→  sprint.org
#+end_src

The Python script:
- Takes =--sprint <N>=, =--start-date=, =--end-date= (or auto-detects from sprint doc).
- Queries git log for commit/merge data between those dates.
- Queries doc state files for story/task transitions.
- Calls =gh pr view= for cycle time of each PR merged in the range.
- Writes one CSV per chart (=prs_per_day.csv=, =cumulative_stories.csv=, =pr_cycle_time.csv=, =line_churn.csv=).

The gnuplot script:
- Reads each CSV and produces a PNG under =assets/images/sprint_<N>/=.

The CMake target (=sprint_charts=):
- Depends on Python3 and gnuplot being installed.
- Outputs a warning with install instructions if they're missing.

* Decisions

- Commits are bucketed by author date (not committer date) to match when work actually happened.
- PR cycle time = mergesAt − createdAt. Weekends are not excluded — a PR open over a weekend correctly shows longer cycle time.
- Story state transitions are detected by grepping =git log -p= for changes to the =State= row in =* Status= tables. This is noisy but works without a structured database.
- CSV files use tab separators (gnuplot's default) for simplicity.

* Out of scope

- Live chart generation during the sprint (charts are generated once, on-demand, not continuously).
- Team velocity / effort estimation (we don't use story points).
- Contributor-level metrics per sprint (individual commit counts).
- A full dashboard or web UI — charts are static PNGs embedded in the sprint doc.
