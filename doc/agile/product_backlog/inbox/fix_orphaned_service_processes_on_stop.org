#+title: Fix orphaned child service processes after compass services stop
#+description: compass services stop only reliably kills the top-level controller and nats-server; child services it spawned can survive as orphans, competing with freshly-started replacements on the same NATS queue groups.
#+type: capture
#+level: cross
#+filetags: :compass:devops:reliability:
#+created: 2026-07-13
#+updated: 2026-07-13

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

=compass services stop= only kills the top-level =ores.controller.service=
and =nats-server=; it does not reliably kill the individual child
services (=ores.refdata.service=, =ores.compute.wrapper.{0..4}=, and
every other fleet member) that the controller spawns. Fix: =compass
services stop= should track and kill every child PID it spawned (or
query the OS for all processes matching the service's binary path +
environment before declaring stop complete, not just the controller's
own PID file), and =compass services status= should detect and warn
when more than one process matches a given service for the current
environment, rather than only reporting whichever PID happens to be
in the tracked PID file.

* Why

Discovered while diagnosing an intermittent, hard-to-reproduce failure
during [[id:E1F9C83E-645D-4CFC-B1C9-656651C55BCD][Commission: contact_type]]'s manual QA: every service in the
=prime_origin= fleet had an orphaned duplicate process left over from
an earlier =compass services stop=, still bound to the same NATS
queue groups as the freshly-started replacements. Requests routed
unpredictably between the old and new instances — symptoms included a
stale Qt controller that silently never constructed (menu action
present but its handler was a no-op) and a provisioning workflow
request that completed its DB write but never sent a NATS reply,
hanging the caller for a full 300s timeout on every retry. The root
cause was invisible from =compass services status=, which only ever
reported one PID per service (the newest), masking the duplication
entirely — the only way to find it was manually cross-checking
=pgrep -af prime_origin= process counts against the reported service
list.
