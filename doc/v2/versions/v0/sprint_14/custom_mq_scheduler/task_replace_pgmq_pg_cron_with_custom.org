* Goal

Remove the external pgmq + pg_cron extensions and replace them with custom in-process infrastructure.

* Acceptance

- pgmq + pg_cron extensions removed.
- Custom =ores_mq_*= tables: =mq_queues=, =mq_messages=, =mq_message_archive=, =mq_queue_stats=, =mq_channel_messages= (stub). Three scope levels (party / tenant / system); two queue types (task / channel); native RLS.
- In-process C++ =scheduler_loop= using =boost::asio::steady_timer=; =sql_action_handler= for executing SQL; =mq_action_handler= for sending MQ messages.
- =ores_scheduler_job_instances_tbl= TimescaleDB hypertable replaces =cron.job_run_details=.
- =ores.mq= component significantly updated with new domain types, repositories, =mq_service= facade.
- MQ Monitor UI shows scope_type / queue_type / description / pending_count / processing_count.
- Report scheduling uses the new =mq_service=.
- Protocol 48.0 (breaking).

Decision driven by tighter integration with our RLS architecture + control over data structures + unified codebase. The pgmq + pg_cron extensions had served their purpose as a proof of concept.
