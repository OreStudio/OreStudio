#+title: Component catalogue
#+description: Literate source for components.json. Maps each codegen component name to its model-discovery roots: where to find legacy JSON models (models_dir, entity_glob, exclude_suffix), where to find co-located org models (modeling_dir), and which codegen org types to skip within modeling_dir (exclude_org_types).
#+type: ores.codegen.component_catalogue

Consumed directly by =src/codegen/manifest.py= at startup via =_load_components()=. This file is the authoritative source; there is no intermediate JSON.

* Components

The SQL-side components (=trade=, =dq=, =iam=) use =*_entity.json= /
=*_table.json= globs to pick up schema models. The C++ components use
=*_domain_entity.json= with no exclusion (=exclude_suffix: null=) since the
=_domain_entity.json= suffix /is/ the target for that discovery path.

=refdata= discovers org models only — its legacy JSON =models_dir= is
retired (the org migration deleted every JSON model), and =--address=,
not =--component=, already selects the technical space (SQL vs C++ vs
Qt) to generate, so no separate =refdata-cpp= component is needed.
=exclude_org_types: junction= keeps junction-typed entity files out of
bulk =refdata= regeneration; junction codegen (domain/generator/
repository) is being brought up to parity separately and currently
errors on some addresses when bulk-regenerated. The module index
(=ores.codegen.module=) is never picked up by org discovery regardless
of component, since it isn't in the codegen org type set.
