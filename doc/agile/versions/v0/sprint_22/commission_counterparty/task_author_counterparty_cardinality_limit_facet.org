* Acceptance

- A new facet (e.g. =domain_entity.sql.cardinality_limit_validations=)
  is added to
  =projects/ores.codegen/library/templates/ores.sql.schema.domain_entity_create.org=,
  parameterised at least by: the column holding the scheme value, the
  table/column to look up the max cardinality from, and the
  table/column set to count existing rows against.
- =party_identifier.org= and =counterparty_identifier.org= both adopt
  the new facet, replacing their hand-written cardinality checks.
- Regenerating both entities (scoped, per [[id:90BD6F75-FFCC-4D27-B92F-5EBCB923B4AB][How do I regenerate all
  layers for a domain entity?]]) produces SQL functionally equivalent
  to the current hand-written version — same error message/errcode
  behaviour, same query shape — verified by diff review, not just
  "it compiles".
- This task can land independently of and in either order relative to
  =Populate existing codegen facets for counterparty models= — they
  touch different facets and don't conflict, though both need to have
  landed before counterparty_identifier's regen is fully clean.
