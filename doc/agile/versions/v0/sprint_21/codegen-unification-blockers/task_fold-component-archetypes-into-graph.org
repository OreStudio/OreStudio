* Notes

** Layer (b) generation-wiring design (2026-06-28)

Grounding: =resolve_targets= (generate.py:55) is the single resolver —
=get_model_type= -> supported set (facets whose =#+model_types= admit the type)
∩ target set (=--address=) -> walk =graph.facet_archetypes= -> resolve
=#+output= per model. =resolve_output_path= (core.py:592) already branches on
model type, including =component= (line 676). The B5 algebra is therefore more
general than per-entity; the two non-entity scopes attach to it as follows.

*** Decision A — component scope: kind-driven variability (SUPERSEDED first call)

First decided as =--address= subsets, then *reversed* during layer-(a) discovery
(2026-06-28) when the migration map revealed *output collisions* the address
model can't express: several component templates write the *same* output,
selected by component kind —
- =projects/{cf}/src/CMakeLists.txt= <- =cmake_component_src= | =_api_src= |
  =_core_src= | =_service_src= (4-way), and
- =projects/{cf}/tests/stub_tests.cpp= <- =cpp_component_stub_test= |
  =cpp_service_stub_test= (2-way).
The walk de-dups by resolved output (generate.py:113), so two variants cannot
coexist in one targeted set — the first silently wins. =--address= can't pick
"the service variant for service components" without a base/variant facet
explosion + multi-invoke orchestration.

*Resolution — kind discriminator (variability):* the component model declares
its kind (flat | api | core | service); generation routes through the EXISTING
walk =model_type= filter (generate.py:99-100). Because each variant archetype
declares exactly the kind(s) it serves, only one variant is ever in the
supported set, so the output collision never arises. Facets stay as the forks
named (=ores.cmake.component=, =ores.cmake.service=, =ores.cpp.component=,
=ores.cpp.service-app=).

Engine seam (layer b): =get_model_type= / the walk must understand the kind.
Two shapes to pick at implementation:
- kind as a refinement of =model_type= (=component-service= etc.) — reuses the
  walk filter verbatim, but =resolve_output_path='s =component= branch must
  match all =component-*= kinds; or
- =model_type= stays =component= for output resolution + a separate
  =#+component_kind= discriminator the walk filters on (archetypes declare
  served kinds). Cleaner separation of "what output shape" vs "which variant".

This also re-unifies the two non-entity scopes: BOTH component and data select
via the model's own declaration (kind / drawer), not =--address=.

*** Decision B — data scope: supported-set-driven, dataset-as-org-model

Root cause of the misfit: data lived in JSON files, which carry no
=:PROPERTIES:= drawer / =#+type=, so they cannot participate in the graph the
way an entity does. Fix: promote each dataset to a literate org *model* —
evolve =dataset_overview.org= from *describing* the data to *being* it
(payloads in source blocks, methodology/tags/description in sections, plus a
drawer).

- The dataset doc declares its supported set via the drawer, consistent with
  B5: =:ores.sql.populate.country.enabled: true= etc. (the entity declares what
  it enables).
- All populate archetypes sit in the single =ores.sql.populate= facet (fork 3),
  so selection is *archetype-granular*, not facet-granular.

Engine additions required:
1. Archetype-granularity activation. =_enabled_overrides= (physical_space.py:98)
   already parses arbitrary-depth addresses; extend =compute_supported_set=
   (physical_space.py:119) and the =resolve_targets= walk (generate.py:97-116)
   to honor archetype-level enables, not just facet-level.
2. Default-off for =ores.sql.populate=. B5 facets default *on* for any admitting
   model type; data must be opt-in. Add =#+default: disabled= (facet or
   archetype) so a dataset generates a populate archetype only when its drawer
   enables it.
3. =data= branch in =resolve_output_path= (core.py:592) — encode populate
   outputs as =#+output= patterns with data placeholders (dataset/table name,
   seeder prefix), replacing the bespoke prefix/code-derived logic
   (core.py:2428-2464).
4. Dataset org-model loader: =load_model=/org_loader extracts the JSON payloads
   from the doc's source blocks so mustache receives the same structure it gets
   from the JSON files today (templates barely change).

*** Execution finding (2026-06-28): layer (a) is NOT independent of layer (b)

Started layer (a). The *only* group that folds in cleanly & generation-neutral
is the *service-registry* group — DONE & verified (commit: new TS =ores.shell= +
facet =ores.shell.service=; populated =ores.sql.service=; =service_registry.org=
resolves to exactly its 5 archetypes via the graph; no =.mustache= touched).
service-registry is clean precisely because it is a singleton =model_type= that
generates *all* its archetypes *unconditionally*.

Every other group needs layer-(b) activation semantics to migrate WITHOUT
changing output:
- *Component* archetypes — kind-variant collisions (Decision A note above);
  need the kind discriminator before they generate correctly.
- *Folds* =cpp_protocol.cpp= and =sql_schema_artefact_create= — NOT
  unconditional: =cpp_protocol.cpp= is an audit candidate (*no profile
  references it*; folding into =ores.cpp.protocol= would generate a new .cpp for
  every entity), and =sql_schema_artefact_create= is conditionally invoked on
  the dataset/artefact path (gated on =has_artefact_insert_fn= etc.). Folding
  either into a flat-=model_types= facet over-generates.
- *Populate/data* — data-scope, deferred.

Conclusion: the clean "(a) first, then (b)" split only held for service-registry.
For components and the conditional folds, doc-graph migration and the
archetype-granularity / kind activation engine must land *together* (or the
engine first), else generation is not neutral. Recommend: build the layer-(b)
activation seam next, then migrate components+folds against it.

Also filed: capture =stale-generated-iam-sql= — committed IAM SQL is stale vs
current templates (pre-existing drift, unrelated to this migration).

*** Layer (b) activation seam — DONE (2026-06-28, commit follows)

Implemented the pure algebra in =physical_space.py= + wired into the
=generate.py= walk, with tests (25 pass):
- =#+default: disabled= on a facet/archetype => off unless an =ores.*= override
  re-enables it (=_parse_default=, =Graph.facet_default=, archetype
  =default_enabled= resolved in a load_graph second pass).
- =is_enabled(address, facet, ts, overrides, default)= — most-specific-wins now
  to *archetype* depth (=_is_enabled= kept as a facet-level shim).
- =kind_matches(archetype_kinds, model_kind)= — component-kind discriminator;
  empty kinds serve every kind, so it is a no-op outside component generation.
- Walk applies model_types -> kind -> enabled per archetype;
  =compute_supported_set= honours facet default-off. Verified neutral:
  service_registry still 5 units, entities unchanged.

Remaining wiring (small, before component/data migration): resolve_targets must
read the model's =:PROPERTIES:= drawer into =properties= (the long-standing
B5 TODO at generate.py) and the component model's =#+component_kind= into
=model_data["component"]["kind"]=. The seam accepts both via params today;
tests pass them directly.

*** BLOCKER for kind tagging / byte-identical verification (2026-06-28)

Real repo component models cannot be loaded at all — pre-existing, independent
of B8. All 99 =component_overview.org= use bare =#+type: component= (the resolver
only accepts =ores.codegen.component=; entities already use =ores.codegen.table=
etc.), and 60 of 99 lack =#+name=/=#+full_name=. So =codegen.sh generate= raises
=ValueError= for every component — legacy =--profile= path AND graph path alike.
Component generation is therefore *already broken* in the repo, and kind tagging
+ byte-identical verification cannot proceed until the models are normalised:
(1) =#+type -> ores.codegen.component=; (2) add =#+name=/=#+full_name=; (3) add
=#+component_kind=. Filed capture =normalise-component-overview-models=. The
graph+kind engine is proven correct (synthetic models above); only the malformed
repo models block real-world verification. Recommend a dedicated normalisation
story as a prerequisite before facet_catalogue retirement.
