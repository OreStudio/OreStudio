* Goal

Make =compass build --direct= (and =compass sprint charts=) the canonical way to
build documentation/agile artefacts, so light environments (no C++ toolchain, no
vcpkg) become a first-class path. Migrate every doc/agile call-site that has a
compass equivalent away from raw =cmake --build --target deploy_*=. *Retain* the
=ORES_VCPKG_ENABLED= bypass and the =linux-doc-only= preset — they are
load-bearing (see Decision below), not hacks to remove. cmake stays responsible
for C++ builds plus the handful of doc targets that have no compass equivalent
(diagrams, plan, qch, the =deploy_org= aggregate) and for the CDash site
submission.

* Notes

** Decision: keep the vcpkg bypass; migrate call-sites instead

*Context.* The task was originally written (2026-06-14) as "remove the
last-sprint cmake vcpkg-bypass hacks … leave cmake purely responsible for C++
builds". On scouting (2026-06-18) we reversed that framing.

*What the "hacks" actually are.* A single cache option =ORES_VCPKG_ENABLED=
(default =ON=, PR #1270 / task EFA4E251). When =OFF= it skips the vcpkg toolchain
setup, defines the emacs/org-babel doc targets, then early-=return()=s before any
C++ configuration. Companion pieces: the =linux-doc-only= preset(s) in
=CMakePresets.json= and a 3-segment-preset guard in =CTestSite.cmake=.

*Why we keep it.*
1. /CDash stays fast./ =site-cdash.yml= builds the site through
   =ctest --preset linux-doc-only --script CTestSite.cmake= and submits to CDash.
   That preset sets =ORES_VCPKG_ENABLED=OFF=, so the CDash job installs *no*
   vcpkg today. Removing the option (the original plan: "always require vcpkg")
   would force a full vcpkg install on every CDash/PR site run — slow, for zero
   benefit, since the site build needs no C++ dependencies.
2. /CDash needs cmake anyway./ =compass build --direct= runs emacs directly and
   has *no* CDash submission path; only =ctest=/=CTestSite.cmake= can submit.
   So the CDash call-site cannot move to compass regardless.
3. /Light-env fallback for cmake-only targets./ Some doc targets have no compass
   equivalent (=make_all_diagrams= needs java/plantuml; =deploy_plan= needs
   =tj3=; =deploy_help_qch= needs a python script; =deploy_org= is an aggregate).
   The bypass is what lets these still be driven through cmake without dragging
   in vcpkg. It advances the light-environment goal rather than obstructing it.

*Revised objective.* Reduce *reliance* on cmake for docs by pointing every
call-site that has a compass equivalent at =compass build --direct= /
=compass sprint charts=, making the light-env path the default. The bypass
remains as the fallback for cmake-only doc targets and for CDash.

| =build-site.yml= build job, template-drift, shell-script-drift | site / templates / scripts | already emacs-direct — out of scope |

*Possible follow-ups (not this task).* Add =--direct=-style or first-class
compass paths for the remaining cmake-only targets (diagrams via a compass
plantuml command, a =compass build --direct org= aggregate, plan, qch) and
consider normalising the three already-emacs-direct CI steps to call compass so
there is a single canonical build path.
