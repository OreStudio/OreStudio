#+title: codegen --component run generates all entities not just the target
#+description: Running codegen.sh with --component generates all matching entity files across the component, not just the ones in scope; out-of-scope files must be manually deleted after each targeted run.

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *next* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

Add an =--entity= (or =--entities=) filter flag to =codegen.sh regenerate=
so a targeted run only generates files for the named entity/entities, not
all entities in the component. Running =--component refdata-cpp --profile
protocol= currently regenerates all 25 domain-entity protocol headers and
writes them to disk, leaving ~22 out-of-scope untracked files that the
caller must manually delete after each run.

* Why

Every targeted codegen run in a focused task (e.g. syncing three currency
auxiliary entities) silently produces 20+ extra untracked files for
entities not touched by the task. The cleanup step is error-prone: if a
file is accidentally committed the PR grows far beyond its scope, and if
it is accidentally left on disk it pollutes subsequent =git status= output.
Observed twice during the =commission_currency= sprint work (once for
=protocol=, once for =nats-handler=).

The minimum fix is a filter flag: =--entity rounding_type,monetary_nature,currency_market_tier=
would restrict generation to only those three entities across the requested
profiles. An alternative is a dry-run mode combined with an explicit commit
step, but a filter flag is simpler to integrate.

Related: [[id:A72E617F-5443-4009-8173-A605D8F01485][codegen profile/model-type incompatibility errors]] captures the companion
issue of spurious ERROR messages for incompatible model types.
