* Goal

=projects/ores.iam/core/include/ores.iam.core/messaging/tenant_handler.hpp:198-203=
(=complete_tenant_provisioning=) directly constructs a
=variability::service::system_settings_service= using IAM's own
database context and calls =set_bootstrap_mode(false, ...)= to clear
the tenant's bootstrap flag on activation. This cross-domain call fails
with =ERROR: permission denied for table
ores_variability_system_settings_tbl= because that table's grants
(=projects/ores.sql/create/variability/variability_system_settings_create.sql=)
only give DML access to =ores_solid_dirac_variability_service= (plus
=ddl_user=/=owner=/=rw=/=ro=), not the IAM service's own DB role. The
call is wrapped in a try/catch that only logs a =warn= and continues
provisioning, so the failure is silent from the operator's point of
view — the tenant is marked active, but its =bootstrap_mode= setting is
left stuck =true= in the DB.

Observed while provisioning the Barclays Plc tenant via
=barclays_system_provision.ores= against a fresh DB
(=ores.iam.service.0.log:1856-1857=):
#+begin_example
[ERROR] [ores.variability.repository.system_settings_repository] Query execution failed: ERROR:  permission denied for table ores_variability_system_settings_tbl
[WARN] [ores.iam.messaging.tenant_handler] Failed to clear bootstrap mode for tenant 9ef2c435-3c2b-4f36-8cad-1dcbe126d373: Repository error: Query execution failed: ERROR:  permission denied for table ores_variability_system_settings_tbl
#+end_example

* Acceptance

- IAM can clear a tenant's =bootstrap_mode= on activation without a
  permission-denied failure.
- Tenant-scoped system settings other than =bootstrap_mode= (e.g.
  =synthetic_data_generation=) are seeded, or fall back to the system
  tenant's default, for tenants provisioned after this story ships —
  no more silent feature-vanishing for non-system tenants.

** Why

Whatever downstream behaviour keys off a tenant's =bootstrap_mode=
setting (flagged in the code comment as independent of the
=variability::flags:create= permission, i.e. meant to work
regardless of user permissions) silently never activates for any
tenant provisioned this way, since the flag never actually flips to
=false=. This was flagged while investigating a separate, still-open
"eventing is broken" report on the Barclays tenant — not confirmed as
the root cause of that report, but a real, silent correctness bug in
its own right: either grant the IAM service role write access to
=ores_variability_system_settings_tbl=, or route the write through
=ores.variability.service= over NATS instead of a same-process,
cross-crate repository call using the wrong DB role.
