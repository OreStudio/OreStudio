* Goal

Introduce =ORES_ENV_TYPE= (values: =development=, =staging=, =production=) as a first-class property across the full stack: .env template, postgres database-level GUC, and a colour-coded chip in the ores.qt status bar.

* Acceptance

- [X] =ORES_ENV_TYPE= appears in the generated .env file (defaults to =development=, preserved on re-run).
- [X] =compass db setup= propagates the value to postgres via =ALTER DATABASE … SET ores.env_type=.
- [X] ores.qt reads =ORES_ENV_TYPE= from the environment (CLI =--env-type= overrides); shows a colour-coded chip in the status bar: production = red, staging = orange, development = neutral.

* Plan

Three-layer propagation, each layer independent:

1. *.env template* — =env_init.py=: read any existing =ORES_ENV_TYPE= before the overwrite-guard, emit it in the =# Checkout identity= block (defaults to =development=).
2. *Postgres* — =compass_db.py= cmd_setup: after schema seed, run =ALTER DATABASE "db" SET ores.env_type = 'value'= so every new session can call =current_setting('ores.env_type')=.
3. *ores.qt UI* — =CommandLineParser=: add =--env-type= flag. =MainWindow=: add =setEnvType()= method that colours the existing instance-label chip and updates the tooltip. =main.cpp=: wire CLI → =ORES_ENV_TYPE= env var → ="development"= fallback; always call =setEnvType= so the chip is visible even without an instance name.

* Notes

The env-label chip (=envLabelWidget_=) was already created in =MainWindow= for the instance name. =setEnvType= re-uses it: it sets the chip's background colour, updates the tooltip to include both the label and env type, and forces the chip visible regardless of whether an instance name was supplied.
