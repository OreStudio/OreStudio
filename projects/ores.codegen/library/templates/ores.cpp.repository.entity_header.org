* Design notes

** Domain entity vs junction: =db_timestamp= vs =sqlgen::Timestamp=

The ={{#domain_entity}}= section uses the =db_timestamp= type alias
(=ores::database::repository::db_timestamp=, imported via =db_types.hpp=) for
=valid_from= and =valid_to=. The ={{#junction}}= section retains the raw
=sqlgen::Timestamp<"%Y-%m-%d %H:%M:%S">= type directly.

This divergence is intentional for now but represents a gap that should be
resolved in a future template cleanup task (see capture linked below).

*Why they differ:*

- Domain entities were migrated to =db_timestamp= (PR #1316) because the mapper
  template passes =valid_from= directly to =timestamp_to_timepoint(v.valid_from)=,
  which takes a non-optional argument. The =db_timestamp= alias makes the type
  opaque and consistent with the mapper helper API.

- Junction entities were not migrated at the same time because: (1) the junction
  mapper template takes a different code path and was not broken in the same way;
  (2) junctions have a composite PK (left + right + valid_from) where =valid_from=
  is part of the key, and the interaction between sqlgen's PK mechanism and the
  timestamp alias was not validated; (3) no junction entity was being actively
  synced at the time of the fix, so the risk of inadvertently breaking something
  was not justified.

*Pros of unifying to =db_timestamp= in the junction section:*

- Consistency: both entity kinds would use the same type for temporal columns.
- Future mapper helpers may depend on the alias.
- Avoids confusion for readers of the generated code.

*Cons / risks:*

- The composite-PK interaction with sqlgen needs to be verified: sqlgen uses
  =valid_from= as a PK column in junction tables, and the alias must be
  transparent to sqlgen's PK deduction logic.
- Requires validation across all existing junction entities (=party_currency=,
  =party_country=, etc.) before the template change is safe to merge.
- A mechanical template change could silently break junction entities that do not
  have active codegen tests.

*Recommendation:* unify in a dedicated template-cleanup task after verifying
the sqlgen PK alias behaviour in a test junction entity. Do not do this
incidentally during an entity-sync task.
