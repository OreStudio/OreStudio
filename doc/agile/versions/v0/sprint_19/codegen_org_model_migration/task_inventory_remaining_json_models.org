* Goal

After the per-component domain-entity sweep merges (PR #989), the
target end-state for =projects/ores.codegen/models/= is *empty*:
every JSON model becomes a literate =.org= file under the appropriate
component's =modeling/= directory, or is retired entirely. This task
takes inventory of every remaining JSON file, groups them by kind,
decides each kind's org-mode shape (or retirement), and writes the
follow-on per-kind migration tasks that close out =models/=.

The deliverable is a written inventory + sequenced plan committed
to the story. Actual conversion happens in the follow-on tasks this
one scaffolds.

* Acceptance

- Every file currently under =projects/ores.codegen/models/= is
  catalogued under exactly one of these dispositions:
  - *Migrate.* Has an org-mode counterpart in this story. The
    inventory names the kind, the org section shape, and the
    follow-on task that owns the conversion.
  - *Retire.* No longer needed (orphan / superseded). The inventory
    explains why and the deletion lands in a follow-on cleanup task.
  - *Out of scope.* Stays JSON (e.g. third-party data, raw fixtures).
    The inventory states why each one is kept and where it lives.
- Per-kind migration tasks scaffolded for each *Migrate* group, each
  with its own plan body covering: which org section it lands in,
  how the org_loader reads it, what the converter (if any) needs to
  emit, and what the follow-on tasks (PRs) look like.
- A retirement task scaffolded if there's a non-trivial *Retire* set.
- =projects/ores.codegen/models/= empty-target plan recorded in the
  story's =* Decisions= section so future readers see the closing
  goal.
- Inventory committed to this task's =* Result= section as the
  authoritative reference (CSV / org table) so the follow-on tasks
  can cross-reference it.

* Plan

** 1. Enumerate

#+begin_src sh :results verbatim
find projects/ores.codegen/models -maxdepth 3 -name '*.json' | sort
#+end_src

For each file, capture: path, suffix pattern, top-level JSON shape
(domain_entity / table / junction / enum / field_group / component /
service / other), whether codegen actually reads it (search for the
filename / pattern in =projects/ores.codegen/src/=), and the
templates that consume it.

** 2. Group by kind

Visible kinds today (from a quick scan of the tree at the time of
writing):

- =*_table.json= — SQL-only lookup tables (refdata, several others).
- =*_entity.json= — non-domain lookup-entity models.
- =*_field_group.json= — trading field groups composed into trade
  variants.
- =*_junction.json= — junction tables for many-to-many relationships.
- =*_enum.json= — pure enum models.
- =ores_*_component.json= — component manifests (one per package).
- Per-package =service.json= / registry models under
  =models/<package>.service/=.
- Standalone files (=tags.json=, =plantuml_er_model.json=,
  =profiles.json=, ...).

For each kind, decide:
- *Migrate to org.* What does the org file look like, where does it
  live, what =#+type:= identifies it, does it need a converter
  extension?
- *Retire.* Is the model still consumed? If not, delete.
- *Out of scope.* Some kinds are third-party data or generator
  fixtures and can stay JSON for now.

** 3. Sequence follow-on tasks

One task per migrate-able kind, ordered by:
1. Number of files in the kind (small kinds first as warm-ups).
2. Whether new converter / org_loader work is needed (no-converter
   tasks land first).
3. Cross-kind dependencies (e.g. junctions reference the
   domain-entities already migrated).

A final cleanup task removes =projects/ores.codegen/models/=
entirely once the per-kind tasks all merge.

** Per-kind detail

*** component (32)

Pattern: =projects/ores.codegen/models/<package>/ores_<package>_component.json=
(or =/<package>.<role>/ores_<package>_<role>_component.json=).

Each file describes a CMake / C++ component (one per library, app,
or service). The root object lives under the =component= key.
Codegen consumes them via =is_component_model= and the component
template family. Org-mode shape: =ores.<package>.component.org=
(or per-role variant) with a small property drawer + section per
sub-template. Lives under each component's =modeling/= dir alongside
the entity models.

*** entity (12)

=*_entity.json= (without the =_domain_= infix). These are SQL-only
lookup tables that don't get a C++ domain class — they back columns
that reference them via text codes. Codegen consumes via
=is_entity_model= and the entity template family. Org-mode shape:
similar to =ores.codegen.entity= but with a narrower section set
(no Qt, no C++ repository). Could share the same =#+type:= or carve
a new =#+type: ores.codegen.lookup_entity=.

*** table (10)

=projects/ores.codegen/models/refdata/*_table.json= — all refdata.
SQL DDL for lookup tables. Root key =table=. Org-mode shape: a
small table-model org file under =projects/ores.refdata/modeling/=.

*** junction (7)

=*_junction.json=. Many-to-many join tables. Distributed across
refdata, dq, iam, compute. Root key =junction=. Org-mode shape: a
junction-model org file under the owning component's =modeling/=.

*** field_group (5)

=projects/ores.codegen/models/trading/*_field_group.json=.
Composable field sets that several trading instruments inherit
(trade_audit, trade_classification, trade_identity, trade_lifecycle,
trade_parties). Root key =field_group=. Org-mode shape: a
field-group org file under =projects/ores.trading/modeling/=.

*** enum (3)

=workflow/workflow_status_enum.json=, =comms/message_type_enum.json=,
=comms/compression_type_enum.json=. Pure C++ enums. Root key =enum=.
Org-mode shape: tiny enum org file under the owning component's
=modeling/= (or under =projects/ores.comms/modeling/= for the
comms ones).

*** service_registry (1)

=projects/ores.codegen/models/services/ores_services_service_registry.json=.
Catalogue of services / ports. Root key =service_registry=. Org-mode
shape: probably a single registry org file under a top-level
=projects/ores.services/modeling/=.

*** Retire (1)

=projects/ores.codegen/models/plantuml_er_model.json= is *generated*
by =projects/ores.codegen/plantuml_er_generate.sh= and shouldn't
live under =models/=. Move the output to a build-only path, add it
to =.gitignore=, and clean up the existing check-in.

** Sequencing

Follow-on tasks are ordered roughly easy → hard:

1. =enum= (3 files, simple shape, no Qt) — warm-up.
2. =field_group= (5, trading-only, no SQL DDL).
3. =junction= (7, structurally small).
4. =table= (10, SQL DDL only).
5. =entity= (12, SQL-only lookup entities).
6. =service_registry= (1).
7. =component= (32, biggest but identical shape).
8. =slovaris= (6 reference-data JSONs; load_data extension).
9. =plantuml_er_model.json= retirement.

Once all nine follow-on tasks land, =projects/ores.codegen/models/=
is empty (=slovaris/methodology.txt= is plain text — not under
inventory).

** Per-kind task scaffolds

Each follow-on task carries the same skeleton:

- *Goal*: convert the N JSON files of kind =K= to org-mode files
  under each owning component's =modeling/= dir; delete the JSON.
- *Acceptance*: org_loader extension (if needed); converter
  extension (if needed); regen byte-identical output; build the
  affected libs cleanly against the production tree.
- *Plan*: settle the org-mode shape; extend tooling; convert;
  delete; commit; PR.

The detail body of each scaffold mirrors the entity-migration tasks
already in the story.
