* Goal

Give LLM agents the neighbourhood of each search hit — the docs that link to it and the docs it links to — ranked so the most relevant and most authoritative neighbours surface first, reducing follow-up compass show round-trips.

* Acceptance

- Each hit in compass search output includes up to N incoming links (ranked by inbound-link-count) and up to N outgoing links (ranked by FTS score against the search query).
- N defaults to 3 and is configurable via =--related-limit=.
- Feature is opt-in via =--related= (or =--related-limit N=); off by default with a hint in the configuration block.
- Sections are omitted entirely when a doc has no links in that direction.
- Link graph is built at query time from =doc_index.load_all()= / =doc_index.build_inbound()= (regex-based outbound parsing of org files); no SQLite schema changes required.

* Result

Delivered on =feature/search-related-docs-via-link-graph=.

Added =--related= and =--related-limit N= flags to =compass search=.

When =--related= is active, each search hit gets a =🔗  Related= block
showing up to N outgoing and N incoming neighbours from the link graph:

- *Outgoing (↳)*: links from the hit doc, ranked by the neighbour's own
  FTS score against the query (falls back to inbound-link count when the
  neighbour is not in the result pool).
- *Incoming (←)*: docs that link to the hit, ranked by their inbound-link
  count as a cheap PageRank proxy; de-duped against outgoing (with
  backfill from lower-ranked candidates so the cap is always met).

=--related-limit N= (default 3) controls the cap per direction and implies
=--related=, so =compass search "foo" --related-limit 5= works without
spelling out both flags.

When =--related= is off (the default), the configuration block shows a
one-line hint so the flag is discoverable.

The link graph is built at query time from the existing =doc_index.load_all()=
and =doc_index.build_inbound()= infrastructure — no schema changes.
