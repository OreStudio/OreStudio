* What

Each dataset in the project (e.g. =external/facestudio=, =external/crypto=,
and the new =projects/ores.compass/tests/data/= scorer corpus) is currently
described by two plain-text files: =manifest.json= (machine-readable
catalogue entry) and =methodology.txt= (human-readable sourcing rationale).
These files are invisible to the documentation system — they don't appear in
compass search, the rendered site, or the doc graph.  The fix is to author a
single =dataset.org= file per dataset that uses org-babel tangle blocks to
generate both files from one source of truth.  The org file carries the full
narrative (what the dataset contains, how it was collected, how to refresh it,
review cadence) as prose sections, with =#+begin_src json :tangle manifest.json=
and =#+begin_src text :tangle methodology.txt= blocks that produce the
machine-readable outputs on =C-c C-v t=.  Additionally, an external catalogue
org doc should be added under =doc/knowledge/= (or =doc/reference/=) listing
all datasets with a one-line description and a link to each =dataset.org= —
this makes the full dataset inventory discoverable from a single compass search.

* Why

=manifest.json= and =methodology.txt= are written in isolation today, so the
narrative and the machine-readable fields can drift.  More importantly, they
are invisible to compass: an LLM agent researching "what datasets do we have"
gets no results.  Moving authorship to org gives us site visibility,
searchability, inbound links from stories and tasks, and a single place to
update when a dataset changes.  The external catalogue doc provides a high-level
entry point for humans and LLM agents alike.
