* Goal

Break this story into its full set of implementation tasks, so work can
proceed task-by-task rather than as one undifferentiated blob. Expect
tasks covering roughly: the codegen entity model(s) for the synthetic
data configuration container and its FX sub-config; the SQL schema and
migrations; librarian logic that defaults a configuration on party
creation; NATS publishing/eventing for generated FX observations; and
end-to-end verification. Review the existing
=ores.synthetic.market_data_generation_config= entity model and the
sprint 21 FX spot PoC/approach-analysis stories before enumerating tasks,
to reuse what already exists rather than re-deriving it.

* Plan

Investigated the existing DQ Librarian/bundle publication pipeline
before enumerating tasks, since the story's "librarian support" phrase
turned out to refer to that existing bundle-apply mechanism rather
than a new concept. Wrote up the pipeline as a durable knowledge doc
([[id:F05B779F-769B-49D5-B369-D2AAC029B36D][DQ Bundle Publication Pipeline]]) with a worked example
(=report_definitions= in the =ore_analytics= bundle) tracing every
file involved, then rated each stage by codegen value. Finding: the
bundle/junction/artefact-type rows and the per-entity shadow artefact
table are already codegen-generated (via =ores.codegen.lookup_entity=
models); the real gap is the =publish_..._from_dq_fn= SQL
function plus its NATS handler and registrar wiring, which are 100%
hand-written across all 25 existing instances. Also found that
=party_service::save_party= has no lifecycle hook and the Librarian's
bundle Apply is a manual user action — there is no existing mechanism
to auto-provision anything on party creation, which the story's
follow-on tasks now need to explicitly resolve rather than assume.
