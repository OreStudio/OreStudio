* Result

All 11 scenario steps passed on the third manual QA run — list, create,
read, update, delete, history, and cross-client eventing (create/
update/delete all landing live in a second instance without a manual
reload).

The first two runs surfaced real infrastructure bugs, not code
defects in the regenerated entity: (1) a stale build/link — the
=ContactTypeController= constructor never ran because the running
=ores.qt.refdata.so= was linked before the source changes were in
place; a clean rebuild fixed it — and (2), found while re-diagnosing
after the clean rebuild, every service in the =prime_origin= fleet had
an orphaned duplicate process left over from an earlier
=compass services stop= that killed only the top-level controller and
not the child services it had spawned. Old and new instances of every
service were bound to the same NATS queue groups, so requests
routed unpredictably between them — explaining both the missing menu
action/constructor log and a third symptom found along the way: a
=refdata.v1.countries.publish-from-dq= provisioning request that
completed its DB work but never replied, hanging the GLEIF tenant
workflow at the same step on every attempt. Killed all orphaned PIDs,
verified 1:1 process-to-service parity via =pgrep=, recreated the DB,
reprovisioned Barclays Plc cleanly, and the retest passed.

Filed a capture for the orphaned-process bug in =compass services
stop= — see backlog.
