* Result

*Recommendation, both allowed-currency and allowed-product-type
lists*:

- *Data shape*: many-to-many junction table, not a multi-value
  column — following the existing =currency_currency_group= and
  =party_currency= junction precedent (a first-class codegen
  "junction" facet already exists and generates the full stack).
  Gives referential integrity and direct SQL joinability instead of
  serialisation tricks.
- *Currency side*: FK to =currencies.iso_code=, same as the existing
  junctions.
- *Product side*: reference =ores.trading='s existing =product_type=
  C++/Postgres enum (8 fixed structural families) rather than
  inventing a new lookup entity — it is already the canonical
  trade/instrument-type discriminator elsewhere in the codebase.

*Two open questions, deliberately left unresolved* — no existing
codebase precedent for either, so guessing at an answer here would be
premature:

1. How an *'All'* wildcard (no restriction) is distinguished from a
   deliberately empty set (nothing allowed currently permitted).
2. The Qt widget shape for editing junction membership (transfer list
   vs checkbox list) — no junction-membership editing UI exists
   anywhere in the Qt layer yet to follow.

Recorded this analysis in =books_and_portfolios.org= under "System
checks and permissions", where the allowed-currency/allowed-asset
concept was already mentioned. No code changes; implementation is
follow-up work once the two open questions get a product decision.
