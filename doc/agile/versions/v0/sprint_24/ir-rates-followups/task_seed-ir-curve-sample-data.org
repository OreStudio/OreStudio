This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:C29FE5A3-61F0-4D2F-BE4F-8A02223EABEF][IR Rates synthetic data: dataset seeding, index cleanup, dual-curve, quoting conventions]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Give IR curve synthetic feeds the same "grounded in real data, easy to
demo" quality FX spot feeds already have. Today an
=ir_curve_generation_config= must be hand-entered with arbitrary
=kappa=/=theta=/=sigma=/=initial_rate= values — nothing stops (and
nothing guides away from) unrealistic numbers, since there's no
reference. Manual verification of the tick-batch-publishing task
surfaced this concretely: an ad hoc test config using annual-scale
Vasicek parameters (=kappa=0.5=, =sigma=0.01=) under the resolver's "1
tick = 1 calendar day" convention produced ~150% published rates and a
FRA point that never moved (its discount factors decay to
r_t-independence within days at that reversion speed) — mechanically
correct, but not demoable.

Mirrors =synthetic_fx_spot_configs_basic_populate.sql= /
=synthetic_fx_spot_configs_realistic_populate.sql=, which both seed
=initial_price= from a real vintage (2016-02-05 Fed H.10), differing
only in per-pair-calibrated vs exaggerated-uniform volatility.

** Already delivered (tick-batch-publishing task, not this one)

While verifying =ir_curve_feed= manually, this task's DQ-dataset and
day-scaling scope turned out to be needed immediately (there was
nothing else to test against), so it was built directly rather than
deferred:
- =synthetic.ir_curve_configs.basic= and
  =.realistic= DQ datasets, each with catalog/dataset registration, a
  Curve Template entries artefact table, and an
  =ores_synthetic_publish_ir_curve_configs_from_dq_fn= SQL function
  mirroring the FX one (container/folder resolution, replace-all
  sync), wired into the =synthetic_realistic=/=synthetic_basic=
  bundles and the Barclays provisioning flow.
- Day-scaled default =kappa=/=sigma= for the "1 tick = 1 calendar day"
  convention (per-currency-calibrated in =.realistic=, uniform in
  =.basic=).

Manual verification of the published =synthetic.ir_curve_configs.basic=
dataset itself (not just the earlier ad hoc test) still produced
unrealistic (~150-250) published rates despite day-scaled
=kappa=0.5/365=. Suspected cause: at that small a =kappa=, the Vasicek
discount-factor recursion's variance term (proportional to
=sigma^2/kappa^2=) is numerically unstable over the tick counts a 2Y
entry needs (730 daily ticks) -- a genuine calibration/numerical-
stability problem, not a mechanical one (the tick-batch pipeline
itself -- generation, batching, publish, ingest, persistence -- is
proven correct via this same dataset). Needs a properly-conditioned
=kappa=/=sigma= relationship, not just "divide the annual number by
365", before this task's own day-scaling claim is actually true.

Manual verification also surfaced a second gap this task now owns:
=ir_curve_feed= only ever auto-starts at service startup, for
system-tenant configs read via the service's own unscoped context --
=market_data_generation_config_repository= has no cross-tenant read
method at all (confirmed: FX's =auto_start_enabled_feeds= has the
identical limitation, so Barclays' provisioned FX configs don't
auto-start either -- this is existing, expected Phase 1 behaviour, not
new). FX works around it with a manual per-tenant *Start* button
(=market_feed_config_handler=, driven from Qt's Market Simulator
window, scoped to the caller's own JWT/tenant); IR curves have no
equivalent, so a config published to a real tenant (e.g. Barclays) can
never actually tick without restarting the whole service under that
tenant's context -- not a viable path for a real UI workflow.

** Sourcing historical IR data

No vintage IR/yield dataset exists anywhere in this codebase today
(only =marketdata_fx_driver_rates_populate.sql= for FX) — this is new
data sourcing, not a lookup of something already half-done. Candidate
public sources for a small set of currency/tenor points on one
reference date, to evaluate for licensing/reuse terms and data shape
before picking one:
- *FRED (St. Louis Fed)* — SOFR and historical USD OIS/Treasury
  series, freely redistributable, CSV/API access.
- *ECB Data Portal* — €STR and historical euro-area yield curve
  points.
- *Bank of England Statistical Interactive Database* — SONIA history.
- *ICE Benchmark Administration* — historical IBOR-successor fixings,
  licensing needs checking before use.

Only a handful of (currency, tenor, date) points are needed to seed
one vintage row per Curve Template entry — not a full historical
curve archive.

* Acceptance

- A small set of real historical (currency, tenor, date) IR points is
  sourced and added as a new vintage populate script (mirroring
  =marketdata_fx_driver_rates_populate.sql=), covering enough
  currencies/tenors to seed the =DEPO=/=FRA=/=IRS= entries a basic
  Curve Template needs.
- =ir_curve_feed=/auto-start gains a vintage-aware seeding path
  (mirroring =feed_controller::start()='s =vintage_source=/
  =vintage_date=), so =initial_rate= comes from a real imported
  observation rather than an arbitrary config value.
- IR curve configs gain a manual per-tenant start/stop/list control
  plane (mirroring =market_feed_config_handler=), with a UI entry
  point in Qt (mirroring the Market Simulator's Start button), so a
  config published to a real tenant (not just system) can actually be
  started without restarting the whole service.
