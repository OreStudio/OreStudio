This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:B38B3869-02FD-4CC7-99BD-9A77904ACA19][Cross-rates matrix (CRM)]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

The Qt CRM matrix panel hand-formats rates itself
(=QString::number(rate, 'f', 5)=, fixed precision, no base/quote
canonical ordering) -- the same "logic in the UI" problem the
sibling task [[id:60B9E7A1-1CFC-4A33-A3AC-C36078D53FDB][Move CRM
matrix's numeric logic (inversion, deltas) out of the Qt client]]
already fixed for inversion and %-change. Formatting is the
remaining piece.

Add a stateless, plain C++ presentation overlay -- a new
=presentation= namespace/subfolder inside =ores.marketdata.client=
(thin, DB-free, matching its existing role) -- that maps a batch of
=crm_rate_item= (or =ores.analytics.quant::domain::crm_rate_view=)
plus a resolved =currency_pair_convention= into ready-to-bind
display strings (rate text, %-change text, tooltip text). The
formatter itself takes the convention as a plain parameter -- it has
no NATS/cache dependency of its own; the caller (the Qt window)
resolves the convention via
[[id:17CAD54D-0C46-427F-9117-E2B82346BAE5][ores.refdata.client's
currency_pair_convention_cache]] first. Qt's own cell widget already
accepts =QString=, not a raw double, at its display boundary -- see
=CrmCrossRatesMatrixMdiWindow::reload()= -- so this is a drop-in
replacement for that formatting call, not a new UI contract.

* Acceptance

- New =presentation= namespace/subfolder inside
  =ores.marketdata.client= (e.g.
  =ores.marketdata.client/presentation/crm_rate_formatter.hpp=)
  provides a stateless mapper from raw CRM rates + a resolved
  =currency_pair_convention= to display-ready strings; no NATS or
  Qt/Wt dependency in the mapper itself.
- =CrmCrossRatesMatrixMdiWindow= calls
  =currency_pair_convention_cache= then the formatter instead of
  formatting rates itself; the fixed ='f', 5= precision and lack of
  base/quote canonical ordering are replaced by convention-driven
  formatting (=pip_factor=/=tick_size=/=decimal_places=,
  =currency.base_precedence=).
- No new convention entity/type is introduced -- uses the existing
  =currency_pair_convention=/=currency= fields.
- Real Catch2 coverage of the formatter itself: precision rounding,
  base/quote reordering, inverted-pair display, missing/unavailable
  status, independent of any NATS/Qt harness.

* Plan

(Implementation strategy. Written when work starts; key decisions
are distilled into the parent story's =* Decisions= at close, but the
plan itself stays — it is the historical record of what we did.)

- =ores.marketdata.client/presentation/crm_rate_formatter=: stateless
  =crm_rate_formatter::format(crm_rate_item, optional<int>
  decimal_places)= returning =crm_rate_display{rate_text, change_text,
  tooltip_text}=. No convention type crosses into the formatter --
  it only takes the already-resolved =decimal_places=, keeping it
  free of any refdata/NATS dependency.
- =currency.base_precedence= turns out not to be needed at format
  time: a CRM cell's (base, quote) orientation is already fixed
  upstream (=rate_inverter=/=crm_rate_view=), and
  =base_precedence= only matters when a convention is *captured*
  (deciding which direction becomes the stored =pair_code=), not
  when it's looked up for a fixed pair. What "base/quote reordering"
  actually needs is a direction-agnostic *lookup*: since a
  convention may be stored as either "EUR/USD" or "USD/EUR",
  =CrmCrossRatesMatrixMdiWindow::resolveDecimalPlaces()= tries the
  cell's own (base, quote) pair-code first, then the reversed one,
  before falling back to =std::nullopt= (the formatter's own fixed
  default precision).
- =CrmCrossRatesMatrixMdiWindow= keeps status-driven colour/glyph
  selection (Qt-specific, not part of "ready-to-bind display
  strings") but delegates rate/%-change/tooltip text entirely to the
  formatter.

* Result

Shipped a stateless =crm_rate_formatter= in
=ores.marketdata.client/presentation= (header + source + Catch2
tests, no NATS/Qt/Wt dependency) mapping =crm_rate_item= +
=optional<int> decimal_places= to =crm_rate_display{rate_text,
change_text, tooltip_text}=. =CrmCrossRatesMatrixMdiWindow= now
resolves each cell's convention via
=currency_pair_convention_cache::lookup= -- trying the pair-code in
both directions, falling back to the formatter's own fixed default
precision on a miss -- and delegates rate/%-change/tooltip text to
the formatter; only status-driven colour/glyph selection (Qt-only
concern) stays in the window. All acceptance criteria met, with one
scoping refinement recorded in =* Plan=: =currency.base_precedence=
turned out unnecessary at format time (a cell's base/quote is already
fixed upstream by =rate_inverter=/=crm_rate_view=; base_precedence
only matters when a convention is *captured*, not looked up) --
"base/quote reordering" is instead handled by the direction-agnostic
direct/reverse pair-code lookup. 10 new Catch2 cases (precision
rounding, default-precision fallback, fresh/inverted/stale/
unavailable tooltips, up/down/none/negligible delta text); local
build clean, =ores.marketdata.client.tests= 100% passed.
