* Goal

Close the read_for_cache auth gap for real: add the same
=make_request_context= check the other generated handler methods use,
wire the generated cache's =load()= to attach a service-account
=Authorization: Bearer <jwt>= header, and fix party's own
hand-written =read_for_cache= (never regenerated — its messaging
layer is separately known-drifted) to match. Along the way, discovered
that IAM authenticating against *itself* (party_cache calling
=refdata=, but the token comes from IAM's own =service-login=) has a
startup-ordering hazard, which produced a new architecture doc:
[[id:1D02DB5B-1E50-4AA0-9780-A3F1CF237349][Service Bootstrap Phases]].

* Acceptance

- The generated =read_for_cache= handler requires a valid signed JWT
  (=make_request_context=), like =list=/=history= — no specific
  permission check, since the caller legitimately reads across
  tenants under one service identity.
- The generated cache class (=nats-event-cache= facet) accepts an
  optional token provider — via constructor and/or
  =set_token_provider= — and attaches an =Authorization: Bearer <jwt>=
  header on every =load()= when one is set.
- Party's hand-written =read_parties_for_cache= (never regenerated)
  is fixed to match, since it's the one real consumer today.
- IAM's =party_cache= is armed with a real service-account token
  (via =make_service_token_provider=) and continues to warm/reload
  correctly.
- No startup-ordering hazard: IAM's own subjects (including
  =service-login=, which the token mint calls) are fully registered
  before anything tries to call them — documented as the general
  "Service Bootstrap Phases" pattern, not a one-off fix.
- =ores.iam.core= depends on =ores.iam.client.lib= (new CMake edge,
  carefully checked for cycles — none: =ores.iam.client= depends only
  on =ores.iam.api=/=ores.nats=).

* Plan

1. Add =make_request_context= check to the =read_for_cache= handler
   archetype (=ores.cpp.nats-handler.nats_handler_header.org=); no
   permission check (cross-tenant access is the legitimate access
   pattern for this caller).
2. Add an optional =token_provider= (constructor param +
   =set_token_provider= setter) to the =nats-event-cache= cache
   archetype; attach =Authorization: Bearer <jwt>= on =load()= when
   set.
3. Hand-fix party's un-regenerable =read_parties_for_cache= handler
   to match (same auth check).
4. Regenerate party's =party_cache.hpp=/=party_cache_registrar.hpp=.
5. Investigate how IAM should authenticate its own =party_cache=
   calls: discovered =make_service_token_provider= eagerly logs in
   synchronously, and IAM would be calling *itself* — found this is
   an ordering hazard, not a threading deadlock (NATS subscriptions
   dispatch on the NATS client's own thread, independent of =io_ctx=,
   confirmed by reading =ores.nats/src/service/client.cpp='s =on_msg=
   comment and =request_sync='s =cnats= implementation).
6. Wrote [[id:1D02DB5B-1E50-4AA0-9780-A3F1CF237349][Service Bootstrap Phases]] (+ sequence diagram) documenting
   the general Phase 1 (wire/subscribe everything) vs. Phase 2
   (mint tokens, warm caches — anything that calls a subject,
   including the service's own) pattern, worked through this exact
   case; cross-linked from "Anatomy of a Service" and
   "Service-to-Service Auth".
7. Applied the fix: =ores.iam.core/messaging/registrar.cpp=
   restructured into the two phases — =party_cache= constructed
   token-less in Phase 1 (needed by =auth_handler= et al.'s
   constructors), armed via =set_token_provider= and warmed only
   after every IAM subject (including =service-login=) is registered
   in Phase 2. =registrar.hpp='s =register_handlers= gained a
   =service_password= parameter; =application.cpp= passes
   =cfg.database.password()=, matching the established pattern used
   by reporting/synthetic/marketdata/workflow/ore's =application.cpp=
   files.
8. Added =ores.iam.client.lib= to =ores.iam.core='s =CMakeLists.txt=
   (PRIVATE — only =registrar.cpp= needs the header; the cache class
   itself only needs =std::function=, no new public dependency).

* Result

Closed the read_for_cache auth gap end-to-end: the generated handler
and party's hand-written copy both now require a valid signed JWT;
the generated cache attaches one via an optional token provider; IAM's
own registrar mints a real service-account token for party_cache.
Along the way, produced a durable architecture doc, [[id:1D02DB5B-1E50-4AA0-9780-A3F1CF237349][Service Bootstrap
Phases]] (+ sequence diagram), naming the Phase 1 (wire/subscribe
everything) vs. Phase 2 (mint tokens, warm caches) split that made the
self-referential IAM case correct rather than a startup hazard.
