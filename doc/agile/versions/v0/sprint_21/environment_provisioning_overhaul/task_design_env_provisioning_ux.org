* Notes

** Structural problems in the current scheme (from env_init.py audit)

1. *DB name uses raw label*: =db_name = f"ores_dev_{label}"= where label can be =festive-hawking= →
   =ores_dev_festive-hawking=. Hyphens are not valid in unquoted PostgreSQL identifiers. The code
   already computes =label_lower= (replacing hyphens and dots with underscores) but does not use it
   for =db_name=. Latent bug exposed by GCP naming.

2. *Port allocation is a hardcoded dict*: ={​"remote": 50000, "local1": 51000, "local2": 52000, ...}=.
   Any environment whose name is not in this dict falls through to the default (51000). Adding a
   new adjective-noun environment requires a code change.

3. *NATS port derived from trailing digit*: =nats_port = 42220 + int(label_suffix)= where
   =label_suffix= is the trailing digit of the label. Works for =local1=, =local2= etc but produces
   =42220= (no offset) for adjective-noun names with no trailing digit.

4. *No light/full distinction*: Every environment is assumed to be full (C++ toolchain, vcpkg,
   NATS, Postgres). There is no way to provision a docs-only environment.

5. *No genesis concept*: There is no canonical first-clone path or convention. Users are expected
   to know which checkout is the "parent" for =git worktree add=.

** vcpkg sharing across environments (preliminary findings)

vcpkg has two tiers of caching:

1. *Binary archive cache* (=~/.cache/vcpkg/archives=): already machine-global by default. When any
   checkout installs a package, subsequent installs on the same machine unpack from the archive
   rather than recompiling. Cost: O(seconds) not O(minutes). This is already in effect.

2. *Installed packages directory* (=vcpkg_installed/= in each checkout root): per-checkout. Contains
   the final headers and libraries that CMake links against. Currently one copy per worktree.

Sharing =vcpkg_installed/= is possible via =--x-install-root=<path>= (vcpkg flag) or a symlink at
checkout root. Risks to assess:
- *Concurrent builds*: two worktrees building simultaneously would clobber each other's installs.
- *Branch version divergence*: if =vcpkg.json= differs between branches, one checkout's install
  step would overwrite the other's packages.
- *Overlay ports*: =build/cmake/overlays/= are per-checkout; shared install root must resolve them
  consistently.

Light environments don't need =vcpkg_installed/= at all (they skip C++ entirely). The vcpkg sharing
question is only relevant for full environments. The full investigation is tracked as a separate
task in this story.

* Result

See =* Decisions= in the parent story for the authoritative record. Summary:

- *Types*: =light= (docs/agile, no C++) vs =full= (complete stack).
- *Naming*: GCP-style =adjective-noun= stored as =ORES_ENV_NAME=; DB uses underscored form; NATS uses dot-separated form.
- *Genesis*: canonical first clone at =~/Development/OreStudio/OreStudio.genesis=; always =full= type; all worktrees provisioned from it.
- *Port allocation*: =ORES_BASE_PORT= and =ORES_NATS_PORT= assigned at =compass env create= time by scanning existing envs; =compass env configure= reads them from =.env= if already set.
- *Commands*: rename =compass env init= → =compass env configure= (keep =init= as alias); add =compass env create <name> [--type light|full]=.
