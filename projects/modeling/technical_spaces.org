This document is the index of ORE Studio's [[id:787A7429-71D5-4295-829B-A9B56275B1F8][physical space]] — the concrete
instantiation of the MASD [[id:E7B74138-9ADA-4B4B-A983-23B3FDB55F44][technical space]] hierarchy across every language
ecosystem the product touches. It provides the linkage between MASD's
abstract concepts (TS, Part, [[id:B079DA67-8408-4BD0-A93C-D108AC780872][Facet]], Archetype, [[id:CE95C751-2125-4BA9-A78D-C30F9540FC5E][Profile]]) and the
actual components, directories, templates, and [[id:D7D53D71-C9E8-43AE-AA42-E58E57CD76C8][ores physical-space graph]]
nodes that implement them.

A [[id:4F24EEB9-58F1-42FD-BF4F-121D890F2D85][logical entity]] (a =value= such as =currency=) projects simultaneously
into every enabled [[id:E7B74138-9ADA-4B4B-A983-23B3FDB55F44][technical space]] — C++ domain + repository + Qt,
SQL DDL + trigger, CMake build target — each producing its own
[[id:787A7429-71D5-4295-829B-A9B56275B1F8][physical artefacts]] via the [[id:B079DA67-8408-4BD0-A93C-D108AC780872][facet]] transformation.

For the meta-model concepts behind this structure (meta-layer,
meta-component, PMM, SRPP) see [[id:BA0E3273-DC4B-7CD4-447B-06227B7D09A8][MASD]].

* Layer and component topology (system dimension)

The physical space has two orthogonal views:

1. *The TS→Part→Facet→Archetype dimension* (codegen structure) — documented
   in this file and its per-TS companions.
2. *The Layer→Component dimension* (system architecture) — documented in
   [[id:D773166D-0C91-8CB4-3323-42166BC07687][System Model]] and its sub-docs.

In MASD terms, the layers (foundation / infrastructure / domain /
application) and components (=ores.refdata.api=, =ores.qt.refdata=, etc.)
are physical-space instances of the meta-layer and meta-component concepts
from the [[id:4F24EEB9-58F1-42FD-BF4F-121D890F2D85][logical space]]. The two views are complementary: the system model
shows /who owns what/; the TS model shows /how it is generated/.

* Physical Metamodel in ORE Studio

The [[id:787A7429-71D5-4295-829B-A9B56275B1F8][PMM]] is instantiated by two complementary artefacts:

- The [[id:D7D53D71-C9E8-43AE-AA42-E58E57CD76C8][ores physical-space graph]] — each facet node's Archetypes table maps a
  [[id:CE95C751-2125-4BA9-A78D-C30F9540FC5E][profile]] name to the Mustache templates that belong to it; read directly
  by =generator.py= (no intermediate JSON).
- =projects/ores.codegen/library/templates/= — the archetype catalogue,
  where each =.mustache= file is one archetype within a facet. The
  directory layout mirrors the TS→Part→Facet hierarchy.

The literate template org files (=cpp_domain.org=, =sql_schema.org=, etc.)
are the authored sources from which the =.mustache= artefacts are tangled.
They document each archetype's SRPP inline, alongside the template code.
The [[id:3B6B9F75-BDC9-4AE9-B16E-5A05FFC3A4B5][template library overview]] describes the three-level literate hierarchy
(archetype ← facet doc ← group doc ← overview).
