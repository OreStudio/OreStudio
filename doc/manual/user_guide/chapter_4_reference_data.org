** What is Reference Data?

@@latex:\lettrine[lines=2,nindent=0em]{R}{eference} \textsc{data is the static or slowly-changing}@@@@html:Reference data is the static or slowly-changing@@ master data that all
financial activity depends on. It defines the vocabulary of the system
— the currencies a trade is denominated in, the counterparty on the
other side of a deal, the book it settles into, the country a legal
entity is incorporated in. Without it, nothing else can be described
precisely.

#+caption[The Reference Data menu]: The Reference Data menu showing the full set of entity types managed by OreStudio. The Audit Trail sub-menu provides access to Change Reasons and Change Reason Categories.
[[../../../assets/images/reference_data_menu_audit_trail.png]]

OreStudio's reference data covers a wide range of entity types:

- *Currencies* — ISO 4217 codes, display formatting, and rounding rules.
- *Countries* — ISO 3166 alpha-2 codes and names.
- *Trading conventions and calendars* — holiday calendars, settlement
  day rules, date rolling conventions, and tenor definitions that drive
  date arithmetic in trade processing and valuation.
- *Counterparties* — the external legal entities your organisation
  trades with.
- *Books* — the internal trading books that own positions.
- *Business units and portfolios* — the organisational structure within
  a party.
- *Party types and statuses* — the classification vocabulary for
  parties.
- *Classifications* — supplementary taxonomies such as currency market
  tiers and monetary natures.

Reference data is distinct from the other two main categories of data
in the system:

- *Market data* — prices, FX rates, yield curves, and volatility
  surfaces that update continuously throughout the trading day.
  Market data is time-stamped and immutable once recorded; a new
  observation does not overwrite an old one.
- *Trades* — specific financial transactions between parties with
  defined cashflows, valuation models, and lifecycle states. Trades
  reference the reference data (they need a currency, a counterparty,
  a book) but they are not reference data themselves.

The key property of reference data is that it changes slowly and
deliberately. A currency's rounding convention is not something that
shifts intraday — it is corrected through a controlled, audited
process. OreStudio enforces this with a comprehensive data quality
framework described in this chapter.

** Data Quality

Financial calculations are only as reliable as the data they consume.
A misspelled currency name is a cosmetic nuisance; an incorrect
rounding rule, a wrong counterparty identifier, or a stale market tier
classification can silently propagate into risk figures, margin calls,
collateral calculations, and regulatory reports. The consequences range
from minor reconciliation breaks to significant financial loss or
regulatory censure.

OreStudio's reference data layer is designed around the definition of
data quality from ISO 8000 and the DAMA Data Management Body of
Knowledge (DAMA-DMBOK): /data quality is the measure of how well a
dataset satisfies the requirements of its intended business use/. In
financial markets this means data must be not only accurate but also
complete, consistent, timely, valid, and unique — the six
industry-standard DQ dimensions described below.

*** The Six Dimensions

*Accuracy* is the degree to which data values agree with their
authoritative golden source. OreStudio seeds currencies from ISO 4217,
countries from ISO 3166, and counterparty identifiers from the GLEIF
Legal Entity Identifier registry. Where a discrepancy arises between
a record in OreStudio and its source, the source takes precedence; the
system provides tooling to re-import from authoritative datasets.

*Completeness* means all mandatory attributes are present. The service
layer rejects saves that omit required fields and returns a validation
error; the Qt UI marks missing mandatory fields visually before a save
is attempted. Bulk imports from external sources are validated against
a mandatory-field checklist before being committed.

*Consistency* is the absence of contradictions between related
entities. A currency's rounding type must be drawn from the governed
rounding-types table; its market tier from the currency-market-tiers
table. Foreign-key constraints in the database enforce this, so a save
that references a non-existent classification code is rejected at the
persistence layer rather than silently stored.

*Timeliness* means data is available when calculations need it. The
tenant provisioner imports standard catalogues — currencies, countries,
market conventions — at setup time, so the foundational reference data
is ready before any trading activity begins. NATS events propagate
updates to all connected clients in real time, so the Qt UI reflects
changes within milliseconds of a save.

*Validity* is adherence to business rules and formats. ISO code
lengths, numeric code ranges, currency format strings, and rounding
precision bounds are all validated at save time. Invalid values are
rejected with a diagnostic message.

*Uniqueness* ensures no duplicate records exist for the same entity.
Primary key constraints at the database level prevent duplicates; the
service layer surfaces a clear error if a duplicate is attempted.

*** Bitemporality

The most important architectural property of OreStudio's reference
data layer is its use of /bitemporality/ — the recording of two
independent time dimensions for every stored fact.

The first dimension is *valid time*, recorded in =valid_from= and
=valid_to= columns on every row. Together they form an open interval
=[valid_from, valid_to)= bounding the period during which this version
of the record is authoritative. The live (current) row carries
=valid_to = 9999-12-31 23:59:59= — a sentinel meaning "no known
expiry". When an update is saved, the database trigger closes the
existing row by setting =valid_to= to the current time, and inserts a
new row with =valid_from = now= and =valid_to = sentinel=. The
application never writes =valid_from= or =valid_to= directly — the
trigger owns both columns. This pattern means the table always contains
the complete history of every entity: no rows are ever deleted or
overwritten.

The second dimension is *transaction time*, surfaced to the application
as the =recorded_at= field. It records when this version was written to
the database — independently of what period the version is valid for.
Transaction time answers the question "what did the system believe at
a given moment?"

All timestamps in OreStudio are stored and processed as UTC throughout
— the database, the service layer, and the NATS protocol all use UTC
exclusively. The Qt UI converts timestamps to your local timezone for
display only; no local-time arithmetic is performed anywhere in the
system. Every timestamp you see in the application — in list windows,
detail dialogs, history dialogs, and the Provenance tab — is a
local-time rendering of a UTC value stored in the database.

The combination of the two time dimensions allows OreStudio to answer
questions that neither alone could answer:

- /What is the current record for USD?/ — the live row has
  =valid_to = sentinel=.
- /What did the USD record look like on a specific past date t?/ —
  find the row where =valid_from ≤ t < valid_to=.
- /When did we first record the current name for a currency?/ —
  inspect the =recorded_at= of the version where the name field
  changed.
- /Revert to an earlier version/ — the revert operation inserts a new
  row copied from the historical version, creating a new current record
  with its own transaction time and leaving all intermediate versions
  intact.

Bitemporality is particularly important for *calculation
reproducibility*. Financial valuations depend on reference data — a
holiday calendar, a rounding convention, a market tier — that can
change after the fact. Being able to reconstruct the exact state of all
reference data as it existed on a past valuation date is a prerequisite
for explaining why a historical calculation produced the result it did,
for regulatory back-testing, and for resolving disputes about
previously reported figures.

*** Versioning and Immutable History

Every save to a reference data entity creates a new, numbered version.
The version counter starts at 1 on the first save and increments
monotonically. History is immutable: no version is ever deleted or
modified. Even a "revert" operation creates a new version — it does
not roll the counter back or remove intermediate history.

This immutability guarantee is deliberate. Compliance frameworks such
as MiFID II and EMIR require that firms be able to demonstrate the
state of their data at any past point in time. If a regulatory
inquiry asks "what was your EUR rounding rule on a specific date?",
the answer must be provable from the data, not reconstructed from
memory. OreStudio's history tables provide this proof unconditionally.

#+caption[The Currency History dialog]: The Currency History dialog for the Angolan Kwanza, showing three versions. Version 1 was imported from the ISO 4217 dataset; version 2 introduced a typo for testing purposes; version 3 corrects it. The Changes tab shows a field-by-field diff for the selected version against its predecessor.
[[../../../assets/images/currency_history_dialog.png]]

The history dialog, accessible from every detail dialog's title bar or
from the list window's right-click menu, exposes the full version
history of any entity. Selecting a version in the upper table
populates a detail panel below it. The *Changes* tab shows a
field-by-field diff between the selected version and its predecessor
— exactly the fields that changed, with old and new values side by
side. The *Full Details* tab shows the complete record as it existed
at that version. The *Revert* toolbar button reinstates a historical
version as a new current version, creating a new entry at the top of
the history rather than removing subsequent entries.

*** Provenance

Every version of every reference data record carries six standard
provenance fields. Some are written by the application at the time of
the save request; others are written by the database trigger and are
never set by the application:

#+latex: \begin{fullwidth}
#+attr_latex: :environment tabularx :width \linewidth :align lllX
| Field                | UI Label      | Set by       | Meaning                                                                          |
|----------------------+---------------+--------------+----------------------------------------------------------------------------------|
| =version=            | Version       | DB trigger   | Monotonically increasing integer; starts at 1, incremented on each update.       |
| =modified_by=        | Modified By   | Application  | Username of the account that submitted the change request.                       |
| =performed_by=       | Performed By  | Application  | Username on whose behalf the change was performed; may differ from =modified_by= in delegated workflows. |
| =recorded_at=        | Recorded At   | DB trigger   | UTC wall-clock timestamp at the moment the row was written (displayed in local time, like all UI timestamps). |
| =change_reason_code= | Change Reason | Application  | Structured code drawn from the change reasons table explaining the business justification. |
| =change_commentary=  | Commentary    | Application  | Optional free-text note accompanying the change.                                 |
#+latex: \end{fullwidth}

These fields are displayed read-only in the *Provenance* tab of every
entity detail dialog. The tab is disabled (greyed out, not hidden) in
create mode — provenance does not exist for a record that has not yet
been saved.

The distinction between =modified_by= and =performed_by= matters in
automated workflows where a service account submits a change on behalf
of a human user. In those cases =modified_by= identifies the service
and =performed_by= identifies the human, preserving accountability at
both layers.

*** Change Reasons

Before every save OreStudio prompts for a /change reason/ — a
structured code that identifies the business justification for the
change. Change reasons are organised into categories; both windows
are accessible from /Reference Data → Audit Trail/.

#+caption[Change Reason Categories]: The Change Reason Categories window (Reference Data → Audit Trail → Change Reason Categories) showing the three built-in category groups and their regulatory alignment.
[[../../../assets/images/change_reasons_categories_main_window.png]]

reason codes with their category, description, and display order.
#+end_note

There are three standard categories:

- *common* — universal data quality reasons aligned with BCBS 239 and
  FRTB standards. These apply across all entity types. Examples:
  =common.rectification= for correcting a data-entry error.
- *trade* — trade lifecycle reasons aligned with FINRA and MiFID II
  standards. These apply when a change is driven by a trade event or
  a regulatory reporting obligation.
- *system* — reasons for automatic, system-generated operations that
  are not user-selectable. Examples: =system.external_data_import= for
  a bulk import from an authoritative data source, and
  =system.new_record= for the initial creation of an entity.

#+caption[Change Reason Category detail]: The detail dialog for the =trade= category, showing the code and regulatory description. Each category carries its own provenance tab and is itself subject to change management.
[[../../../assets/images/change_reasons_categories_details.png]]

Change reasons serve two purposes. First, they act as a forcing
function: requiring an explicit reason prevents silent, undocumented
changes and encourages operators to think deliberately before saving.
Second, they make the audit trail machine-readable and regulatorily
aligned. A compliance officer can query all records modified under
=common.rectification= across an entire tenant in a single query,
and the category structure maps directly to reporting obligations
under MiFID II, FRTB, and BCBS 239.

Each reason code has three properties that govern when it may be used
and what it requires: whether it applies to new record creation, to
amendments of existing records, or to deletions; and whether a
free-text commentary is mandatory or optional. These constraints are
enforced by the application layer before a save is submitted.

**** The System Category

System reasons are assigned automatically by the application and
services. They are not presented to users in the change reason
prompt and cannot be selected manually. They exist solely to
provide traceability for machine-driven operations.

#+latex: \begin{fullwidth}
#+attr_latex: :environment tabularx :width \linewidth :align llX
| Code                           | Applies to      | Description                                                                     |
|--------------------------------+-----------------+---------------------------------------------------------------------------------|
| =system.initial_load=          | Create          | Initial system bootstrap or database migration. Used once during deployment.    |
| =system.new_record=            | Create          | Normal operational record creation by the application or a service.             |
| =system.external_data_import=  | Amend           | Import from an external data source (ISO feed, GLEIF, vendor file). Commentary mandatory — data lineage must be recorded. |
| =system.import=                | Create, Amend   | Data loaded via the CLI import command.                                         |
| =system.test=                  | Create, Amend, Delete | Test data created by automated test suites.                               |
| =system.tenant_terminated=     | Amend           | Applied when a tenant is marked as terminated.                                  |
| =system.admin_reset=           | Amend           | Data reset by a system administrator during re-provisioning.                    |
#+latex: \end{fullwidth}

**** The Common Category

Common reasons are the universal data quality reasons, aligned with
BCBS 239 (Risk Data Aggregation) and FRTB standards. They apply
across all entity types and all tenants. These are the reasons most
frequently used by operators during day-to-day data maintenance.

#+latex: \begin{fullwidth}
#+attr_latex: :environment tabularx :width \linewidth :align llX
| Code                           | Applies to    | Description                                                                      |
|--------------------------------+---------------+----------------------------------------------------------------------------------|
| =common.non_material_update=   | Amend         | A cosmetic or administrative change with no economic impact — a "touch" to refresh a timestamp or correct capitalisation. Commentary optional. |
| =common.rectification=         | Amend, Delete | Correction of a user or booking error. The most commonly used reason for fixing a data-entry mistake. Commentary optional. |
| =common.duplicate=             | Delete        | Removal of a duplicate record where an equivalent entry already exists.          |
| =common.stale_data=            | Amend, Delete | Data that has not been updated within the required liquidity horizon, requiring a refresh or removal. Commentary optional. |
| =common.outlier_correction=    | Amend, Delete | Manual override following a plausibility check failure — when a value falls outside acceptable bounds and is overridden by an operator. Commentary mandatory. |
| =common.feed_failure=          | Amend, Delete | Correction caused by an upstream vendor or API data issue. Commentary mandatory — the failed feed must be identified. |
| =common.mapping_error=         | Amend, Delete | Incorrect identifier translation — for example a wrong ISIN-to-FIGI mapping. Commentary mandatory. |
| =common.judgmental_override=   | Amend, Delete | Expert judgment applied when market prices or reference values are unavailable and an operator must supply a value manually. Commentary mandatory. |
| =common.regulatory=            | Amend, Delete | Mandatory compliance adjustment required by a regulatory obligation or instruction. Commentary mandatory. |
| =common.other=                 | Amend, Delete | Exceptional changes that do not fit any other category. Commentary mandatory — the reason must be fully explained in the note. |
#+latex: \end{fullwidth}

**** The Trade Category

Trade reasons are aligned with FINRA and MiFID II trade lifecycle
reporting requirements. They apply specifically to changes driven by
trade events or trade-processing workflows.

#+latex: \begin{fullwidth}
#+attr_latex: :environment tabularx :width \linewidth :align llX
| Code                      | Applies to    | Description                                                                            |
|---------------------------+---------------+----------------------------------------------------------------------------------------|
| =trade.fat_finger=        | Amend, Delete | Erroneous execution — a trade entered with the wrong quantity, price, or instrument due to a keying error. Commentary optional. |
| =trade.system_malfunction= | Amend, Delete | Change caused by a technical glitch or algorithmic error in an execution system. Commentary mandatory. |
| =trade.corporate_action=  | Amend, Delete | Adjustment following a corporate action such as a stock split, dividend reinvestment, or merger. Commentary optional. |
| =trade.allocation_swap=   | Amend, Delete | Reallocation between a house account and a client sub-account, or between sub-accounts. Commentary optional. |
| =trade.re_booking=        | Amend, Delete | Correction of a wrong legal entity booking — a trade entered under the wrong counterparty or book. Commentary mandatory. |
| =trade.other=             | Amend, Delete | Exceptional trade lifecycle changes that do not fit any other code. Commentary mandatory. |
#+latex: \end{fullwidth}

**** Extending Change Reasons

Change reason categories and individual codes are stored as versioned
reference data and can be added or modified through the same Qt UI
and CLI tools used for other entities. In principle, a tenant
administrator can create additional categories and codes to cover
firm-specific workflows.

In practice, extending the built-in taxonomy should be done
cautiously and sparingly:

- The standard codes are aligned with regulatory reporting
  frameworks (BCBS 239, FRTB, MiFID II, FINRA). Adding non-standard
  codes risks creating audit trail entries that do not map cleanly
  to regulatory reports, complicating compliance submissions.
- The =common= and =trade= categories are designed to cover the vast
  majority of change scenarios. Before adding a new code, check
  whether =common.other= or =trade.other= (with mandatory commentary)
  would adequately document the change.
- Custom codes cannot be distinguished from standard codes by the
  system — there is no "built-in" flag — so any tenant that adds
  codes must maintain its own record of which codes are standard and
  which are custom.

If you do add codes, use the =category.reason_name= naming convention,
document the code's meaning and commentary requirements thoroughly,
and treat it as a long-lived commitment: changing or removing a
reason code after it has been used in the audit trail would leave
historical records referencing a code whose meaning is no longer
documented.
