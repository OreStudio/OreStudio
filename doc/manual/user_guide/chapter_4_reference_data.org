** What is Reference Data?

Reference data is the static or slowly-changing master data that all
financial activity depends on. It defines the vocabulary of the system
— the currencies a trade is denominated in, the counterparty on the
other side of a deal, the book it settles into, the country a legal
entity is incorporated in. Without it, nothing else can be described
precisely.

Examples of reference data in OreStudio include:

- *Currencies* — ISO 4217 codes, display formatting, and rounding rules.
- *Countries* — ISO 3166 alpha-2 codes and names.
- *Counterparties* — the external legal entities your organisation trades with.
- *Books* — the internal trading books that own positions.
- *Business units and portfolios* — the organisational structure within a party.
- *Party types and statuses* — the classification vocabulary for parties.

Reference data is distinct from the other two main categories of data
in the system:

- *Market data* — prices, FX rates, yield curves, and volatility
  surfaces that update continuously throughout the trading day.
  Market data is time-stamped and immutable once recorded; a
  new observation does not overwrite an old one.
- *Trades* — specific financial transactions between parties with
  defined cashflows, valuation models, and lifecycle states. Trades
  reference the reference data (they need a currency, a counterparty,
  a book) but they are not reference data themselves.

The key property of reference data is that it changes slowly and
deliberately. A currency's rounding convention is not something that
shifts intraday — it is corrected through a controlled, audited
process. OreStudio enforces this by requiring a /change reason/ for
every modification and recording a full version history for every
entity.

** Data Quality

Financial calculations are only as good as the data they consume. A
misspelled currency name is a cosmetic nuisance; an incorrect rounding
rule or a wrong counterparty legal entity identifier can propagate
into risk figures, margin calls, or regulatory reports. OreStudio's
reference data layer is built around six industry-standard data quality
dimensions, codified by ISO 8000 and DAMA:

- *Accuracy* — values match their authoritative source. OreStudio
  seeds currencies, countries, and counterparty identifiers from
  ISO 4217, ISO 3166, and the GLEIF LEI registry respectively.
- *Completeness* — all mandatory attributes are present. The service
  layer rejects saves that omit required fields; the Qt UI marks them
  visually.
- *Consistency* — no contradictions across entities. Foreign-key
  constraints in the database enforce that a currency's rounding type
  and market tier are drawn from valid classification tables.
- *Timeliness* — data is available when calculations need it. The
  tenant provisioner imports standard catalogues at setup time so that
  currencies and countries are ready before any trading activity.
- *Validity* — values conform to business rules and formats. ISO code
  lengths, numeric code ranges, and format strings are validated at
  save time.
- *Uniqueness* — no duplicate records for the same entity. Primary key
  constraints prevent duplicates at the database level.

*** Provenance

Every reference data record in OreStudio carries six standard
provenance fields that together answer the "who, when, and why" of the
last change:

| Field         | UI Label      | Meaning                                                    |
|---------------+---------------+------------------------------------------------------------|
| =version=     | Version       | Increments from 1 on every save.                           |
| =modified_by= | Modified By   | Account that submitted the change.                         |
| =performed_by= | Performed By | Account on whose behalf the change was made (may differ in delegated workflows). |
| =recorded_at= | Recorded At   | UTC timestamp when the row was written.                    |
| =change_reason_code= | Change Reason | Structured reason code drawn from the change reasons table. |
| =change_commentary= | Commentary  | Optional free-text note accompanying the change.           |

These fields are displayed in the read-only *Provenance* tab of every
detail dialog and are stored in the current record. In addition, the
database writes a copy of every version into a companion history table
so that the full audit trail — including the values of all fields at
every past version — is always recoverable.

*** Versioning and History

Saves never overwrite data. Each change creates a new row in the
history table and increments the version counter on the live record.
The history dialog for any entity shows the complete version list,
allows field-by-field comparison between any two versions, and
provides a revert button that reinstates any past version as a new
current version — preserving the audit chain rather than rolling it
back.

*** Change Reasons

Before every save, OreStudio prompts for a /change reason/ — a
structured code drawn from a configurable table (for example,
=common.rectification= for a data-entry correction, or
=system.external_data_import= for an automated import). Change
reasons make the audit trail machine-readable: a compliance officer
can filter the history of all records modified under
=common.rectification= across an entire tenant. Some reason codes
require a free-text commentary; others make it optional.
