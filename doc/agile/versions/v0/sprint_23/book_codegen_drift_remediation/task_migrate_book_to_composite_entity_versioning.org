* Goal

Register Book as a composite child of Portfolio (not of book_status/
regulatory_book_type) so that adding/editing/deleting a book bumps its
parent portfolio's version, and "portfolio as of version N" can
correctly compose the books it contained at that point in time.

** Recursive (multi-level) bump

Manual QA correctly identified a real gap: Portfolio has its own
self-referencing =parent_portfolio_id= (a portfolio tree), and the
initial implementation only bumped the *immediate* parent -- a
grandparent portfolio would never see a grandchild book's change,
exactly the multi-level composition case the target-state doc flagged
as a deferred follow-up ("out of scope for the first cut... a natural
extension").

Fixed by declaring Portfolio as composite parent *and* child of
itself: added a =* Foreign keys= section to portfolio.org for its own
=parent_portfolio_id= (previously undeclared -- a pre-existing gap,
this column had *no* existence validation at all) with
=:bump_parent_version: true=. Since every portfolio write already
re-fires Portfolio's own insert trigger (including the touch
function's re-insert when a *child* bumps it), that trigger now also
recursively bumps *its own* parent. A root portfolio
(=parent_portfolio_id IS NULL=) safely terminates the recursion: the
touch function's row lookup on a NULL id matches nothing and no-ops.

This surfaced a genuine template bug, not just a missing model flag:
=CREATE RULE= (the soft-delete rule) is parsed eagerly by PostgreSQL,
unlike a PL/pgSQL function body -- so a self-referencing entity's
delete rule, which calls its own touch function, failed with
"function ... does not exist" because the touch function was
generated *after* the trigger/rule in the same file (fine for
cross-entity parent/child, since the child's file always runs after
the *entire* parent file, but broken for self-reference). Fixed the
shared template (both the tangled =.mustache= and its literate =.org=
source, re-tangled via =compass build --direct
tangle_codegen_templates=) to generate the touch function *before* the
insert trigger/delete rule, not after. Confirmed safe for the existing
party/counterparty pilot too (diffed both -- content unchanged, only
relocated).

Verified via SQL with a 3-level tree (grandparent -> parent -> book):
adding a book to =parent= correctly bumped =parent= (v1->v2) *and*
=grandparent= (v2->v3) in the same transaction.

* Result

Book registered as composite child of Portfolio (=:bump_parent_version:
true=, =:list_by_as_of: true= on book.org's =parent_portfolio_id= FK;
=:generate_touch_function: true= on portfolio.org). Recursive
multi-level bump implemented by declaring Portfolio composite parent
*and* child of itself, uncovering and fixing a real codegen-template
bug (touch-function generation order vs. eager =CREATE RULE=
resolution) along the way -- confirmed safe for the existing
party/counterparty pilot. Mechanism verified directly via SQL (not
just generated code) at 2 and 3 levels deep, and =ores.refdata.api/core/
service.tests= pass 3/3.

Manual QA scenario PASSED. Its one reported FAIL (delete-side bump not
visible in the Portfolio History dialog) was run down to ground: a
direct SQL reproduction against the exact same real portfolio/tenant
showed the bump firing correctly (v5->v6 on insert, v6->v7 on delete),
proving the backend is correct and the UI symptom is the already-known
Portfolio-window live-refresh gap (Portfolio's Qt layer doesn't react
to NATS events from a sibling entity's write -- out of scope here,
captured separately).
