* Notes

Earlier run (2026-07-13, commit 69ea5f7be) found no eventing on Create
and stale reads on the list until reload. Root cause: the committed
=ContactTypeController.cpp= and other Qt files were drifted, hand-adjusted
output that predated the current codegen templates (never regenerated),
and the entity had no NATS event registrar wired at all. Rather than
hand-patch the symptom, =ores.refdata.contact_type.org= was brought up
to the latest meta-model and every facet regenerated (including moving
contact_type's Qt ownership from =ores.qt.party= to =ores.qt.refdata=,
matching its domain component) — see [[id:BD717BF3-DFD6-41C6-9A88-90B0470A37A7][Update contact_type codegen model
and regenerate all layers]]. Second run (commit 456b512df) still found
=ContactTypeController= missing from the "Organisation Codes" menu and
its "created" log line entirely absent. Root cause turned out to be
deeper than a stale link: every single service in the =prime_origin=
fleet had an orphaned duplicate process left over from an earlier
=compass services stop= that only killed the top-level controller,
not the child services it had spawned — including a stale
=ores.refdata.service= and stale =ores.compute.wrapper= processes
predating the clean rebuild. Both the old and new processes were
bound to the same NATS queue groups, so requests routed
unpredictably to whichever answered first: this explains the missing
menu action, the missing constructor log line, and a third symptom
found along the way (a =refdata.v1.countries.publish-from-dq=
provisioning request that completed its DB work but never sent a
reply, hanging the GLEIF tenant-provisioning workflow at the same
step on every attempt). Killed all orphaned PIDs, verified 1:1
process-to-service parity via =pgrep=, recreated the DB, and
reprovisioned — this third run passed cleanly end to end.
