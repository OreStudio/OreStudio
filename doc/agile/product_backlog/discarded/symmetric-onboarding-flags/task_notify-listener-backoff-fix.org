* Goal

The NOTIFY listener's =listen_loop= holds the mutex for the entire
=consume_input()= + =get_notifications()= + callback chain. During a heavy
bulk import (e.g. GLEIF party dataset — hundreds of INSERTs each firing a
NOTIFY trigger), rapid-fire notification processing causes mutex contention
that blocks the 100ms poll cycle for up to ~30 seconds. The callbacks run
slowly under load, so the 100ms sleep never gets a chance to fire and the
listener appears to have died.

Refactor to hold the mutex only for the drain step, then process callbacks
outside the lock:

#+begin_src cpp
std::vector<Notification> batch;
{
    std::lock_guard lock(mutex_);
    if (!(*connection_)->consume_input()) { ... }
    batch = (*connection_)->get_notifications();
}
for (const auto& n : batch)
    handle_notification(n);
#+end_src

Also reduce =max_backoff= from 30s to 5s (the 30s ceiling was appropriate
for a remote DB; a local PostgreSQL should reconnect in under a second).

Note: suspending the listener during wizard execution was considered but
rejected — it requires cross-process coordination (Qt client → service)
and leaves the listener permanently paused if the client crashes mid-wizard.
