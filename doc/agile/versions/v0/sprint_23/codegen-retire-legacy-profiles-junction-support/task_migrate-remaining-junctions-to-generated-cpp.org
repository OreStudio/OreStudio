This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:9023BDE0-979F-422F-BC73-985D6A80E7CB][Retire legacy codegen profile system; add junction support to physical-space codegen]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Regenerate every remaining =has_tenant_id: true= junction's domain/
generator/repository via the junction codegen path this story added
(validated so far only against =tenor_convention_resolution=),
retiring their hand-authored equivalents. Fixes a real, independently-
discovered class of bug along the way: hand-authored junction
repositories that forget the tenant_id filter on their "list all" query
return duplicate rows to the client (both the requesting tenant's own
rows and any other tenant's, since Postgres RLS on these tables is
deliberately permissive -- =tenant_id = current_tenant() OR tenant_id =
system_tenant()= -- to support a "global registry + tenant override"
pattern; the app-level query is the only thing narrowing that back
down, and hand-written code has already gotten this wrong four times
independently).

* Notes

- 2026-07-18: Discovered while debugging duplicate rows in the Badge
  Mappings tab ([[id:98AC38DD-E800-4961-A6BE-9F018097DAB2][Add a
  badge_mapping browser UI]] task, badge colour scheme story).
  =badge_mapping_repository::read_all()= had no =tenant_id= filter at
  all, relying solely on RLS -- which for this table (and every other
  "system-owned global registry, tenant-overridable" table) permits
  =tenant_id = current_tenant() OR tenant_id = system_tenant()=, so any
  tenant with its own published copy of a row (e.g. via a
  publish-from-dq bundle) sees it twice. Hand-patched
  =badge_mapping_repository.cpp= directly to add the filter (matching
  the working pattern below) as an immediate fix, separate from this
  task -- see the badge task's PR.
- Full inventory of all 11 =has_tenant_id: true= junctions, checked by
  reading each one's =read_all=/=read_latest= query:
  | Junction | Component | Tenant filter present |
  |----------+-----------+------------------------|
  | currency_calendar | refdata | yes |
  | currency_country | refdata | yes |
  | currency_pair_convention_calendar | refdata | yes |
  | party_currency | refdata | yes |
  | party_country | refdata | yes |
  | tenor_convention_resolution | refdata | yes (generated by this story) |
  | app_version_platform | compute | yes |
  | badge_mapping | dq | no -- hand-patched as an immediate fix, see above |
  | dataset_bundle_member | dq | no -- still broken |
  | account_party | iam | no -- still broken |
  | party_counterparty | refdata | no -- still broken |
  | currency_currency_group | refdata | n/a -- no C++ repository exists yet, SQL-only |
- =compass codegen entity generate badge_mapping= (the junction path
  this story added) currently writes to a *stale, pre-=ores.dq= api/
  core/service-split path*
  (=projects/ores.dq/include/ores.dq/...=/=projects/ores.dq/src/...=),
  not the real location
  (=projects/ores.dq/core/include/ores.dq.core/...=). This is why
  =compass codegen entity show badge_mapping= reports the C++ files as
  =❌ MISSING= -- it's comparing against the wrong path, not flagging a
  real gap. The generated files at the wrong path were deleted (never
  committed). Path resolution for junctions in the =dq= component needs
  fixing before this task can actually regenerate =badge_mapping= (or
  =dataset_bundle_member=, same component) via codegen; the =iam= and
  =refdata= junctions may or may not have the same issue -- untested.
