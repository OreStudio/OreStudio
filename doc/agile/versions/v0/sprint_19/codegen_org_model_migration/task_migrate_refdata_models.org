* Goal

Convert the 24 remaining =*_domain_entity.json= files in
=projects/ores.codegen/models/refdata/= (the 25th, =party=, is the
pilot) into literate =ores.refdata.<entity>.org= files under
=projects/ores.refdata/modeling/=. Codegen reads them through the
=modeling_dir= discovery path landed in PR #967. Once every entity is
migrated and the regenerated output matches HEAD, the JSON files are
deleted.

Approach: write a *one-shot conversion script* (=migrate_json_to_org.py=)
that reads a directory of =*_domain_entity.json= files and emits the
matching =ores.<component>.<entity>.org= files. The script targets a
byte-identical regenerator round-trip for the *structured* parts of
the model. Manual touch-up is fine for prose (=brief=, descriptions,
detail) and for any custom-repository-method sections that need to
land via the =:implements <kind-UUID>= mechanism.

* Plan

** 1. Write =migrate_json_to_org.py=

Single file, =argparse=, no external dependency beyond stdlib.

#+begin_src sh
python3 projects/ores.codegen/src/codegen/migrate_json_to_org.py \
    --input-dir projects/ores.codegen/models/refdata \
    --output-dir projects/ores.refdata/modeling \
    --component refdata
#+end_src

Maps each JSON section to the meta-model section:

| JSON path                          | Org location                                  |
|------------------------------------+-----------------------------------------------|
| =domain_entity.{entity_singular,...}= | Frontmatter                                  |
| =domain_entity.description=        | Body prose before first heading              |
| =domain_entity.{schema,product,component,has_tenant_id}= | =* Flags= drawer       |
| =domain_entity.primary_key=        | =* Primary key= drawer + body                |
| =domain_entity.natural_keys[*]=    | =** <column>= under =* Natural keys=         |
| =domain_entity.columns[*]=         | =** <name>= under =* Columns=                |
| =domain_entity.sql.tablename=      | =* SQL >> ** Flags= drawer                   |
| =domain_entity.repository=         | =* C++ >> ** Repository= drawer              |
| =domain_entity.cpp.includes.domain= | =** Domain includes= babel block            |
| =domain_entity.cpp.includes.entity= | =** Entity includes= babel block            |
| =domain_entity.cpp.iterator_var=   | =** Conventions= drawer                      |
| =domain_entity.cpp.table_display=  | =** Table display= table                     |
| =domain_entity.qt= flat keys       | =** Qt= drawer                               |
| =domain_entity.qt.detail_fields=   | =*** Detail fields= table                    |
| =domain_entity.qt.columns=         | =*** Columns (Qt model)= table               |

Out of scope for the script (left for hand-authoring later):
- =* Custom repository methods=. The block-paste mechanism requires
  per-method UUIDs and prose context the JSON cannot supply.
- Service flags introduced by the refdata-cpp drift work
  (=service_find_by_uuid=, etc.). Those were never in the JSON.

** 2. Run on refdata, inspect

Run the script. Spot-check 3-4 produced files against the party pilot
for shape. Read one file end-to-end to confirm the structure reads
naturally.

** 3. Round-trip verify

Snapshot refdata-cpp regen output with the JSON files present.
Delete the JSONs, leave only the org files in place, regen, diff. The
two outputs must be byte-identical. Iterate on the script until they
are.

** 4. Manual touch-up

For each entity, briefly read the generated org file and:
- Add a =#+description= field if the script's auto-derived one is
  bland or wrong.
- Reword =brief= if it reads awkwardly.

This step is the gate where the user has eyes on every entity.

** 5. Build + test

- =cmake --build --preset linux-clang-debug-make=
- =ctest -R "ores\.refdata"=

** 6. Commit + push + raise PR

Three commits:
1. Add the conversion script.
2. Add the 24 generated org files.
3. Delete the 24 JSON files.
