* Plan

Analysis (see [[id:74AA46EB-64ED-4FD7-B212-AEC164648B84][Book classification]], "How the classification axes
relate to each other") showed the original 5-boolean scope doesn't
hold up: =is_manual_ledger_book= is not independent of
=is_ledger_book= (invalid state possible: manual=true, ledger=false),
and the task's stated precedent (=is_trading_book= as a checkbox) is
stale — that field was already turned into the =regulatory_book_type=
lookup entity earlier in the sprint, precisely because a fixed boolean
turned out to be an extensible category. The same treatment applies
here: Reserve and Funding are two instances of a wider "risk role"
category (with Wash/Write-off/Test/Sales/Central-Remittance already
named in the domain notes as future members), and "fed from a source
book" vs "fed manually" vs "not fed" is a 3-state axis, not two
booleans.

Further teased apart against real process detail (see the new
[[id:B0902FF2-138F-4234-A473-1371549CE219][Funding process]], [[id:9E364BEF-12EE-42F8-902C-B1E68A3095B8][Wash books and risk routing]], [[id:CA299D08-AC97-4A37-B3F2-A5927F379326][Spot sweep and ledger
sweep]], and [[id:DFC04627-FB0F-4FFF-973E-E3EC9E05AD87][Central remittance]] knowledge docs): =book_purpose_type=
values are *mutually exclusive* (a book plays exactly one risk role),
including two more values discovered from the process write-up — Sweep
target and Remittance target — which look combinable at first glance
(both are singleton "central" books) but are not: different owners
(Treasury vs Finance), different cadence, and different P&L treatment
(Sweep realises trading P&L; Remittance is balance-sheet-only only).
Mixing them on one book would blur the audit/reconciliation separation
these processes exist to provide.

Full revised design — 2 new lookup FKs + 1 boolean, no cross-axis
constraints (per the Basel/FRTB analysis: no regulatory rule ties risk
role to trading/banking classification, so regulatory type, ledger
feed, and purpose combine freely with each other; only *within*
=book_purpose_type= are the values mutually exclusive):

| Field                 | Type                         | Values                                          | Task |
|-----------------------+-------------------------------+--------------------------------------------------+------|
| =book_purpose_type=   | FK -> new lookup entity      | Trading (default/ordinary), Reserve, Funding, Wash, Write-off, Test, Sales, Sweep target, Remittance target | this task |
| =ledger_feed_type=    | FK -> new lookup entity (or 3-value enum) | None, Automatic, Manual                        | [[id:DC13CC83-2807-4AA9-B56E-BA7D6BA896D2][Add ledger_feed_type lookup entity to book]] |
| =is_sweepable=        | bool                         | independent of the above; governs spot-sweep eligibility only, not ledger sweep; UI may default false for Reserve purpose but no DB constraint | [[id:FA28F852-012F-47A7-94E7-FEDAEF2F2DF9][Add is_sweepable flag to book]] |
| Wash paired risk book | nullable self-FK to book     | set only when purpose = Wash                    | [[id:077F2C20-50AE-4E23-A4E1-0FC053854313][Add paired risk-book link for Wash-purpose books]] (BACKLOG, discovered scope) |

=regulatory_book_type= (Trading/Banking) is unchanged — already
correctly modeled, and confirmed independent of purpose/ledger-feed by
the FRTB analysis.

*This task's scope is now =book_purpose_type= only.* Follow the
=regulatory_book_type= implementation as the pattern (domain/SQL/
repository/protocol/Qt CRUD + badge-coloured combo), not the
originally-scoped checkbox pattern. The companion fields above split
into their own sibling tasks under this story.

** Implementation

Reviewed the pre-scaffolded =ores.refdata.book_purpose_type.org=
against =regulatory_book_type.org= and proactively added
=:has_pagination: true= and =:has_change_reason_cache: true= (the
same gap that caused real bugs for book_status/regulatory_book_type
earlier in the sprint -- caught this time before generating). Ran
=compass codegen entity generate book_purpose_type= (full/supported
set) -- all layers generated cleanly, byte-for-byte following
regulatory_book_type's shape.

Manual QA surfaced two real gaps, both fixed:

- *NATS handler never wired.* Codegen generates
  =book_purpose_type_registrar.cpp/hpp= (the request handlers) but
  nothing calls =register_book_purpose_type_handlers()= --
  =projects/ores.refdata/core/src/messaging/registrar.cpp= is a
  hand-maintained dispatch list, not itself generated, and wiring a
  new entity's CRUD handlers into it is a manual step the codegen
  workflow doesn't cover (only the *eventing* registrar,
  =event_registrar.cpp=, was wired proactively, which is why the
  Postgres NOTIFY -> NATS event subscription showed up in the service
  log but the actual list/detail/save requests silently failed with
  "Failed to fetch ... from server" -- the client's request had no
  handler to answer it). Added the missing =append(...)= call.
- *New system reference data wasn't copied into existing tenants.*
  =iam_tenant_provisioner_create.sql= has a hand-written, per-entity
  copy block that seeds a newly-provisioned tenant's lookup tables
  from the system tenant's rows at provisioning time --
  book_purpose_type wasn't in that list (it didn't exist when that
  function was last touched), so Barclays' tenant only ever saw the 0
  rows it started with. This is the same publish-at-provisioning
  mechanism book_statuses/regulatory_book_types/purpose_types already
  use -- confirmed via direct SQL comparison (regulatory_book_types
  showed 2 system-tenant rows *and* 2 Barclays-tenant rows; book_purpose_types
  showed 9 system-tenant rows but 0 Barclays-tenant rows before the fix).
  Added the matching copy block; re-provisioned Barclays and confirmed
  exactly 9 rows landed under its tenant.

Neither gap is specific to book_purpose_type -- both are pre-existing,
undocumented manual steps in the "add a new lookup entity" workflow
that happened to not bite book_status/regulatory_book_type/purpose_type
only because whoever added them (in earlier sessions/sprints) happened
to remember. Worth a backlog item to either codify these as codegen
steps or document them explicitly in the entity-creation runbook.
