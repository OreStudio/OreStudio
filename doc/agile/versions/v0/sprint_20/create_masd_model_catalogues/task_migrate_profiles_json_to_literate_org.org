* Goal

Replace the hand-edited =profiles.json= with a literate org-mode source =facet_catalogue.org= that tangles to it. The org file is structured in three sections matching the MASD taxonomy derived from the analysis in the parent story's =* Decisions=:

1. *Facets* — one org heading per facet (=sql=, =domain=, =repository=, =qt=, etc.), each with its archetypes (template→output pairs) as sub-entries, the model types it binds to, and a description of its role in the physical space.
2. *Facet groups* — composite entries (=all-cpp=, =all=, =non-temporal=) that activate multiple facets in one invocation; documented as facet-group compositions.
3. *Component archetypes* — scaffold templates (=component=, =component-api=, =component-core=, =component-service=) that instantiate new physical components; these operate on the =component= model type rather than on entity types.

=profiles.json= becomes a generated artefact produced by tangling =facet_catalogue.org=. CI fails if =profiles.json= is edited directly without updating the org source.

*** profiles.json shape (the tangle target)

- File: =projects/ores.codegen/library/profiles.json=, 572 lines,
  22,880 bytes, 2-space indent, trailing newline. Top-level
  ={ "profiles": { <name>: {...}, ... } }=.
- A leaf profile has =description= (string), =model_types= (array of
  strings), =templates= (array of ={ "template", "output" }= objects).
- A composite profile has =description=, =model_types=, and =includes=
  (array of profile names) instead of =templates=.
- *Formatting gotcha (key risk for the zero-diff check):* =model_types=
  is emitted /inline/ (=["domain_entity", "junction", ...]=) while
  =templates= is emitted /multi-line/ (one object per line). A naive
  =json.dump(indent=2)= will NOT reproduce this — it expands
  =model_types= onto multiple lines. So either: (i) the converter
  custom-formats to match exactly; or (ii) — preferred — the drift
  check compares /parsed JSON/ (=json.load both, assert equal=) rather
  than bytes, which is what actually matters since =generator.py= reads
  it via =json.load=; or (iii) canonicalise =profiles.json= formatting
  once as a separate prep commit so the converter's natural output is
  the target.

*** Stale output patterns — corrected characterisation

The earlier "28 stale" figure conflated /model substitution variables/
with /stale paths/. =\{component\}=, =\{component_include\}=,
=\{component_core\}= are legitimate generator variables — they are not
stale in themselves. What is stale is specific /path structures/ that
assume the pre-regroup layout. Raw occurrence counts of the
=projects/ores.{component...}= forms: =\{component_core\}= ×15,
=\{component_include\}= ×13, =\{component\}= ×3, =ores.qt.\{component\}=
×12. Capture [[id:12450E70-EF24-49D9-B156-808B7529932A][12450E70]] confirms at least the =ores.qt.\{component\}= and
=projects/ores.\{component_include\}= forms target directories that no
longer exist; the /precise/ stale set needs the per-pattern audit that
capture describes — do NOT trust a regex. This is why the recommendation
is to migrate /byte-/ (or JSON-) /identical first/ and fix paths as a
separate pass on =facet_catalogue.org= (closing 12450E70).
