* Goal

Build two 100%-handcrafted, reusable Qt classes: a generic
self-referencing HierarchyTreeWidget, and a hierarchy-model-builder

codegen templates so a generated dialog can bind them.

* Acceptance

- HierarchyModelBuilder and HierarchyTreeWidget exist, 100% handcrafted, entity-agnostic.
- Unit tests cover HierarchyModelBuilder's tree-building logic (multi-level, multi-root, orphan cases).
- Qt detail-dialog codegen templates have a paste-block seam for binding the widget.
- Full ores.qt.api.lib and ores.qt.api.tests build cleanly; tests pass.

* Plan

Surveyed existing conventions first: =ores.qt/api= layout (headers
under =include/ores.qt/=, sources under =src/=, CMakeLists globs
recursively), the =EntityDetailDialog=/=EntityDetailOperations=
Doxygen/license-header style, and the existing =:implements <UUID>=
paste-block convention (=d4e8e136-...= in the SQL templates) as the
model for the new Qt seam.

Implemented =hierarchy_node= (id/parent_id/label POD, no party or
counterparty knowledge) + =HierarchyModelBuilder::build()= producing a
=QStandardItemModel*=, and =HierarchyTreeWidget= (a =QTreeView=-based
=QWidget= taking any model via =setModel()=). Added one new paste-block
seam (=9B165431-2921-4CAC-A2E8-2C186741E523=) to the generated
=<Entity>DetailDialog.cpp= constructor, right after
=setupConnections()=, documented in =org_entity_meta_model.org=
alongside the existing seams. No entity implements the seam yet —
binding party/counterparty is task 3.
