A codegen *entity model* is a single literate org-mode file that
describes one entity (e.g. [[id:8F3A1C2D-4B5E-4A6C-8F9D-2E3F4A5B6C7D][party]]) and drives code generation across
every technical space that entity touches: [[id:E39C0E47-768F-4DE2-8436-08C4A8A06CFA][SQL]] and several independent
C++ facets (e.g. [[id:62342007-5080-43BF-8207-CD05B6568BBE][ores.cpp.domain]], [[id:50246CF5-EB61-4261-890F-4E75D3241863][ores.cpp.repository]],
[[id:E1898A0F-EC04-4CC3-A547-951F84827432][ores.cpp.qt]]). In [[id:BA0E3273-DC4B-7CD4-447B-06227B7D09A8][MASD]] terms, this hub — together with its
segment pages — is the *metamodel* for the entity model: it does not
describe any one entity, it describes what any entity model file is
allowed to contain. See [[id:43FF3591-AD18-4CE3-8239-A220862C9B5A][Applied MASD]] for how ORE Studio instantiates
MASD's logical/physical space, metatype, facet, and archetype
concepts — those concepts are exactly the organising principle this
hub uses below.

This page stays deliberately short: an overview, the file-level
conventions every entity model shares regardless of which facets it
uses, and a table routing to the segment page that documents each
facet's knobs, worked example, and paste blocks in full. Go to a
segment page for the exhaustive per-property reference; come back
here only for the shape of the file as a whole.

* About this document

The =:ignore:= tag (not applied to this heading itself — this is a
knowledge doc, not an entity model, so the tag has no functional
effect here) asks the codegen loader to skip a heading when reading
an actual entity file (it means "skip this heading", not "this
document is unimportant" — an unfortunate naming collision if you're
skimming the table of contents of an entity model that uses it).
Sections without that tag may be consumed by the loader.

* Notes

** Terminology: knobs are (mostly) MASD variability

Every "knob" documented across these segment pages is, precisely, a
[[id:CE95C751-2125-4BA9-A78D-C30F9540FC5E][MASD Variability]] *feature* (a single configuration point) or belongs
to a *feature bundle*. But not uniformly: MASD reserves "variability"
strictly for *non-structural* configuration — it doesn't change the
object graph, only how it's rendered. Most Qt knobs fit that exactly
(they gate generated code paths without changing the entity's own
logical shape). =:has_tenant_id:= does not — it adds an actual
column/field, which is *structural* variability, living in the
logical model itself, not MASD's Variability Metamodel. This hub does
not yet consistently distinguish the two, and MASD's =profile=
concept (a reusable, named bundle of feature values, composing
base/component/element) looks like a close match for what task
=EE237306= separately calls a domain_entity "shape" — resolving both
of these properly is tracked by capture [[id:31A66613-8584-42AE-97EB-D524F19C8574][Formalize the codegen entity
knob system as a proper MASD/MDE feature model]], not decided
piecemeal across these pages.

** Why two levels of UUID (paste blocks)

See [[id:40AD7EFB-AACE-472F-B767-8FD3FF2F5C03][Paste blocks]] for the full explanation; in short, a *block kind* is
a stable, template-referenced UUID (a concept); a *block instance* is
one entity's per-block =:ID:= (an implementation). This lets
templates and entity files evolve independently.
