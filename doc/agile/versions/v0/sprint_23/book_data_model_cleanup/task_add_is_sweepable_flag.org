* Plan

Straightforward boolean column addition to book.org, following the
=check_box= widget pattern already used by cds_convention/
deposit_convention/etc. (=type: boolean=, =cpp_type: bool=,
=nullable: false=, =default_value: false=). Badge rendering reuses the
existing =is_badge=/=badge_key= mechanism (=dq_badge_system_populate.sql=)
with a new =is_sweepable= code domain and =true=/=false= mappings onto
the existing active/inactive colour definitions -- the Qt model's
=is_bool= formatter already emits =tr("true")=/=tr("false")= strings,
which is what the badge lookup keys off.

=validate_schemas.sh= (run for the first time as part of this task,
per user direction) surfaced two real, pre-existing gaps unrelated to
is_sweepable itself: book_purpose_type and ledger_feed_type (both
merged in earlier tasks) were missing RLS policies entirely --
tenant-isolation =ENABLE ROW LEVEL SECURITY=/=CREATE POLICY= is 100%
hand-maintained per component, no codegen facet generates it (matches
an existing backlog capture, =codegen_rls_policy_facet.org=, updated
with this fourth recurrence). Fixed both by hand, matching the
existing convention in =refdata_rls_policies_create.sql=.

Manual QA surfaced a second real, pre-existing bug: the check_box
widget type was scaffolded into the Qt detail-dialog codegen template
(populate/collect field logic existed) but never wired into
=setupConnections()= -- so toggling a checkbox never marked the dialog
dirty via =onFieldChanged()=, and Save stayed disabled until some
other field was also touched. This affects every entity using
=check_box= fields (cds_convention, deposit_convention,
currency_pair_convention, ibor_index_convention, ois_convention, and
two synthetic/marketdata entities), not just book. Fixed in the shared
=.org= source (=ores.cpp.qt.detail_dialog_impl.org=) using
=&QCheckBox::toggled= (the version-portable signal, matching existing
hand-written checkbox code elsewhere in the codebase) -- deliberately
did *not* regenerate the other affected entities, since their whole Qt
layer is independently stale/drifted already and pulling that in would
have been unrelated scope creep for this task.

Also found and fixed: =refdata_publish_from_dq_create.sql='s
explicit-column =INSERT INTO ores_refdata_books_tbl= (used by the
"Publish Datasets" / Barclays-provisioning workflow) didn't include
=is_sweepable=, which is =not null= with no DB default -- would have
broken provisioning outright. Rather than hardcoding =false= for every
published book, added =is_sweepable= as a real column on
=ores_dq_books_artefact_tbl= (the DQ staging table) and threaded it
through, so individual seed books can be marked sweepable.

Per user direction, also replaced the testdata books/portfolios
artefact scripts' sequential fake UUIDs
(=20000000-0000-4000-a000-...=/=30000000-0000-4000-a000-...=) with
real random ones (verified no other file in the codebase references
the old fake values before replacing), and added two realistic
sweep-participating books (=GBP Nostro Sweep=, =USD Cash Management
Sweep=, both =is_sweepable = true=) under the existing GBP/USD Rates
portfolios, per the [[id:CA299D08-AC97-4A37-B3F2-A5927F379326][Spot and ledger sweep]] domain doc's naming.
