#+title: Party re-provisioning fails on duplicate report definitions
#+description: provision party --reports all in ores.shell/Qt wizard throws a raw Postgres duplicate-key error instead of skipping/upserting cleanly when re-run against an already-provisioned party.
#+type: capture
#+level: cross
#+filetags: :shell:qt:provisioning:reporting:
#+created: 2026-07-13
#+updated: 2026-07-13

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

=ores.shell='s =provision party <name> --reports all= (and the
equivalent Qt =PartyProvisioningWizard= path) creates report
definitions unconditionally in its "Creating report definitions"
phase, with no existence check or upsert semantics. Re-running
provisioning against an already-provisioned party fails immediately
with a raw Postgres error surfaced to the operator: ~Repository
error: Executing INSERT failed: ERROR: duplicate key value violates
unique constraint "report_definitions_name_uniq_idx"~. Every other
phase in the same command (counterparty import, organisation bundle,
synthetic FX configs, FX driver rates, CRM topology bundles) is a DQ
=publish-from-dq= call and is idempotent by construction (upsert
mode, skip-if-unchanged); report-definition creation is the one
phase that is not, and it aborts the whole command before later
phases run.

* Why

Found while manually verifying [[id:6D953558-A61C-461F-859D-CBD28AD5D8F5][the CRM topology bundle publish task]]
(PR #1543): re-provisioning the same Acme test party a second time to
probe idempotency of the new CRM/FX phases aborted at the
report-definitions phase before reaching them, forcing a
=--reports none= workaround. This is a real rough edge for anyone
re-running provisioning against an existing party during manual QA or
demo-environment refresh -- the error message doesn't even name which
report definition collided, let alone offer a clean skip/upsert path.
