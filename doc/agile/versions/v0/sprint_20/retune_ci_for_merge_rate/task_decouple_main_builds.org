* Goal

Remove push-to-main triggers from the three continuous workflows;
add cron schedules (Linux 2h, macOS 3h, Windows 4h) with
=cancel-in-progress: false=; prepend a relevance guard comparing
HEAD with the last successful run's SHA, skipping when only
doc-class paths changed. Coverage stays on the Linux debug leg.

* Acceptance

- No compile workflow has a push trigger.
- Schedules fire at 2h/3h/4h and never cancel each other.
- A doc-only merge stream produces zero scheduled builds.
- First run after the change builds (no prior green run edge case).

* Plan

1. Converted all three continuous workflows from =push: branches: [main]=
   to =schedule: cron= (Linux 2h, macOS 3h, Windows 4h) plus =push: tags=
   and =workflow_dispatch=. Removed all push/branches triggers.

2. Changed =cancel-in-progress: true= → =false= in all build jobs so
   staggered runs queue rather than cancel each other.

3. Added a =guard= job (ubuntu-24.04) before =build= in each workflow that:
   - On non-schedule events, always outputs =should_build=true= and exits.
   - Queries the GH API for the workflow's last successful run SHA on main.
   - If no prior run or SHA matches HEAD, short-circuits.
   - Diffs HEAD vs last SHA; filters out doc-class paths
     (=doc/=, =assets/=, =.claude/=, =.gemini/=, =projects/ores.codegen/=,
     =*.md=, =*.org=). If only doc-class paths changed, outputs
     =should_build=false=.

4. Wired =build= with =needs: guard= and
   =if: needs.guard.outputs.should_build == 'true'=.

5. Used =GITHUB_WORKFLOW_REF= to extract the workflow filename dynamically
   rather than hard-coding it.
