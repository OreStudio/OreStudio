MASD is the [[https://uhra.herts.ac.uk/id/eprint/17031/][Model Assisted Software Development]] methodology developed as
part of a PhD programme and documented in full on the [[https://mcraveiro.github.io/dogen/index.html][Dogen site]]. ORE Studio
does not use the original code generator Dogen (which was decommissioned) but
ports MASD's core ideas to [[https://en.wikipedia.org/wiki/Large_language_model][LLMs]] and [[https://en.wikipedia.org/wiki/Natural_language_processing][NLP]]. The conceptual model governs how
the project is organised, how code generation is approached, and what the
long-term generative objective is.

* Conceptual model

MASD starts from the premise that a software project is a collection of
/physical artefacts/ — files and directories — that live in /physical space/,
and that these artefacts can be thought of as projections from a
higher-dimension space called the /logical space/. For a thorough formal
treatment of these spaces, models, and transformations see
[[https://mcraveiro.github.io/dogen/docs/models_and_transformations.html][Models and Transformations]] on the Dogen site.

The distinction between logical and physical is adapted from Lakos's
/Large Scale C++ Software Design/:

#+begin_quote
/Logical design/ addresses all the type-related aspects of design — classes
and their relationships. /Physical design/ addresses the placement of logical
entities, such as classes and functions, into physical ones, such as files and
libraries. All design has a physical aspect.
#+end_quote

For ORE Studio:

- /Logical entities/ have well-defined properties in logical space which we
  call /facets/.
- /Transformations/ can be defined that convert each facet into its physical
  representation.

Physical artefacts are /projections/ of logical entities. The same logical
entity projects into many physical artefacts across many components: a C++
struct in an =.api= component, a repository in a =.core= component, a NATS
message type in a messaging header, a SQL table in a create script, a Qt MDI
window in a UI plugin. Each projection is a distinct expression of the same
underlying logical truth.

* Taxonomic analysis: products, components, and product families

MASD provides a precise vocabulary for classifying artefacts. Understanding
this taxonomy is essential for knowing where generated code should live and
how the project is decomposed.

** Product

A /product/ is the overall container for physical artefacts — a complete
software system composed of executables and libraries. ORE Studio is a
product. The GitHub repository is the canonical boundary of the product.

** Component

A /component/ is a library or executable that is part of a larger product.
Each component follows MASD's component directory structure layout and is the
/native home/ for exactly one layer of a logical entity's projection. In ORE
Studio, every =ores.*= CMake target is a component:

- =ores.refdata.api= — domain types and NATS protocol for reference data
- =ores.refdata.core= — repository, mapper, service layer
- =ores.qt.refdata= — Qt MDI windows and dialogs for reference data
- =ores.codegen= — the generator itself

The component structure directly encodes the facet catalogue (see below).

** Product family

A /product family/ is a collection of related products unified by a shared
methodology and reference implementation. ORE Studio is a single product at
present but is designed to be extensible along the same axis — the generator,
the reference implementation (=currency=), and the [[id:897A7156-504D-4581-AD4C-026D643C0838][entity evaluation checklist]]
together constitute the embryonic product family machinery.

** SRPP: the unit of extraction

MASD defines a /Schematic and Repetitive Pattern in the Physical space/ (SRPP)
as the class of patterns found in infrastructure code that vary predictably
across entities but are otherwise mechanically identical. An SRPP is what a
Mustache template encodes. Identifying an SRPP is the analytical step that
precedes writing a template.

In ORE Studio, commissioning the =currency= entity produced the canonical set
of SRPPs for reference-data entities: the C++ domain struct, the rfl-based
JSON I/O, the repository entity and mapper, the service class, the NATS
protocol types, the SQL DDL and trigger scripts, and the full Qt UI (12 files:
MDI window, controller, detail dialog, history dialog, client model, and their
headers).

* Model transformations

MDE distinguishes three categories of /model transformation/ (per
[[https://mcraveiro.github.io/dogen/docs/models_and_transformations.html][Models and Transformations]]):

- *Model-to-Model (M2M)* — transforms one model into another at the same or a
  different abstraction level.
- *Model-to-Text (M2T)* — produces a string representation (source code,
  documentation, configuration) from a model. This is the category that code
  generators occupy.
- *Text-to-Text (T2T)* — converts one textual representation into another
  without an intermediate model step.

[[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]] is a /Model-to-Text/ transformer: it takes JSON or org-mode model
files (the source models), applies Mustache templates parameterised by a
profile, and produces physical artefacts. The entity model format is the
implicit meta-model; the profile catalogue is the transformation configuration;
the Mustache templates are the transformation rules.

ORE Studio takes a pragmatic position: the meta-model is implicit in the model
file schema and in the skill recipes that LLMs use to author model files,
rather than in a formal meta-meta-model such as MOF or Ecore. The underlying
concepts are encoded in skills and recipes that an LLM can understand and
apply.

** Two codegen routes: authored and automated

[[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]] runs in two distinct routes that share the same machinery but
take their input model from different sources:

| Route     | Where the model comes from              | Format     | Where the model lives                     |
|-----------+-----------------------------------------+------------+-------------------------------------------|
| Authored  | A human (or LLM) writes it.             | =.org=     | =projects/ores.<group>/modeling/=         |
| Automated | A script parses already-generated artefacts and derives it. | =.json=    | =build/output/codegen/= (gitignored)      |

The *authored route* is the common case. Domain entities, lookup
tables, junctions, components — everything that a human is expected
to maintain — uses literate org-mode under each component's
=modeling/= directory. Literate format earns its place there because
the model carries intent that other readers need: descriptions,
custom-method prose, generator expressions inside =#+begin_src=
blocks. The [[id:D85E2B30-8C64-46DF-819D-B50D2E70B73C][org-mode migration story]] tracks moving the entire
authored set off legacy JSON.

The *automated route* applies when the model is not authored, just
derived. The canonical example today is the [[id:496DC90B-8951-42A6-A63E-B1762DF496C6][PlantUML ER diagram]]: the
parse stage reads the already-generated SQL DDL under
=projects/ores.sql/create/= and produces an intermediate JSON model;
the render stage feeds it to a Mustache template and emits
=ores_schema.puml=. The intermediate model is regenerated on every
run, carries no information a human is expected to read or edit, and
lives under =build/output/= so it stays out of the input tree. JSON
is the right shape here precisely *because* it's machine output:
literate prose, drawers, and block-paste mechanisms have no readers.

The distinction matters for the input tree. =projects/ores.codegen/models/=
holds authored input only. Intermediate / derived artefacts go under
=build/output/=.

* The six MASD principles

The principles below are defined in full in
[[https://mcraveiro.github.io/dogen/docs/the_masd_methodology.html][The MASD Methodology]]. Each is applied here to ORE Studio specifically.

1. *Focus Narrowly (P-1)* — target infrastructure code only, not business
   logic. The generator produces mechanical layers (C++ plumbing, SQL DDL, Qt
   boilerplate) but never touches domain logic inside a service class.

2. *Integrate Pervasively (P-2)* — the generator fits the existing toolchain
   without disruption. =ores.codegen= is invoked via =compass=, produces files
   that slot into CMake targets and the existing directory layout, and carries
   the standard licence header and editor modeline on every output file.

3. *Evolve Gradually (P-3)* — coverage grows incrementally, driven by concrete
   practice. Each new profile is added when a second real entity needs it. The
   commissioning process surfaces new SRPPs; those SRPPs become templates;
   templates become profiles.

4. *Govern Openly (P-4)* — all templates, models, and profiles are in the open
   repository and subject to the same review process as production code.

5. *Standardise Judiciously (P-5)* — de facto standards (Mustache, JSON,
   Python) at the core; physical artefacts conform to project conventions (C++
   style, SQL naming, org frontmatter contract).

6. *Assist and Guide (P-6)* — the automation spectrum runs from Level 0 (stub
   generation) to Level 4 (full product-line generation). ORE Studio currently
   operates between Level 2 and Level 3.

* The reference-implementation backout strategy

Rather than designing an abstract generator first, ORE Studio extracts the
generator from a proven hand-written implementation. This is analogous to
the bootstrapping / dogfooding approach described in
[[https://mcraveiro.github.io/dogen/docs/masd_reference_implementation.html][MASD Reference Implementation]] for Dogen itself, grounded in a working
system rather than an idealised top-down design:

1. *Commission the reference entity by hand.* The =currency= entity is the
   reference implementation — every layer written by hand without generator
   involvement.

2. *Identify the SRPPs.* The [[id:897A7156-504D-4581-AD4C-026D643C0838][entity evaluation checklist]] formalises what
   "fully commissioned" means at each layer; patterns that appear identically
   across all checklist items are candidates for extraction.

3. *Encode patterns as Mustache templates.* Each SRPP becomes a =.mustache=
   file parameterised by the entity model (entity name, field list, namespace,
   component path).

4. *Add a profile entry.* The template set is registered in
   =projects/ores.codegen/library/profiles.json=.

5. *Verify zero drift.* The generator is run against the =currency= model and
   its output is diffed against the hand-written files. A zero diff confirms
   the template faithfully encodes the SRPP.

6. *Extend to new entities.* Commissioning a new entity reduces to authoring
   its model file and running the generator.

This strategy avoids the central failure mode of top-down code generation: a
generator that produces code nobody has actually used.

* Automation spectrum

| Level | Description                                     | ORE Studio status                      |
|-------+-------------------------------------------------+----------------------------------------|
| 0     | Stub generation only; human writes all content  | Exceeded                               |
| 1     | One layer generated (e.g. SQL only)             | Exceeded                               |
| 2     | Multiple layers generated; some human editing   | Achieved for SQL, domain               |
| 3     | Full entity generated with no human editing     | Partial (SQL + domain + protocol)      |
| 4     | Product-line generation: new entity = new model | Not yet reached                        |

The immediate goal is to reach Level 3 for the complete entity stack (all
seven profiles) using =currency= as the fitness function.

* Facet catalogue

The facets of a domain entity are generated by [[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]] profiles.
Each profile corresponds to one or more facets, each documented in its own
facet reference:

| Facet | Profile | Reference |
|---|---|---|
| Type definition (struct, JSON I/O, table I/O) | =--profile domain= | [[id:429AE3F5-A825-8834-0353-9B98F235AE84][Type definition facet]] |
| Test data generator | =--profile generator= | — |
| Repository entity and mapper | =--profile repository= | [[id:608675E1-4284-4902-99AC-4DA565390AFB][Entity lifecycle]] |
| Repository CRUD operations | =--profile repository= | [[id:608675E1-4284-4902-99AC-4DA565390AFB][Entity lifecycle]] |
| Service layer | =--profile service= | [[id:608675E1-4284-4902-99AC-4DA565390AFB][Entity lifecycle]] |
| NATS messaging protocol | =--profile protocol= | [[id:A7B3C9D2-E5F1-4A8B-9C6D-3E2F1A0B8C7D][ORE Studio Messaging Reference]] |
| SQL schema, triggers, drop scripts | =--profile sql= | [[id:5CB31A0E-4E38-4521-B0EE-E4396C3E2936][SQL facet]] |
| Qt UI (12 files: window, dialog, model) | =--profile qt= | [[id:2FDDC2F6-C510-4A86-8C50-D79C83EDDFFF][Qt facet]] |
| Shell commands | manual | [[id:E38360B6-99FF-4AF0-937F-58F31BD573F5][Shell facet]] |
| CLI commands | manual | [[id:05891220-2503-48A0-ADF5-13EC95B605D7][CLI facet]] |
| HTTP endpoints | manual | [[id:579910D3-89FF-4820-9A9B-57508FF39096][HTTP facet]] |
| Wt UI | manual | [[id:ACF334D8-703B-4020-A48E-F8A8E5090E72][Wt facet]] |

MASD using NLP and LLMs does not require the rigid formalism of meta-models
and meta-meta-models. Instead, the underlying concepts are ported into [[id:F663DF66-4F62-0884-DB23-BB2D6A42B644][skills]]
which the LLM can understand, and these are used to generate the target
physical artefacts.

The key attributes of the logical element we intend to model are kept within
the /type/ facet — regular C++ code that defines a =struct= with the domain
type. Each logical element lives on its "native" component (e.g. =account= in
=ores.iam=). They are defined via the [[id:B1450696-8C51-4CC5-8910-84912A411AB6][Domain Type Creator]] skill, which bundles
a number of facets together. Other facets then refer to the domain type files
to extract required information.
