This page documents a test scenario verifying [[id:615FD100-BCD9-4CF5-B9DF-197447769AA9][Tick-batch publishing and persistence for curve instruments]] in [[id:3ECC4BA8-6ACA-4028-9E42-AE7F25FC3B98][IR Rates synthetic data generation]]. It is filled in with the target dialog and checklist of steps before testing starts; the QA Validation Runner panel rewrites =* Results= in place on save.

This scenario supersedes the original =payment_frequency= UI check: the
data model was reworked mid-task to consolidate the duplicate
=ores.trading.payment_frequency_type= entity into a single
=ores.refdata.payment_frequency=, move the badge from the self-referencing
=code= column onto the FK =period_unit= column (reusing =tenor_unit='s
existing badge domain), and migrate all four =ores.trading= instrument
tables that store a payment-frequency code to validate against it.

* Notes

The FAILED run above surfaced three real, pre-existing bugs unrelated
to the payment_frequency data model itself (history provider not
registered, ORE importer storing raw tenor strings instead of
frequency names, missing =trade_type_code= on swap-family instruments,
and =stamp()= silently not overwriting nested =.audit.modified_by=
fields). All four are fixed — see the task's =* Plan / Migration
completed / Bugs found and fixed= subsection for detail — and
confirmed via =ores.trading.service= logs: "Invalid modified_by:
ores", "Invalid payment_frequency: 3M...", and "Invalid trade_type:
..." no longer appear after rebuild. A handful of unrelated
check-constraint and empty-date errors remain on other instrument
types (composite, scripted, commodity, swaption, credit tenor, bond,
equity_swap, vanilla_swap, cap_floor) — explicitly out of scope for
this task, left for a future fix. This scenario should be re-run
end-to-end (PENDING steps still open) to confirm PASS across the
board.
