* Goal

PR #1014 made =component_overview.org= the single source of truth
for component identity (humans + codegen). The new-component
flow — doc scaffold template, =component-creator= skill,
=component-model-creator= skill, runbook, recipe — still
describes the pre-merge JSON-driven flow and will silently
produce overviews that codegen can't read (no =#+name:=,
=#+full_name:=, =#+brief:=).

This task reconciles the entire new-component creation flow with
the merged-overview model, and proves it end-to-end with two
sample components (flat + composite).

Design decisions agreed before implementation:

- *Groups are codegen-readable too.* The 5 group-level overviews
  landed by PR #1014 (=ores.compute=, =ores.controller=,
  =ores.reporting=, =ores.trading=, =ores.workflow=) keep their
  codegen scalars; the loader can read them. Whether any profile
  *generates* code from a group is determined by the profile (no
  group-targeting profile exists today; that's future work).
- *No explicit subcomponent declaration.* The directory layout
  =projects/ores.iam/{api,core,service}/= is the composite
  declaration. Each subcomponent's =component_overview.org=
  is fully self-describing. The group overview is for
  navigation + group-level codegen scalars.

* Acceptance

- =doc_component.org.mustache= (the template used by
  =compass add component=) includes the three new codegen
  keywords (=#+name:=, =#+full_name:=, =#+brief:=) — overviews
  scaffolded after this PR are codegen-readable out of the box.
- The =component-creator= skill describes the
  overview-driven flow: scaffold the overview, then run codegen
  against it (no JSON intermediate).
- The =component-model-creator= skill aligns: scaffold via
  compass, fill in the sections, run codegen.
- The runbook =Create a new component= reflects the new ordering.
- The recipe =How do I create a component overview?= shows the
  new keywords.
- Two sample components scaffolded as end-to-end proof:
  - =ores.sample_flat= — a flat component with no
    subcomponents.
  - =ores.sample_composite/{api,core,service}= — a composite
    with its parent overview + three sub-overviews.
- Verified by running codegen against each sample's overview
  and inspecting the generated scaffold.
- Samples are removed before merge (they're throwaway test
  fixtures; the verification output is captured in the PR
  description).
