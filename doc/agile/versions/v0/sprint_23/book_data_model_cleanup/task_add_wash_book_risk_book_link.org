* Result

*Field shape*: nullable self-referencing =book.paired_risk_book_id=
FK to =books.id= — directly follows the existing
=business_unit.parent_business_unit_id=/=portfolio.parent_portfolio_id=
precedent, so the field addition itself is small and well-trodden.

*Two things push this past a trivial field-add*, both sized as small
but real, and both are why this needed its own sizing pass rather than
folding straight into a sprint:

1. *Conditional validation* — must be set iff
   =book_purpose_type = Wash=, plus a self-loop guard. The
   trigger/=validate_*_fn= plumbing already exists for single-field
   validation (e.g. =regulatory_book_type=); extending it to a
   cross-field condition is straightforward but genuinely new — no
   existing validation function in the codebase checks one field
   against another's value yet.
2. *Qt UI* — the field should only be enabled when
   =book_purpose_type= is Wash. No existing generated detail dialog
   conditionally enables a sibling form field based on another field's
   live value (existing =setEnabled= calls only gate toolbar
   actions/save buttons); needs either a codegen extension or a
   hand-written override.

Recorded this sizing in =wash_books_and_risk_routing.org=. No code
changes; implementation is follow-up work, tracked as a separate
capture.
