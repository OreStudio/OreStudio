what /it is/ and what it's /for/ live in [[id:2F71292F-CDB0-4E2E-B50F-4F02E10597C4][Product identity]]. This
page is the index from which each component's
=component_overview.org= hangs, organised into the four
architectural layers below.

The system is organised into four architectural layers:

** Foundation Layer

- [[id:00E33570-F7BE-7F94-FD83-2FDC9CB04463][ores.utility]]: UUID v7 generation, Base64/Base32 encoding, string conversion,
  datetime helpers, faker utilities, =rfl= type reflectors. No ORE Studio
  dependencies; base of the dependency tree.
- [[id:7F3A2E81-B5C9-4D6A-8E12-9C4B3F5A7D0E][ores.platform]]: Filesystem utilities, environment variables, network info
  (hostname, MAC, machine ID), cross-platform time utilities. Depends on:
  utility.
- [[id:1FBCC86B-8E04-4F81-8558-85C0B6C54836][ores.logging]]: Core Boost.Log infrastructure — =make_logger()= factory,
  severity levels, lifecycle manager for console/file sinks. Extracted from
  =ores.telemetry= to break circular dependency with =ores.database=. Depends
  on: platform.
- [[id:8A2B3C4D-5E6F-7A8B-9C0D-1E2F3A4B5C6D][ores.telemetry]]: Extends logging with distributed tracing (trace_id, span_id),
  OpenTelemetry integration, JSON Lines log export, and service identity.
  Depends on: logging, platform.
- [[id:3A8F2E91-D4C7-8AB4-5E23-91D6F8C04A72][ores.database]]: PostgreSQL access layer — connection pool, tenant/party
  context, bitemporal CRUD, repository helpers, health monitor, and
  LISTEN/NOTIFY integration. Depends on: telemetry.
- [[id:1EE1B07C-3EC4-9BC4-1C4B-5BD458A8B2E5][ores.variability]]: Feature flags and runtime configuration management via
  the =variability.v1.>= NATS namespace.
- [[id:8A1C5E2D-9B4F-4E8A-A7C3-6F2D1E0B9A5C][ores.geo]]: IP-to-location geolocation using MaxMind GeoLite2-City stored in
  PostgreSQL. Provides country, city, and coordinate lookups for IAM session
  tracking.
- [[id:B7E9F3A1-C8D2-4E6B-A1F5-9D3C7E8B2A4F][ores.sql]]: PostgreSQL schema, migration scripts, role provisioning, and
  database lifecycle management. Enforces strict service-table isolation.
- [[id:D7BCB9B1-43C9-4DEF-8B1D-BABD23494DC2][ores.security]]: Shared cryptographic primitives — scrypt password hashing,
  AES-256-GCM encryption, JWT parsing/validation, OWASP password and email
  validators. Used by =ores.iam= and =ores.connections=.
- [[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]]: Code generation utility for domain types and documentation.
- Dependency order: utility → platform → logging → telemetry → database.

** Infrastructure Layer

- [[id:C9C24C99-F16C-45BE-A262-1C0F4502765E][ores.nats]]: NATS messaging transport — request/reply, queue-group subscribe,
  and JetStream for durable events. All payloads are JSON via =rfl=. Subject
  convention: ={domain}.v1.{entity}.{operation}=. Replaced the former
  =ores.comms= binary protocol stack.
- [[id:9CFAC175-C28F-4C03-9655-1CDC6E8C3EE0][ores.service]]: Shared NATS domain service runner — =domain_service_runner=
  lifecycle, heartbeat publisher, per-request context, and handler helpers.
  All =*.service= entrypoints build on this library.
- [[id:7F58FF13-5A85-49C4-09BB-EED86B874ABB][ores.testing]]: Catch2 infrastructure — unique database per test process,
  logging and database lifecycle listeners, tenant RLS helpers, coroutine
  test support.
- [[id:DDBC94C4-F350-478B-9E4C-643DEEA803B0][ores.eventing]]: In-process publish/subscribe event bus with PostgreSQL
  LISTEN/NOTIFY bridge for cross-service domain events.
- [[id:DF32FBB0-84E3-4679-A4ED-1E2A9FD9CADF][ores.assets]]: Binary image storage with hierarchical tagging and
  currency-to-image linking. Served via the =assets.v1.>= NATS namespace.
- [[id:B0C9FDC8-F0B2-48FE-AE37-8FD79E6FD164][ores.http.api]]: HTTP server infrastructure — Boost.Beast/Asio async server,
  JWT authentication middleware, path-based routing, and OpenAPI registry.
  See also [[id:869526B6-6C82-430E-95E1-776D4A11A56B][ores.http.core]] (route implementations) and [[id:A8B318BC-8653-4D33-A58C-60793E3596D5][ores.http.server]]
  (entrypoint).
- [[id:E75DE282-4FDE-4B18-B23B-692A2F783B61][ores.connections]]: Client-side server connection bookmarks with hierarchical
  folders, tags, AES-256-GCM encrypted credentials (PBKDF2), and SQLite
  persistence. Used by =ores.qt= and =ores.shell=.
- [[id:ED632650-BFB1-4904-BE34-CFA4B6170B8B][ores.storage]]: File storage abstractions — local filesystem paths, HTTP-based
  file transfer, and archive pack/unpack for binary asset bundles.
- Depends on: Foundation (telemetry, database, nats).

** Domain Layer

Each domain is structured as =.api= (shared types/NATS protocol), =.core=
(business logic and persistence), and =.service= (thin NATS entrypoint).

- [[id:204DB292-C32B-03E4-9DDB-BFE634F7CE91][ores.refdata]]: Reference data — currencies, countries, coding schemes, FPML
  product types. XML/CSV/JSON I/O, temporal versioning. =refdata.v1.>=.
- [[id:D00BEA0D-E501-C534-A013-E6F40C1A6097][ores.iam]]: Identity and access management — account lifecycle, login with
  scrypt verification, RBAC, session tracking with geolocation. =iam.v1.>=.
- [[id:D3A9F7C2-8B14-4E5D-A6C9-1F7E2B0D8C4A][ores.dq]]: Data quality — provenance, validation metadata, change management,
  staging artefacts, and the publish-from-DQ pattern. =dq.v1.>=.
- [[id:7C2A8B4E-1F5D-4C9E-B8A2-7F1C3D5E6A9B][ores.fpml]]: FpML processing infrastructure for parsing, validating, and
  archiving derivatives trade representations. In early development.
- [[id:9A71F1F5-C3ED-4C07-9D7D-C5B42D4A1332][ores.ore]]: ORE XML import/export — converts between ORE Engine XML format and
  internal domain types (currencies, curve configs). Uses =rfl= for XML I/O.
- [[id:E9A3F7B2-6C14-4D8E-A5B9-3F2D1C0E7A6B][ores.trading]]: Trade booking and lifecycle — FSM-enforced trade status (new,
  live, expired, cancelled), ~30 activity types across five categories for P&L
  attribution, cancel-rebook via =successor_trade_id=, full temporal versioning.
  =trading.v1.>=. Depends on: refdata, fpml, dq.
- [[id:759530D8-2314-44F3-A50E-71CE7AD02558][ores.marketdata]]: Market data — series, observations, fixings, import options.
  =marketdata.v1.>=.
- [[id:DC252F72-1BB0-4CC8-B558-C191FFA5E826][ores.synthetic]]: Synthetic market data and scenario generation for stress
  testing. =synthetic.v1.>=.
- [[id:2F7E5268-1ECF-4F5B-B90F-EC916559DE54][ores.scheduler]]: Task scheduling for compute jobs (run ORE scenarios,
  generate reports). =scheduler.v1.>=.
- [[id:D65B6932-7A33-471C-98C6-6AC345D4684C][ores.reporting]]: Report execution and result persistence — runs ORE output
  through defined report templates. =reporting.v1.>=.
- [[id:BC804C31-D6F7-4E4E-9C4F-F4578BEAF4F5][ores.analytics]]: Risk analytics results and aggregation — NPV, sensitivities,
  VaR/CVaR. =analytics.v1.>=.
- [[id:440294D7-385D-41EE-92CB-CAB937E65E81][ores.workflow]]: Workflow management for multi-step compute pipelines. Tracks
  job state across scheduler, compute, and reporting stages. =workflow.v1.>=.
- Depends on: Foundation, Infrastructure.

** Application Layer

Client applications and compute processes that consume domain services.

- [[id:D0876A24-69BA-F484-ED9B-71CD3AA8710C][ores.shell]]: Interactive REPL connecting to ORE Studio via NATS. Currency
  queries, account management, auto-login from config. Developer/ops tool.
- [[id:F0DAACD0-8EAB-46C4-6B53-640921E3F225][ores.cli]]: Command-line import/export of reference data (JSON/XML/CSV)
  against the database directly. Used for admin and data-migration tasks.
- [[id:25812E6F-7AD1-36E4-697B-EEC8015C4F1F][ores.qt]]: Qt 6 desktop application with MDI interface. Manages currencies,
  accounts, parties, trades, and other entities via NATS services. Composed
  of the core shell and feature modules (=ores.qt.*=).
- [[id:869526B6-6C82-430E-95E1-776D4A11A56B][ores.http.core]] + [[id:A8B318BC-8653-4D33-A58C-60793E3596D5][ores.http.server]]: RESTful HTTP API server. =http.core=
  implements all route handlers (IAM, assets, risk, storage, variability);
  =http.server= wires them and runs the Boost.Asio event loop.
- [[id:FE36240A-7A64-4332-9E82-8D976D458C73][ores.wt.service]]: Wt C++ Web Toolkit frontend — account management, currency
  and country views, feature flags. Alternative to the Qt desktop client.
- [[id:152C4A5E-9947-4EE1-A5D4-893305D235B5][ores.compute.service]]: ORE Engine compute wrapper — receives compute job
  requests via NATS, spawns ORE child processes, and publishes results.
- [[id:C871C5DF-A521-4815-9572-72D07DA62CC7][ores.controller.core]]: Orchestration controller — assembles compute
  pipelines, sequences scheduler/compute/reporting stages, and tracks run
  state.
- Depends on: All layers.

* Database Isolation

Each domain service runs as a dedicated PostgreSQL role
(~ores_<env>_<service>_service~) and holds DML only on its own component
tables (~ores_<service>_*~). This is the strict service table isolation
invariant.

** Service table isolation

Each domain service user holds DML only on its own ~ores_<service>_*~ tables.
Cross-service reads inside trigger functions use ~SECURITY DEFINER~; cross-service
writes go through NATS. All tracked cross-component grants have been removed.

The synthetic service holds broad SELECT on all domains and is classified
as ~role: "tooling"~ — a permanent exception, not a service.

Full rules, patterns, and examples are in the
[[id:B7E9F3A1-C8D2-4E6B-A1F5-9D3C7E8B2A4F][ORE Studio SQL Schema]] document
under "Service Table Isolation".

** DQ publish-from-DQ pattern

Target services (refdata, assets, reporting, trading) expose
~<service>.v1.<entity>.publish-from-dq~ NATS subjects. Each handler reads DQ
artefact tables via a ~SECURITY DEFINER~ SQL function (owned by the DQ schema)
and writes only to the target service's own tables. This satisfies the isolation
invariant while allowing DQ-sourced seed data to be materialised into service
tables.

Publish exception: the ~SECURITY DEFINER~ function runs as the DQ schema owner,
so it may read ~ores_dq_*~ tables. The calling service user never receives a
SELECT grant on DQ tables. This pattern is the only permitted cross-service read
path from service SQL functions; ordinary DML handlers must not query
~ores_<other_service>_*~ tables directly.
