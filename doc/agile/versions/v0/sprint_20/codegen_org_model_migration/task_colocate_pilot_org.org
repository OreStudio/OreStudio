* Plan

** 1. Decide the discovery mechanism

Two parts:

- *Glob*: =projects/ores.<component>/modeling/*.org= picks up all org files
  in the modeling directory.
- *Frontmatter filter*: of those, keep only =#+type: ores.codegen.entity=.
  The dotted namespace prefix avoids collision with any future generic
  =entity= type. Anything else (knowledge docs, component overviews,
  plantuml org files) is ignored by codegen but still navigable in
  Emacs / via compass.

This makes the modeling directory a literate-friendly space where the
codegen consumes only the files explicitly marked as models.

** 2. Update =manifest.py=

The =Component= dataclass grows a =modeling_dir= field pointing at the
component's =modeling/= directory. =entity_glob= and =exclude_suffix= still
apply to the legacy =models_dir=; the new =modeling_dir= is globbed as
=*.org= and filtered by =#+type:= via a small helper.

** 3. Update =generate.py= driver

For each component, union the (legacy JSON) matches and the (new org)
matches. Pass both into the existing model-by-model loop.

** 4. Move the pilot file

#+begin_src sh
mkdir -p projects/ores.refdata/modeling
git mv projects/ores.codegen/models/refdata/party_entity.org \
       projects/ores.refdata/modeling/ores.refdata.party.org
#+end_src

** 5. Regenerate and verify

=codegen.sh regenerate --component refdata-cpp --profile all-cpp= produces
the same =party.hpp= (and other party files) as before the move.

** 6. Document

Add a recipe under =doc/recipes/codegen/= for "How do I add a new codegen
entity model?". Points at =compass add entity-org= and the meta-model.

** 7. Commit + push + raise PR

* Notes

** Frontmatter as a model discriminator

This is the same idiom that makes ORE Studio's existing literate docs
work: each org file carries =#+type:= and tooling dispatches on it.
Codegen joins the club. No bespoke discovery mechanism beyond reading the
frontmatter — the same lightweight regex the org loader uses anyway.

** Backward compatibility

While other components still have their models under
=projects/ores.codegen/models/<short>/=, codegen unions both locations.
Once all components migrate, the legacy directory is deleted in the
story-final cleanup task.
