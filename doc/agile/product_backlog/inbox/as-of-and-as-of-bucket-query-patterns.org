#+title: Generalize as-of and as-of-bucket queries across repositories
#+description: As market_observations_repository::read_as_of()/read_as_of_buckets() now show for IR curve snapshots: latest-value-at-time-T (as-of) and N-snapshots-over-time (as-of buckets) are general patterns any bitemporal/timeseries repository needs, not just curve data. Capture the pattern and scope generalizing it.

#+level: cross
#+filetags: :architecture:database:timeseries:

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

Two distinct, general-purpose query shapes were built bespoke for
=market_observations_repository= (curve-snapshot-builder-viewer task)
and should be recognised and generalised as reusable patterns rather
than staying one-off:

1. *As-of*: for a set of independently-updating rows sharing a key
   family (e.g. one row per =point_id= within a =series_id=), return
   the latest row per key at or before a given time —
   =SELECT DISTINCT ON (key) ... WHERE ... <= as_of ORDER BY key, time
   DESC=. Needs a supporting index ordered *key-first*, not
   time-first (the existing =observations_series_datetime_idx= is
   time-first and doesn't help =DISTINCT ON (point_id)=; a new
   =observations_series_point_datetime_idx= (key-first) was added
   specifically for this).
2. *As-of buckets*: N as-of snapshots at regular time intervals ending
   at a given boundary — "the curve every 30 minutes for the last 5
   buckets" — computed as *one* SQL statement (=generate_series()= for
   the boundaries, a =LATERAL= subquery doing the per-bucket as-of
   reduction), not N round trips of the as-of query. This is
   deliberately *not* implemented as a C++ loop over the as-of
   primitive — the bucket generation and per-bucket reduction both
   belong in the database for one query plan and one round trip.

This is distinct from the *bitemporal* "as-of" already documented in
[[id:4AAD3581-CFA7-4B44-A2DD-0236784E939F][Time and Timestamps]] and
[[id:BA159E88-74EB-4401-91CC-1DE7DF5AE50A][temporal_composite_entity_versioning]] (=valid_from <= t <
valid_to= — "what did this row look like at transaction-time t",
i.e. correction history for one entity instance). The pattern here is
about *multiple independently-ticking rows sharing a key family*
(financial valid-time, =observation_datetime=), reconstructing a
*snapshot across a set*, not a single row's correction history.
=compute_telemetry_repository::latest_node_samples()= already used
the identical =DISTINCT ON (key) ... ORDER BY key, time DESC= shape
for per-host latest telemetry before this task existed — so the shape
recurs, but nobody has named or captured it as reusable.

* Why

Any timeseries-shaped repository (market data, telemetry, sensor/tick
data generally) is likely to need both queries eventually: "what does
this look like right now, reconstructed from independently-updating
parts" and "how has it evolved recently, in regular buckets". Building
this bespoke each time means re-deriving the =DISTINCT ON= + index
shape, and re-deciding "generate the buckets in the DB, not a loop"
each time — a decision that's easy to get wrong in the direction of
"just loop the single-row query N times" (which was the first draft
here, before being corrected to a single generate_series/LATERAL
query).

Candidate generalisation, not yet decided in detail:

- A shared =ores.database= helper (e.g. under
  =repository/timeseries_as_of.hpp=) that takes a table name, key
  column, time column, extra WHERE filters, and produces the
  parameterised =DISTINCT ON= / =generate_series+LATERAL= SQL, so a
  new repository wires columns rather than re-deriving the SQL shape.
- Or, more conservatively: keep hand-written per-repository SQL (as
  =market_observations_repository= does today) but document the
  *shape* here as the reference implementation to copy from, and only
  build the shared helper once a second repository actually needs it
  (avoiding premature abstraction from a single caller).
- Either way: the "buckets are one DB query, not a C++ loop over N
  as-of calls" principle should be captured as a hard rule, not left
  to be rediscovered per repository.
