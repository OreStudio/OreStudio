* What

Investigate and fix a crash in =ores.controller.service=: shortly after
=process_supervisor= launches a supervised service (reproduced with
=ores.wt.service[0]=), =service_instance_repository::update_phase()=
checks out a fresh pooled connection, and =tenant_aware_pool='s
=SELECT set_config('app.current_service', '<value>', false)= hook fails
with =ERROR: invalid byte sequence for encoding "UTF8": 0x<byte>= — a
different byte each run (=0x99= then =0x8f= across two consecutive
=compass services start= attempts in environment =solid_dirac=, both on
a freshly recreated, empty database). The =<value>= being set is the
per-process constant =service_account_= string (e.g.
=ores_solid_dirac_controller_service=), which is identical and clean on
every one of the four earlier successful calls to the same statement
earlier in the same process's lifetime — only this specific call site
(the connection checked out for the first =update_phase= write after a
launch) corrupts it, and the byte value is non-deterministic across
runs. The controller process then goes silent and dies (no crash log,
no further output) — =compass services status= reports it and the
child service as dead moments later.

* Why

A non-deterministic single-byte corruption at a fixed call site, varying
between runs, is the signature of a memory-safety bug (uninitialised
read, use-after-free, or small-buffer overrun) rather than a data
problem — worth an ASAN/valgrind pass on =ores.controller.service=,
focused on =process_supervisor.cpp= (=update_phase= call path) and
=tenant_aware_pool.hpp='s connection-checkout hook
(=projects/ores.database/include/ores.database/domain/tenant_aware_pool.hpp:227-236=).
This blocks any workflow that needs the controller-supervised fleet up
(e.g. the =solid_dirac= environment's party_type Qt UI verification
task), so likely affects other developers/environments once hit.
