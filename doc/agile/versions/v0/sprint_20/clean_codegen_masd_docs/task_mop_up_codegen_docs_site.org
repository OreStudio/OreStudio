* Items

** 1 — Drop Architecture nav menu

Move the "System Model" page into the Project nav menu. Delete the
Architecture top-level menu entry. Tech Spaces remains under the
Modeling (formerly MASD) menu.

** 2 — Rename MASD menu → Modeling; MASD entry → Methodology

In the site nav config rename the top-level menu from "MASD" to
"Modeling". The first child entry, currently titled "MASD", is
renamed to "Methodology".

** 3 — Deduplicate parts/facets note

Search for "Parts and facets are not subdivisions of a technical
space." Keep the instance in the facets page; remove all others.

** 4 — Rename Developer Links → Developer Guide

Change =#+title:= in =doc/developer.org= to "Developer Guide". Update
the nav entry. Audit every internal cross-section reference in the
file and convert any that use bare anchors or raw text into proper
=[[id:...][...]]= or =[[*Section][...]]= org-mode links.

** 5 — Split inlined archetypes into separate org files

For every facet document that contains inlined archetype content:

1. Create a new =<archetype-slug>.org= file alongside the facet doc,
   using the standard knowledge document template.
2. Move the archetype content into that file, structured with MASD
   terminology headings (=* Preamble=, =* Helper includes=, etc.) and
   separate =#+begin_src= blocks for each logical section.
3. Replace the inlined content in the facet doc with a table:

   | Archetype | Description | Link |
   |...

** 6 — Template variable reference document

Create =doc/codegen/template_variables.org= (or equivalent). Content:
- Intro paragraph explaining where variables come from (Stencil
  context, model metadata, etc.)
- Tables grouped by subject area, each row: variable name,
  description, example value.
- Cover at minimum: =cpp_license=, =component=, and any others
  present in the current templates.

Add an =[[id:...][Template variable reference]]= link in:
- Every org-mode model file in =doc/codegen/=.
- Every archetype document.

** 7 — Enrich archetype descriptions

Each archetype document must open with a paragraph (not just a
one-liner) that explains:
- What the archetype emits (e.g. a C++ struct, a factory method).
- Which MASD element type or profile it applies to.
- Any key library dependency (e.g. reflect-cpp for serialisation,
  with a link to their site).
- Any notable conventions (naming, include path, generated file
  location).
