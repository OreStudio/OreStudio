* Goal

Add a new =archetypes= subcommand to =compass codegen entity= that makes the physical space
self-documenting. Users can discover valid =ores.*= addresses for property drawers and for
=--address= CLI arguments without reading =facet_catalogue.org= by hand.

** Command forms

#+begin_src sh
# List the full physical space catalogue in ores.* notation, grouped by TS and facet
compass codegen entity archetypes

# Narrow to one TS or facet (always the full ores.* form, matching --address
# everywhere else in codegen entity)
compass codegen entity archetypes --address ores.cpp
compass codegen entity archetypes --address ores.cpp.qt

# Show the supported set for a specific entity (S_e), with enabled/disabled status per archetype
compass codegen entity archetypes --entity currency
compass codegen entity archetypes --entity currency --address ores.cpp
#+end_src

** Output format: catalogue mode (no --entity)

#+begin_example

    ...
#+end_example

** Output format: entity mode (--entity <name>)

Shows the entity's supported set S_e computed from its =ores.*= bindings and model-types
filter. Each archetype shows whether it is in S_e (✅) or excluded (❌) and why.

#+begin_example
Supported set for: currency  (domain_entity, refdata)
  ores.* bindings: :ores.cpp.qt.enabled: false

sql  ✅ [technical space — in supported set]
  sql.schema  ✅
    sql.schema.entity_create          ✅  sql_schema_domain_entity_create.mustache
    sql.schema.entity_drop            ✅  sql_schema_domain_entity_drop.mustache
    ...

cpp  ✅ [technical space — in supported set]
  cpp.domain  ✅
    cpp.domain.class_header           ✅  cpp_domain_type_class.hpp.mustache
    ...
  cpp.qt  ❌ [facet — excluded by :ores.cpp.qt.enabled: false]
    cpp.qt.client_model_header        ❌
    ...
#+end_example

** Implementation notes

- Reads the facet catalogue including the new =* Technical spaces= section added by B5.
- Uses =compute_supported_set()= from =codegen/physical_space.py= (B5 dependency) when
  =--entity= is supplied; otherwise prints the raw catalogue.
- Address names are derived algorithmically from the catalogue: TS name from =* Technical
  spaces=, facet name from the facet heading, archetype name from the template filename
  (strip =.mustache= suffix and convert underscores/dots).
- The archetype naming convention (template → address) is: strip the TS prefix and
  extension from the template filename. Examples:
  - =sql_schema_domain_entity_create.mustache= → =sql.schema.entity_create=
  - =cpp_domain_type_class.hpp.mustache= → =cpp.domain.class_header=
  The mapping from template filename to archetype name must be explicit in the catalogue
  (a new =:archetype:= property on each template row) — not inferred by heuristic.

* Acceptance

- =compass codegen entity archetypes= prints all archetypes grouped by TS and facet in
  =ores.ts.facet.archetype= notation; each line also shows the source template name.
- =--address <addr>= filters the listing to that TS, facet, or archetype prefix.
- =--entity <name>= prints the supported set with per-archetype ✅/❌ status; ❌ entries
  show which =ores.*= property caused the exclusion.
- =--address= and =--entity= may be combined.
- All addresses printed are valid inputs to =--address= on =compass codegen entity generate=.
- No new modules beyond what B5 adds; all physical-space logic delegated to
  =codegen/physical_space.py=.

* Plan

B5 (=codegen/physical_space.py=) was already DONE from the closed
"Resolve codegen model unification blockers" story, so the "Waiting on
B5" note was stale — =load_graph=, =compute_target_set=,
=compute_supported_set=, and =is_enabled= already exist and needed no
changes. This task is presentation only: a new =_cmd_archetypes= in
=compass_codegen_entity.py= that walks the loaded graph and prints it,
delegating every resolution decision to =physical_space.py=.

1. Add =_cmd_archetypes= + an =archetypes= subparser in
   =compass_codegen_entity.py=, wired into the existing dispatch table.
2. Catalogue mode (no =--entity=): walk =graph.ts_facets= /
   =graph.facet_archetypes= directly, filtered by =compute_target_set=
   when =--address= is given.
3. Entity mode (=--entity=): resolve the entity via the existing
   =_resolve_entity=, get its model type and =:ores.*.enabled:=
   overrides (=_read_drawer_properties= + =_enabled_overrides=, both
   already in =codegen.generate=/=physical_space=), then annotate the
   same tree with ✅/❌ per TS/facet/archetype. Added a small
   =_enable_reason= helper (not in =physical_space.py=) that mirrors
   =is_enabled='s most-specific-wins walk but also returns *which* key
   decided it, so exclusion reasons can name the exact
   =:ores.*.enabled:= override or fall back to "facet/archetype
   default".
4. Add a recipe (=how_do_i_discover_the_physical_space_catalogue.org=)
   and wire it into =doc/recipes/codegen/codegen.org= and the sibling
   =how_do_i_list_codegen_entities.org='s =* See also=.

* Notes

- The task's own illustrative examples used unprefixed addresses
  (=--address cpp=, =--address cpp.qt=), but every other =--address=
  flag in this codebase (=generate=, =show=, =diff=) requires the full
  =ores.*= form — =compute_target_set=/=graph.facets_under= key off
  the node's =#+title:= verbatim, which always carries the prefix.
  Kept =archetypes= consistent with that rather than adding a
  special-cased unprefixed shorthand; updated the Command forms
  examples above to match.
- =--address= only resolves to a technical space or a facet (whatever
  =graph.facets_under= accepts) — there is no archetype-level
  =--address= filter, matching every other =codegen entity= subcommand.
- Verified end-to-end against a real disabled-facet entity
  (=market_observation=, =:ores.cpp.qt.enabled: false=) — every
  =ores.cpp.qt.*= archetype correctly shows ❌ with that exact key
  named as the reason, and the facet-level exclusion cascades to every
  archetype under it without a separate per-archetype override needed.
- Verified per-archetype model-type filtering independent of the
  facet's aggregate =#+model_types=: =ores.sql.schema= admits
  =domain_entity junction schema table= at the facet level, but a
  =domain_entity= entity correctly shows =sql.schema.junction_create=/
  =sql.schema.table_create= as ❌ (not admitted) while
  =sql.schema.domain_entity_create= shows ✅.
- Acceptance calls for =--address= to filter to an archetype prefix
  too, but =physical_space.compute_target_set=/=facets_under= only
  resolve TS- and facet-level addresses (the graph's own discovery
  roots). Added archetype-address resolution directly in
  =_cmd_archetypes= instead of extending =physical_space.py= — it is a
  presentation-only narrowing (which single archetype line to print
  within its facet), not a generation-set decision, so it stays out
  of the shared module B5 owns.

* Result

=--entity=) lists every archetype grouped by TS/facet with its source
template; =--entity <name>= annotates the same tree with ✅/❌ and

in =physical_space.py=, since B5's =compute_target_set= only handles
the graph's own discovery roots). All resolution — graph loading,
target/supported-set computation, most-specific-wins override

recipe (=how_do_i_discover_the_physical_space_catalogue.org=) wired
into the codegen recipe topic and its sibling entity-list recipe.

=:ores.*.enabled:= overrides, and confirmed printed addresses are
valid =--address= inputs to =generate=. All six acceptance criteria
met. Full local build clean; ctest 74/74; codegen pytest 55/55.
