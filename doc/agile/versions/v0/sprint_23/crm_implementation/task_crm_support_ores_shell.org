This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:B38B3869-02FD-4CC7-99BD-9A77904ACA19][Cross-rates matrix (CRM)]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

=ores.shell= has no command exposing the CRM cross-rates matrix
(=marketdata.v1.crm.rates=) at all today -- every other consumer
(Qt) had to be built first before this was viable. Add a shell
command (e.g. =crm rates <crm-name>= / =crm rates --all=) that
issues the same NATS request as the Qt matrix panel, prints the
resolved rates (base/quote/rate/status/as_of, inverted flag,
delta_pct), and -- once
[[id:4F2FC58A-D715-4159-A93F-E372AD720D0D][the convention-aware
formatter]] exists -- reuses it for display formatting instead of
ad hoc =printf=-style formatting, so =ores.shell= doesn't repeat the
inversion/delta/formatting mistakes the Qt client already had fixed.
This is the second real consumer of both
[[id:60B9E7A1-1CFC-4A33-A3AC-C36078D53FDB][the resolved-rates NATS
response]] and the formatter, concretely validating the "shared
across UIs" design those tasks were built for rather than leaving it
theoretical.

* Acceptance

- A shell command lists CRM rates for a party (all enabled CRMs, or
  one by name), via the existing =get_crm_rates_request=/=response=
  protocol -- no new NATS subject.
- Output uses the convention-aware formatter (once available) for
  display; falls back to a plain numeric dump only if this task lands
  before the formatter task, with a follow-up note to switch over.
- Manual QA scenario or shell integration test covering at least one
  named CRM and the "all enabled CRMs" case.

* Plan

(Implementation strategy. Written when work starts; key decisions
are distilled into the parent story's =* Decisions= at close, but the
plan itself stays — it is the historical record of what we did.)

- New =crm_commands= in =ores.shell=: =crm rates [<crm-name>] --party
  <id-or-full-name> [--inverted]=, sending the same
  =get_crm_rates_request= the Qt matrix uses -- no new NATS subject.
  =--party= accepts a UUID or exact full name (resolved the same way
  =provision party <party>= does), since ores.shell has no
  "current party" concept at the session level.
- Reuses =crm_rate_display_service= (constructed with lambdas wrapping
  the shell's own =do_auth_request= calls, not the raw NATS
  =client&= =crm_client= wants -- no need for ores.shell to plumb a
  service-account token provider just for this) for convention
  resolution + =crm_rate_formatter= formatting, so the shell's numbers
  match the Qt matrix exactly.
- Added =crm_name= to =crm_rate_display_service::row= (missing before
  this task): needed to label each row when =--all= mode mixes rates
  from several enabled CRMs.
- Output rendered via a new =crm_rate_table=/=crm_rate_table_io= pair
  in =ores.marketdata.client/presentation= (fort-based table,
  following the =convert_to_table=/=operator<<= convention every
  domain entity list already uses), so any future non-Qt consumer
  gets the same table for free.
- Added one shell recipe per enabled CRM plus one for the grid view,
  under =doc/recipes/shell/marketdata/crm/= ->
  =projects/ores.shell/scripts/library/marketdata/=:
  =crm_rates_majors=, =crm_rates_exotics=, =crm_rates_scandies=,
  =crm_rates_matrix=. Also split the pre-existing
  =doc/recipes/shell/marketdata/= imports into their own
  =import/= subfolder for readability, alongside the new =crm/=
  subfolder (source-tree organisation only -- the tangled library
  category, taken from the recipes' own filetag, is unchanged).
- =--matrix= (=crm_rate_table::convert_to_matrix_table=) renders the
  same rows as an NxN grid (base rows, quote columns, dash on the
  diagonal) instead of the flat list, mirroring the Qt matrix
  window's own layout. Requires a named CRM -- rejected with an error
  for the all-CRMs case, since mixing rows from more than one CRM
  into one grid is ambiguous.
- Manual QA (first =crm= scenario, live-environment run) surfaced a
  real precision bug fixed on this branch: a convention resolved via
  the *reverse* pair-code direction was reusing that convention's
  decimal_places/tick_size verbatim, collapsing every such cell (e.g.
  the whole JPY row) to a flat low-precision value. Fixed by deriving
  reciprocal-preserving decimal places
  (=crm_rate_formatter::inverted_decimal_places=) instead of reusing
  them, with a small floating-point epsilon guard on the underlying
  =log10=-based order-of-magnitude helper (caught in PR review) so a
  reciprocal landing exactly on a power-of-ten boundary doesn't
  under/over-shoot by one digit.
- A second manual QA pass (Qt-embedded Shell window/script library,
  see second scenario below) surfaced two more real defects, both
  fixed on this branch:
  1. Unicode arrow glyphs (▲/▼/—) in =crm_rate_formatter='s
     change-text broke libfort's byte-based column-width calculation
     in =ores.shell='s table output ("wonky" alignment) -- replaced
     with plain ASCII +/- signs (the sign alone already conveys
     direction; a separate glyph was redundant even before the
     alignment bug).
  2. =--party= being *required* on every =crm rates= call was masking
     a real gap: =ores.shell= had no "current party" concept because
     none of its three login paths
     (=accounts_commands::process_login=,
     =application.cpp::auto_login=, =ShellMdiWindow='s internal
     Qt-embedded re-login) captured =default_party_id= from the login
     response into =nats_client::login_info=, unlike Qt's
     =ClientManager=. Fixed by adding =default_party_id= to
     =login_info=, populating it at all three sites, and having
     =crm rates= default to it -- =--party= is now purely an
     override for an account managing more than one party.
- Considered (not done, scoped as a follow-up task
  [[id:A4F67A24-D6B1-42D8-A842-4A094E2C84DA][elsewhere]]): renaming
  "inverted" to "reciprocal" throughout, the correct FX-market term,
  and distinct from "derived"/triangulated rates.
- Considered and explicitly rejected: colouring the Change column
  (ANSI escape codes) -- both the Qt-embedded Shell window (a plain
  QTextEdit-style pane with no ANSI parsing) and non-TTY output paths
  (piped =compass shell -f=, captured recipe =#+RESULTS= blocks)
  would render raw escape codes instead of colour.

* Result

Shipped =crm rates [<crm-name>] [--party <id-or-full-name>]
[--inverted] [--matrix]= in =ores.shell=, sending the same
=get_crm_rates_request= (subject =marketdata.v1.crm.rates=) the Qt
Cross-Rates Matrix window sends and formatting through the same
=crm_rate_display_service=/=crm_rate_formatter=, so the shell's
numbers match Qt's exactly rather than re-deriving inversion/delta/
precision logic. Defaults to the logged-in account's own default
party (fixed a real gap along the way: none of ores.shell's three
login paths captured =default_party_id=, unlike Qt's
=ClientManager=); =--party= is only needed to override for an
account managing more than one party. =--matrix= renders an NxN grid
mirroring the Qt window's own layout, rejecting the all-CRMs case as
ambiguous. Output goes through a new =crm_rate_table=/
=crm_rate_table_io= pair in =ores.marketdata.client/presentation=
(fort-based, following the =convert_to_table=/=operator<<= convention
every domain entity list already uses).

Two rounds of manual QA against a live environment (see =* Test
Scenarios=) caught and fixed three real defects beyond the original
acceptance criteria: reciprocal-direction rate precision collapsing
to near-zero significant figures, Unicode glyphs breaking libfort's
column alignment in ores.shell's own table output, and the missing
default-party plumbing across all three of ores.shell's login paths.
All acceptance criteria met; 49 =ores.marketdata.client.tests=, 80
=ores.shell.tests=, and 6 =ores.nats.tests= assertions passing after
every fix.
