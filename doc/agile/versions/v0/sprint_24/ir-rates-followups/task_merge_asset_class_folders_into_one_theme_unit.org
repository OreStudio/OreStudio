* Goal

Make a synthetic data theme (e.g. "2016 ORE Samples") one logical
unit in the Market Simulator tree and in the backend folder model,
instead of one duplicate top-level node per asset class. Today
=ores_synthetic_publish_fx_spot_configs_from_dq_fn= and
=ores_synthetic_publish_ir_curve_configs_from_dq_fn=
(=projects/ores.sql/create/synthetic/synthetic_publish_from_dq_create.sql=)
each resolve/create the "collection" folder keyed by
=collection_id = v_config_id=, and =v_config_id= is scoped to that
call's own =dataset_id= -- so FX and IR curves for the same theme,
published from two separate DQ datasets, each get their own
collection folder that happens to share a display name. This already
forces the tester through this sprint's theme test scenario to
special-case "appears twice" as expected, and will only get worse as
more asset classes (rates, credit, equity, ...) are added to a
theme -- a party operator should be able to start/stop/browse a whole
theme as one unit, not click once per asset class.

Change both publish functions to resolve the collection folder by
=(tenant_id, party_id, name = v_config_name)= instead of by
=collection_id=, so every asset class published under the same theme
name reuses one shared collection folder with per-asset-class
children underneath (mirroring the existing FX/Rates ->
FX Rates/IR Curves sub-folder shape). Because
=promptThemeAndStart()= and folder-scoped start/stop
(=projects/ores.qt/synthetic/src/MarketSimulatorWindow.cpp=) already
operate on the folder hierarchy (via each generation config's own
=folder_id=), not on =collection_id=, merging the folder should not
require Qt-side changes -- confirm that during implementation.
Existing already-provisioned tenants (including this one) will keep
their duplicate folders until a one-off data migration merges them;
scope that migration as part of this task or split it out
explicitly, but don't leave it undocumented.

* Acceptance

- A theme published with both FX and IR curve datasets (e.g. "2016
  ORE Samples") shows as exactly one top-level Collection node in
  the Market Simulator tree, with FX and IR curve asset-class
  sub-folders underneath it -- not two sibling top-level nodes with
  the same name.
- Starting/stopping the merged theme node still starts/stops every
  asset class under it (regression check against
  =promptThemeAndStart()= and the Start-at-Root theme picker from
  the "Synthetic data themes" test scenario).
- A future third asset class published under the same theme name
  joins the same folder instead of creating a fourth top-level node.
- Existing duplicate folders in already-provisioned tenants are
  either migrated or the gap is explicitly called out as a known
  follow-up, not silently left broken.
