* Goal

Add =entity_type_of(const T&)=, a =constexpr= free function returning
the entity's dispatch-key string (e.g. ~"ores.refdata.currency"~),
for every =ores.refdata= domain entity — the single source of truth
the generic history-diff dispatch table (server side) and action
registry (client side) will key off, per
[[id:85892C17-BB81-4724-9B84-20E21D41C499][History Diff Architecture]].

* Acceptance

- =entity_type_of(const T&)= generated for every clean =ores.refdata=
  domain entity, returning ~"ores.{component}.{entity_singular}"~.
- Kept off the domain struct itself (a free function, not a member),
  per the earlier design discussion — the struct stays "the data,"
  not "the data plus its own metadata."
- =constexpr=, so it costs nothing at the call site.
- Representative test coverage (not exhaustive per-entity, since the
  codegen facet is mechanical): correctness, distinctness, stability
  across instances, and a domain_entity + a would-be junction case
  (junction not currently reachable — see Notes).
- Local build clean; full ctest suite green.

* Notes

*Scope, per direction mid-task*: generation was deliberately scoped to
=ores.refdata= only, not the whole repository — this story only
touches that component; other components pick up the trait when their
own history-dialog work reaches them.

*Pre-existing drift, deferred, not fixed here*: regenerating the
=domain= facet for =ores.refdata= surfaced real, unrelated drift
between several entities' org models and their generated code —
=business_unit= (lost =std::optional=-wrapped fields), =contact_type=
(missing default value), =currency_market_tier=, =monetary_nature=,
=party=, =party_id_scheme=, =rounding_type=. These entities did *not*
get =entity_type_of()= in this task — fixing their drift is out of
scope here and belongs to whoever owns those models. Notably *party*
is affected; the generic-history-dispatch task inherits this as a
known gap to close before party's own history dialog can migrate.

*Junction models are out of scope for entity_type_of() today*: the
=domain= facet's =:model_types:= is =schema domain_entity= only —
junction models (e.g. =party_counterparty=) are not generated by it at
all, so the =entity_type_of()= junction template branch this task
added is currently unreachable in practice. Pre-existing gap, not
introduced here; noted for whoever eventually brings junctions under
the =domain= facet.

*Codegen tooling gap*: =--address=/=--profile= only resolve at facet
granularity (confirmed against =src/codegen/physical_space.py=), not
per-archetype — there is no way to regenerate just one archetype
(e.g. only the class header) without also re-running every other
archetype in the same facet. This is why isolating this change
required diffing and selectively reverting files rather than a single
scoped regenerate command.
