* Goal

Add 'provision party <party> [--dataset-size small|large] [--reports all|none|<name,...>]' where <party> is a UUID or exact full name resolved via refdata.v1.parties.list: publish the base bundle with the gleif.lei_counterparties.<size> dataset and wait; publish the organisation bundle for the party and wait; create the selected report definitions (default all templates, system party as owner); save the party with status Active. Activation failure is a hard failure (divergence from the wizard, by design).

* Acceptance

- provision party by exact full name and by UUID both work; ambiguous or missing parties are a clear error.
- Defaults (small dataset, all reports) reproduce the wizard's output; --reports none skips report creation.
- All four phases report progress; any failure including activation is fatal and resumable via plumbing.

* Plan

1. =provision party <party>= in =provision_commands=: resolve the
   party by UUID or exact full name over =parties list='s protocol.
2. Phase 1: publish bundle =base= opted into
   =gleif.lei_counterparties.<size>= (default small) and wait;
   phase 2: publish =organisation= with the party id and wait;
   phase 3: create report definitions from the templates (=all=
   default as the wizard pre-checks, =none= or a comma list of
   names; owner = System party, fallback first; fatal on failure);
   phase 4: re-fetch the party and save it =Active=.
3. Activation failure is a hard failure — the recorded divergence
   from the wizard's warn-and-continue.

* Result

Implemented per plan. Notes:

- The counterparty bundle code is =base=, hardcoded as in the wizard
  (its =selectedBundleCode_= is a constant; there is no page to
  change it).
- =--reports <name,...>= errors on unknown template names rather than
  silently skipping; =--reports none= skips phase 3 entirely.
- Report definitions are owned by the System party with fallback to
  the first party, exactly the wizard's resolution.
- Verified by build (user-confirmed green) + unit tests via
  compass test results (ores.shell.tests 23/23) and menu/validation
  smoke in the binary. Live verification lands with the e2e script.
