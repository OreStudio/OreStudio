* Goal

The [[id:6D953558-A61C-461F-859D-CBD28AD5D8F5][CRM DQ publishing task]] created three new tables
(=ores_marketdata_crm_topology_configs_tbl=,
=ores_marketdata_crm_driver_pairs_tbl=,
=ores_marketdata_crm_enabled_derived_pairs_tbl=) with no entity layer
above raw SQL -- no domain type, repository, service, HTTP endpoint,
shell command, or Qt UI exists for any of them (confirmed: zero Qt
references to =crm_topology= in the codebase today). Use the
=codegen-add-entity= orchestrator to generate the full stack for all
three entities (domain JSON/table I/O, repository, service,
=ores.http.server= REST endpoints, =ores.shell= CLI commands, Qt list
and detail views), then wire the new list view(s) into the
reference-data menu, so the DQ-published "majors"/"exotics" CRM
bundles are inspectable and testable through the app UI ahead of --
and independent of -- the still-blocked cross-rates matrix UI panel.

* Acceptance

- Full-stack entities exist for =crm_topology_config=,
  =crm_driver_pair=, and =crm_enabled_derived_pair=: domain type with
  JSON/table I/O, repository, service, HTTP CRUD endpoints, shell
  list/history commands, Qt list window + detail/history dialogs --
  following the same codegen conventions as other refdata entities.
- The =crm_topology_config= list view (and its driver/derived pair
  detail) is reachable from the reference-data menu in the Qt client,
  not just creatable via raw SQL or DQ publish.
- After a synthetic party publish, the UI shows both "majors" and
  "exotics" configs for the party with correct driver/derived pair
  counts (8+3 and 3+0 respectively), matching what was already
  confirmed at the SQL layer.
- Full local build clean, =ctest= green.
