* When to use this skill

When implementation of a feature or task is otherwise complete and it's
time to verify it before raising a PR: a confirmed build, a fresh
Barclays-provisioned database, the full test suite, a written test
scenario, and a client handed to the user to execute it.

* How to use this skill

Run [[id:BE325E3E-1BAB-4D9D-964E-78333CDB97D7][How do I test a feature after implementation?]] end to end. The
first four steps are prerequisites, non-negotiable and in order —
never skip straight to writing the scenario:

1. *Ask the user whether a new build is necessary.* If yes, build the
   affected targets and run the full test suite (=rat=) — treat any
   failure as a blocker.
2. Validate SQL schemas, and codegen docs if the feature is a codegen
   entity.
3. *Check the database is recent* (=compass db status=) — recreate it
   only if drift/staleness says so, not unconditionally.
4. *Provision the Barclays Plc tenant yourself*, via =compass shell -f
   barclays_system_provision.ores= — never hand this to the user. Any
   failure here is usually a real bug (schema drift, a stale function);
   diagnose and fix it, redeploy, and re-run from a fresh =db recreate=
   until it completes cleanly before moving on.

Then write and run the scenario:

5. Create a =test_scenario= doc for the feature, with a clear
   description stating what it verifies. Set *Target dialog* to the
   dialog class(es) *and* the menu path to reach them — a bare class
   name leaves the tester hunting through menus. Every step title is
   five to seven words (fits the QA Validation Runner's list without
   wrapping); every step body is a bullet-point checklist giving the
   tester all the context to execute it, not prose. Give every client
   its own leading connect/login step sourced from the provisioning
   script's credentials — never hardcode or guess them, and never
   assume a second client instance inherits the first one's session.
   Cover *every* aspect of the feature that needs testing — for an
   entity/CRUD feature that means the full
   Create/Read/Update/Delete/History/Eventing set, not a subset; the
   eventing step needs a second client instance to actually exercise
   the NATS relay, not just the repository layer ctest already covers.
6. Start the Qt client with =--open-scenario= pointing at the new
   scenario (absolute path) and ask the user to execute it — do not
   run the UI steps yourself.
