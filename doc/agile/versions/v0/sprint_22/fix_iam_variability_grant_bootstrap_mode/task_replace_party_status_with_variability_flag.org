This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:7356C36E-2633-4043-9747-27E61593EA4F][Fix tenant-scoped variability system settings: permissions and per-tenant seeding]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Login's =party_setup_required= (=ores.iam.core/messaging/auth_handler.hpp:317=)
is derived from =party.status == "Inactive"= via IAM's in-memory
=party_cache_=, not a live DB read. If the cache is stale relative to
a party activation (or the party's status legitimately hasn't
flipped for some other reason), the party provisioner wizard
re-fires on every login even though setup already completed.
Independently of caching, using a domain-meaningful status field
(Active/Inactive/Suspended) as a *provisioning-complete* signal
conflates two different concerns.

Replace it with a dedicated =onboarding.party= variability flag,
scoped per-party via a new =party_id= column on
=ores_variability_system_settings_tbl=, checked with a direct DB read
at login (same pattern as the existing tenant =bootstrap_mode=
check) — immune to the party cache and decoupled from the party's
actual status.

This absorbs the pre-existing, never-started backlog story
[[id:C2BC9C1E-4AAA-42FC-A62F-CE4E5C4FA160][Symmetric onboarding flags for provisioner wizards]] (sprint 19
inbox), which already designed this exact fix plus the symmetric
=onboarding.system=/=onboarding.tenant= flags for the other two
wizards. That story is retired in favour of this task; its 4 tasks'
content is folded in below as the implementation plan.

* Acceptance

- =ores_variability_system_settings_tbl= has a non-nullable =party_id=
  column; system/tenant-scoped settings use the system party's UUID.
- =onboarding.system=, =onboarding.tenant=, =onboarding.party= exist
  in =system_setting_definitions= with clear descriptions.
- System, tenant, and party provisioner wizards each write their
  =onboarding.*= flag =true= on completion (server side, same pattern
  as the =bootstrap_mode= clear fixed in the sibling task).
- Login's =party_setup_required= is derived from a direct DB read of
  =onboarding.party=, not =party.status= and not the party cache.
- If the party wizard ran (=onboarding.party = true=) but the party
  is still =Inactive=, login returns =party_setup_required = false=
  plus a new =party_setup_warning= field so the client shows a
  message instead of re-launching the wizard.
- =system.bootstrap_mode= is kept alongside =onboarding.tenant= for
  this task (removal is an explicit follow-on, not in scope here).
- NOTIFY listener =max_backoff= reduced from 30s to 5s (unrelated
  mitigation carried over from the absorbed story; keep or split out
  as a separate quick task if it muddies review).

* Plan

Folded in from the 4 tasks of the absorbed story
[[id:C2BC9C1E-4AAA-42FC-A62F-CE4E5C4FA160][Symmetric onboarding flags for provisioner wizards]]:

** 1. Schema + C++ layer: party_id column on variability settings

Alter =ores_variability_system_settings_tbl= to add a non-nullable
=party_id= (uuid) column. Update =system_setting_entity=, its mapper,
repository read/write/remove queries, =system_settings_service=, and
provisioner seed calls to pass the system party's UUID for
system/tenant-scoped settings (avoids a nullable FK, keeps index
cardinality well-defined — same rationale the absorbed story used).

** 2. onboarding.* flag definitions

Add =onboarding.system=, =onboarding.tenant=, =onboarding.party= to
=system_setting_definitions= with clear descriptions. Add
=is_onboarding_complete=/=set_onboarding_complete= accessors on
=system_settings_service=. Seed =onboarding.tenant = false= in the
tenant provisioner (system-party scoped); seed =onboarding.party =
false= per party in the party provisioner flow.

** 3. Wire wizards + fix login's party_setup_required

- =TenantProvisioningWizard=: write =onboarding.tenant = true= in the
  =complete_provisioning= handler, server side — same NATS-routed
  pattern as the =bootstrap_mode= clear fixed in the sibling task
  (=clear_bootstrap_mode_request=), not a same-process cross-crate
  call.
- =PartyProvisioningWizard=: write =onboarding.party = true= in Phase
  4, server side, same pattern.
- =auth_handler.hpp=: replace the =party.status == "Inactive"= /
  =party_cache_= check with a direct DB read of =onboarding.party=
  (bypasses the cache entirely — same pattern as the existing
  =auth_is_tenant_bootstrap_mode= check).
- Add =party_setup_warning= to =login_response= for the "wizard ran
  but party still Inactive" case, so the client shows a message
  instead of re-launching the wizard.

** 4. NOTIFY listener mutex contention + max_backoff (carried over, may split out)

=listen_loop= currently holds the mutex for the entire
=consume_input()= + =get_notifications()= + callback chain. Under a
heavy bulk import (e.g. GLEIF party dataset — hundreds of NOTIFY-
firing INSERTs), this causes contention that can block the 100ms
poll cycle for up to ~30s, making the listener appear dead — which is
exactly the kind of staleness this task is designed to make
irrelevant for onboarding flags, but is a real underlying bug in its
own right. Fix: drain notifications into a local vector under the
lock, then process callbacks outside it:

#+begin_src cpp
std::vector<Notification> batch;
{
    std::lock_guard lock(mutex_);
    if (!(*connection_)->consume_input()) { ... }
    batch = (*connection_)->get_notifications();
}
for (const auto& n : batch)
    handle_notification(n);
#+end_src

Also reduce =max_backoff= from 30s to 5s (30s suited a remote DB; local
Postgres reconnects in under a second). Document the rationale in the
PostgreSQL architecture doc. This sub-item is only loosely coupled to
the rest of the task — split into its own task if it makes review
unwieldy.

(Key decisions are distilled into the parent story's =* Decisions= at
close, but this plan stays — it is the historical record of what we
did.)
