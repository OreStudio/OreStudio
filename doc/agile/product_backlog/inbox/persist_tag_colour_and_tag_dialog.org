#+title: Persist per-tag colour and add a tag create/edit dialog
#+description: Store a colour per connection tag in the DB and provide a dialog to create/edit tags with a colour picker, instead of hashing the tag name into a fixed palette at render time.

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

Give connection tags a persisted colour and a proper way to create/edit
them. Add a =colour= column to the =tags= table in the connections SQLite
DB (=~/.local/share/ores.qt/connections.db=), and add a "create/edit tag"
dialog in the Connection Browser that lets the user name a tag and pick its
colour. Both the Connection Browser tree (=ConnectionItemDelegate=) and the
login quick-connect label rendering should read the stored colour instead of
deriving one. Default a colour at creation time (e.g. next unused palette
entry) but let the user override it.

* Why

Today tag colours are derived at render time by hashing the tag name into a
fixed 8-colour palette:
=colorForTag(name)= → =tag_colors[qHash(name) % 8]= in
[[proj:projects/ores.qt/application/include/ores.qt/ConnectionTypes.hpp][=ConnectionTypes.hpp=]] (lines 41–58). With more than a handful of tags this
collides — distinct tags (e.g. the per-environment instance-id tags
=bright_faraday=, =brave_hopper=, …) map to the same colour — and because
=qHash(QString)= is seed-randomised per process, a tag's colour is not even
stable between runs. Colour is a user-meaningful attribute of a tag, so it
should be authored and persisted, not computed.
