This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:037B474D-84ED-4888-9054-D58AB1FB10D2][Consolidate history dialogs onto HistoryDialogBase]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Make the entity history "Changes" tab render diffs like GitHub: the old
value on a red background, the new value on a green background, with the
actually-changed characters/tokens highlighted in a darker shade
(GitHub dark-theme palette — line bg red =rgba(248,81,73,.15)= / green
=rgba(63,185,80,.15)=; intra-line highlight red =rgba(248,81,73,.40)= /
green =rgba(46,160,67,.40)=). Implement centrally in
=HistoryDialogBase::displayChangesTab= (=projects/ores.qt/api/src/HistoryDialogBase.cpp=,
~line 208–254) — it already produces a 3-column =[field | old | new]=
table from =calculateDiffAt()='s =DiffResult=, so this is a render-only
change in one place: no per-entity code, no templates, no regen, and
every entity's history view upgrades at once. Optionally add a =+=/=−=
gutter glyph for the full GitHub look.

*Must handle multiline.* Short single-line fields (codes, names, dates)
get an inline char highlight via common prefix/suffix. Long multi-line
fields (e.g. =change_commentary=) must render a real line-by-line diff
with intra-line (word/char) highlighting — i.e. a proper line diff
(LCS over lines) plus per-line token diff, not just prefix/suffix on the
whole string. Render via =QLabel= rich-text HTML (or =QTextEdit=) so
backgrounds and spans show; size the cells to the content.

* Plan

- =HistoryDialogBase::displayChangesTab= already produces a flat
  =[field | old | new]= =DiffResult= from =calculateDiffAt()=,
  identical in shape whether that override is the generic
  =HistoryDialog= (server-computed diff, spans discarded when
  flattened to =DiffResult=) or one of the ~30 remaining legacy
  per-entity dialogs (client-side =checkString=/=checkInt= helpers,
  never had spans at all). Rather than threading span data through
  every per-entity override, recompute it centrally: for each diff
  row, call =ores::diff::engine::compute()= (already
  exhaustively tested, entity-agnostic, cheap for one field) with a
  single-field =field_value= on each side to get that field's
  =diff_entry= — same algorithm the server already uses, just run
  again client-side for legacy dialogs.
- Render each side as a =QLabel= (rich text, word-wrapped) instead of
  a plain =QTableWidgetItem=: an outer =<div>= carries the line-level
  background (faint red/green tint), spans inside are wrapped in
  =<span>= with the darker highlight shade. Byte-offset spans are
  sliced directly on the =std::string= before converting to
  =QString= (never on the UTF-16 string), since spans are
  byte-offsets into UTF-8 and are guaranteed not to split a
  multi-byte code point — converting slice-by-slice sidesteps any
  UTF-8-byte-offset-to-UTF-16-code-unit-index math entirely.
  Newlines become =<br/>= so multiline diffs render as real line
  breaks, not one long wrapped line.
- =changeCellWidget()= keeps first refusal: an entity-specific custom
  widget (e.g. a flag icon) still wins; the new rendering is only the
  default fallback, previously a plain =QTableWidgetItem=.
- =ores.diff.lib= is already transitively available in =ores.qt.api=
  via =ores.history.api.lib='s =INTERFACE= link (confirmed during
  PR #1553 review, where an explicit redundant link was removed) — no
  CMake change needed.
- Manual verification surfaced a row-sizing bug: a fresh =QLabel= cell
  widget's =sizeHint()= isn't based on its actual wrapped-at-final-
  width content until Qt lays it out, so calling
  =resizeRowsToContents()= in the same call as =setCellWidget()=
  under-sizes multiline rows (visible as truncated commentary diffs).
  Fixed with the standard idiom for this: defer a second
  =resizeRowsToContents()= call via =QTimer::singleShot(0, ...)= so it
  runs after Qt has laid the widgets out and can query their real
  height.

** Why

The current changes tab shows old/new as plain text, which makes it hard
to see exactly what changed — especially for near-identical values
(=2025= → =2026=) or long commentary. A GitHub-style diff makes the delta
obvious at a glance and matches a UI users already know. Because the
history widget is fully abstracted into =HistoryDialogBase=, the
improvement is cheap and universal.
