* Goal

Add a test that walks every =.org= document in the repo, parses each
with =ores.orgmode=, and dumps the resulting structure to JSON. This is
a scale/robustness check against the real corpus (nothing should crash
or hang the parser on any doc actually in the repo, not just curated
fixtures) and, as a byproduct, gives a debuggable, diffable view of
what the parser extracts from any given doc — useful on its own while
developing or debugging the parser, independent of the test's
pass/fail result.

* Plan

Serialization used =rfl::json::write= (reflect-cpp), already this
project's standard for domain-type JSON I/O — confirmed it handles
=heading='s self-referential =children: vector<heading>= without any
special-casing needed; recursive aggregate types just work.

=collect_org_files()= walks the repo with
=recursive_directory_iterator=, pruning =build=, =external=, =.git=,
=vcpkg=, =vcpkg_installed= by disabling recursion into those
directories (not filtering paths after the fact, so pruned subtrees
are never even traversed). Guarded against a silently-empty corpus
(=REQUIRE(files.size() > 100)=) so a broken exclusion list or a moved
repo root fails loudly instead of reading as "covered everything".
Repo root and JSON output directory both come in as compile
definitions (=ORES_REPO_ROOT=, =ORES_TEST_OUTPUT_DIR=, the latter set
to =CMAKE_RUNTIME_OUTPUT_DIRECTORY= — same place test-result XMLs
already land) rather than environment variables, so the test runs the
same way under =ctest=, a direct binary invocation, or an IDE runner.

Result: found 3565 =.org= files; all parsed without a single exception
across the full corpus. JSON dumps land in
=<build>/publish/bin/orgmode_corpus_json/=, one file per doc, named
from its repo-relative path. Confirms both this component's own docs
and the wider corpus (agile docs, knowledge docs, codegen templates,
recipes, ...) actually exercise the parser correctly, not just the
hand-picked fixtures in =parser_tests.cpp=.
