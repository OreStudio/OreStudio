* Goal

After =compass db recreate= the database is in bootstrap-mode: the
schema exists but there are no tenants, accounts, or reference data.
Getting it to a usable state currently requires two manual steps:

1. Start services: =compass services start=
2. Run the provisioning script: =compass shell -f acme_system_provision.ores=

Add a =compass db provision= subcommand that wraps this flow so
developers can go from a fresh database to a seeded system with one
command. The provisioning recipe to invoke should be selectable (e.g.
=--recipe acme= or =--recipe barclays=) with =acme= as the default.

The standard provisioning recipes are:
- [[id:B0A8E259-5B88-40AE-BA27-BBC3679186E7][How do I provision the system with Acme (synthetic)?]] — the canonical
  synthetic recipe; deterministic (=--seed 42=), no external data required.
- [[id:87246342-8B5C-4345-9BAD-7A2F40985DF4][How do I provision the system with Barclays (GLEIF)?]] — the GLEIF
  variant; requires LEI data.
- [[id:BAADE064-43A3-4620-92CF-B8AF7F732C9B][How do I bootstrap a smoke-test system?]] — the lightweight multi-tenant
  smoke for quick iteration.

* Acceptance

- =compass db provision= (with =acme= as default) starts services if not
  running, invokes =compass shell -f acme_system_provision.ores=, and
  confirms success.
- =compass db provision --recipe barclays= invokes the Barclays GLEIF
  variant.
- =compass db provision --recipe smoke= invokes =bootstrap_smoke.ores=.
- Running =compass db provision= on an already-provisioned system
  (non-bootstrap state) exits cleanly with a warning, not an error.
- The DB lifecycle recipe (wherever documented) is updated to show the
  one-command post-recreate flow.
