* Goal

Settle whether =FxSpotGridWindow='s per-row feed connection status
(PENDING/LIVE/STALE/DISCONNECTED) should be a reused badge
(=BadgeCache=/=BadgeLabelUtils=, as originally captured) or a
purpose-built compact status dot/icon, before any DB/wiring work
starts. Options to weigh:

- *Badge (as captured)*: reuses existing =BadgeCache=/
  =BadgeLabelUtils= infra, tenant-configurable colours/labels,
  consistent with the rest of the app's categorical-state UI —
  but wide colored pills may be visually noisy across many grid
  rows scanned at a glance.
- *Compact status dot/icon*: smaller footprint, closer to the
  conventional "connection health" pattern in monitoring/dashboard
  UIs — but needs new componentry rather than reusing badge infra
  as-is, and loses tenant-configurability unless built in.

Look at how many rows =FxSpotGridWindow= typically shows at once and
whether other parts of the app already have a precedent for
live/transient (as opposed to categorical/persistent) per-row state
indicators, before deciding.

* Result

Abandoned badges. Reviewed the original mockup screenshot — it
already used icon+text inline in the Status column (no pill), with
the icon *shape* differing by state (pulse/hourglass/disconnect),
not just color. That's a better fit than a badge for a live per-row
indicator scanned down a grid, and lighter weight than
=BadgeCache=/=BadgeLabelUtils='s pill rendering. Picked icons from
the app's existing =IconUtils= dual-theme convention (FluentUI +
Solarized pairs) — see the story's =* Decisions= for the full table.
