* What

Two related fixes so a new domain entity can actually be generated via codegen,
specifically for =ores.synthetic=:

1. *Update the stale new-entity codegen docs.* The recipe
   [[id:B6C5D4E3-F2A1-4B0C-8D9E-0F1A2B3C4D5E][How do I create a new entity?]] and the =codegen-add-entity= skill instruct
   =compass codegen entity generate <entity> --profile sql|all-cpp|qt=, but
   =compass codegen= only exposes =generate= (single model file) and
   =regenerate= (per component) — there is no =entity= subcommand. They also
   describe JSON models (=projects/ores.codegen/models/<comp>/<entity>_domain_entity.json=),
   while the live/authoritative convention is *org* models under each
   component's =modeling/= dir (see =projects/ores.codegen/modeling/org_entity_meta_model.org=).
   Fix the recipe + skill to the real commands and the org model format.

2. *Register =ores.synthetic= as a codegen component.* It is absent from
   =projects/ores.codegen/library/component_catalogue.org=, so codegen cannot
   discover synthetic models. Add the catalogue row(s) (sql + cpp), create
   =projects/ores.synthetic/modeling/=, so entities can be authored and run via
   =compass codegen regenerate --component synthetic=.

* Why

Blocks the synthetic market data config foundation task (and any future
synthetic entity): we cannot follow the codegen-first path while the recipe is
wrong and the component is unregistered. Surfaced while attempting to author
=synthetic_config= + =synthetic_fx_spot_config= entities.
