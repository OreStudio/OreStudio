#+title: Three-level hierarchy in Market Simulator config tree
#+description: Restructure the Market Simulator tree to show named config → asset-class group → individual rates, with start/stop propagating down from any level.
#+type: capture
#+level: cross
#+filetags: :synthetic:qt:ui:market-simulator:
#+created: 2026-07-01
#+updated: 2026-07-01

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

The Market Simulator tree currently has two levels: =market_data_generation_config= (named
container, e.g. "FX Rates") → =fx_spot_generation_config= children (individual pairs). The desired
structure is three levels:

#+begin_example
Flat                              ← market_data_generation_config
  FX Rates                        ← asset-class group (UI-derived from sub-config type)
    EUR/USD                       ← fx_spot_generation_config
    GBP/USD
  IR Rates                        ← group for future ir_rate_generation_config
    EUR
Realistic
  FX Rates
    EUR/USD
    GBP/USD
  IR Rates
    EUR
#+end_example

The intermediate "asset class" group node is purely a UI construct — no new database entity is
needed. The group label is derived from the sub-config type (=fx_spot_generation_config= → "FX
Rates"; future =ir_rate_generation_config= → "IR Rates"). Start/Stop must propagate hierarchically:
clicking Start on a named config starts everything under it; clicking Start on an asset-class group
starts only that group's rates; clicking on an individual rate starts only that rate.

The implementation lives in =projects/ores.qt/synthetic/src/MarketSimulatorWindow.cpp= and its
header. The current tree uses =QStandardItemModel= with =NodeType= enum values =Feed= and =FxPair=
(see =NodeTypeRole= / =NodeIdRole= user-data slots). A third value =AssetClassGroup= must be added.
The =buildTree()= method must be updated to insert group nodes between the feed node and the pair
nodes, keyed by asset class. =selectedFxPairs()=, =fxPairsForFeed()=, =onStartAllClicked()=,
=onStopAllClicked()=, and the toolbar enable/disable logic in =updateToolbarState()= all need to
handle the new node type. The feed details panel (=showFeedSummary()=) should summarise counts per
asset class group. The FX pair details panel (=showFxPairSummary()=) is unaffected.

* Why

Without the grouping, all sub-configs under a named config are presented as a flat list. Once IR
rates, vol surfaces, and other asset classes are added, the list becomes unmanageable. The three-level
tree also makes the "start everything in this config" and "start just the FX rates" workflows
discoverable and consistent — a single click at any level triggers the right scope. Named configs
("Flat", "Realistic") are the natural unit of work for scenario testing; being able to start an
entire scenario with one click is the primary user need.
