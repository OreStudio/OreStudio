* Goal

The first-pass component migration ([[id:ADBD1D70-F62F-468E-A4FB-26DF28FBA527][Migrate component JSON models
to org-mode]]) landed the standalone =*_component.org= shape — one
file per component carrying only the four scalars (=name=,
=full_name=, =brief=, =description=). This was option B in the
original brainstorm and consistent with every other migration
(=_table.org=, =_lookup_entity.org=, etc.), so suffix-routing
discovery is uniform.

The brainstorm preferred option A: collapse the codegen-side
identity into the existing =component_overview.org= per component
(no duplicated identity, single source of truth for both humans
and codegen). This task does that second pass.

Concretely:

1. Add a property drawer + =#+brief:= to each
   =projects/ores.<component>/modeling/component_overview.org=
   so codegen can read the four scalars off the overview.
2. Extend codegen to recognise =#+type: component= overviews as
   component models (re-using the existing =load_org_component_model=
   logic where possible, or a small variant).
3. Delete the standalone =ores.<component>_component.org= files
   landed by the first pass.
4. Regen produces byte-identical output.

Out of scope: the cross-link work (overview → entities,
entity → overview, entity → schema) which is [[id:95031F88-44AB-4705-BA99-05A874800EC5][its own task]].

* Acceptance

- =load_org_component_overview_model= reads =#+name:=,
  =#+full_name:=, =#+brief:=, =#+description:= from frontmatter
  and returns the JSON-equivalent =/component= dict.
- 26 existing =component_overview.org= files (the PAIRs that
  had a sibling =*_component.org=) gain the three new
  =#+name:=, =#+full_name:=, =#+brief:= keywords; their existing
  =#+description:= becomes the codegen description.
- 5 ORPHAN group-level overviews (=ores.compute=,
  =ores.controller=, =ores.reporting=, =ores.trading=,
  =ores.workflow=) are created with frontmatter scalars +
  =* Summary= seeded from the JSON description.
- All 31 standalone =*_component.org= files deleted.
- =is_component_model= recognises =component_overview.org=;
  =generator.load_model= and =codegen.generate._generate_single=
  dispatch on that suffix to the new loader.

* Note on regen drift

The merged overviews' =#+description:= differs from what the
original JSON descriptions said for the 26 PAIRs (the overviews
already carried their own short tagline-shaped descriptions
while the JSON descriptions were longer, separately-authored
prose). Regen output would therefore drift on the
=cpp_component_header.hpp= and similar generated files.

This PR does *not* commit the regen drift because some of those
generated files carry hand-edits in HEAD (e.g.
=projects/ores.nats/include/ores.nats/ores.nats.hpp= has
subject-naming-convention prose that's not in the org source).
A clean regen would wipe those hand-edits — a separate
concern, tracked by the [[id:0BD89DB5-B7F7-4C18-8B00-D6C651264D71][post-regroup codegen validation
task]] which already needs to deal with stale generated paths
and re-baseline the scaffolds.

This task's scope is the source-of-truth refactor: the codegen
description now flows from a single org file per component.
