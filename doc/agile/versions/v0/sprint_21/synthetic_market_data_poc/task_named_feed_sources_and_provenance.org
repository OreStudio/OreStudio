* Goal

Today fx_spot_feed stamps every market_observation with source = 'SYNTHETIC' (a single opaque value), so provenance is lost — you cannot tell which feed or configuration produced a given observation. Introduce named sources: a feed configuration is a persisted entity with a stable source name (scheme synthetic.<id>, e.g. synthetic.gmm123) holding its parameters (ore_key, GMM means/stdevs/weights, initial price, ticks/hour, target tenant). The start_market_feed_config request references/creates a named source; fx_spot_feed stamps each observation's source field with that name. Persist the configurations (where: marketdata or synthetic schema — decide) so they can be listed, reopened, and re-run. This is the backend/data-model half; the configuration/control UI is task (see Feed configuration + market data control panel UI). Coordinate with the system-tenant fix so the persisted config also carries the target tenant.

* Acceptance

- A feed configuration is persisted as a named entity with a stable source
  name (=synthetic.<id>=).
- =start_market_feed_config= references a named source (by name or id) rather
  than carrying anonymous parameters only; defaults still work for the PoC.
- =fx_spot_feed= stamps =market_observation.source= with the source name, not
  ="SYNTHETIC"=.
- Existing observations/series remain valid; the change is additive.
- The persisted config can be queried (list + fetch by source name) over NATS
  so the future UI can populate from it.
- Decision recorded: persistence location (schema/table) and the
  source-naming scheme.
