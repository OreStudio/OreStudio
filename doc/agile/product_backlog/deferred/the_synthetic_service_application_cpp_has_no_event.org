#+title: Codegen gap: service application.cpp eventing pipeline not generated for new entities
#+description: The synthetic service application.cpp has no eventing pipeline wired up for gmm_component, fx_spot_generation_config, or market_data_generation_config. The refdata service wires this by hand: postgres_event_source + event_bus.subscribe<changed_event> + nats.publish, one block per entity. Codegen generates the changed_event header and the SQL notify trigger but does NOT generate or update application.cpp to register the entity in the eventing pipeline. This means saves/deletes in synthetic entities never trigger a stale-indicator in the Qt list window. The fix for now is to hand-write the eventing wiring in ores.synthetic/service/src/app/application.cpp mirroring ores.refdata/service/src/app/application.cpp. The longer-term fix is a codegen target or template that generates (or appends to) the service application eventing registration block, so new entities get eventing for free without a hand-edit gap.

* What

=ores.refdata.service='s =application.cpp= has the same gap, at much
larger scale: of the entities with a codegen-generated
=<entity>_changed_event.hpp=, only 14 are registered in the
=register_mapping<...>= / =event_bus.subscribe<...>= pipeline
(=book=, =business_centre=, =business_unit=, =counterparty*=,
=country=, =currency*=, =monetary_nature=, =party*=, =portfolio=).
At least 15 more have a generated event header but were never wired
in: =book_status=, =cds_convention=, =contact_type=,
=deposit_convention=, =fra_convention=, =fx_convention=,
=ibor_index_convention=, =ois_convention=,
=overnight_index_convention=, =party_id_scheme=, =party_status=,
=party_type=, =rounding_type=, =swap_convention=, =zero_convention=.
=party_type= was fixed inline while investigating a live "eventing is
broken" report during Qt UI verification (see
[[id:59583AFF-7657-4F79-BB7E-2086BCA04A91][Commission: party_type]]);
the other 14 are still unverified and presumably still broken.

* Why

Confirms this is not synthetic-specific: any entity commissioned after
=application.cpp= was last hand-updated silently has no
eventing — SQL trigger and service-layer publish calls are both
correct, and =compass codegen entity show= reports zero drift, so
nothing in the normal commissioning flow flags it. Also added a line
item to the [[id:897A7156-504D-4581-AD4C-026D643C0838][domain entity
evaluation checklist]] naming this exact file/mechanism, since the
existing checklist's "NATS event firing on mutation" criterion reads
as being about the service class's own publish call, not this separate
hand-maintained relay — only the Qt-layer manual cross-session check
would have caught it, and that's easy to skip.
