* Acceptance

- =BookMdiWindow= toolbar shows 4 new buttons after a separator
  following History: Statuses, Regulatory Types, Purpose Types,
  Ledger Feed Types.
- Each button opens its respective list window (=BookStatusMdiWindow=,
  =RegulatoryBookTypeMdiWindow=, =BookPurposeTypeMdiWindow=,
  =LedgerFeedTypeMdiWindow=), reusing the existing controllers (no
  duplicate windows/controllers created).
- Pattern matches =CurrencyMdiWindow=/=CurrencyController=/
  =RefdataPlugin='s existing RoundingType/MarketTier/MonetaryNature
  relay wiring exactly (signal on MdiWindow -> relay signal on
  Controller -> connect in RefdataPlugin to the target controller's
  =showListWindow()=).
- Builds clean (=ores.qt.refdata.lib=, =ores.qt.exe=).

* Plan

This is hand-maintained boilerplate, not codegen-driven -- confirmed
no facet/template exists for cross-entity toolbar wiring (same
finding as the earlier =rates_centre_code= flagged-combo task).
=CurrencyMdiWindow.hpp='s "Extra signal declarations seam" comment
references a =<<paste:UUID>>= mechanism that isn't actually populated
here either; the signals are hand-added directly above the seam
comment, same as Currency's own =showRoundingTypesRequested= etc.
Followed that exact convention:

1. =BookMdiWindow.hpp=/=.cpp= — 4 new signals + toolbar actions
   (separator, then one =Icon::Tag= button per child, matching that
   all 4 children's own list windows also use =Icon::Tag= for their
   own window icon, same as RoundingType).
2. =BookController.hpp=/=.cpp= — 4 matching relay signals, connected
   to the =BookMdiWindow= signals in the constructor.
3. =RefdataPlugin.cpp= — 4 =connect()= calls after all 5 controllers
   (=bookController_= plus the 4 targets) are constructed, each
   calling the target controller's =showListWindow()=.
