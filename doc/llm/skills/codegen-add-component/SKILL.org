* When to use this skill

When the user asks to create a new ORE Studio component — a new
project under [[proj:projects][=projects/=]].  Most domain components are split into
=.api=, =.core=, and =.service= parts; decide which parts are needed
before starting.  See [[id:F27E5DF7-6223-4B6F-80A5-CEFBEB1BC756][Component architecture]] for the split rules,
facet placement, and CMake dependency chain.

* How to use this skill

Codegen is *mandatory* for all boilerplate.  Do not hand-write
CMakeLists or stub files.

Since [[https://github.com/OreStudio/OreStudio/pull/1014][PR #1014]] the
=component_overview.org= per component is the single source of
truth for both humans and codegen — the standalone
=*_component.json= and =*_component.org= models are retired.

1. /Gather requirements/: component short name, parts needed
   (=.api=, =.core=, =.service=, or a subset), one-line brief, longer
   description.

2. /Scaffold the per-part overview/ via compass.  For each part
   (=.api=, =.core=, =.service=) run:
   #+begin_src sh
   ./projects/ores.compass/compass.sh add component \
     --slug component_overview \
     --parent-dir projects/ores.COMPONENT/PART/modeling \
     --title "ores.COMPONENT.PART" \
     --description "<one-liner ≤ 120 chars>" \
     --brief "<short tagline>"
   #+end_src
   The scaffold includes the codegen-readable keywords
   (=#+name:=, =#+full_name:=, =#+brief:=) automatically.  Fill in
   the =* Summary= and other sections per
   [[id:B9D3D469-9E59-465F-8C18-34546B680D13][Component Documentation Guide]].

3. /Run codegen against each overview/ to scaffold the C++ + CMake
   project. Each overview's =#+component_kind:= frontmatter (=api=,
   =core=, =service=, or the default =flat=) selects the scaffolding
   variant; the address is the same =ores.cpp.component= for all of them:
   #+begin_src sh
   ./projects/ores.codegen/codegen.sh generate \
     --model projects/ores.COMPONENT/api/modeling/component_overview.org \
     --address ores.cpp.component
   #+end_src
   Repeat per part (=.api=, =.core=, =.service=), each against its own
   overview file.

4. /(Optional) Add a group-level overview/ at
   =projects/ores.COMPONENT/modeling/component_overview.org=.
   The group overview is navigation + group-level codegen scalars;
   the directory layout =projects/ores.COMPONENT/{api,core,service}/=
   is the composite declaration (no =:subcomponents:= keyword needed).

5. /Register/ each part in [[proj:projects/CMakeLists.txt][=projects/CMakeLists.txt=]] in dependency
   order (=.api= → =.core= → =.service=).

6. /Add a PlantUML stub/ (=modeling/ores.COMPONENT.PART.puml=) for each
   part — codegen does not yet produce this.  See
   [[id:B0B99088-D88E-4980-A4EE-7D6A9A8E96A5][component-model-creator]] for the diagram conventions.

7. /Build and verify/ the stub tests with [[id:654D4A70-E63D-4C18-B5A5-886066E36314][cmake-runner]], via
   =compass build= (not raw =cmake --build=, so it takes the
   host-wide build lock — see [[id:F176FCA1-68D5-420D-B076-75A65FEE5EBB][How do I build the system?]]):
   =./compass.sh build ores.COMPONENT.api.tests=

8. /Fill in the architecture sections/ of each overview using
   [[id:B0B99088-D88E-4980-A4EE-7D6A9A8E96A5][component-model-creator]] (Inputs, Outputs, Entry points,
   Dependencies, See also, and the PlantUML diagram).

* Templates

All scaffolding runs via the single address =--address ores.cpp.component=;
the overview's =#+component_kind:= frontmatter picks the variant (each
generates 9 files for =.api=/=.core=; 18 for =.service=):

| =#+component_kind:= | Key templates |
|---|---|
| =flat= (default) | Flat component (no =.api=/=.core=/=.service= split) — produces all 10 files including =stub.hpp= / =stub.cpp= |
| =api= | =cmake_component_root.mustache= + =cmake_component_api_src.mustache= + 7 others |
| =core= | =cmake_component_root.mustache= + =cmake_component_core_src.mustache= + 7 others |
| =service= | =cmake_component_root.mustache= + =cmake_component_service_src.mustache= + 16 others |

All four kinds consume the same model type =component=, read
from a =component_overview.org= via =load_org_component_overview_model=.
Full template list is in [[id:08FBD248-257A-A474-DD43-DB08949978F1][ORE Studio Codegen]]§"Profile catalogue".

The codegen-side scaffold flow is exercised by
[[proj:projects/ores.codegen/tests/component_scaffold/verify.py][=projects/ores.codegen/tests/component_scaffold/verify.py=]] — two
fixture components (flat + composite) plus prior-art checks
against =ores.nats= (flat) and =ores.controller= (composite
group + 3 subs).
