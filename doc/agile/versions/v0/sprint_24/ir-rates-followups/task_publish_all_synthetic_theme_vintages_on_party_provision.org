* Goal

Every synthetic theme published to a party during standard
provisioning should come up with valid vintage data out of the box,
not just the first/default one. =party_provisioning_bundle_plan()=
(=projects/ores.dq/api/include/ores.dq.api/messaging/party_provisioning_plan.hpp=)
publishes exactly one FX driver-rates bundle,
=marketdata.reference_vintage_2016_02_05=, unconditionally -- but the
2026 Realistic synthetic theme's FX configs are seeded with
=vintage_source = 'fed.h10.2026-05-05'=
(=projects/ores.sql/populate/synthetic/synthetic_fx_spot_configs_realistic_2026_populate.sql=),
which that bundle never provides. The result: every 2026 Realistic FX
feed shows a red vintage-invalid badge immediately after a fresh
=barclays_system_provision.ores= run, discovered while testing the
"Synthetic data themes" scenario, and only fixable today by knowing
to separately run =bundles publish synthetic_ore_samples_2016=/
=synthetic_uniform_demo=-style commands by hand.

Make the provisioning plan (or the synthetic themes' own publish
step) aware of which vintage bundle each synthetic theme it
publishes actually needs, and publish all of them -- not just one
hardcoded default. Keep it data-driven enough that a future theme
introducing a new vintage doesn't require another hand-edit here
(see the related "data-driven bundle list" tech-debt story/task
already tracked for this same publish-phase pattern).

* Acceptance

- A fresh =barclays_system_provision.ores= run (or equivalent Qt
  wizard flow) against a clean database leaves every synthetic
  theme's FX (and IR curve, once vintage-gated) feeds showing a
  valid, not red, vintage badge -- no manual follow-up
  =bundles publish= commands required.
- Adding a new synthetic theme with its own vintage requirement
  doesn't require a hand-edit to
  =party_provisioning_bundle_plan()=, only new data (dataset/bundle
  rows).
- No regression to parties that don't use any vintage-gated
  synthetic theme -- provisioning stays fast and doesn't publish
  bundles nothing needs.
