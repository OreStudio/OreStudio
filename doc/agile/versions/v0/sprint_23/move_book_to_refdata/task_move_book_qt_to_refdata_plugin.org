#+description: Architectural decision: Book, BookStatus, and RegulatoryBookType (backend already in ores.refdata) must have zero cross-component leakage on the Qt side -- CRUD Qt code and plugin wiring belong entirely in ores.qt/refdata under RefdataPlugin, not ores.qt/trading under TradingPlugin. Book's Qt classes already physically moved in PR #1449, but TradingPlugin.cpp still constructs and menu-wires all three (BookController, BookStatusController, RegulatoryBookTypeController) -- move that construction and Trading Codes menu wiring to RefdataPlugin.cpp, contributing to the shared trading_codes_menu the same way RefdataPlugin already does for Purpose Types. TradingPlugin keeps only the composite trading-workflow views (PortfolioExplorerMdiWindow, OrgExplorerMdiWindow, TradeController) that consume BookController via a pointer but do not construct/own it.
