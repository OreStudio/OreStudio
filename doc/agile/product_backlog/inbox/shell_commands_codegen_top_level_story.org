#+title: Shell entity commands — top-level commissioning story
#+description: ores.shell entity commands have no codegen profile and are hand-written per entity. Lift shell-command commissioning out of per-entity commission stories into one top-level story: establish the per-entity command pattern, modernize, templatize into a codegen facet, then roll out across entities.
#+type: capture
#+level: cross
#+filetags: :ores.shell:codegen:refdata:
#+created: 2026-07-03

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

Make *shell entity commands* a top-level story (with one task per component /
entity group), rather than an "implement shell commands" task buried in each
=commission_*= story. =ores.shell= has *no codegen profile* (confirmed in the
=codegen-add-shell-entity= skill's "Codegen gap" section) — every entity's REPL
commands are hand-written as =<entity>_commands.{hpp,cpp}= under
=projects/ores.shell/src/app/commands/= (e.g. =currencies_commands.cpp=,
=countries_commands.cpp=, =parties_commands.cpp=), each defining a submenu and
registering list/add/history/delete subcommands, then hand-wired into
=application.cpp=. Consolidate the per-commission shell tasks: they are
superseded by this top-level story and should be abandoned/closed in favour of
it when picked up (e.g. the =commission_purpose_type= "Implement and verify
purpose_type shell commands" task).

*Pick up after all entities are commissioned* — shell is a horizontal layer best
templatized once across entities, not implemented entity-by-entity.

** Target architecture (codegen-first)

Mirror the SQL/C++/Qt facets and the sibling HTTP/Wt stories: adding an entity's
shell commands = a codegen run, touching no shared file.

1. *Establish the pattern* on one reference entity (currency — its
   =currencies_commands.cpp= already exists): a thin per-entity command unit,
   parse → in-process =<entity>_service= call (or NATS request) → =table_io=
   render, with declarative submenu + subcommand registration.
2. *Generated aggregator = the wiring*: a generated
   =register_refdata_shell_commands(menu, ctx, …)= that instantiates and
   registers each entity's command unit; =application.cpp= calls the one
   aggregator, which iterates the entity model list rather than growing a
   hand-edited block (same approach as the messaging registrar and the HTTP
   aggregator).
3. *Templatize* into a codegen shell facet (=<entity>_commands= + the
   aggregator); regenerate currency and prove zero-diff against the hand-written
   unit; per the skill, phase as list+add, then history, then delete.
4. *Per-component/per-entity rollout tasks*: generate shell commands for each
   commissioned entity. One task per component keeps it reviewable.

* Why

Shell is a horizontal entity-facing layer with neither codegen nor a coherent
home: every entity's commands are hand-written and wired by hand, so each
commission story carries an identical "implement shell commands" task that only
defers the real work. A single top-level story — establish the pattern on
currency, templatize into a facet, then roll out — gives every entity a
consistent, generated REPL surface and removes the per-entity toil, exactly as
was done for HTTP and Wt. =ores.cli= has the *identical* gap and warrants a
separate sibling story.
