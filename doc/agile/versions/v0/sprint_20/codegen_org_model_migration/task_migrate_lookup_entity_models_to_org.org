* Goal

Convert the 12 SQL-only =*_entity.json= lookup-entity models
(analytics/pricing_engine_type, dq/{artefact_type, lei_entities,
lei_relationships, report_definitions}, iam/{account_type,
tenant_status, tenant_type}, trading/{lifecycle_event,
party_role_type, trade_id_type, trade_type}) into literate
=ores.<component>.<entity>_lookup_entity.org= files under each
owning component's =modeling/= dir. Extend the codegen
pipeline with a new =ores.codegen.lookup_entity= kind: dedicated
=load_org_lookup_entity_model= producing the =sql_schema_table_create.mustache=
shape (JSON root key =entity=), filename-suffix dispatch on
=_lookup_entity.org=, and exclusion from the generic entity match
so routing is unambiguous.

The org file preserves *all* scalar metadata from the JSON
(=entity_singular_upper=, =entity_title=, =component_include=,
...) so that future C++/Qt/protocol profiles work directly
against the org file without re-introducing the JSON.

* Acceptance

- =migrate_lookup_entity_to_org.py= one-shot converter committed.
- 12 =ores.<component>.<entity>_lookup_entity.org= files; 12
  JSON files removed.
- =load_org_lookup_entity_model= returns the JSON-equivalent
  =/entity= dict (verified end-to-end:
  =json.load(... .json) == load_org_lookup_entity_model(... .org)=
  for all 12).
- =is_entity_schema_model= accepts the new =_lookup_entity.org=
  suffix; =is_domain_entity_model= excludes it.
- =_CODEGEN_ORG_TYPES= accepts =ores.codegen.lookup_entity= so
  the per-component =modeling/= discovery picks it up.
- Both =generator.load_model= and =codegen.generate._generate_single=
  dispatch on the new suffix to the lookup-entity loader.
- =codegen.sh generate --model X_lookup_entity.org --profile sql=
  produces byte-identical output to the JSON path for all 12
  lookup entities. (Note: =trading/lifecycle_events_create.sql=
  in HEAD has pre-existing drift from when the model was a
  =_domain_entity.json= — both JSON path and org path produce
  the new =sql_schema_table_create.mustache= output, but neither
  matches the stale HEAD file. Out of scope for this PR.)

* Plan

Org file shape:

- Frontmatter scalars: =product=, =schema=, =component= +
  optional =component_include= / =component_core=, all
  =entity_*= scalars (=entity_singular=, =entity_plural=, plus
  derivatives like =entity_singular_upper=, =entity_title=,
  =entity_plural_words_cap= where the JSON carries them),
  =has_tenant_id=, =system_tenant_validation=,
  =has_display_order=, =has_coding_scheme=, =has_image_id=,
  =has_artefact_insert_fn=.
- Body: free-form description.
- =* Primary key= drawer (=column=, =type=, =is_text=, optional
  =cpp_type=).
- =* Columns= sub-headings (one per column) with drawer
  (=type=, =nullable=, optional =default=).
- =* Validations= org table (=column= / =validation_function=).
- =* Indexes= sub-headings when present (none currently).
- =* Artefact indexes= sub-headings when present (used by the
  3 dq files with =has_artefact_insert_fn=).
