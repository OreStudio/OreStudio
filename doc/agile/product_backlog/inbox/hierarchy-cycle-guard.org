#+title: No cycle guard in hierarchy CTEs and build_tree()
#+description: PR #1450's review (4 independent Claude runs) consistently flagged: the recursive hierarchy CTEs (party/counterparty *_hierarchy_fn, and the codegen twin in sql_schema_domain_entity_create.mustache) and ores::utility::domain::build_tree() have no cycle guard. A parent_id cycle (A's parent is B, B's parent is A, no external root) would either spin the recursive CTE until Postgres's safety net kicks in, or silently drop both nodes from build_tree()'s output forest (neither qualifies as a root). Low severity given DB-level one-root-per-tenant conventions constrain real data, but worth a defensive fix (cycle detection in the CTE, and detecting unreached ids in build_tree to promote them to extra roots like the existing orphan-handling guarantee) plus a test case in domain_hierarchy_tests.cpp.
