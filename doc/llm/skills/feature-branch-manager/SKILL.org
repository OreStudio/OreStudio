* When to use this skill

When a piece of work is too large for a single PR and needs to be
split into a sequence of phase-sized PRs. The skill keeps each
phase on its own branch from a fresh =main= so the history stays
linear and each phase is reviewable in isolation.

* How to use this skill

1. /Plan the phases up front/. A typical entity implementation
   splits as e.g. domain type → generator → repository → tests, or
   the Qt sequence model+window → detail dialog → history dialog →
   controller. Each phase is one branch, one PR.

2. /Name branches consistently/:
   =feature/<entity>_<phase_description>=. Examples:
   - =feature/currency_domain_type=
   - =feature/currency_generator=
   - =feature/session_history_dialog=

3. /Run one phase to merge/ via [[id:545D6B43-17FC-0234-FF2B-AB6565695616][PR Manager]] (open) and [[id:57B7D4D9-AAA5-427D-8D16-0E82DA9D3763][PR
   Shepherd]] (watch until green and merge).

4. /Transition to the next phase/ — once the current PR is merged,
   use [[id:479E7E27-4060-4BFD-AC38-F11EFA3BB306][How do I start a new feature branch phase?]]. This stashes
   any WIP, branches from the freshly-fetched main, restores the
   stash, and deletes the old branch both locally and on the
   remote.

5. /Never rebase + force-push/ — confuses reviewers, breaks the
   per-phase history this skill exists to maintain.
