A codegen *entity model* is a single literate org-mode file that
describes one entity (e.g. [[id:8F3A1C2D-4B5E-4A6C-8F9D-2E3F4A5B6C7D][party]]) and drives code generation across
every technical space that entity touches: [[id:2E4DC627-64F4-4CD8-9B26-BD974C648F07][SQL]] and several independent
[[id:5293A5FE-9150-4C0A-92F2-5FE53325FFE0][C++]] facets (e.g. [[id:62342007-5080-43BF-8207-CD05B6568BBE][ores.cpp.domain]], [[id:50246CF5-EB61-4261-890F-4E75D3241863][ores.cpp.repository]],
[[id:E1898A0F-EC04-4CC3-A547-951F84827432][ores.cpp.qt]]). In [[id:BA0E3273-DC4B-7CD4-447B-06227B7D09A8][MASD]] terms, this hub — together with its
segment pages — is the *metamodel* for one metatype, =ores.codegen.entity=
(=domain_entity=); it is one entry in the [[id:884C6B00-0849-430D-9D32-F3544269DC56][Codegen meta-model]] registry of
all eight metatypes ores.codegen recognises. It does not describe any
one entity, it describes what any entity model file is allowed to
contain. See [[id:43FF3591-AD18-4CE3-8239-A220862C9B5A][Applied MASD]] for how ORE Studio instantiates MASD's
logical/physical space, metatype, facet, and archetype concepts, and
for why an entity model file looks the way it does at all (§ "The
entity model file as a unified literate document") — those concepts
are exactly the organising principle this hub uses below.

This page stays deliberately short: an overview, the file-level
conventions every entity model shares regardless of which facets it
uses, and a table routing to the segment page that documents each
facet's fields, worked example, and paste blocks in full. Go to a
segment page for the exhaustive per-property reference; come back
here only for the shape of the file as a whole.

* File-level conventions

See [[id:18D99C09-22CD-4487-B7A3-FA0612A8B154][Codegen entity meta-model — file-level conventions]]: file naming,
required frontmatter, and the prose body — the conventions every
entity model shares regardless of which facets it uses, and the
document-type archetype ([[id:C0B89074-D107-46E5-B9D2-146AE14ADDFA][ores.doc.modeling.entity_org]]) that scaffolds
them.

=compass add entity_org --shape <name>= scaffolds a new entity model
with a named domain_entity shape's knobs already set correctly
(=simple-text-key=, =fk-scoped=, =hierarchical-composite=, =richest=,
=timeseries= — sampled from the reference entities in the shape
table below), rather than the bare, unknobbed skeleton a plain
=compass add entity_org= produces. Anything the shape can't derive
(actual columns, foreign-key targets, display fields, ...) is left as
an explicit =TODO= in the scaffolded file, never silently omitted.
Individual =--entity-<knob>= flags override any preset value.

* Segments

=Entity= specializes [[id:884C6B00-0849-430D-9D32-F3544269DC56][Base]] with its own frontmatter type and one typed
field per segment below — [[id:884C6B00-0849-430D-9D32-F3544269DC56][Codegen meta-model]] documents =Base= and
=CommonFrontmatter= on their own, without any metaclass's
specialization. An entity is modelled as a set of segments, which loosely correspond
to the properties needed to support the various aspects of the
technical spaces the entity targets — [[id:0BD99407-1CBE-469B-B2D3-1C0AE3AA18DE][Keys and columns]] and [[id:729E44CA-6AB7-4ADB-BA54-B1AD5A12DD27][SQL]]
for the database layer, one segment per independent C++ facet
(domain, repository, service, protocol, NATS, Qt, generator,
presentation) for the application layer. Segment classes are drawn
empty here — each has its own structural fields, documented on its
own page, not duplicated in this overview. Several segments also
document their own [[id:40AD7EFB-AACE-472F-B767-8FD3FF2F5C03][paste blocks]].
