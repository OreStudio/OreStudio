#+title: How do I manage the checkout environment with compass?
#+description: Use compass env init/diff/list/bump to generate and inspect the .env: behaviour-preserving credential generation, the .env.old diff, masked variable listing, and recording an env-format version bump.
#+type: recipe
#+level: cross
#+filetags: :compass:env:provision:credentials:recipe:
#+created: 2026-06-02
#+updated: 2026-06-02

=compass env= is the *Provision* pillar of the [[id:6F7A122F-BC0F-4F3E-BA46-8B793FF15D10][developer toolkit]]: it sets the
checkout up so the system can run. It owns the =.env= at the checkout
root — the file every service, the CLI, the shell, and the test suite
read their database and NATS credentials from. =compass env= replaces the
old =build/scripts/init-environment.sh= and =diff-environment.sh=.

* Question

How do I generate the =.env= for a checkout, inspect what's in it, see
what a regeneration changed, and record an env-format version bump?

* Answer

** Generate or refresh the .env — =compass env init=

#+begin_src sh :results verbatim
./projects/ores.compass/compass.sh env init --preset linux-clang-debug-make -y
#+end_src

What it does, in one idempotent pass:

- Derives the checkout identity (label, database name, NATS subject
  prefix) and the per-environment ports from the checkout directory name.
- Generates NATS mTLS certificates and the IAM RSA signing key if absent.
- Reads the NATS domain-service list from the authored registry (see
  [[id:CB7713BA-544A-43CD-AEB3-30C2157CBBCC][How do I add a domain service?]]) and writes each service's DB
  credentials.
- Writes the =ORES_ENV_VERSION= from the canonical version file (below).

*Behaviour-preservation guarantee.* On a re-run, existing secrets are
*never* changed — only missing variables are generated. Derived values
(role/user names, ports, paths) are always recomputed from the label, so
they track a moved or relabelled checkout. A backup of the previous file
is written to =.env.old= before the new one is written.

Flags:

| Flag | Effect |
|------+--------|
| =--preset P= | Build preset (required locally; CI passes it / sets =CI=) |
| =-y=, =--yes= | Skip the overwrite confirmation prompt |
| =--with-diff= | After writing, print the =.env.old= → =.env= diff |
| =--enable-logging [level]= | Add test-logging vars only (default level =debug=) |
| =--disable-logging= | Remove the test-logging vars only |

The postgres superuser password is taken from =$PGPASSWORD= if set, else
reused from the existing =.env=, else prompted for.

** See what a regeneration changed — =compass env diff=

#+begin_src sh :results verbatim
./projects/ores.compass/compass.sh env diff
#+end_src

Prints a unified diff of =.env.old= versus =.env= (exit 0 identical,
1 differ). =compass env init --with-diff= runs this automatically after
writing. Output contains secrets in plain text — do not share it.

** Inspect the environment — =compass env list=

#+begin_src sh :results verbatim
./projects/ores.compass/compass.sh env list                 # secrets masked
./projects/ores.compass/compass.sh env list --show-secrets   # reveal everything
#+end_src

Variables are grouped under the =.env='s own section headings; the
per-service credential block is sub-grouped by service (=[iam]=,
=[refdata]=, …) so it's easy to scan. Secret values (passwords, the JWT
key, =PGPASSWORD=) are masked by default; =--show-secrets= reveals them.

** Record an env-format version bump — =compass env bump=

The =.env='s shape is versioned by =ORES_ENV_VERSION=, checked at the
start of =recreate_database.sh=. When you make a change that alters the
=.env= — add a service to the registry, add or rename a variable — record
a bump:

#+begin_src sh :results verbatim
./projects/ores.compass/compass.sh env bump --note "added analytics service DB creds"
#+end_src

This increments the canonical version and appends a dated changelog entry
in =build/config/env_version.json= — the single source read by both
=compass env init= (which writes =ORES_ENV_VERSION=) and the validator. Then
re-run =compass env init= to write the new version into your =.env=. Any
checkout still on the old version now fails =recreate_database='s
validation with a clear "re-run =compass env init=" message, instead of a
later missing-password error.

* Script

=projects/ores.compass/compass.sh env {init,diff,list,bump}= →
=src/compass.py= (=cmd_env=) → =src/env_init.py=. The canonical
env-format version + changelog live in =build/config/env_version.json=.

* Tested by

Manual. =env init= against an existing =.env= changes no secret and no
key set (verify with =compass env diff= or by diffing a backup); the
generated =.env= round-trips through =recreate_database.sh=.
