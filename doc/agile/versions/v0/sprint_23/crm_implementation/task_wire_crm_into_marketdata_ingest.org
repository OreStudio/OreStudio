This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:B38B3869-02FD-4CC7-99BD-9A77904ACA19][Cross-rates matrix (CRM)]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Add the persisted config surface the CRM needs in =ores.marketdata=
(topology + driver pairs + curated on-demand derived pairs), fully
codegen-generated, laying the groundwork to wire the
=ores.analytics.quant= CRM engine into =ores.marketdata='s ingest path
(the follow-up task) following the [[id:DF2B616F-E112-4C57-A73B-69F85BE96EF4][Market Data Architecture]] doc's
"drivers in, derived rates out" section -- without the naive design of
republishing every derived pair as an official tick on every driver
movement (an O(n^2) fan-out for no real benefit, since triangulation is
cheap and deterministic and anyone with the driver ticks + topology can
derive the same value locally).

* Acceptance

- Two new codegened entities in =ores.marketdata= (tenant *and* party
  scoped independently -- a party's CRM config is its own, not shared
  across the tenant):
  - =crm_driver_pair=: the topology edges (=base_currency_code=,
    =quote_currency_code=, =enabled=). One row per driver pair; the set
    of enabled rows for a (tenant, party) is fed straight into
    =topology_builder::build= as its =ccy_pair_input= list.
  - =crm_topology_config=: the pivot currency + whether the CRM is active
    at all for this (tenant, party) (=pivot_currency_code=, =enabled=).
  - =crm_enabled_derived_pair=: the curated list of *derived* (non-edge)
    pairs a consumer may ask for on demand (=base_currency_code=,
    =quote_currency_code=, =enabled=) -- this is what makes "which pairs
    are enabled" config-driven rather than "every reachable pair",
    directly per this story's decision.
  - Both driven entirely by codegen (domain/api/core/sql, following the
    =feed_binding= entity as the template) -- CRUD + history + eventing
    for free; the Qt/Wt management UI is the separate, later story task.
- Tenant and party independence: two parties in the same tenant may have
  completely different topology/driver config -- enforced at the schema
  level (RLS_001 tenant-isolation policy; the party-level RESTRICTIVE
  policy is the same documented RLS_002 stopgap the four existing
  marketdata tables already carry).

Deferred to the service-wiring follow-up task (identified in review, see
=* Review= below): cross-party FK validation on =config_id=, and a
uniqueness constraint on at most one =enabled= =crm_topology_config= per
(tenant, party) -- both matter once something actually reads "the"
config for a party, not before.

* Plan

** Architecture decision: compute locally where possible, pull on demand otherwise -- never broadcast the full derived set

Considered and rejected: recomputing and republishing every reachable
derived pair as an official tick whenever any driver ticks. Real market
data systems (vendor feeds, and this story's own
[[id:DF2B616F-E112-4C57-A73B-69F85BE96EF4][Market Data Architecture]] doc) publish only *observed* legs at full tick
rate; derived/calculated values are either computed client-side by a
shared calc library (cheap, zero added server load, zero added latency --
exactly what =ores.analytics.quant::rate_engine= already is, embeddable
by any C++ consumer) or served *on demand* to consumers that can't embed
the library, never broadcast. This task implements exactly that split:

1. Drivers: unchanged, full tick rate, via the existing remap path.
2. CRM config (topology + enabled-derived-pairs): published as ordinary
   low-churn reference data (the entities' own NATS CRUD/eventing,
   generated for free) -- any consumer that wants to run the CRM locally
   subscribes to the driver stream (already public) + this config, and
   gets the exact same =rate_engine= this task uses server-side.
3. Consumers that can't embed the library: a request/reply endpoint,
   computed on demand from the service's own per-party engine. No
   subscribe/push path for derived rates in this task; add one later only
   if a concrete consumer needs it, and even then throttled/coalesced,
   never 1:1 with driver ticks.

** Entities (see =projects/ores.marketdata/modeling/= once scaffolded)

Modelled on =ores.marketdata.feed_binding= (=modeling/ores.marketdata.feed_binding.org=): tenant-scoped natural keys, =enabled= soft-disable
column, =read_latest_all_tenants=-style repository method for the
service's own startup/refresh reads. Both new entities add
=party_id= alongside =tenant_id= for independent per-party config (feed_binding
today is tenant-only; the CRM's is the first tenant+party-scoped entity in
this component, matching the story's tenant/party-independence
requirement).

=crm_driver_pair.enabled = false= deliberately does *not* delete the row
-- disabling and re-enabling a driver pair should not lose its history,
same rationale as =feed_binding=.

** Service wiring (moved to the follow-up task)

The service-side design (a =crm_ingest_bridge= class owning one
=service::rate_engine= per (tenant_id, party_id), a =feed_ingest_loop=
hook that also feeds driver ticks into it, and the
=marketdata.v1.crm.rate= request/reply endpoint) is carried forward
unchanged into the new follow-up task -- see its own =* Plan= for the
full design, this section is intentionally left as history of where
that design was first written.

* Result

Shipped three new codegened entities in =ores.marketdata=
(=crm_topology_config=, =crm_driver_pair=, =crm_enabled_derived_pair=),
tenant+party-scoped, modelled on =feed_binding=/=market_series=, Qt UI
deferred. Fully codegen-generated -- verified via =git status= that
every file under =ores.marketdata/{api,core,service}= and
=ores.sql/{create,drop}= is untouched generated output.

Wired the three tables into the two things codegen doesn't touch:
=marketdata_rls_policies_create.sql=/=_drop.sql= (tenant-isolation RLS)
and =marketdata_create.sql=/=_drop.sql= (the =\ir= aggregator that makes
generated SQL reachable from =setup_schema.sql= at all -- missing this
isn't caught by any validation, only by actually running
=db recreate=). Filed both as a generalised codegen-gap capture
(=inbox/codegen_rls_policy_facet.org=) and extended the existing RLS_002
backlog item to cover all seven affected tables; folded in a third gap
found in review (unique-index identifiers exceeding Postgres's 63-byte
=NAMEDATALEN= limit, inherited from the existing codegen template, not
introduced here).

Review surfaced two real, non-blocking-for-this-PR gaps -- recorded in
=* Review= and carried forward as acceptance items on the new
follow-up task, [[id:88FD9001-F2BA-4061-994C-5EBE8BBD590A][Wire the CRM into ores.marketdata's ingest path (service
wiring)]], since both matter once something actually reads "the"
config for a party rather than before: cross-party FK validation on
=config_id=, and a uniqueness constraint on at most one =enabled=
=crm_topology_config= per (tenant, party).
