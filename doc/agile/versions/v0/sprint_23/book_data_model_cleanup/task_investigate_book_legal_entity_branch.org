* Result

*Recommendation: no new fields or entities needed.* The existing
=party=/=business_unit= pair already cover the domain notes'
legal-entity/branch requirement:

- =book.party_id= (mandatory) *is* the Legal Entity — =party= is
  documented as "an internal legal entity participating in financial
  transactions".
- =book.owner_unit_id= (optional FK to =business_units=) covers
  Entity, Business Unit, *and* Branch — =business_unit= is
  deliberately generic ("desks, departments, branches" per its own
  model doc) and self-referencing via =parent_business_unit_id=, so a
  branch is just a business unit at the right depth in that chain, not
  a distinct entity type. (=owner_unit_id= was already wired up to
  =book= in an earlier task, per the =books_belong_to_business_unit=
  backlog capture.)
- Cost Centre is separately tracked as free-form ledger metadata on
  the book itself, not as its own lookup entity — out of scope here.

Recorded this mapping in =book_and_the_ledger.org= (the doc that
already describes the Legal Entity → Entity → Business Unit → Cost
Centre → Ledger Book hierarchy) so the connection between the
real-world hierarchy and the ORE Studio domain model is explicit.
No code changes.
