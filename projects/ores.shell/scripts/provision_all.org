#+title: provision_all — fresh system to fully provisioned
#+description: End-to-end provisioning: super_admin, base system, tenant with tenant_admin, synthetic organisation, party setup — logins and logoffs included.
#+property: header-args:ores-shell :tangle provision_all.ores

This is a [[file:README.org][library]] script; the runnable artefact is the tangled
=provision_all.ores=, regenerated by the build — do not edit it by
hand.

* Purpose

Reproduce, from the command line, exactly what an operator does
manually through the three Qt provisioning wizards: bootstrap the
system with a =super_admin= account, install the base system and
first tenant, provision the tenant with synthetic reference data, and
provision its root party. =load='s stop-on-error semantics guarantee
that an error-free run ends with a fully provisioned system: tenant
active, parties active, report definitions in place, two admins able
to log in.

* Expectations

- All services are up (=compass services start=).
- The session is already connected: run the script via =compass shell=
  (=compass shell -f provision_all.ores=), which auto-connects using
  the environment's =.env= — the script deliberately contains no
  =connect= line so it is not bound to any environment's NATS port.
- The database is fresh (=compass db recreate -y -k=) so the system
  is in /bootstrap mode/ — stage 1 refuses otherwise.
- Nobody is logged in; the script performs every login and logout
  itself.
- The synthetic organisation is generated with a *fixed seed* so the
  party names are deterministic; stage 3's hard-coded party name is
  in lock-step with that seed. If you change =--seed=, discover the
  new root party with =parties list --category Operational= and
  update stage 3.

* Stage 1: system provisioner

One command spans the SystemProvisionerWizard: confirms bootstrap
mode, creates =super_admin= (password =Secure-Password-123=), logs in
as it, and provisions the first tenant with a =tenant_admin= account
(here sharing the same password for development convenience). The
tenant hostname is =default= (the wizard's hostname-from-code
derivation) rather than the single-tenant default =localhost=: the
seeded Root Tenant already occupies =localhost=, and tenant hostnames
are unique.

The command leaves the session logged in as =super_admin=, exactly as
the wizard does; the explicit =logout= mirrors the operator logging
off before the tenant stage.

#+begin_src ores-shell
provision system super_admin Secure-Password-123 admin@localhost.com --tenant-admin-password Secure-Password-123 --tenant-hostname default
logout
#+end_src

* Stage 2: tenant provisioner

Log in as the tenant admin — principals are =username@hostname=,
hence =tenant_admin@default=. The login lands in the tenant's
/bootstrap mode/, which is what =provision tenant= requires.

One command spans the TenantProvisioningWizard: publishes the first
available dataset bundle (waiting on its workflow), generates the
synthetic organisation with a fixed seed, associates =tenant_admin=
with every operational party, clears the bootstrap flag and completes
provisioning. The logout afterwards is required: tenant state is
re-evaluated at login, exactly as the wizard's summary page instructs.

#+begin_src ores-shell
login tenant_admin@default Secure-Password-123
provision tenant --source synthetic --seed 42
logout
#+end_src

* Stage 3: party provisioner

Log back in (the tenant is now active) and provision the root
operational party: import the small GLEIF counterparty dataset,
publish the party's organisation bundle, create every standard report
definition, and activate the party. The party's full name is pinned
by =--seed 42= above.

#+begin_src ores-shell
login tenant_admin@default Secure-Password-123
provision party "Lloyds Wealth Management Ltd" --reports all
logout
#+end_src

* Done

An error-free run ends here with a fully provisioned system. Log in
as =tenant_admin@default= in the Qt client or the shell to use it.

#+begin_src ores-shell
exit
#+end_src
