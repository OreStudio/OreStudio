* Goal

Importing the full ORE reference vintage file
=external/ore/examples/Legacy/Example_56/Input/market.txt= via
=ores.marketdata.core::service::import_service= (subject
=marketdata.v1.import=) hits a Postgres unique-constraint violation
on =_hyper_*_market_observations_observations_current_uniq=:
duplicate key =(tenant_id, party_id, series_id, observation_datetime,
point_id)=, e.g. =point_id = '6M'= at =2016-02-05=. Suspect the root
cause is in =ores.ore.core='s series key decomposition
(=projects/ores.ore/core/src/market/series_key_registry.cpp=,
=qualifier_depth= table): some distinct source quote keys are being
decomposed to the same =(series_id, point_id)= pair when they
shouldn't be — e.g. two different =series_type=/qualifier
combinations both ending up with =point_id = "6M"=. Needs
investigation into which =series_type= entries collide and whether
=qualifier_depth= is wrong for one of them, or whether the file
genuinely contains a duplicate quote.

Confirmed reproducible on a freshly recreated database (not an
artefact of earlier partial/failed import attempts): re-running
=marketdata import --file
external/ore/examples/Legacy/Example_56/Input/market.txt --source
ore.reference= against a clean Barclays-provisioned tenant hits the
exact same error, same key shape.

By contrast, =market.txt=/=fixings.txt= import against two smaller
ORE example vintages succeeds cleanly, including real fixings
import, which rules out fixings import itself as a suspect and
narrows this to =market.txt='s larger/denser quote sets specifically:
- =external/ore/examples/Academy/TA002_IR_Swap= (market + fixings):
  =✓ Imported 4 series, 52 observation(s), 0 fixing(s)= — this
  vintage's =fixings.txt= is present but empty, so the =0= is
  expected, not a fixings bug.
- =external/ore/examples/MarketRisk/Input/HistSimVar= (market +
  fixings): =✓ Imported 217 series, 7172 observation(s), 904
  fixing(s)= — real, non-empty fixings imported correctly. This is
  good evidence fixing import itself works; the collision is
  specific to =market.txt='s series/observation decomposition, not to
  fixings handling.

* Decisions

- Corrected the original hypothesis: this was never a
  =series_key_registry= decomposition collision — the file itself
  contains genuine raw duplicate =(date, key)= pairs with differing
  values. Recording this here so the capture's original "suspect
  qualifier_depth" framing isn't mistaken for what actually happened.
- Fix lives in the importer (=ores.ore.core::market::parse_market_data=/
  =parse_fixings=), not =import_service=, per explicit steer:
  duplicate handling is a parsing concern, reusable by any caller —
  not something specific to how the service happens to persist the
  result.
- Duplicate handling is a policy (=duplicate_policy::warn= / =error=),
  not a hardcoded behaviour: last-line-wins de-duplication always
  happens (matches how a sequential loader would naturally behave),
  but the caller decides whether repeats are reported as warnings
  (import proceeds) or errors (affected content is not persisted).
  Default is =warn=, since real ORE example data contains repeats and
  a strict default would make this vintage permanently unimportable.
- Exposing =duplicates_are_errors= as an =ores.shell= flag is
  deliberately out of scope here — the request field exists and
  works over NATS, but wiring a =--strict= flag through the shell
  command is left as follow-up since nothing currently needs it.

** Why

Surfaced while testing the new =ores.shell= =marketdata import=
command (task E10830F5-1596-4748-99BB-41B8057CEFA4) against the full
file rather than a curated subset. Prior testing of the import
mechanism ([[id:32D9596A-ADB6-4ECE-9307-025B58C4C82A][Review and extend ores.marketdata import for ORE reference vintages]])
likely only exercised a handful of curated quotes (EURUSD/GBPUSD
spot), not the full file, so this gap went unnoticed. Blocks a clean
full-file import of the 2016-02-05 ORE reference vintage.
