* When to use this skill

When implementation of a feature or task is otherwise complete and it's
time to verify it before raising a PR: a full build, the full test
suite, schema/docs validation, a fresh database, the Barclays Plc
tenant, a written test scenario, and a client handed to the user to
execute it.

* How to use this skill

Run [[id:BE325E3E-1BAB-4D9D-964E-78333CDB97D7][How do I test a feature after implementation?]] end to end:

1. Build the affected targets, then run the full test suite (=rat=)
   — treat any failure as a blocker.
2. Validate SQL schemas, and codegen docs if the feature is a codegen
   entity.
3. Recreate the database from scratch so the feature is exercised
   against a clean load.
4. Provision the Barclays Plc tenant (default; ask if a different
   tenant fits better).
5. Create a =test_scenario= doc for the feature. Set *Target dialog* to
   the dialog class(es) *and* the menu path to reach them — a bare
   class name leaves the tester hunting through menus. Give every
   client its own leading connect/login step sourced from the
   provisioning script's credentials — never hardcode or guess them,
   and never assume a second client instance inherits the first one's
   session. For an entity/CRUD feature, include the
   Create/Read/Update/Delete/History/Eventing test plan the recipe
   lays out; the eventing step needs a second client instance to
   actually exercise the NATS relay, not just the repository layer
   ctest already covers.
6. Start the Qt client with =--open-scenario= pointing at the new
   scenario (absolute path) and ask the user to execute it — do not
   run the UI steps yourself.
