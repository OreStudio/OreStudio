* What

Andrej Karpathy proposes an "LLM Wiki" pattern: a persistent, self-compounding
markdown knowledge base maintained by an LLM, structured in three layers —
immutable *raw sources*, an LLM-maintained *wiki* (entity pages, topic
summaries, cross-references, syntheses), and a *schema* config defining
conventions and workflows. It runs on three operations: *ingest* (process one
source at a time, updating 10–15 related pages and flagging contradictions),
*query* (search pages, synthesise an answer, file valuable results back as new
pages), and *lint* (health-check for contradictions, orphans, stale claims and
missing cross-references). Two special files anchor navigation: =index.md=
(catalogue by category) and =log.md= (chronological ingest/query record).

There is a lot of overlap with compass: the org-roam doc graph (entity/knowledge
pages + cross-links), =compass index= / search, the index docs, captures, and
the journal (≈ log.md). We should mine the gist for ideas worth folding into
compass — concretely: a compounding *ingest* operation that updates many docs
per source rather than just filing one; a *lint* command for graph health
(orphans, stale claims, missing links — note we already have an orphan-docs
story in Sprint 21); and query-results-file-back-as-docs.

* Why

Compass already implements much of this pattern by accident; Karpathy's framing
makes the missing pieces explicit (compounding ingest, automated lint, query
write-back). Mining it could sharpen compass's knowledge-maintenance story and
surface concrete, low-cost enhancements that fit the existing graph + index +
journal substrate.
