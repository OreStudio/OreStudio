* Plan

Linked =ores.orgmode.lib= into =ores.qt.api= (both the shared lib and,
for the headless-friendly test target, the three new =.cpp= files
compiled in directly — same pattern already used for
=parse_trade_instrument.cpp=/=HierarchyModelBuilder.cpp= to avoid
pulling in Qt Widgets/a display environment just to test Qt6::Core-only
logic).

Three new pieces, all in =ores.qt.api=:

- *=OrgDocRenderer=*: turns an =ores.orgmode= document into HTML. Hit a
  real bug during testing — doing the four inline-marker substitutions
  (bold/italic/code/verbatim) as four sequential
  find-and-replace passes corrupts the output, because a tag emitted
  by an earlier pass can contain a character a later pass's regex
  matches on (e.g. the =/= inside a =</b>= closing tag gets consumed by
  the italic pattern). Fixed with a single left-to-right scan over the
  *original* text using one alternation regex, escaping plain spans
  and emitting tags as we go — never re-scanning already-emitted HTML.
  Caught by a dedicated regression test, not by inspection.
- *=TestScenarioResultsWriter=*: a targeted text splice, not a
  round-trip parse-then-reserialize — find each heading's line span
  (up to the next same-or-shallower heading, or EOF) and replace just
  that span, so everything else in the file survives byte-identical.
  Deliberately not built into =ores.orgmode= itself (per the story's
  decision) since no other consumer needs this narrow, scenario-
  specific capability.
- *=EnvironmentMetadata=*: branch/commit via shelling out to =git
  rev-parse= (best-effort — empty fields, not an error, if =git= isn't
  on PATH or it's not a repo), worktree identity from the
  =ORES_CHECKOUT_LABEL= environment variable the shell/client launchers
  already set (confirmed via =ores.qt.application/src/main.cpp=, which
  reads the same variable for the instance-label feature).

Also addressed a requirement raised mid-task: some scenarios need
several running clients at once (e.g. verifying a NATS notification
propagates to a second instance). Rather than change anything in
=ores.orgmode= or the writer's core shape, extended the
=test_scenario= template (from the prior task) with an optional
=Clients= field in =Scenario Info= and a documented convention
(comment, hidden from site export) for structuring =* Steps= as one
sub-heading per client when needed — ores.orgmode's existing generic
heading-nesting already handles this with no parser changes. The
writer's step-results table gains a =Client= column only when at least
one step actually sets one, so single-client scenarios (the common
case) stay exactly as simple as before.
