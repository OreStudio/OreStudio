#+title: How do I add a domain service?
#+description: Add a NATS domain service by editing the single service registry (a codegen model) and regenerating; the registry is the one source for service_vars.sh, SQL users/roles/grants, and the per-service .env credentials.
#+type: recipe
#+level: cross
#+filetags: :codegen:services:registry:database:recipe:
#+created: 2026-06-02
#+updated: 2026-07-12

The list of NATS domain services has a *single source of truth*: the
service registry, a [[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]] model. Everything else is derived
from it — never hand-maintain a service list anywhere else, and never
infer the service set from the [[proj:projects][=projects/=]] directory layout.

* Question

How do I add (or remove) a NATS domain service so that its database
user, roles, grants, generated shell vars, and =.env= credentials are
all created consistently?

* Answer

1. /Edit the registry/ — the one authored source, a literate org model:

   #+begin_src sh :results verbatim
   $EDITOR projects/modeling/service_registry.org
   #+end_src

   Add (or remove) a service as a top-level heading (=* <name>=) with the
   per-service properties (=psql_var=, =env_key=, =iam_role=,
   =description=, =email=) and its =DML prefixes= (the =ores_<name>_*=
   tables it owns); =Select prefixes= for cross-service reads.

2. /Regenerate/ the artefacts the registry drives via the
   =service-registry= codegen profile (see [[id:20D0A77D-E0C0-492E-B0EC-45C7638DD220][How do I run codegen?]] for the
   current invocation). This (re)writes [[proj:projects/ores.sql/service_vars.sh][=projects/ores.sql/service_vars.sh=]]
   (=SERVICE_NAMES=) and the SQL for service users, accounts, roles, and
   DB grants.

3. /Regenerate credentials/ so the new service gets its =.env= entries
   (=ORES_<NAME>_SERVICE_DB_USER= / =_PASSWORD= / =_DATABASE=):

   #+begin_src sh :results verbatim
   ./projects/ores.compass/compass.sh env configure --preset <preset> -y
   #+end_src

4. /Recreate the database/ to provision the new role and grants:

   #+begin_src sh :results verbatim
   ./projects/ores.compass/compass.sh db recreate -y
   #+end_src

5. /Commit/ the registry edit and all regenerated files together.

* Who reads the registry

| Consumer | How |
|----------+-----|
| [[proj:projects/ores.sql/service_vars.sh][=projects/ores.sql/service_vars.sh=]] (=SERVICE_NAMES=) | generated by the =service-registry= profile |
| SQL service users / accounts / roles / grants | generated by the same profile (4 templates) |
| =compass env configure= | reads the registry to emit per-service =.env= credentials |
| [[proj:projects/ores.lisp/src/ores-prodigy.el][=projects/ores.lisp/src/ores-prodigy.el=]] | derives the launch list from =ORES_*_SERVICE_DB_USER= in =.env= (downstream of =compass env configure=) |

=wt= is intentionally /not/ in the registry — it uses =rw_role= /
=ORES_WT_DB_*= naming rather than the service-role naming.

* Script

Generation: =./compass.sh codegen generate --model projects/modeling/service_registry.org --profile service-registry=.
Credentials: =compass env configure=. Provisioning: =compass db recreate=.

* Tested by

Manual: after step 2, =git diff projects/ores.sql/service_vars.sh= shows
the new =SERVICE_NAMES= entry; after steps 3–4 the new
=ORES_<NAME>_SERVICE_DB_*= vars exist and =compass db recreate= reports
no missing-password errors. CI exercises the same path in the canary
"Setup Database" step.
