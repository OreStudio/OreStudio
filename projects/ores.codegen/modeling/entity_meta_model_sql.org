This page is a segment of the [[id:BDE309BA-71DA-4A52-AA10-00F17AFDA476][Codegen org-entity meta-model]] hub, covering
the =* SQL= section. It generates from the [[id:E39C0E47-768F-4DE2-8436-08C4A8A06CFA][ores.sql.schema]] facet:
bi-temporal table create/drop, insert triggers, and notify triggers.
The columns it operates on are declared once, shared with C++, on the
[[id:0BD99407-1CBE-469B-B2D3-1C0AE3AA18DE][Keys and columns]] page — this page is only the SQL-specific knobs
layered on top.

* Flags

The section always opens with a =** Flags= drawer:

#+begin_src org
,* SQL
,** Flags
,:PROPERTIES:
,:tablename: ores_refdata_books_tbl
,:END:
#+end_src

| Property        | Required | Meaning                                          |
|-----------------+----------+-----------------------------------------------------|
| =:tablename:=   | yes      | The generated table's name. Convention: =<product>_<component>_<entity_plural>_tbl=. |

Generates:

#+begin_src sql
create table if not exists "ores_refdata_books_tbl" (
    "id" uuid not null,
    "tenant_id" uuid not null,
    "version" integer not null,
    -- ... columns from Keys and columns ...
);
#+end_src

this =Flags= sub-heading.

* Hierarchy: =has_parent_id=

Some entities form a tree via a self-referencing soft foreign key —
a =book= belongs to a =portfolio=, a =counterparty= can have a parent
=counterparty=. Setting =:has_parent_id: true= tells the SQL facet
to generate a recursive-CTE function that walks that tree, so
consumers don't each hand-write their own recursive query.

| Property                | Required | Meaning                                                    |
|--------------------------+----------+-----------------------------------------------------------|
| =:has_parent_id:=        | no       | Turns the knob on.                                          |
| =:parent_id_column:=     | if set   | The self-referencing FK column name, e.g. =parent_party_id=. |
| =:hierarchy_name_field:= | if set   | An existing column used as the display label in the tree.  |

Model:

#+begin_src org
,* SQL
,** Flags
,:PROPERTIES:
,:tablename:            ores_refdata_counterparties_tbl
,:has_parent_id:        true
,:parent_id_column:     parent_counterparty_id
,:hierarchy_name_field: full_name
,:END:
#+end_src

Generates a
=<product>_<component>_<entity_plural>_hierarchy_fn(p_tenant_id, p_root_id, p_from_root)=
recursive-CTE function returning flat =(id, parent_id, name)= rows —
the subtree rooted at =p_root_id= when =p_from_root= is false, or
(resolving up to the real root first) the whole tenant tree when
true. Consumers turn the flat rows into a tree client-side via the
shared, hand-written =ores::utility::domain::hierarchy_node=/
=build_tree()= — that assembly step is entity-agnostic generic logic,
not per-entity template output, so it is not itself generated.

Two things this knob does *not* do, both still hand-written today:
the =parent_<entity>_id= column itself and its insert-trigger
validation. An entity's SQL file has to be fully migrated onto the
unified template before that becomes safe to generate (hand-written
logic like codename generation currently has no paste-block coverage
on the SQL side — see =Commission: party='s "Bring party under
codegen" task). The hierarchy function composes independently of
that column either way.

* Paste block: extra drop statements
:PROPERTIES:
:ID: 5E47F108-1350-4540-B3C2-E83DD5379B2D
:END:

See [[id:40AD7EFB-AACE-472F-B767-8FD3FF2F5C03][Paste blocks]] for the general mechanism this section assumes.

Additional =DROP= statements inserted into the generated
=<component>_<entity_plural>_drop.sql=, just before =drop table if
exists=. Use this when the entity's companion =_table.org= model
generates database objects (e.g. a validate function) that the
entity template itself has no way to know about. If no entity
implements this kind, the marker collapses to nothing.

Template usage (=sql_schema_domain_entity_drop.mustache=):

#+begin_src text
drop function if exists {{product}}_{{component}}_{{entity_plural}}_insert_fn;
<<paste:5E47F108-1350-4540-B3C2-E83DD5379B2D>>
drop table if exists "{{product}}_{{component}}_{{entity_plural}}_tbl";
#+end_src

Entity attachment (=** SQL > ** Extra drops=), expecting the =body=
block name:

#+begin_src org
,** Extra drops

,#+begin_src sql :name body :implements 5E47F108-1350-4540-B3C2-E83DD5379B2D
drop function if exists ores_refdata_validate_rounding_type_fn;
,#+end_src
#+end_src

Multiple =drop= statements can appear in one block or across several;
codegen concatenates them in order with a blank line between blocks.
