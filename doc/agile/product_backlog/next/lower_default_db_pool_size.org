#+title: Lower default per-service DB connection pool size
#+description: Reduce the hardcoded pool_size (4) in the service-app codegen template to shrink each environment's steady-state Postgres connection footprint.
#+type: capture
#+level: cross
#+filetags: :codegen:database:postgres:ops:
#+created: 2026-07-03
#+updated: 2026-07-11

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *next* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

The per-service DB connection pool size is hardcoded to =pool_size = 4=
in the =component-service= codegen archetype
(=projects/ores.codegen/library/templates/ores.cpp.service-app.application_impl.org=,
tangled to
=projects/ores.codegen/library/templates/cpp_service_app_application.cpp.mustache=),
and generated identically into all 17 domain services' =application.cpp=
(=ores.refdata=, =ores.trading=, =ores.iam=, =ores.dq=, etc). Lower the
literal to a smaller default (e.g. =2=), re-tangle with
=compass build --direct tangle_codegen_templates=, then regenerate each
service with =codegen.sh generate --model
projects/ores.<component>/service/modeling/component_overview.org
--profile component-service= and rebuild.

* Why

Each running environment holds a steady-state connection floor of
roughly (services × pool_size) Postgres connections regardless of
actual load — about 80 connections/env at pool_size=4. When several
ORE Studio dev environments run concurrently (observed ~10 worktrees
in the fleet), this floor multiplies fast and exhausts
=max_connections=, surfacing as "remaining connection slots are
reserved for roles with the SUPERUSER attribute" during ordinary
service startup or synthetic-data workflow runs. Halving pool_size to
2 roughly halves that floor (~80 -> ~40/env) with no code beyond the
template change, buying headroom without needing to raise
=max_connections= as aggressively on a host that's already memory
constrained (observed 2.9GB free / 16GB swap in use).
