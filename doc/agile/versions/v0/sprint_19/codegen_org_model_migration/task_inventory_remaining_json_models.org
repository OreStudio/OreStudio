* Goal

After the per-component domain-entity sweep merges (PR #989), the
target end-state for =projects/ores.codegen/models/= is *empty*:
every JSON model becomes a literate =.org= file under the appropriate
component's =modeling/= directory, or is retired entirely. This task
takes inventory of every remaining JSON file, groups them by kind,
decides each kind's org-mode shape (or retirement), and writes the
follow-on per-kind migration tasks that close out =models/=.

The deliverable is a written inventory + sequenced plan committed
to the story. Actual conversion happens in the follow-on tasks this
one scaffolds.

* Acceptance

- Every file currently under =projects/ores.codegen/models/= is
  catalogued under exactly one of these dispositions:
  - *Migrate.* Has an org-mode counterpart in this story. The
    inventory names the kind, the org section shape, and the
    follow-on task that owns the conversion.
  - *Retire.* No longer needed (orphan / superseded). The inventory
    explains why and the deletion lands in a follow-on cleanup task.
  - *Out of scope.* Stays JSON (e.g. third-party data, raw fixtures).
    The inventory states why each one is kept and where it lives.
- Per-kind migration tasks scaffolded for each *Migrate* group, each

* Plan

** 1. Enumerate

#+begin_src sh :results verbatim
find projects/ores.codegen/models -maxdepth 3 -name '*.json' | sort
#+end_src

For each file, capture: path, suffix pattern, top-level JSON shape
(domain_entity / table / junction / enum / field_group / component /
service / other), whether codegen actually reads it (search for the
filename / pattern in =projects/ores.codegen/src/=), and the
templates that consume it.

** 2. Group by kind

Visible kinds today (from a quick scan of the tree at the time of
writing):

- =*_table.json= — SQL-only lookup tables (refdata, several others).
- =*_entity.json= — non-domain lookup-entity models.
- =*_field_group.json= — trading field groups composed into trade
  variants.
- =*_junction.json= — junction tables for many-to-many relationships.
- =*_enum.json= — pure enum models.
- =ores_*_component.json= — component manifests (one per package).
- Per-package =service.json= / registry models under
  =models/<package>.service/=.
- Standalone files (=tags.json=, =plantuml_er_model.json=,
  =profiles.json=, ...).

For each kind, decide:
- *Migrate to org.* What does the org file look like, where does it
  live, what =#+type:= identifies it, does it need a converter
  extension?
- *Retire.* Is the model still consumed? If not, delete.
- *Out of scope.* Some kinds are third-party data or generator
  fixtures and can stay JSON for now.

** 3. Sequence follow-on tasks

One task per migrate-able kind, ordered by:
1. Number of files in the kind (small kinds first as warm-ups).
2. Whether new converter / org_loader work is needed (no-converter
   tasks land first).
3. Cross-kind dependencies (e.g. junctions reference the
   domain-entities already migrated).

A final cleanup task removes =projects/ores.codegen/models/=
entirely once the per-kind tasks all merge.
