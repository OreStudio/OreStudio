* Result

Applied two fixes to =refdata_party_types_create.sql= (PR [[https://github.com/OreStudio/OreStudio/pull/1310][#1310]]):

1. Added =security definer set search_path = public, pg_temp= to both
   =ores_refdata_party_types_insert_fn= (trigger function) and
   =ores_refdata_validate_party_type_fn= (validation function), preventing
   search-path injection attacks.

2. Tightened the bootstrap guard from a bare =select 1 ... limit 1= to
   filter =tenant_id = ores_utility_system_tenant_id_fn() and valid_to =
   ores_utility_infinity_timestamp_fn()= (active rows for the system tenant
   only), preventing soft-deleted historical rows from satisfying the guard
   and causing spurious "must be one of: (empty list)" validation errors
   during bootstrap.

Database recreated cleanly after both fixes. Two review observations were
acknowledged but not actioned: style placement is a template-level concern
(deferred to reconciliation pass); broader sweep is covered by the
commissioning plan applied per entity.
