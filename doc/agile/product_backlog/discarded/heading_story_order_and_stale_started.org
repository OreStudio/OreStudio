Both gaps described are addressed.

* What

Two related gaps in =compass heading=. First, when the active story has multiple BACKLOG tasks,
heading scores them freely and can surface task N+3 while N, N+1, N+2 are still untouched —
it does not prefer the first BACKLOG entry in story table order. Second, heading has no mechanism
to detect that a STARTED task's PR has merged and the task is effectively done; the entry
persists as a ghost candidate at reduced score until someone runs =compass task done= manually.
Improvements: (1) when ranking tasks within the active story, prefer the first BACKLOG row in
table order rather than open scoring; (2) after =compass pr merge=, prompt or auto-invoke
=compass task done= on any STARTED task whose branch was just deleted.

* Why

Developers following the story sequence expect heading to guide them through tasks top-to-bottom.
Free scoring breaks that expectation and forces the user to cross-check the story table manually.
The ghost STARTED entry also inflates the in-flight count and can mislead sprint audits.
