(=load_profiles()=, =_load_profiles_from_org()=, =_parse_facet_table()=,
plus their exclusive regex helpers) was already deleted by task "Delete
legacy codegen profile system" (id 386BED71-A76B-43EF-8A72-0D19704C5151)
-- grep confirmed zero live callers anywhere, so it was safe to remove
independently of the file/link work below. That task deliberately did
*not* delete =facet_catalogue.org= itself: a grep for inbound =id:=

estimate -- real, separate blast radius that belongs here with its own
content-porting precondition, not folded into a "should be zero diff"
deletion task.

*Precondition — port useful content first (do NOT delete before this):* the
legacy docs carry substantive hand-written prose that the scaffolded ores.*
nodes lack (they only have the brief #+description). Before any deletion,
port each doc's real content to its corresponding new node:
- group doc (e.g. =cpp_group.org=) Summary + shared-conventions prose →
  the technical-space node (=ores.cpp.org=).
- facet doc (e.g. =cpp_qt.org=, =sql_schema.org=) Summary, "The <facet>
  facet" projection prose, and Mustache-incantations notes →
  the facet node (=ores.cpp.qt.org=, =ores.sql.schema.org=).
- =facet_catalogue.org= per-archetype descriptions → already captured as
  =#+description= on the archetype/facet nodes; verify none are lost.
Audit every legacy doc for non-placeholder prose and confirm it has a home
in the graph before removing the source.
