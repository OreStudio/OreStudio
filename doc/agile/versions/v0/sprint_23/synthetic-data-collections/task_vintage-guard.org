* Goal

Both collections' feeds seed their initial spot from imported
reference-vintage data (per the story's Decisions). Before such a feed
starts,
=ores.synthetic= should confirm the data actually exists in
=ores.marketdata= (queryable via =source=/=point_id=/
=observation_datetime=) and fail with a clear, actionable error — "No
vintage data found for source=ore.reference, date=2016-02-05 — run the
reference-data import first" — rather than silently defaulting to a
wrong value or failing deep inside feed-start logic. Provide this two
ways: an implicit check at feed-start, and an on-demand "validate all"
toolbar action in Market Simulator so a user can audit an entire
collection proactively.

* Acceptance

- =fx_spot_generation_config= (schema, domain, JSON/table I/O) carries
  a =vintage_source=/=vintage_date= association so a feed can declare
  which vintage it depends on — the prerequisite this task discovered
  was missing.
- Starting a feed with missing vintage data (either collection — both
  seed initial spot from the same vintage, per the story's Decisions)
  fails with a clear, actionable message naming the missing
  source/date and pointing at the import step.
- A Market Simulator toolbar action validates every configured entry's
  vintage-data availability on demand, surfacing results without
  requiring each feed to be started individually.

* Plan

(Implementation strategy. Written when work starts; key decisions
are distilled into the parent story's =* Decisions= at close, but the
plan itself stays — it is the historical record of what we did.)

Prerequisite gap found while starting this task: neither
=fx_spot_generation_config= nor its parent
=market_data_generation_config=
(=projects/ores.synthetic/api/include/ores.synthetic.api/domain/=)
carries any field associating a feed with the vintage it should be
validated/seeded against. =fx_spot_generation_config= has =ore_key=
(e.g. =FX/RATE/EUR/USD=) and a manually-entered =gmm_initial_price=,
but no =vintage_source= (e.g. =ore.reference=) or =vintage_date=
(e.g. =2016-02-05=). Without this, there is nothing for a guard to
query — =ores_marketdata_market_observations_tbl= is keyed on
=(series, observation_datetime, point_id)=, and the feed config has no
=observation_datetime= to look up. This must be closed before the
guard logic itself can be written:

1. Add =vintage_source= and =vintage_date= columns to
   =fx_spot_generation_config= (SQL schema, domain struct, JSON I/O,
   table I/O — via codegen, see
   [[id:5B670808-D28B-4818-A2FA-EA5220D35E8D][How do I add a doc with compass?]]-adjacent =codegen-add-sql-schema=/
   =code-add-domain-type= conventions) so each feed declares which
   vintage it depends on. Populate them for any existing configs (none
   yet — =Seed Basic and Realistic dataset bundles= is still BACKLOG
   and depends on this task).
2. Implicit check: before =fx_spot_feed= starts, query
   =ores_marketdata_market_observations_tbl= for
   =(source = vintage_source, observation_datetime = vintage_date,
   point_id = 'SPOT')= matching the feed's =ore_key= series. Fail with
   the specified actionable message if absent.
3. On-demand check: Market Simulator toolbar action that iterates all
   configured feeds (enabled or not) and runs the same query per feed,
   surfacing pass/fail per entry rather than requiring each to start.
4. Confirm with the story owner whether =vintage_source=/
   =vintage_date= belong on =fx_spot_generation_config= specifically
   or should be hoisted to the parent =market_data_generation_config=
   (both collections currently seed from the *same* vintage per the
   story's Decisions, so a per-config field may be redundant — but
   per-config keeps the door open for mixed-vintage collections
   later). Default to per-config unless that proves awkward.

Steps 1-3 done:

- Step 1: =vintage_source=/=vintage_date= added to
  =fx_spot_generation_config= via codegen (=ores.cpp= + hand-applied
  =ores.sql.schema= column/check additions — the wholesale SQL
  regeneration would have clobbered pre-existing custom validation-
  function logic unrelated to this change, so that part was applied
  by hand instead). Went with per-config (not hoisted to the parent),
  per the default in point 4 above — not revisited since nothing made
  it awkward.
- Step 2: =market_data_client= gained =find_series=/
  =list_observations= (no dedicated lookup-by-key request existed).
  =feed_controller::start()= now takes optional =vintage_source=/
  =vintage_date= and checks =market_observation= before spawning the
  feed, returning a new =vintage_data_missing= result with the
  specified actionable message; =start_market_feed_config_request=
  carries the two new fields, populated in
  =MarketSimulatorWindow::startPairsAsync= from the
  =fx_spot_generation_config=.
- Step 3: new =validate_market_feed_config_request/response= (subject
  =marketdata.v1.market_feed_configs.validate=) exposes the same check
  standalone; a new "Validate Vintage" Market Simulator toolbar action
  iterates every configured FX rate and shows pass/fail per entry.

Verified: =ores.synthetic.core.tests= (59 cases) and

Example_56's =market.txt= stores GBP/USD under the reversed key
=FX/RATE/USD/GBP= (confirmed by triangulation: EUR/GBP÷EUR/USD =
0.717157, and the given value 1.394610179594994 is its reciprocal to
6 significant figures) — a real upstream vendor bug, not an import
decomposition bug. Patched around it for now with a direct SQL
special-case and a one-off bitemporal update of the already-seeded
=synthetic.gbpusd= row; the generic fix (detect and auto-correct any
such reversed key at import time, using refdata's =currency_pair=
canonical order) is tracked separately as
[[id:F3CB17FB-6472-4DEB-BC5B-EC764C1368F8][Detect and correct reversed FX spot quotes during market data import]].

* Result

Delivered as planned: =fx_spot_generation_config= carries
=price_source=/=vintage_source=/=vintage_date=; =feed_controller=
checks vintage-data availability (delegated, tenant-scoped) before a
feed starts, overriding the manually-entered spot with the real
imported observation on success; a "Validate Vintage" Market Simulator
toolbar action runs the same check on demand across every configured
rate. The FX spot rate editor gained a Fixed/From-vintage-data radio
selector (two groups stacked vertically, each disabling the other's
fields — iterated twice on the layout per review feedback) and a
"Browse available…" picker so a user can pick a real (source, date)
vintage instead of typing one blind (though it doesn't yet know about
the reversed-key quirk below — see the follow-up task).

Also fixed two independent stale-Qt-plugin-vs-exe ABI-mismatch
crashes, permanently closed via =ores.qt.exe='s new
=add_dependencies()= on every plugin lib (previously plugins were only
=QPluginLoader=-loaded, never a CMake dependency of the exe).

Found along the way, not part of the original scope: Legacy/
Example_56's =market.txt= files GBP/USD under the reversed key
=FX/RATE/USD/GBP= (confirmed via EUR-cross triangulation — see the
story's Decisions). Patched around it for now with a one-off SQL
special-case in the DQ-seeding function; the generic fix is
[[id:F3CB17FB-6472-4DEB-BC5B-EC764C1368F8][Detect and correct reversed FX spot quotes during market data import]],
raised as its own task and started next.
