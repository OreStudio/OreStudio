#+title: CLI Trades Import / Export
#+options: <:nil c:nil ^:nil d:nil date:nil author:nil toc:t html-postamble:nil

* Overview

=ores.cli= currently has no support for actual trade data — only
reference-data entities (day count types, leg types, etc.) under the
=trading= domain. This plan adds two new trade-level operations to the
CLI:

- *Export*: fetch a book's trades (with instruments) from the server
  and write a standards-format ORE XML portfolio file.
- *Import*: read a local ORE XML portfolio file and load it into a
  target book via the server-side import saga.

Both operations follow the existing entity-parser pattern established
by currencies/countries and the trading reference-data entities.

* Architecture overview

The CLI dispatches commands as:

#+begin_example
  ores.cli <domain> <entity> <operation> [flags]
  ores.cli trading trades export --book <uuid> --output portfolio.xml
  ores.cli trading trades import --file portfolio.xml --book <uuid>
#+end_example

The =trading_parser= routes the =trades= entity to a new
=trades_parser= that parses import/export options and returns a
populated =config::options=. =application.cpp= then calls the
appropriate action handler which communicates with the server over NATS.

*** 1b — Options structs

*=include/ores.cli/config/export_trades_options.hpp=* (new):

#+begin_src cpp
struct export_trades_options {
    std::string book_id;      ///< UUID of the book to export
    std::string output_file;  ///< Path to write the ORE XML portfolio
};
#+end_src

*=include/ores.cli/config/import_trades_options.hpp=* (new):

#+begin_src cpp
struct import_trades_options {
    std::string file;         ///< Path to the ORE XML portfolio
    std::string book_id;      ///< Existing target book UUID
    std::string trade_date;   ///< Override trade date (ISO 8601, optional)
    std::string activity_type_code; ///< Override lifecycle event (optional)
};
#+end_src

Add both to the =add_options= variant in =config/options.hpp=
(alongside the existing per-entity add structs), or introduce a
separate =import_options= / =export_options= variant — match whatever
pattern the existing options struct uses.

** Phase 2 — Trades entity parser

Create =include/ores.cli/config/entity_parsers/trades_parser.hpp= and
=src/config/entity_parsers/trades_parser.cpp=.

The parser handles two operations:

| Operation | Flags |
|-----------+-------|
| =export=  | =--book <uuid>= (required), =--output <path>= (required) |
| =import=  | =--file <path>= (required), =--book <uuid>= (required), =--trade-date=, =--activity-type= (optional) |

Pattern: mirror =currencies_parser.cpp= — parse with
=boost::program_options=, validate required flags, populate and return
=config::options= with the appropriate entity and operation fields set.

** Phase 3 — Trading domain router extension

Extend =src/config/domain_parsers/trading_parser.cpp=:

1. Add =trades= to the entity name constants.
2. Add =trades= to =validate_entity_name=.
3. Add a =trades= branch in =handle_trading_command= that delegates to
   =entity_parsers::handle_trades_command=.
4. Update =print_domain_help= to list the new entity.

** Phase 4 — Application handlers

Extend =src/app/application.cpp= with two new private methods:

*** 4a — Export handler

#+begin_src
void application::export_trades(const config::export_trades_options& opts)
#+end_src

Algorithm:

1. Send =export_portfolio_request= with =node_id = opts.book_id=.
2. Receive =export_portfolio_response=; check =success=.
3. Pass =response.items= through =ores::ore::xml::exporter::export_portfolio=
   to produce an ORE XML string.
4. Write the string to =opts.output_file=.
5. Print trade count and output path to stdout.

The exporter already exists in =ores.ore=; =ores.cli= needs to add
=ores.ore= as a dependency in its =CMakeLists.txt= if not already
present.

*** 4b — Import handler

#+begin_src
void application::import_trades(const config::import_trades_options& opts)
#+end_src

Algorithm:

1. Read the local ORE XML file into a string.
2. Upload it to object storage via the existing storage client, using a
   freshly generated UUID as the request ID / storage key root. The
   storage path convention is =ore-import/<request_id>/portfolio.xml=.
3. Build =ore::planner::import_choices= with:
   - =existing_target_book_id= = =opts.book_id=
   - =defaults.trade_date= / =defaults.activity_type_code= from opts
   - =currency_mode= = =missing_only= (safe default)
   - =party_id= from the authenticated session
4. Serialise choices to JSON.
5. Send =ore_import_request= with =request_id= and =import_choices_json=.
6. Receive =ore_import_response=; print success/error summary including
   any =item_errors=.

Wire =export_trades= / =import_trades= into =application::run()= via
the same dispatch pattern as existing import/export actions.

** Phase 5 — CMakeLists and dependencies

- Add =ores.ore= to the =target_link_libraries= of =ores.cli= if the
  exporter is not already transitively available.
- Add =ores.ore.api= if =ore_import_request= is not already reachable.

Check with a dry-run configure; add only what is missing.

** Phase 6 — Manual smoke test

#+begin_src sh
# Export a book to XML
ores.cli trading trades export \
  --book <book-uuid> \
  --output /tmp/portfolio_export.xml

# Verify file looks like ORE XML
head -5 /tmp/portfolio_export.xml

# Import it back to a different book
ores.cli trading trades import \
  --file /tmp/portfolio_export.xml \
  --book <target-book-uuid>
#+end_src

Confirm trade count printed by export matches trade count confirmed by
import response.
