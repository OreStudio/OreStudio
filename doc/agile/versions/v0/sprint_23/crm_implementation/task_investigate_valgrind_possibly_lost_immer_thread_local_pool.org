"Possibly lost" (not "definitely lost") pointing at
=thread_local_free_list_storage= is the classic signature of a
thread-local allocator pool: valgrind flags memory still referenced only
via a thread-local pointer as possibly-lost when the owning thread has
exited without the pool being explicitly torn down, even though the
pool itself would still be reachable/reused had the thread continued.
Need to confirm that's what's happening here (worker threads in
=rate_engine_tests.cpp= are short-lived, matching the pattern) rather
than a genuine leak in =rate_engine=/=vertex_state='s own use of
=immer::vector_transient=.

* Acceptance

- Root cause determined: either (a) confirmed thread-local-pool false
  positive -- documented and suppressed via a targeted valgrind
  suppression entry (not a blanket immer suppression), or (b) a genuine
  leak in =rate_engine=/=vertex_state='s =immer= usage, fixed.
- If (a), the suppression is scoped tightly enough that it would not
  hide a real leak introduced later in the same call path.
- Verified clean (or only the suppressed record) on a valgrind run of
  =ores.analytics.quant.tests= after the fix/suppression.

* Plan

- Reproduce locally: build =ores.analytics.quant.tests= and run it
  directly under =valgrind --leak-check=full --show-reachable=yes=
  (both filtered to =[rate_engine]= and unfiltered), rather than the
  full nightly =CTest.cmake= dashboard script -- much faster iteration
  for a single component.
- Check the =LEAK SUMMARY=: every run showed *0 bytes definitely lost,
  0 indirectly lost* -- only "possibly lost", all funnelling through
  the same =immer= thread-local free-list allocator chain
  (=cpp_heap::allocate -> debug_size_heap::allocate ->
  free_list_heap::allocate -> unsafe_free_list_heap_impl<thread_local_free_list_storage>::allocate
  -> split_heap::allocate=), for both single-threaded and worker-thread
  test cases. That's the confirming signal: a genuine leak in
  =rate_engine=/=vertex_state='s own code wouldn't route through the
  allocator's own internal pool-allocate frames for every single
  occurrence, and definitely-lost/indirectly-lost would be nonzero.
- Confirmed hypothesis (a): thread-local-allocator-pool false positive,
  not a real leak.
- Found =build/valgrind/custom.supp= (a real tracked source file, not
  build output -- only =build/output/= is gitignored) is already wired
  as =CTEST_MEMORYCHECK_SUPPRESSIONS_FILE= in =CTest.cmake= and used by
  nightly's =with_memcheck=1= dashboard run, with an existing catalogue
  of scoped suppressions (Boost.Log, OpenSSL, Catch2, glibc TLS, etc.)
  to follow the same style/rigour for.
- Added one new targeted suppression entry, matched on the immer
  allocator's own 5-frame =allocate()= chain for this specific
  280-byte/1024-slot pool instantiation (the one =rate_engine='s
  =vertex_state= vector uses), with =match-leak-kinds: possible= only
  -- deliberately *not* a blanket =immer::*= or =possibly lost=
  suppression, so a genuine definite leak in the same call path (or a
  possibly-lost report through a different immer allocation site)
  still surfaces normally.
- Verified: re-ran valgrind with =--suppressions=build/valgrind/custom.supp=
  against the full =ores.analytics.quant.tests= suite (147 test cases) --
  0 definitely/indirectly/possibly lost, all 4 hits of the new
  suppression entry accounted for as "suppressed".
