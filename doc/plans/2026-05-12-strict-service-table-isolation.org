#+TITLE: Strict Service Table Isolation
#+SUBTITLE: Mandatory role drops + zero cross-component DB access
#+DATE: 2026-05-12
#+AUTHOR: ORE Studio Team
#+STATUS: REVISED 2026-05-13
#+OPTIONS: toc:t num:nil

* Context

Two findings in
[[file:../analysis/bootstrap-and-architecture-violations.org][Bootstrap and
Architecture Violations]] motivate this plan:

1. ~recreate_database.sh~ does not drop cluster-level roles, so historical
   group memberships (notably the old broad ~rw_role~ on domain service
   users) silently mask missing per-service GRANTs. The system is therefore
   not reproducible across machines.
2. The IAM, compute, scheduler, reporting, trading, workflow and ORE
   service users today hold cross-component table grants. The Service
   Account RBAC Design
   ([[file:2026-03-28-service-rbac-design.org][2026-03-28-service-rbac-design.org]])
   stipulated that cross-component access should be the exception; in
   practice it has become the norm.

This plan establishes a strict architectural invariant — *a domain service
user holds DB privileges on its own component's tables only* — and lays
out an incremental, low-risk migration to reach it. It also makes role
drops mandatory in ~recreate_database.sh~ so that the invariant cannot
silently regress.

* Revision Note (2026-05-13)

The original Phase 2 strategy proposed per-service tenant cache tables
populated via NATS subscriptions. This was revisited after a thorough
audit of the C++ codebase and SQL trigger definitions.

** Root cause of most cross-component grants

The audit revealed that the overwhelming majority of the cross-component
~SELECT~ grants are not required by application-level queries or FK
constraints. They exist because the shared trigger validation functions
~ores_iam_validate_tenant_fn~ and ~ores_dq_validate_change_reason_fn~ are
defined as ~SECURITY INVOKER~ (the PostgreSQL default). When any service's
INSERT/UPDATE trigger fires these functions, they execute as the calling
session user — which means that user must hold ~SELECT~ on the tables the
functions read (~ores_iam_tenants_tbl~,
~ores_dq_change_reasons_tbl~).

Converting these functions to ~SECURITY DEFINER~ makes them run as their
definer (the DDL owner), eliminating the requirement for every service user
to hold cross-component ~SELECT~. No cache tables, no NATS subscriptions,
no C++ changes required for Phases 2 and 3.

** Grants confirmed as errors

Code audit found no trigger, query, or C++ code justifying:
- ~SELECT ON ores_refdata_parties_tbl~ for ~compute_service_user~
- ~SELECT ON ores_scheduler_*~ (prefix) for ~reporting_service_user~

These will be removed in Phase 1 as part of the grant cleanup.

** Grants confirmed as real application cross-reads

The following grants reflect genuine cross-component coupling that requires
architectural work to remove:
- IAM reads ~ores_variability_system_settings_tbl~ (bootstrap mode, signup
  policy, token settings) — Phase 4
- IAM writes ~ores_refdata_parties~ (system party during provisioning) —
  Phase 4
- Workflow writes ~ores_iam_*~ and ~ores_refdata_parties~ (tenant onboarding
  flow) — Phase 5
- ORE import writes ~ores_workflow_*~ (uses workflow as job queue) — Phase 5

Trading's broad ~SELECT ON ores_refdata_*~ grant exists because trading
triggers contain inline SELECTs against refdata tables (~books~,
~portfolios~, ~counterparties~, ~parties~) for soft-FK validation. These
are also SECURITY INVOKER contexts and will be fixed the same way as the
shared validator functions — by wrapping inline trigger SELECTs in
SECURITY DEFINER functions (Phase 5.2).

* Goal and Invariant

** Invariant (target state)

For every domain service ~S~ with DB user ~ores_<env>_<S>_service~:

1. ~S~ has ~SELECT/INSERT/UPDATE/DELETE~ only on tables matching
   ~ores_<S>_%~ in the ~public~ schema (its own component).
2. ~S~ has ~USAGE/SELECT~ on sequences only for its own tables (default
   privileges).
3. ~S~ has *no* grants on any table prefixed with another component.
4. ~S~ has *no* membership of the broad ~rw_role~ or ~owner_role~. The
   only group membership is ~service_role~ (CONNECT on DB, USAGE on
   schema).
5. *Cross-component validation* (tenant existence, change reason validity,
   refdata entity existence) is enforced by ~SECURITY DEFINER~ functions
   owned by the defining service, so no ~SELECT~ grant on cross-component
   tables is needed by the calling service user.

Equivalently: removing every line in
~create/iam/iam_service_db_grants_create.sql~ except its own
~_ores_grant_dml_fn('ores_<S>_', ...)~ call must not break any service.

** Non-goals

- Splitting ~public~ into per-component PostgreSQL schemas. Today every
  table lives in ~public~ with a ~ores_<component>_~ prefix; moving to
  one schema per component is desirable but orthogonal to this plan and
  carries its own migration cost.
- Removing the C++ link between ~ores.iam.core~ and ~ores.variability.core~
  for *types and configuration shapes*. Only the DB-reading code paths
  must move to NATS or to IAM-owned tables.
- Touching the in-process unit tests that talk directly to a shared
  ~public~ schema as the DDL user.

* Phase 1 — Establish the Invariant and CI Enforcement

*Status: COMPLETE* (committed in ~[sql] Phase 1: annotate cross-component
grants and add invariant validator~)

The service registry JSON carries ~migration_exception: {until, reason}~
on every existing cross-component grant. The codegen validator
~validate_service_registry.py~ enforces that no new unannotated
cross-component grant can be added. A SHA-256 snapshot of the generated
grants file detects out-of-band edits.

** Step 1.1 — Remove the two confirmed-incorrect grants

As part of the Phase 1 grant cleanup, remove:
- ~grant select on ores_refdata_parties_tbl to :compute_service_user~ —
  no code path in compute triggers or C++ reads this table
- ~select _ores_grant_select_fn('ores_scheduler_', :'reporting_service_user')~ —
  ~scheduler_job_id~ in ~ores_reporting_report_definitions_tbl~ is a soft
  UUID column; no trigger or C++ code queries scheduler tables from
  reporting

These can be removed immediately with no application changes.

* Phase 2 — Convert ~ores_iam_validate_tenant_fn~ to SECURITY DEFINER

*Effort: S (≤ 1 week). Risk: Low.*

Today ~ores_iam_validate_tenant_fn~ is ~SECURITY INVOKER~. Every service
table trigger that calls it therefore requires the service user to hold
~SELECT ON ores_iam_tenants_tbl~. Making the function ~SECURITY DEFINER~
causes it to run as its DDL-user definer, removing the requirement from
all 16 service users.

** Step 2.1 — Add SECURITY DEFINER to ~ores_iam_validate_tenant_fn~

In ~projects/ores.sql/create/iam/iam_tenant_functions_create.sql~:

#+begin_src sql
create or replace function ores_iam_validate_tenant_fn(p_tenant_id uuid)
returns uuid
language plpgsql
security definer     -- ← add this
stable
as $$ ... $$;
#+end_src

Add a ~SET search_path = public, pg_temp~ clause to the function definition
to prevent search-path hijacking (standard SECURITY DEFINER hygiene).

** Step 2.2 — Remove the 16 tenant SELECT grants

Remove all 16 instances of:

#+begin_src sql
grant select on ores_iam_tenants_tbl to :<S>_service_user;
#+end_src

from the service registry and regenerate ~iam_service_db_grants_create.sql~.

** Step 2.3 — Update the invariant snapshot

Update ~projects/ores.sql/tests/service_db_grants.sha256~ to the new
file hash after regeneration.

* Phase 3 — Convert ~ores_dq_validate_change_reason_fn~ to SECURITY DEFINER

*Effort: S. Risk: Low.*

Same pattern as Phase 2. ~ores_dq_validate_change_reason_fn~ is
~SECURITY INVOKER~ and SELECTs from ~ores_dq_change_reasons_tbl~. The
four services that call it (scheduler, reporting, trading, compute) each
hold ~SELECT~ grants on the DQ change reason tables for this reason alone.

** Step 3.1 — Add SECURITY DEFINER to ~ores_dq_validate_change_reason_fn~

In ~projects/ores.sql/create/dq/dq_change_reason_functions_create.sql~:

create or replace function ores_dq_validate_change_reason_fn(
    p_tenant_id uuid, p_change_reason_code text)
returns text
language plpgsql
security definer
as $$ ... $$;
#+end_src

** Step 3.2 — Remove the 4+3 DQ change reason SELECT grants

Remove from the service registry:

- ~grant select on ores_dq_change_reason_categories_tbl~ for scheduler,

* Phase 4 — IAM-Specific Cross-Reads

*Effort: M (1-3 weeks). Risk: Medium.*

IAM is the most entangled service. It reaches into *variability* (reads)
and *refdata* (writes).

** 4.1 — IAM-owned settings table (replaces ~ores_variability_system_settings_tbl~)

IAM reads these flags from ~ores_variability_system_settings_tbl~:
~system.bootstrap_mode~, ~system.user_signups~,
~system.signup_requires_authorization~, and JWT token settings.

These are IAM concerns residing in a generic variability table. Move them
to IAM-owned tables:

- ~ores_iam_bootstrap_state_tbl~
- ~ores_iam_token_settings_tbl~
- ~ores_iam_signup_policy_tbl~

Steps:
1. Create the three tables under ~ores_iam_~.
2. Backfill from ~ores_variability_system_settings_tbl~.
3. Update ~bootstrap_mode_service.cpp~, ~auth_handler.hpp~,
   ~account_handler.hpp~ to use repositories on the new tables.
4. Remove the ~grant select on ores_variability_system_settings_tbl~
   from the IAM service grants.
5. Remove the CMake link to ~ores.variability.core~ from
   ~ores.iam.core/src/CMakeLists.txt~ once no IAM source references it.

** 4.2 — Provisioning returns the system_party_id

Modify ~ores_iam_provision_tenant_fn~ to ~RETURNS RECORD~ with
~(tenant_id uuid, system_party_id uuid)~. Update
~bootstrap_handler.hpp~ to consume both and remove the cross-service
~party_repository.read_system_party~ call.

** 4.3 — IAM party cache

IAM's hot-path party reads (~auth_compute_visible_party_ids~,
~auth_lookup_party~) query ~ores_refdata_parties_tbl~ via a DML grant.

Add ~ores_iam_party_cache_tbl~ owned by IAM. Populate at startup from
a refdata snapshot and keep warm via JetStream subscription to
~refdata.v1.parties.changed~. Replace ~refdata::repository::party_repository~
usages in IAM handlers with the local cache. Remove the
~_ores_grant_dml_fn('ores_refdata_parties', :'iam_service_user')~ grant.

* Phase 5 — Other Service-Specific Cross-Reads

*Effort: varies. Risk: Low–High.*

** 5.1 — Fix trading inline trigger refdata SELECTs

Trading table triggers directly SELECT from ~ores_refdata_books_tbl~,
~ores_refdata_portfolios_tbl~, ~ores_refdata_counterparties_tbl~, and
~ores_refdata_parties_tbl~ for soft-FK validation. These are SECURITY
INVOKER contexts requiring the service user to hold the broad
~SELECT ON ores_refdata_*~ prefix grant.

Fix: extract each inline cross-component SELECT into a named
~SECURITY DEFINER~ validation function owned by refdata (e.g.
~ores_refdata_validate_book_fn~, ~ores_refdata_validate_counterparty_fn~).
The trading triggers call the SECURITY DEFINER functions; no ~SELECT ON
ores_refdata_*~ grant is needed.

** 5.2 — Workflow has DML on ~ores_iam_~ and ~ores_refdata_parties~

Workflow orchestrates tenant onboarding by writing IAM accounts and
refdata parties directly. Refactor to call IAM and refdata NATS write APIs
instead (~iam.v1.accounts.create~, ~refdata.v1.parties.create~). This is
potentially a multi-PR effort; bundle with the workflow service milestone.

** 5.3 — ORE Import has DML on ~ores_workflow_~

ORE import uses workflow tables as a job queue. Add NATS write APIs on
workflow (~workflow.v1.runs.create~, ~workflow.v1.steps.append~) and
migrate ORE to use them.

** 5.4 — Synthetic reads everything

The synthetic data generator holds broad SELECT on every domain. It is a
developer-only tool. Two options:

(a) Run as the DDL user — correct model since it needs broad read access.
(b) Keep as a service; accept ~role: "tooling"~ exception permanently.

Recommendation: (a). Either is acceptable.

* Phase 6 — Lock the Invariant

After Phases 0–5:

1. ~create/iam/iam_service_db_grants_create.sql~ contains only one DML
   grant per service.
2. The codegen validator has no remaining ~migration_exception~ entries.
3. The snapshot hash in ~tests/service_db_grants.sha256~ is bumped.
4. The invariant is documented in ~CLAUDE.md~ and ~system_model.org~.

* Bootstrap Concerns Across the Plan

Bootstrap remains the one place where IAM legitimately crosses domain
boundaries. The mechanisms that survive this plan:

1. ~SECURITY DEFINER~ provisioning function continues to handle
   tenant + system-party creation in a single transaction.
2. After Phase 4.2, the function returns ~system_party_id~, so IAM
   never reads ~ores_refdata_parties_tbl~ even during bootstrap.
3. Bootstrap-mode gating remains in IAM's own
   ~ores_iam_bootstrap_state_tbl~ (Phase 4.1).

* Open Questions

1. *Schema-per-component.* Should components move to their own Postgres
   schemas as a follow-on to this plan? It would make GRANTs trivial.
   Recommended: out of scope here; list as natural next step.
2. *Tenant deletion cascade.* Removing cross-component FKs (once any
   remain) requires explicit ~tenant.deleted~ handlers per service.
3. *Trading refdata coupling.* Trading's inline trigger validations against
   refdata represent a view that trading data integrity depends on refdata
   at write time. Evaluate whether SECURITY DEFINER functions are the
   right long-term model or whether trading should cache the lookup sets.
