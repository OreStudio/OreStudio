* Goal

Grant the IAM service's DB role the access it needs to write
=ores_variability_system_settings_tbl= so that
=complete_tenant_provisioning='s cross-domain call to
=variability::service::system_settings_service::set_bootstrap_mode=
stops failing with =permission denied= and =bootstrap_mode= actually
clears to =false= on tenant activation. Decide between (a) a targeted
grant on the specific table in
=iam_service_db_grants_create.sql=/the mustache template that
generates it, or (b) routing the write through
=ores.variability.service= over NATS instead of a same-process
cross-crate repository call using the wrong DB role — prefer (b) if it
is not disproportionate effort, since it avoids widening IAM's DB
footprint into another domain's tables.

- Provisioning a tenant (e.g. re-running the Barclays Plc scenario
  against a fresh DB) no longer logs =permission denied for table
  ores_variability_system_settings_tbl=.

  activation, verified via =ores.shell='s =variability get-setting= (or
  equivalent) against the provisioned tenant.
- No unrelated widening of IAM's DB grants beyond what's needed for
  this write.
