SQL (Structured Query Language) is the standard declarative language
for defining and manipulating relational data — schemas, tables,
constraints, and the queries and procedural functions that operate on
them. ORE Studio runs on PostgreSQL 15+, an open-source relational
database that extends standard SQL with its own procedural language
(=PL/pgSQL=) for functions and triggers, plus extensions the schema
relies on (=pg_trgm=, =btree_gist=, =pgcrypto=).

The SQL [[id:E7B74138-9ADA-4B4B-A983-23B3FDB55F44][technical space]] covers all database artefacts generated by
[[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]]. It contains 3 [[id:B079DA67-8408-4BD0-A93C-D108AC780872][facets]] and approximately 26 [[id:787A7429-71D5-4295-829B-A9B56275B1F8][archetypes]].
Unlike the [[id:5293A5FE-9150-4C0A-92F2-5FE53325FFE0][C++ TS]], the SQL TS has a single implicit [[id:787A7429-71D5-4295-829B-A9B56275B1F8][part]] — SQL files are
not split by compile-time role (there is no header/implementation
distinction in SQL).

The SQL TS projects that live in the repository are placed under
=projects/ores.sql/=. All DDL output is managed by the schema lifecycle
there. For the MASD concepts see [[id:BA0E3273-DC4B-7CD4-447B-06227B7D09A8][MASD]], [[id:787A7429-71D5-4295-829B-A9B56275B1F8][Physical Space]], and [[id:B079DA67-8408-4BD0-A93C-D108AC780872][Facet]].
Group-level literate source: [[id:230D4E62-E2CC-464E-8EFD-E5CF2FA7EB0E][ores.sql]].

* Part

The SQL TS has a single part corresponding to the schema directory:

| Part | Directory | Contains |
|------+-----------+----------|
| =schema/= | =projects/ores.sql/create/= | DDL scripts, trigger scripts, drop scripts |

Population scripts and service setup scripts live under related
directories (=projects/ores.sql/populate/=, =projects/ores.sql/service/=)
but are still part of the same TS.

* Facets and archetypes

** Schema facet

[[id:B079DA67-8408-4BD0-A93C-D108AC780872][Facet]] profile: =--profile sql=. Literate source: [[id:E39C0E47-768F-4DE2-8436-08C4A8A06CFA][ores.sql.schema]].
Modeling reference: [[id:5CB31A0E-4E38-4521-B0EE-E4396C3E2936][SQL facet]].

Generates the DDL lifecycle for an entity: the create table script, the
notify trigger (which fires a =pg_notify= on every mutation, enabling
NATS-based change propagation), and the corresponding drop scripts. The
SRPP is the bi-temporal entity table pattern: every reference-data entity
table has =valid_from= / =valid_to= and =transaction_from= / =transaction_to=
columns, a surrogate primary key, and an auto-notify trigger.

9 archetypes total — see [[id:E39C0E47-768F-4DE2-8436-08C4A8A06CFA][ores.sql.schema]] for the full per-archetype
catalogue (schema create variants, entity drop, notify trigger
create/drop).

** Service facet

[[id:B079DA67-8408-4BD0-A93C-D108AC780872][Facet]] profile: =--profile service-sql=. Literate source: [[id:CD3FF923-9471-428F-BB9E-CB9BE1388A0C][ores.sql.service]].

Generates the PostgreSQL user/permission setup for a service. Each service
that touches the database has a dedicated database user with minimal
privileges — the service facet encodes this SRPP. These are generated once
per service, not once per entity.

4 archetypes total — see [[id:CD3FF923-9471-428F-BB9E-CB9BE1388A0C][ores.sql.service]] for the full per-archetype
catalogue (service users, DB grants, IAM accounts, account roles).

** Populate facet

[[id:B079DA67-8408-4BD0-A93C-D108AC780872][Facet]] profile: =--profile populate=. Literate source: [[id:80890D97-2BDA-4EF7-80F5-C96021A8E537][ores.sql.populate]].

Generates the reference-data seed scripts — the canonical data sets for
entities such as currencies (ISO 4217), countries (ISO 3166), FPML coding
schemes, methodologies, and tags. Unlike the schema and service facets,
the populate facet archetypes are not parameterised per-entity-type; each
archetype encodes a fixed data set. The dataset is driven by an FPML
source and the [[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]] automated route reads FPML XML to derive
the populate scripts.

13 archetypes total — see [[id:80890D97-2BDA-4EF7-80F5-C96021A8E537][ores.sql.populate]] for the full per-archetype
catalogue (batch execute, catalog/country/currency/dataset/flag/
methodology/tag populate, and their supporting functions).

* Component mapping

SQL TS projections land in the following components:

| Facet | Target component |
|-------+------------------|
| Schema | =ores.sql= (=create/= subdirectory) |
| Service | =ores.sql= (=service/= subdirectory) |
| Populate | =ores.sql= (=populate/= subdirectory) |

All SQL artefacts are managed centrally in =ores.sql= rather than
distributed per-entity-group, because the database schema is a shared
resource across all components.
