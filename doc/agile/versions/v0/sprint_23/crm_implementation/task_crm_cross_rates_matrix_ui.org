* Goal

A read-only Qt panel that shows every currently-configured CRM rate for
a party as an NxN cross-rates matrix (currency x currency, direct pairs
and triangulated crosses shown the same way -- the whole point of the
CRM is the caller never needs to know which is which), sourced from the
[[id:88FD9001-F2BA-4061-994C-5EBE8BBD590A][service-wiring task]]'s =marketdata.v1.crm.rates= endpoint. Loosely modelled
on [[file:../../../../../analysis/gemini_cross_rates_matrix.png][a reference mockup]] (grid layout, per-cell rate, base-currency-row /
quote-currency-column headers, connection/status footer) -- not a
pixel-accurate spec, just the shape: a matrix, not a list.

* Acceptance

- A Qt panel/dialog (new =ores.qt.marketdata= widget, alongside the
  existing feed-binding/market-series ones) displaying an NxN grid: rows
  and columns are every currency that appears in any configured pair for
  the current party (derived from the response, not hardcoded), each
  cell showing the rate for that (row, column) pair when one exists in
  the response, blank/dash on the diagonal and for pairs the CRM doesn't
  currently cover -- see the mockup's own dashes on the diagonal and its
  "8 Currencies" footer count as the shape to match, not literal values.
- A visible Reload button that calls =marketdata.v1.crm.rates= on
  click, *plus* a footer "Update Interval" combo (No auto-refresh /
  5s / 10s / 30s / 60s) driving a client-side polling timer -- *on by
  default at 5s*, matching the reference mockup's own default
  behaviour, not an opt-in extra. See the story's =* Decisions= for
  why this doesn't reopen the "never broadcast the derived set"
  decision (that's about server-side push, not client-side polling).
  *No* NATS subscription/push either way.
- Each cell also reflects =rate_status= (fresh/stale/unavailable) from
  the response -- at minimum a visual distinction (e.g. dimmed/greyed)
  for stale or unavailable cells, so the operator can tell a rate is
  live from one that's aged out per the party's =staleness_policy=,
  without needing the raw =as_of= timestamp.
- Party selection: whatever mechanism the existing marketdata Qt panels
  use for picking a party (consistent with =crm_topology_config=/
  =crm_driver_pair=/=crm_enabled_derived_pair='s own tenant+party
  scoping) -- no new party-selection UI pattern invented for this one
  panel.
- Covers *every* pair the response returns, not a fixed/curated subset --
  the grid dimension is however many currencies are actually configured
  for that party (the mockup's G10 filter is a nice-to-have, not
  required for this task).
- Cells are a *fixed size*, not stretched to fill the widget -- the grid
  should not visibly grow/shrink per-cell as the currency count changes
  between reloads (e.g. switching the CRM selector from "majors" to
  "exotics"), matching the mockup's uniform cell sizing.
- Each cell shows the pair code (e.g. "EUR/USD") above the rate, not
  just the bare number, matching the mockup's two-line cells.
- Row/column headers show the currency's flag icon next to its code
  (reuse =currency_flag_icon()= from =ores.qt.api/FlagIconHelper.hpp=,
  the same helper already used by =ClientCrmDriverPairModel= etc.),
  matching the mockup's flagged headers.
- Fixed decimal precision is acceptable for this task (5 d.p.) --
  convention-aware precision/base-quote ordering is a separate,
  cross-cutting gap (see [[id:76362D7D-44E6-4990-8058-2F9C41BF08DD][the FX market-convention capture]]), not blocking
  here.
- Only the %-change indicator per cell is coloured green/up-arrow or
  red/down-arrow (not the rate itself), computed against that same
  cell's value on the *previous* reload of this session (client-side
  comparison, no server-side tick history needed) -- first load and
  unchanged values get neutral "—" styling, matching the mockup.

* Notes

Implemented =CrmCrossRatesMatrixMdiWindow= (custom =QWidget=, following
the =ServiceDashboardMdiWindow= pattern rather than the entity-list
=EntityListMdiWindow= one -- this is a computed NxN grid, not a paginated
row list) plus =CrmCrossRatesMatrixController= and =MarketdataPlugin=
menu wiring ("Cross-&Rates Matrix"). Manual reload only, no timer, no
NATS subscription, per acceptance. Builds clean.

Blocked before manual QA: bringing up the environment and calling
=marketdata.v1.crm.rates= only ever returns an empty result, because no
party in any environment has a real =crm_topology_config= configured --
there is no seed/publish path for CRM data today. Rather than hand-seed
rows in one local database (untestable by anyone else, and not what a
real user would ever see), this surfaced a real product gap: users need
a realistic CRM available after a normal synthetic-data publish. Raised
[[id:BA82072F-0B2B-48F4-9C22-8D6D39337099][multi-named-config engine support]] and
[[id:6D953558-A61C-461F-859D-CBD28AD5D8F5][the DQ publishing task]] to close that gap; this task resumes once
they land.

* Result

Delivered the CRM cross-rates matrix Qt panel (=CrmCrossRatesMatrixMdiWindow=
+ =CrmCrossRatesMatrixController=), reached through several rounds of
real manual QA against live data, not just build/tests -- see the
=* Open Issues= table above (28 items across 6 rounds) for the full
trail. Final shape:

- A picker dialog (custom Icon::Checkmark/Icon::Dismiss buttons, app
  convention) lets the user open "All" or any named CRM
  (=majors=/=exotics=/=scandies=); each name reuses its own window if
  already open, with independently persisted geometry and toggle
  state.
- Fixed-size cells render via =CrmRateCellWidget= (pair code / rate /
  coloured %-change with ▲/▼ arrows; stale/unavailable status shown via
  colour *and* a glyph, never colour alone) with currency flag icons in
  both header directions.
- Footer controls: Base Currency filter, Update Interval (auto-refresh
  on by default at 5s -- a deliberate relaxation of the story's
  "never broadcast" decision, since client-side polling isn't
  server-side broadcast; see the story's =* Decisions=), Hide Empty,
  and Show Inverted (computed 1/rate, pastel-tinted).
- A "FX Pair Overview" side panel with a client-side session sparkline
  for whichever cell was last clicked.

Along the way, fixed several real bugs surfaced only by driving the
actual UI against real data: a missing =crm_enabled_derived_pair_changed_event=
subscription in =crm_ingest_bridge=, an RLS policy that blocked its
cross-tenant startup scan after the CRM tables moved to =ores_refdata_*=,
an always-editable id field in every UUID-primary-keyed entity's Add
dialog (fixed at the codegen template level), and a silent NATS
max-payload timeout in Market Simulator's feed-start vintage lookup.

Follow-up work captured as separate tasks/stories rather than expanded
in place: server-side UUID generation (all entities), moving the
matrix's inversion/delta logic into =ores.analytics.quant=, CSV/ORE
export, a server-side rate-history endpoint for charts, and an FX
market-convention precision facet.
