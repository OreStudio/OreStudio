* Goal

The knowledge graph can be filtered to show only LLM-related documents
(memories, skills, recipes, runbooks) via a one-click view preset in
the org-roam-ui sidebar. The graph export correctly populates the ~tags~
field so the existing tag whitelist/blacklist panel works. The compiled
graph output is no longer tracked as source in =projects/=; it lives in
=external/= like any other vendored binary.

* Acceptance

- =projects/ores.org-js/graph/= is removed; compiled output lives in =external/org-roam-ui/= and is =.gitignore='d as a build artefact.
- The site build script copies from =external/org-roam-ui/= (unchanged behaviour for deployments).
- =ores-org-roam-export.el= populates the ~tags~ field on each node from the =ALLTAGS= property; the existing org-roam-ui tag whitelist panel works against them.
- All skill documents under =doc/llm/skills/= carry =#+filetags: :skill:= so they appear in tag-filtered views.
- The =mcraveiro/org-roam-ui= fork gains a "view preset" control (initially an "LLM" preset) that sets the tag whitelist to =memory skill runbook recipe= in one click.
- Filtering to the LLM preset in a local graph of this repo produces a readable subgraph (~55–120 nodes).

* Decisions

- Tag data is already present in graphdata.json via =properties.ALLTAGS=; the fix is in the exporter, not the graph UI.
- Compiled output moves to =external/= rather than being deleted: it is still needed for local and CI deploys without a Node build step.
- View presets go into the upstream fork (=mcraveiro/org-roam-ui=) rather than as post-build patches, so the change is maintainable.

* Out of scope

- A fully separate LLM dashboard page (deferred; tag-filtered graph may be sufficient).
- Adding a Node/npm build step to the site CI pipeline (the compiled output in =external/= avoids this for now).
