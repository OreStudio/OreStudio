This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:B38B3869-02FD-4CC7-99BD-9A77904ACA19][Cross-rates matrix (CRM)]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Real trading systems tier their cross-rates: a dense direct-quote mesh
for liquid majors/G10 pairs, and a strict USD-pivot star for thin-liquidity
exotics (nobody directly quotes e.g. TRY/ZAR). =crm_topology_config= already
has a =name= natural key unique per (tenant, party), so the entity model
already supports several named configs per party -- the gap is entirely in
the engine and NATS layers, which currently assume at most one *enabled*
config per (tenant, party) and warn-and-pick-first if more than one is
enabled (see =crm_ingest_bridge::refresh()='s duplicate-enabled safeguard,
landed in [[id:88FD9001-F2BA-4061-994C-5EBE8BBD590A][the service-wiring task]]).

This task removes that stopgap: build and hold one =rate_engine= per
enabled =(tenant, party, config name)=, and let =marketdata.v1.crm.rate=
/=marketdata.v1.crm.rates= address a specific named CRM (or, for
=rates=, return the union across every enabled CRM for that party,
tagged by CRM name, when no name is given -- exact shape to be decided
during implementation).

* Acceptance

- =crm_ingest_bridge= keys its engines map by (tenant, party, config
  name) instead of (tenant, party); every currently-enabled config for a
  party gets its own =rate_engine=, built independently (one config's
  ambiguous/invalid topology must not prevent the others from building).
- The duplicate-enabled warn-and-pick-first safeguard is removed (no
  longer applicable) and its stopgap note in
  [[id:88FD9001-F2BA-4061-994C-5EBE8BBD590A][the service-wiring task]]'s
  =* Result= is superseded.
- =get_crm_rate_request=/=get_crm_rates_request= gain a way to select a
  named CRM; =get_crm_rates_response= tags each returned rate (or the
  response as a whole) with which named CRM it came from when serving
  the multi-CRM case.
- Existing single-CRM-per-party behaviour (today's only real usage)
  keeps working unchanged.
- Unit tests cover: two enabled configs for the same party building two
  independent engines, a per-name lookup, and an all-CRMs batch fetch.

* Result

=crm_ingest_bridge= now keys its engines map by =(tenant, party)= ->
a vector of named engines (=name=, =rate_engine=, =configured_pairs=)
instead of one engine per party. =refresh()= builds one engine per
*every* enabled =crm_topology_config= for a party (duplicate names for
the same party still warn-and-skip the second, since names aren't yet
enforced unique at the DB layer). =update()= feeds a driver tick to
every named engine for the (tenant, party) that has it as an edge (a
tick can be a driver for more than one of a party's CRMs at once).

New/changed API:
- =rate(tenant, party, crm_name, base, quote)= -- single named CRM,
  =crm_name= now required (a pair can resolve differently per CRM).
- =rates(tenant, party, crm_name)= -- batch for one named CRM.
- =rates(tenant, party)= -- batch across *every* enabled CRM for the
  party, returned as =std::vector<named_rate>= (a =derived_rate= tagged
  with which CRM produced it).
- =get_crm_rate_request=/=get_crm_rates_request= gained =crm_name=;
  =crm_rate_item= gained =crm_name= in the response. Empty
  =crm_name= on =get_crm_rates_request= means "every CRM, tagged".

Also fixed the macOS build in the same change (touching the exact same
type): replaced =std::atomic<std::shared_ptr<const engines_map>>= with
a mutex-guarded =std::shared_ptr<const engines_map>= -- portable across
toolchains, same lock-free-reads-mostly semantics (critical section is
just a refcount bump). See
[[id:41B8DCE5-AC29-4DDF-94D7-652225B9EE08][the OSX build task]] and
[[id:2CA8864F-336E-4632-A510-FC503213165B][the capture]] to switch back
once macOS's libc++ implements =atomic<shared_ptr<T>>= for this case.

Verified: full local build clean (=linux-clang-debug-make=); full
=ctest= 71/71 suites passed (=ores.marketdata.service.tests= covers the
new multi-CRM behaviour: two independent named engines, per-name
lookup, unknown-name nullopt, all-CRMs tagged batch); =db recreate -y
-k= clean (no schema changes in this task); =validate_schemas.sh
--strict= clean (233 tables, 0 warnings).
