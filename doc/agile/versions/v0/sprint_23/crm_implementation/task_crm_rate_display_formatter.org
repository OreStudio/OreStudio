* Goal

The convention-aware CRM rate display formatter
([[id:4F2FC58A-D715-4159-A93F-E372AD720D0D][a separate task]]) needs
read access to =currency_pair_convention= rows
(pip_factor/tick_size/decimal_places) and =currency.base_precedence=
without a refdata round-trip on every CRM request/reload. The
project now has a generic caching primitive for exactly this shape
of problem -- =ores.eventing.core::service::cache::partitioned_cache=
plus the =nats-event-cache= codegen facet (see =party_cache= in
=ores.iam.core= for the worked exemplar: generated cache class +
registrar wiring, populated via a =read_*_for_cache= NATS request at
startup, refreshed on the entity's changed-event).

Where the cache should live is the interesting question this task
resolves: =party_cache= lives in =ores.iam.core=, the specific
*consumer* of party data for its own authorization logic -- not in
=ores.refdata= itself (refdata has direct DB access to its own data;
it has no need to cache itself via NATS). But unlike party's
single-consumer (IAM authorization) shape, FX pair conventions are a
cross-UI concern: the CRM matrix needs them in Qt today, and a shell
CRM view ([[id:F4D45153-D81E-464D-8D9B-2DBFEE7CB670][Add CRM support to ores.shell]], filed as a
sibling task) will need the same conventions tomorrow. Putting the
cache inside =ores.qt= would just relocate the duplication problem
the sibling formatter task's own "presentation lives in
=ores.marketdata.client=, not =ores.qt=" decision was designed to
avoid.

=client= is an existing, documented specialist sub-component role
(see =projects/modeling/component_architecture.org=: "provides the
consumer-facing API without coupling consumers to core internals",
currently only instantiated as =ores.iam.client=), so
=ores.refdata.client= is the regular, catalogue-consistent name for
this new piece -- not a bespoke ad hoc suffix. Add it, mirroring
=ores.marketdata.client='s existing shape (thin, DB-free, NATS-only),
and put
=currency_pair_convention_cache= there, generated via the
=nats-event-cache= facet exactly as =party_cache= was, so any UI
component (Qt now, shell/Wt later) links one shared cache instead of
each hand-rolling its own. See
[[id:723B0112-4DDC-4C03-924E-2C90C2F19B5C][Where should an
entity-mirror cache live?]] for the general placement rule this task
established.

* Acceptance

- The =nats-event-cache= facet's currently-hardcoded
  =projects/ores.{cached_by}/core/include/ores.{cached_by}.core/...=
  output path is generalised to accept a configurable subcomponent
  segment (not a literal =core=), so =cached_by: refdata=,
  =cache_subcomponent: client= correctly targets
  =ores.refdata.client= (=projects/ores.refdata/client/include/ores.refdata.client/...=),
  not a nonsensical =ores.refdata.core=.
- New =ores.refdata.client= component: thin, DB-free, links only NATS
  + refdata's own API/protocol types, following
  =ores.marketdata.client='s existing shape (its CMakeLists, export
  header, and directory layout are the template).
- =currency_pair_convention_cache= (+ registrar wiring) generated via
  the =nats-event-cache= codegen facet, mirroring =party_cache=:
  populated via a =read_currency_pair_conventions_for_cache= NATS
  request at startup, refreshed on
  =currency_pair_convention_changed_event=, partitioned by tenant,
  keyed by =pair_code=.
- No refdata round-trip on every lookup -- reads are wait-free
  =partitioned_cache::get()= calls against the in-process snapshot.
- Real Catch2 coverage: cache load from a fake/mock NATS reply,
  lookup hit/miss, refresh-on-changed-event replacing only the
  affected tenant's partition.
- =ores.qt= (the CRM matrix's own module) links and instantiates this
  cache -- actual convention-aware *formatting* using it is the
  sibling formatter task's job, not this one; this task only needs to
  prove the cache loads, refreshes, and is queryable end-to-end.
