* When to use this skill

When the change you are planning crosses component boundaries —
adding a cross-cutting concern, introducing a new architectural
pattern, retiring or replacing a component, or anything where the
impact propagates through multiple layers.

* How to use this skill

1. /Read the [[id:D773166D-0C91-8CB4-3323-42166BC07687][System Model]]/ to anchor: foundation, infrastructure,
   domain, and application layers; which components live in each;
   what depends on what.

2. /Pull the affected component overviews/. Each
   =projects/<component>/modeling/component_overview.org= has the
   inputs / outputs / entry points / dependencies of one
   component. Use the doc tools to enumerate:

   #+begin_src sh :results verbatim
   ./projects/ores.compass/compass.sh list --type component
   #+end_src

3. /Assess impact across the four layers/. For each layer, answer:
   - Foundation — new utilities? shared abstractions changed?
   - Infrastructure — protocol / messaging / testing changes?
   - Domain — which entities? new domain types?
     repository changes?
   - Application — which clients (CLI, Qt, Wt, HTTP, shell, service)
     have to follow?

4. /Assess database implications/ if data is touched: schema
   changes, migration path, whether the old and new schemas need
   to coexist, indexes, triggers.

5. /Design a backwards-compatibility strategy/. Versioning of NATS
   subjects (=v1= / =v2=), of CLI flags, of file formats. Feature
   flags or config switches where they help. A timeline for
   retiring old paths.

6. /Sequence the migration/. Lower layers first (foundation →
   infrastructure → domain → application). Each step must:
   - Leave the system buildable.
   - Be independently testable.
   - Be small enough to review.

7. /Plan diagram updates/. Both [[id:271F40DA-D0EA-4D08-A4F8-3D2B835DF5E7][class diagrams]] (per affected
   component) and the system-level diagram in
   =projects/modeling/ores.puml=. Sequence the diagram updates
   alongside the code changes.

8. /File the plan as a story/ (or a small set of stories if the
   migration is large) via [[id:2DF8ADAF-20C9-4757-9399-798EEC6DF828][agile-product-owner]]. Each migration
   step becomes a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] inside the story.

9. /Document trade-offs/. If you deviate from strict layer
   separation (e.g. business logic in a stored procedure for
   performance), record the rationale in the story's
   =* Decisions= section — both the gain and the cost.
