This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:E5EC80BB-789B-4BD6-B6E5-DC7AA790692A][Synthetic data librarian support: FX foundation]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Wire =ores.synthetic.market_data_generation_config= and its FX spot
sub-config into the DQ publish-from-dq pattern so a synthetic FX
config becomes publishable from a DQ bundle: the artefact-type row,
shadow table (an =ores.codegen.lookup_entity= model, same mechanism
=report_definitions= uses today), the publish function
(=ores_synthetic_publish_fx_spot_config_from_dq_fn= or equivalent),
NATS handler, and registrar wiring. Deferred per
[[id:53116563-2667-48CD-941E-8521D3E42CBC][Codegen facet: generate the DQ publish-from-dq pattern]]: this task
hand-writes the publish function/handler/registrar using the
=report_definitions= instance
(=projects/ores.sql/create/reporting/reporting_publish_from_dq_create.sql=,
=publish_from_dq_handler.{hpp,cpp}=,
=projects/ores.reporting/core/src/messaging/registrar.cpp=) as the
copy-and-adapt template, rather than waiting on the codegen facet.

* Acceptance

- Artefact-type row, shadow table (codegen'd lookup_entity model),
  publish function, handler, and registrar entry all exist and are
  wired for the synthetic FX config.
- The publish function writes only into =ores.synthetic='s own
  tables, per the strict service-table isolation exception documented
  in [[id:98B3572E-A9F9-45F5-A4C5-3DC23F3CFF70][DQ Publish Pattern]].
- The hand-written function/handler/registrar are written so they can
  be dropped in favour of generated equivalents once
  [[id:53116563-2667-48CD-941E-8521D3E42CBC][the codegen facet story]] lands, with no other rework required.

* Plan

Modeled the artefact as one denormalized DQ artefact type
(=synthetic_fx_spot_configs=) combining the parent
=market_data_generation_config= fields (name, description, enabled)
with the =fx_spot_generation_config= fields (currency pair, initial
price, tick cadence, process type) in a single shadow row, since the
publish path needs to create both a shared container and one or more
FX sub-configs per party. Added an =ores.codegen.lookup_entity= model
and a hand-written shadow table (codegen's lookup_entity path turned
out to be broken — see Notes), a =SECURITY DEFINER= publish function
that resolves/creates the shared container per (tenant, party) then
upserts FX rows keyed by currency pair, a NATS handler mirroring
=report_definitions='s file-for-file, registrar wiring, and the IAM
execute grant. Verified end-to-end against the live dev DB (container
reuse, version bump on upsert, insert_only skip behaviour) and built
=ores.synthetic.core.lib= successfully. Ran
=projects/ores.sql/utility/validate_schemas.sh= and fixed a missing
drop script it flagged for the new artefact table.

* Notes

Found a pre-existing =ores.codegen= bug while trying to generate the
shadow table: the enrichment step that populates the mustache
=entity= context is gated on =model_type == 'schema'=, but
=ores.codegen.lookup_entity= models map to internal model_type
=enum=, so =sql_schema_artefact_create.mustache= cannot render from
*any* lookup_entity model today — confirmed the same failure against
the pre-existing =ores.dq.report_definition_lookup_entity.org=. Hand-
wrote the shadow table to match the template's expected output
column-for-column instead. Recorded on
[[id:53116563-2667-48CD-941E-8521D3E42CBC][Codegen facet: generate the DQ publish-from-dq pattern]] as a
prerequisite fix.
