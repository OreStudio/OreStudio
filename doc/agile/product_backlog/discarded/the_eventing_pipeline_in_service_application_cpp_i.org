#+title: Codegen approach: generate eventing registrar for service application.cpp
#+description: The eventing pipeline in service application.cpp is hand-written and grows unboundedly as entities are added. The pattern per entity is:

  (1) register_mapping<entity_changed_event>(event_source, "component.entity", "sql_table_name");
  (2) auto entity_sub = event_bus.subscribe<entity_changed_event>([&nats](const entity_changed_event& e) {
          publish_entity_event(nats,
              std::string(event_traits<entity_changed_event>::name),
              entity_change_event{.entity = "component.entity",
                                  .timestamp = e.timestamp,
                                  .entity_ids = e.entity_ids,
                                  .tenant_id = e.tenant_id});
      });

For synthetic, the three missing blocks are:
  - gmm_component: table=ores_synthetic_gmm_components, ids field=gmm_component_ids
  - fx_spot_generation_config: table=ores_synthetic_fx_spot_generation_configs, ids field=fx_spot_generation_config_ids
  - market_data_generation_config: table=ores_synthetic_market_data_generation_configs, ids field=market_data_generation_config_ids

Suggested approach — mirror the messaging registrar pattern:
  - Add a new codegen address (e.g. ores.cpp.service.eventing) that generates a per-component eventing_registrar.hpp header
  - The header exposes a single inline function: register_eventing(event_source, event_bus, nats)
  - application.cpp includes one header per component and calls register_eventing() once
  - Each entity contributes its register_mapping + subscribe block inside that function
  - The entity model already has all required knobs: tablename (SQL), changed_event_class, changed_event_include, entity_plural (ids field name)
  - This eliminates the hand-edit gap entirely: adding an entity to codegen automatically extends the eventing registrar
