This page documents a [[id:31C868B9-306E-4282-BA8C-07E647021B34][task]] in the [[id:DBDFF0B1-E358-4300-B361-644A89FBD169][Migrate all entities onto the generic HistoryDialog]] story. It captures the goal, current status, acceptance, and any notes or results.

* Goal

Retire the last 17 hand-rolled per-entity =*HistoryDialog= classes
left in =projects/ores.qt/refdata= once [[id:9605CFDB-9732-46D2-A8D3-714BD44653A1][the codegen-model-backed
batch]] closed. None of these 17 have a codegen model reachable from
=ores.refdata='s own codegen, which split into two genuinely
different sub-problems investigated when this task was created:

1. *5 entities actually have a codegen model, just under a different
   owning component* — =floating_index_type=, =leg_type=,
   =payment_frequency_type= (=ores.trading=), and =code_domain=,
   =dataset_bundle= (=ores.dq=). Their Qt controller happens to live
   in =ores.qt/refdata= (same organizational pattern as =Book=/
   =BusinessCentre=, whose backend lives elsewhere too), but the
   entity itself is addressable via
   =compass codegen entity generate <entity> --address ores.cpp.qt=
   once invoked with the right model file — these can migrate via the
   exact same full-regeneration process as every other entity in this
   story, no scoping decision needed.
2. *The other 12 entities have no codegen model anywhere in the
   codebase* — =catalog=, =change_reason=, =change_reason_category=,
   =coding_scheme=, =coding_scheme_authority_type=, =data_domain=,
   =dataset=, =methodology=, =nature_dimension=, =origin_dimension=,
   =subject_area=, =treatment_dimension=. These are hand-crafted,
   cross-cutting infrastructure in three families:
   - *Change Reason* (=change_reason=, =change_reason_category=): the
     change-reason-cache system itself, used to audit every other
     entity's save/delete. Self-referential — codegen-ing these would
     need to special-case (or omit) =has_change_reason_cache= on the
     entity that backs that very cache.
   - *Coding Scheme / classification* (=coding_scheme=,
     =coding_scheme_authority_type=, =code_domain= is the one exception
     with a model, above): used to validate other entities'
     classification codes.
   - *Data Catalogue* (=catalog=, =data_domain=, =dataset=,
     =methodology=, =nature_dimension=, =origin_dimension=,
     =subject_area=, =treatment_dimension=): a data-governance/catalogue
     metadata subsystem (subject areas, methodologies, dimensions)
     that may have bespoke domain logic not captured by the generic
     entity model.

   Whether these 12 warrant becoming codegen models (eliminating this
   exact class of bug — a human forgetting a wiring step that codegen
   would emit automatically) or are legitimately bespoke enough to stay
   hand-authored is a real product/architecture decision, not
   something to default silently either way. This task's first job is
   making that decision (or handing it back to the story owner with a
   recommendation per family) before any migration work on the 12
   starts.
