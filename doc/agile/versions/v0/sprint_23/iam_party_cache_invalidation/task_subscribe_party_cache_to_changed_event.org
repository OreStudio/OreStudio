* Goal

A reusable, entity-agnostic =partitioned_cache<PartitionKey, Key,
Value>= template exists (e.g. in =ores.eventing.core=) that any
future consumer-side cache can be built on without hand-writing
concurrency code again. Storage is
=immer::atom<immer::map<PartitionKey, immer::map<Key, Value>>>=:
readers take a wait-free atomic =load()= of the whole structure and
are never blocked by a concurrent reload; a partition replace goes
through the atom's compare-and-swap =update()=, not a blind
=store()=, so concurrently reloading two different partitions never
races into a lost update. An optional aux-index policy (a callable
applied when a partition is replaced) lets entity-specific derived
structure — e.g. party's parent/child hierarchy — stay generic
rather than hardcoded into the primitive.

The hand-written =party_cache.hpp= prototype produced during this
story's investigation (already on =immer::atom=, already correctly
subject-bound) becomes the reference this template is extracted
from — not a shipped deliverable itself.

* Acceptance

- =partitioned_cache<PartitionKey, Key, Value>= implemented with
  wait-free =get(partition, key)=/=snapshot(partition)= reads and a
  CAS-safe =replace_partition(partition, entries)= write.
- Aux-index policy hook supports building a derived structure (e.g.
  a parent→children map) alongside the primary key→value map,
  without the primitive itself knowing about hierarchy.
- Unit tests: concurrent =replace_partition= calls for different
  partitions never lose an update; a reader's =snapshot()= never
  observes a torn mix of old/new entries within one call; a
  concurrent reader is never blocked by a writer or vice versa.
