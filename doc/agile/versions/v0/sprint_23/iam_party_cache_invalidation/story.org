* Goal

When a party is activated (or otherwise changed) in ores.refdata, any
ores.iam.core instance holding that tenant's =party_cache= sees the
change without needing an IAM service restart.

Root cause (confirmed by reading the code, not the capture's
original hypothesis): =ores.refdata='s side is fine. Party writes go
through the standard postgres outbox pattern — =application.cpp=
registers =party_changed_event= against table
=ores_refdata_parties=, and every save already publishes to NATS
subject =ores.refdata.party_changed= (see
=party_changed_event.hpp='s =event_traits=). Party is *not* one of
the 26 refdata entities missing an event registrar.

The bug is entirely on the consumer side, in
=ores.iam.core/service/party_cache.hpp=. Its docstring claims it is
"reloaded on receipt of =refdata.v1.parties.changed= notifications"
— but that subject string appears nowhere except the docstring, and
=party_cache::load()= is only ever called from the read-at-startup /
read-on-miss path (=registrar.cpp=,
=auth_handler.hpp='s =auth_ensure_parties_cached=). There is no NATS
subscription anywhere in =ores.iam.core= to
=ores.refdata.party_changed= (or to anything else). The docstring
describes a mechanism that was never implemented.

Fix: add a NATS subscription in =ores.iam.core= (in =registrar.cpp=,
alongside where =party_cache= is constructed) to
=ores.refdata.party_changed=, and reload the affected tenant's
partition — via =party_cache::load(tenant_id)= or a more targeted
per-party update — when it fires. Correct the docstring's stale
subject name to match reality once fixed.

* Acceptance

- =ores.iam.core= subscribes to NATS subject
  =ores.refdata.party_changed= (the outbox event refdata already
  publishes on every party save/activation) and reloads the affected
  tenant's =party_cache= partition on receipt.
- =party_cache.hpp='s docstring is corrected to name the real subject
  (=ores.refdata.party_changed=), not the nonexistent
  =refdata.v1.parties.changed=.
- Reproduction from the capture no longer occurs: activating a party
  via =save_party_request= (e.g. through
  =barclays_system_provision.ores=) is visible to a subsequent IAM
  login without restarting the IAM service.
- Regression/unit test coverage added for cache
  reload-on-notification in =ores.iam.core=.

* Decisions

- 2026-07-12: Original capture hypothesised the bug was on
  =ores.refdata='s producer side (missing publish). Reading the code
  showed refdata's outbox eventing for party already works correctly
  and publishes =ores.refdata.party_changed= on every save. The bug
  is entirely on =ores.iam.core='s consumer side: =party_cache= never
  subscribes to anything. Story goal/acceptance updated accordingly;
  scope narrows to IAM-side subscription wiring only.

* Out of scope

- General TTL/reload-on-miss fallback strategy for =party_cache=,
  unless the eventing fix proves unreliable during implementation.
- Broader ores.iam/ores.refdata service-boundary cleanup (tracked
  separately as "IAM/Refdata service boundary cleanup").
