This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *next* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* Update — root cause fixed (2026-06-29)

The empty-output root cause was a *filename-suffix misclassification*, not the
renderer itself: =generate_from_model= classified models by filename suffix, so
=ores.synthetic.gmm_component.org= matched the =_component.org= exclusion and was
treated as a component model — skipping the domain enrichment, hence empty files.
Fixed by classifying off the document =#+type:= (=get_model_type=), dropping the
bogus =_component.org= suffix, and marking the filename-suffix predicate family
legacy. gmm_component now generates end-to-end.

What remains for the *simplification* ask (still valid): scaffold should
pre-populate knobs and auto-register the component; codegen should fail loudly
on an empty render; and the filename-suffix legacy family should be retired
once every model carries a =#+type:=.

* What

Onboarding the *first* entity of a *new* component via codegen is far too hard
and currently depends on legacy infrastructure that is being removed. Doing it
for =ores.synthetic= (=gmm_component=) surfaced three separate friction points,
none of which errors — the failure mode is silent empty output:

1. *Component registration (undocumented, easy to miss).* A new component must
   be added to [[id:B3CBBDED-7CD9-423B-94D8-F5C4C0488CEA][component_catalogue]] (the
   =* Components= table) or =compass codegen entity= cannot discover its org
   models ("No entity found"). There is no scaffold step that does this.

2. *Scaffold ships no knobs.* =compass add entity_org= emits a bare skeleton.
   A model missing header/section knobs (=#+coding_scheme:=, =#+image_id:=,
   the uuid-PK / Qt / eventing blocks, ...) still parses but generates empty.

3. *CLI uses the legacy renderer.* Both =compass codegen generate --model= and
   =codegen.sh generate= dispatch to the legacy =generate_from_model=
   (=core.py=). Its enrichment silently empties =domain_entity= for a minimal
   new-style org model, so every generated file is just the licence header
   (~19 lines). The modern path renders the SAME model correctly:
   =render_template(tmpl, load_model(model))= → 110 lines (identical treatment
   to =country=). Even =compass codegen entity generate <e> --diff= renders
   empty because it, too, delegates the write to =generate_from_model=.

Decision from the team: *do not use any legacy infrastructure for new models —
it is being removed soon* (the deprecation warning references "task B11"). So
new-model generation must route through the modern =load_model= +
=render_template= path, which is not yet exposed by any CLI command. Until that
migration lands, a new component's entities cannot be generated via the CLI.

* Why — simplify the process

Adding the first entity of a new component should be one obvious, modern
command — e.g. =compass codegen entity new <component> <entity>= that
registers the component, scaffolds a model pre-filled with sensible knobs, and
generates via the modern renderer, failing loudly (never silently empty) if a
knob is missing. Today it is three undocumented manual steps plus a legacy
renderer that emits empty files without error — which cost a long debugging
session. Retiring =generate_from_model= (B11) and folding registration +
scaffold + generate into one modern command is the simplification.

* Action

- Finish migrating CLI callers off =generate_from_model= onto the modern
  =load_model= + =render_template= path; then retire the legacy renderer.
- Make the scaffold pre-populate knobs and auto-register the component.
- Make codegen fail loudly on an empty render instead of writing empty files.
