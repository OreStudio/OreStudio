* Goal

The story's goal says a party should get a usable synthetic data
config "when it is created." Investigation found no existing mechanism
for that: =party_service::save_party= is a plain upsert with no
lifecycle hooks, and the Librarian's bundle Apply
(=dq.v1.bundles.publish=) is a manual, user-triggered action, not an
automatic one. This task decides — and records in the story's
=* Decisions= — whether "usable config on party creation" is satisfied
by prompting/defaulting the bundle Apply step inside the existing
Party Setup wizard flow (cheap, reuses this story's infrastructure
as-is), or whether a genuine automatic trigger (e.g. a
=party_changed_event= listener that calls the bundle-publish workflow)
is required, with the tradeoffs of each spelled out.

* Plan

Investigated the existing Party Setup flow:
=projects/ores.qt/application/src/PartyProvisioningWizard.cpp=
(=PartyExecutePage=) already runs a fixed, automatic sequence of
backend phases with no user confirmation for most of them: Phase 1
(counterparty import), Phase 2 (publish the =organisation= bundle —
already an automatic =dq.v1.bundles.publish= call, no picker),
Phase 3 (create user-*selected* report definitions — the one phase
that's user-confirmed, via =PartyReportSetupPage='s checklist), Phase 4
(activate). There is no generic "apply bundle" abstraction reused
across phases — each phase is bespoke inline logic.

Decision: add a new automatic Phase 3.5 that publishes the
=synthetic_market_data= bundle, mirroring Phase 2's =organisation=
bundle-publish pattern (automatic, no user picker, uses
=WorkflowStepsWidget= for progress) — not the Phase 3 reports pattern
(no per-item selection makes sense yet, since this story only ships
one seeded FX config). Sequenced after Phase 3 (reports) and before
Phase 4 (activate), per explicit instruction. This satisfies "usable
config on party creation" without needing a genuine
event-driven trigger — going that route would duplicate the wizard's
existing bundle-publish call for no benefit, since party creation in
this app already goes through this one wizard.
