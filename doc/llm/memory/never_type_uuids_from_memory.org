#+title: Never type org-roam UUIDs from memory
#+description: When writing [[id:UUID]] links, always copy the UUID from the target file's :ID: drawer (head -3 or compass show); LLM-generated UUIDs look plausible and silently break the link graph.
#+type: memory
#+memory_subtype: feedback
#+level: cross
#+filetags: :llm:org:links:uuid:memory:feedback:
#+created: 2026-06-05
#+updated: 2026-06-05

When writing an =[[id:UUID][...]]= link, the UUID must be /pasted from
compass output/ — =compass show=, =compass list=, or the path+ID a
=compass add= prints — or from a just-read =:ID:= drawer. Never
compose one: an LLM writing prose will fill the UUID slot with a
plausible-looking value (confabulation under fluency pressure), and a
fabricated UUID is indistinguishable from a real one at a glance while
silently breaking the org-roam link graph.

*Why:* On 2026-06-05 two links were written with fabricated UUIDs
within an hour (currency model link in the messaging reference;
developer-scripts link in a task) — both looked valid and were only
caught by explicit re-verification.

*How to apply:* Every time a UUID is about to be written — an id-link
/or/ a Story-ID/Task-ID commit trailer — and it is not visibly present
in the current context from a tool output, stop and fetch it (a third
incident on 2026-06-05 put a fabricated Task-ID in a pushed commit
trailer). After bulk-writing docs with links, grep the
new ids against the repo to confirm each resolves.
