* Goal

Add a simple script-library panel to the Qt UI's shell window: browse
the =.ores= scripts, preview one, and run it in the embedded shell.
Nothing more — v1 is a browser and runner, not an editor.

Derived from a richer concept (composer with autocomplete, sequence
editor, save, local/grid context). Trimmed to match reality:

- /No editing or composing./ Tangled =.ores= files are build
  artefacts that are never hand-edited — authoring happens in the
  literate org sources and flows through =tangle_shell_scripts=. A
  save button would fight the library's own conventions.
- /Run = =load=./ The shell already has stop-on-error script
  execution; the Run button simply issues =load <script-path>= into
  the existing embedded shell session, whose widget is the live
  output. No new execution engine.
- /Descriptions from the literate sources./ Each =.ores= has a
  sibling =.org= whose =#+title:= / =#+description:= lines the panel
  reads for the list entries.
- /Flat list, no folders or status badges/ until the library is big
  enough to need them.

** What

A Qt UI surface for the ores-shell script library: browse the
available =.ores= scripts (name, description and step commentary
sourced from their literate org files), view a script's contents, and
run one against the current session — likely alongside the embedded
shell widget.

** Why

The provisioning epic creates a script library in
=projects/ores.shell/scripts/= (org-mode tangled). Operators using
the Qt client should be able to discover and execute those scripts
without leaving the UI or knowing the filesystem layout. Idea
recalled from earlier discussions; the backlog had no capture for it
(verified 2026-06-06).
