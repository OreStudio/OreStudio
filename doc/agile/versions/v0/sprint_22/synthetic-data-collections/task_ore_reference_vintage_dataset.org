* Goal

The synthetic FX collections (Basic/Realistic) need a real, internally
consistent 2016-02-05 spot for every major currency, and the future
Cross-Rates Matrix (CRM, [[id:48F6FD8D-56C7-4FF7-A369-CD13D08A6057][see design]]) needs a minimal *spanning-tree* set
of directly-quoted *driver* rates spanning every currency it cares
about — not derived crosses, and not gaps. ORE's own bundled
=Legacy/Example_56/market.txt= was the original candidate source, but
comparing its EUR-anchored values against real Federal Reserve H.10
closes for the same date found it consistently off (1.28%-6.63%
across all 7 pairs, worst on JPY) — plausible 2016 levels, not actual
market closes. It's also missing AUD, CAD, and NZD entirely, so no
amount of triangulation recovers those three currencies from it.

This task publishes a small, purpose-built dataset instead: 8 real
Federal Reserve H.10 rates for 2016-02-05, each individually citable
to its own Fed URL, hub'd on USD — =EUR/USD, GBP/USD, USD/CHF,
USD/JPY, USD/SEK, AUD/USD, USD/CAD, NZD/USD=. That's a minimal
spanning tree over 9 currencies (EUR, USD, GBP, CHF, JPY, SEK, AUD,
CAD, NZD): every pair among them is triangulable once the CRM exists,
and every value in the dataset is real, not derived by us. Published
through the same DQ/librarian mechanism =currency_pairs= already uses
— a new artefact table, a new methodology entity documenting the
sourcing (Fed URLs, retrieval timestamp, target date — queryable, not
a code comment), a catalog, a dataset registration, and a
publish-from-dq function — so a party gets it the same way it gets its
currency-pairs bundle, and the synthetic generator has one consistent,
citable source of truth instead of ad hoc file parsing.

* Result

Delivered, with two design refinements beyond the original plan (both
made mid-implementation, per review):

- *Generic, extensible artefact shape*: =ores_dq_market_data_observations_artefact_tbl=
  mirrors =market_observation='s own decomposed-key columns
  (=series_type=/=metric=/=qualifier=/=point_id=), not anything
  FX-specific — future asset classes (rates curves, vol surfaces, ...)
  are new dataset rows, never a new artefact table. Added =source_url=/
  =retrieved_at= as per-row columns so every future value carries its
  own citation independently, rather than a shared methodology blob
  that would grow unboundedly as more data is added.
- *Bundle grouping*: registered =marketdata.reference_vintage_2016_02_05=
  (via =ores_dq_dataset_bundles_upsert_fn=/=..._bundle_members_upsert_fn=,
  the same bundle mechanism =base=/=slovaris=/etc. already use) with
  =marketdata.fx_driver_rates= as its first member — future datasets
  for the same vintage date become further members of this one bundle,
  not new bundles.

Five pieces, all hand-written (no codegen model exists for a DQ
artefact table — see Notes): the artefact table, a new =artefact_type=
row, the methodology/catalog/dataset/bundle seed populate script, the
=ores_marketdata_publish_market_data_observations_from_dq_fn=
=SECURITY DEFINER= function (finds-or-creates the series, upserts
observations, party-scoped like =market_series=/=market_observations=
require), and a new =ores.marketdata.core= NATS handler
(=publish_from_dq_handler=, mirroring =ores.synthetic.core='s) wired
into the registrar — =ores.marketdata.service='s first inbound
publish-from-dq subject.

Verified end-to-end against a live stack via the generic =bundles
publish <code> --wait --party-id <id>= shell command (no dedicated
provisioning script needed — this mechanism already existed and
required no changes): all 8 pairs land on Barclays with exact curated
values, correct =observation_datetime=, and the distinct
=source='fed.h10.2016-02-05'= tag; a second publish is idempotent
(still exactly 1 current row per pair, no duplicates).

Hit one unrelated environmental issue along the way: =ores.marketdata.service=
was running a deleted-inode binary (the executable file had been
overwritten by a later build while the process was still up, and a
prior =services stop=/=start= cycle silently failed to actually
replace it) — diagnosed via =readlink /proc/<pid>/exe= showing
"(deleted)", fixed by killing the stale PID so the controller
respawned it fresh. Not a code bug; noted here in case it recurs.
