* Plan

Scaffolded the model via =compass add entity_org --component refdata
--slug business_day_convention_type=, then filled it in following
=ores.refdata.currency_market_tier='s aux-type shape
(code/name/description/display_order) rather than trading's leaner
original (which only had =description=), since it's now a first-class
=ores.refdata= lookup type. Generated each facet individually
(=sql=, =domain=, =generator=, =all-cpp=, =nats-eventing=,

=compass codegen generate --model <path> --profile <profile>=,
disambiguating from the still-existing trading model by path since
both shared an entity name during the transition.

** Investigation before removing the trading original

Checked every real consumer before touching anything:
- Qt: =ores.qt.refdata= already had a full, *committed* Qt layer
  (Controller/MdiWindow/DetailDialog/HistoryDialog,
  =RefdataPlugin.cpp= wiring) for this exact entity, dated months
  ago — built ahead of a backend that never existed until this task.
  Compiling it for the first time surfaced two latent bugs: an invalid
  =Icon::Calendar= enum value (fixed to =Icon::CalendarClock=) and a
  delete-request field name/shape mismatch (fixed with
  =:delete_request_id_field: codes= +
  =:delete_request_id_is_plural: true=, matching the native
  batch-delete protocol shape).
- CLI: =ores.cli= had full hand-written wiring (options struct, parser,
  =application.cpp= export/add/delete handlers) — not a codegen gap
  exemption, just hand-code that needed removing along with the
  trading original, per the story's acceptance.
- SQL: *not* zero-consumer as originally believed —
  =trading_swap_legs_create.sql= soft-FKs =business_day_convention_code=
  to it. Repointed that one call site at
  =ores_refdata_validate_business_day_convention_type_fn= before
  deleting the trading version.
- NATS: hand-written combined handler
  (=instrument_ref_handler.hpp=/=registrar_trades.cpp=) alongside three
  other trading aux types — removed just this type's four subscribe
  blocks and handler methods, left the others alone.

** Gotchas hit generating a brand-new entity from scratch

- =needs_counter= (the synthetic-generator boilerplate) is derived from
  =primary_key.is_text=, which needs to be set explicitly
  (=:is_text: true=) — omitting it silently produces a generator that
  references an undeclared =counter= variable.
- New =.cpp= files aren't picked up by CMake's =GLOB_RECURSE= until a
  fresh =cmake --preset ...= configure — a build immediately after
  =codegen generate= will fail to *link* (undefined reference) even
  though the file exists on disk and compiles fine on its own.
- Hand-maintained files with no =AUTO-GENERATED= marker
  (=ores.refdata.core/messaging/registrar.cpp=,
  =ores.refdata.service/app/application.cpp=) must be wired manually —
  codegen does not touch them. Missing the eventing relay wiring in
  =application.cpp= is exactly the kind of silent gap the entity
  evaluation checklist warns about (cf. the =party_type= precedent).
