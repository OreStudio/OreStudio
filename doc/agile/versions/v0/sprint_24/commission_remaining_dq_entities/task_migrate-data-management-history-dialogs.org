* Goal

Retire the legacy hand-rolled per-entity =*HistoryDialog= classes for
all 14 DQ-owned entities currently living in =ores.qt.data_management=
(=CodeDomainHistoryDialog=, =DatasetBundleHistoryDialog=, etc.),
replacing them with the generic =HistoryDialog= widget — same pattern
every other component's migration in this backlog follows (currency,
PR #1553; the full refdata batch, task 9605CFDB).

Discovered while scoping [[id:EE0459FE-8AC1-4D7B-99C2-747FCF0E4499][the ores.qt.refdata story's last straggler
task]]: these 14 entities were originally thought to be
=ores.qt.refdata= entities whose Qt controller "happened to live"
there (same pattern as =Book=/=BusinessCentre=), but
[[id:D5DF4CFA-5316-4825-BC4B-37C96726C00C][a prior task]] already moved all 14 out of =RefdataPlugin= into
=ores.qt.data_management= — so they're not =ores.qt.refdata= work at
all, and belong here instead, under the DQ component's own
commissioning story (which is already standing up =ores.qt.dq= as the
proper plugin =ores.qt.data_management= is a stepping stone toward).

Two of the 14 (=code_domain=, =dataset_bundle=) already have a
codegen model; the other 12 (=catalog=, =change_reason=,
=change_reason_category=, =coding_scheme=,
=coding_scheme_authority_type=, =data_domain=, =dataset=,
=methodology=, =nature_dimension=, =origin_dimension=,
=subject_area=, =treatment_dimension=) have none anywhere in the
codebase and need one authored from scratch before they can migrate
— no entity in this backlog is left hand-authored as a matter of
policy (see the sibling task's Notes for the reasoning, including how
=change_reason=/=change_reason_category='s self-referential
change-reason-cache wrinkle doesn't change that).
