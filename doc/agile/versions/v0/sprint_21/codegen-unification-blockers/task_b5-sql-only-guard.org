* Goal

Implement element-scope physical-space activation for entity model files, following the [[id:CE95C751-2125-4BA9-A78D-C30F9540FC5E][MASD variability]] convention used by Dogen (=:masd.cpp.enabled:=, =:masd.csharp.enabled:=, etc.). The generator reads =ores.*= properties from an entity's =:PROPERTIES:= drawer, resolves them with specificity semantics (more-specific address overrides less-specific), and applies the result as an element-level filter before facet dispatch.

The address hierarchy is =ores.{technical_space}[.{facet}[.{archetype}]].enabled=:

| Address | Level | Example effect |
|---------+-------+----------------|
| =:ores.sql.enabled: false= | technical space | disables all SQL facets |
| =:ores.cpp.enabled: false= | technical space | disables all C++ facets |
| =:ores.cpp.qt.enabled: false= | facet | disables only the =qt= facet |
| =:ores.cpp.domain.class_header.enabled: false= | archetype | disables one archetype |

Specificity resolution: a more-specific address overrides a less-specific one. Setting
=:ores.cpp.enabled: true= and =:ores.cpp.qt.enabled: false= in the same drawer enables all
C++ facets except Qt. This replicates Dogen's property resolution semantics exactly.

Absence of any =ores.*= property leaves the entity's activation unchanged from the current
behaviour (=model_types= filter is the sole gate).

This task replaces the ad-hoc =#+sql_only= and =#+has_qt= boolean flags that were originally
planned as B5 and B6.

** Technical-space to facet mapping

For hierarchical resolution the generator needs the TS→facet map:

| Technical space | Facets |
|-----------------+--------|
| =sql= | =sql=, =non-temporal-sql= |
| =cpp= | =domain=, =generator=, =repository=, =service=, =protocol=, =nats-eventing=, =nats-handler=, =qt=, =non-temporal-domain=, =non-temporal-repository= |

This map must live in =facet_catalogue.org= (as a new =* Technical spaces= section) so it is
the authoritative source consumed by =generator.py= at startup.

** Implementation notes

- Add a =* Technical spaces= section to =facet_catalogue.org= mapping TS names to their facet
  members.
- Implement =resolve_physical_bindings(properties: dict, facet_catalogue: dict) -> set[str]=
  in a dedicated module (=codegen/physical_space.py= or similar) — *not* inline in =core.py=
  or =generator.py=.
- The function takes the raw =:PROPERTIES:= dict from the entity and the facet catalogue, and
  returns the set of profile names that are enabled for this element.
- Unit-test the resolution logic exhaustively: default (no properties), TS-level disable,
  facet-level disable, combined TS-enable + facet-disable override, unknown address (warn and
  ignore), conflicting same-specificity entries (last-wins or raise).
- Wire into =core.py= profile dispatch after the model-type filter.

* Acceptance

- Entity with =:ores.cpp.enabled: false= in its =:PROPERTIES:= drawer produces SQL output
  only; no C++ files are generated.
- Entity with =:ores.cpp.qt.enabled: false= generates all C++ facets except Qt.
- Combined override: =:ores.cpp.enabled: true= + =:ores.cpp.qt.enabled: false= in the same
  drawer enables all C++ except Qt (more-specific wins).
- Entity with no =ores.*= properties behaves exactly as before — no regression.
- =facet_catalogue.org= carries a =* Technical spaces= section mapping each TS to its member
  facets; =generator.py= loads and uses it.
- Resolution logic lives in a dedicated Python module, not inline in dispatch code.
- Unit tests cover: no properties, TS disable, facet disable, combined TS-enable +
  facet-disable, unknown address (warning, no crash), and all existing regression tests pass.
