* Goal

The MASD documentation is cleanly separated into two document sets:

- *Pure MASD* (=doc/knowledge/masd/=): =masd.org= plus the =logical_space=,
  =physical_space=, =technical_space=, =facet=, and =variability= concept
  docs describe only the MASD methodology and its conceptual model — they
  make no reference to ORE Studio, since they are observations of the
  external world. They are augmented with technical content and figures
  from the Dogen [[https://mcraveiro.github.io/dogen/docs/domain_architecture.html][Domain Architecture]] reference.
- *Applied MASD* (=projects/modeling/applied_masd.org=): a new umbrella doc
  that uses the MASD definitions to explain how ORE Studio instantiates each
  concept, tying together =system_model=, =technical_spaces=,
  =variability_model=, the =*_facet= docs, and =entity_lifecycle=.

Throughout, the four canonical MASD terms (TS, Part, Facet, Archetype) are
used precisely, and the =part= definition is corrected: a part (like a
facet) is an arbitrary partitioning/taxonomy of the *logical* space for a
problem domain (e.g. type representation, serialisation), expressed using a
technical space's language — *not* a subdivision of a technical space. The
first-pass terminology fixes (archetype vs facet, no "facet group",
projection over pipeline, disciplined "meta-model" usage) merged in PR #1176
are preserved.

* Acceptance

- =doc/knowledge/masd/*.org= contain no ORE Studio references (no =ores.*=
  components, no =--profile=, no =currency=, no four-layer architecture).
- The =part= definition no longer reads "subdivision of a technical space";
  it is reframed as an arbitrary partitioning of the logical space expressed
  in a technical space's language (corrected in =masd.org= and
  =physical_space.org=, with the "belongs to a TS" wording fixed in
  =technical_space.org=).
- The pure docs incorporate technical content and the key figures from the
  Dogen Domain Architecture reference (including "Figure 3: Transforms from
  logical representation to filesystem").
- A new =projects/modeling/applied_masd.org= umbrella doc absorbs the
  ORE-specific content (facet catalogue, the two codegen routes, the
  =currency= reference implementation, automation-spectrum status) and
  cross-links to the existing applied docs.
- Pure↔applied See-also links are reciprocal and the org-roam index is
  updated.
- Carried over from PR #1176: "facet" is never used to mean "archetype";
  "facet group" does not appear; "pipeline" is replaced with "projection"
  for the logical→physical transformation; "meta-model" is used only for
  actual MASD meta-model concepts (PMM, LMM, VMM).
