* Acceptance

- [X] Root cause identified: =HierarchyModelBuilder.cpp=/=OrgDocRenderer.cpp=
  define =ORES_QT_API=-decorated symbols but are compiled a second
  time directly into =ores.qt.api.tests= (to dodge =ores.qt.api.lib='s
  Widgets/display requirement for headless testing); without
  =ORES_QT_API_LIBRARY= defined there, =export.hpp= expands
  =ORES_QT_API= to dllimport for a TU that also defines the symbol —
  MSVC flags this as inconsistent linkage (GCC/Clang don't).
- [X] Fixed at the root — not by defining the macro in the test
  target (a "quick unblock" alternative was explicitly considered and
  rejected in favour of the real fix): Widgets-free logic extracted
  into a new =ores.qt.headless= component with its own
  =ORES_QT_HEADLESS_API= macro.
- [X] =ores.qt.api.lib= links =ores.qt.headless.lib= =PUBLIC=; every
  existing consumer's =#include "ores.qt/X.hpp"= updated to
  =#include "ores.qt.headless/X.hpp"= (physical move only — no
  behavioural change).
- [X] =ores.qt.api='s =tests/= directory retired (every test file in
  it was exactly the logic that moved); =ores.qt.headless.tests= is
  the new home, linking the real shared library rather than
  duplicate-compiling source files.
- [X] Verified on Linux: =ores.qt.headless.lib=/=tests=,
  =ores.qt.api.lib=, every =ores.qt.*= plugin, and =ores.qt.exe= all
  build and link clean; full test suite passes (198 assertions); the
  QA Validation Runner panel (the feature that most exercises this

* Plan

Investigated rather than immediately patched: the user pushed back on
a proposed quick fix ("define =ORES_QT_API_LIBRARY= for the test
target too") and asked to trace whether this was a genuine
architectural smell first. It was: Widgets-free, headless-testable
logic (=OrgDocRenderer=, =TestScenarioResultsWriter=,
=EnvironmentMetadata=, =HierarchyModelBuilder=, =FontUtils=,
=RepoFileFinder=, =parse_trade_instrument=/=IInstrumentFormPopulator=)
was physically living inside =ores.qt.api=, which as a whole requires
=Qt6::Widgets=/a display — forcing the test target to duplicate-
compile specific =.cpp= files to test headlessly, which is what broke
dllexport/dllimport consistency on MSVC.

Confirmed no existing =ores.qt.*= sub-component was already
Widgets-free (checked every sibling's =CMakeLists.txt= for
=Qt6::Widgets=/=ores.qt.lib=; all of them require it, since they all
link =ores.qt.api.lib=), so a new component was warranted rather than
relocating into an existing one.

Extracted =ores.qt.headless= (own =export.hpp=, macro
=ORES_QT_HEADLESS_API= deliberately distinct from =ores.qt.api='s
=ORES_QT_API= — since =ores.qt.api= consumes this component's headers
directly, both export headers can land in the same TU, and a shared
macro name would silently clash — last =#define= wins — rather than
fail loudly). Moved the header namespace too (=ores.qt.headless/X.hpp=,
not =ores.qt/X.hpp=), matching how every other split component in the
repo (=ores.trading.api= vs =.core=, etc.) uses the component's own
name as its header namespace. =FontUtils.hpp= (header-only, no
=ORES_QT_API= decoration, ~15 consumers across the app) moved too —
otherwise it would sit at the boundary and risk exactly the
same-macro-name clash it was designed to avoid.
