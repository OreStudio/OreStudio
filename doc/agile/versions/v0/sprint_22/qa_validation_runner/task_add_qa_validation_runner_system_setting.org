* Plan

Straightforward seeding, plus one real discovery along the way: system
settings turned out to be tenant-scoped end to end
(=system_settings_service::get_all()= only reads the caller's own
tenant, no fallback to the system tenant), and tenant provisioning only
ever seeded =system.bootstrap_mode= per-tenant. Confirmed live against
the =barclays= tenant — the setting existed (and was =true=) under
=system= but was completely absent under =barclays=, so the Testing
menu item was silently invisible after login despite the "system"-wide
setting saying otherwise.

Fixed at two levels: seeded the setting for =barclays= directly (via
=ores.shell='s =variability save-setting=, since a raw SQL upsert
requires =current_user= to resolve to a valid account username, which a
superuser psql connection doesn't satisfy once a tenant is past
bootstrap) and added it to the tenant provisioner's seeding block so
future tenants get it automatically. Also added
=SettingGatedActionController='s =default_when_missing= parameter as a
second line of defence — a feature meant to be on-by-default should
degrade gracefully if a setting row is ever missing, not vanish.

The broader gap (any =system.*= setting seeded only for =system=, not
per-tenant, silently breaks for every other tenant — almost certainly
also true of =system.synthetic_data_generation=) is out of scope for
this task and captured separately:
=doc/agile/product_backlog/inbox/system_settings_not_seeded_for_new_tenants.org=.

* Result

=system.qa_validation_runner_enabled= is seeded for the system tenant
and for every tenant provisioned from here on, defaulting to =true=.
=SettingGatedActionController= now supports failing open on a missing
setting. Verified live: barclays initially lacked the row (tenant
provisioning only seeds =bootstrap_mode=), the menu item was
confirmed invisible, then confirmed visible after seeding it via
=ores.shell='s =variability save-setting=. The general tenant-scoping
gap is filed as a capture for a future story.
