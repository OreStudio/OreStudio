* Goal

This is task 1 of 4 in the Acme provisioning architecture (see the
parent story's =* Decisions= for the full picture). This task covers
*only* the source data: a Python script that generates a clean,
versioned, extensible JSON dataset describing Acme Bank's four legal
entities —

- Acme Bank Group plc (holding company)
- Acme Bank UK plc (London)
- Acme Bank US Inc (New York)
- Acme Bank HK Ltd (Hong Kong)

— each with its own LEI, contact details, business units (trading
desks + Middle Office + Market Risk), portfolios, books, and staff
accounts. The shape (which desks, how many people) is fixed/hardcoded
per office; the filler data (names, emails, addresses) is generated
via Faker where the locale supports it, or a curated name pool where
it doesn't (Hong Kong). Config-driven over a list of offices, so
adding e.g. a Singapore office later is a config change, not new
code — the dataset is meant to grow as the system grows, to keep
exercising new functionality.

Output lands under =projects/ores.seeder/datasets/acme_bank/=
(manifest.json + methodology.txt + the generated data JSON),
mirroring the =external/facestudio=/=slovaris= convention but
internal, since this isn't third-party sourced data.

*Not* in scope for this task: turning the JSON into SQL (task 2, via
=ores.codegen=), the server-side orchestration to publish it
per-party (task 3), or the =ores.shell=/=ores.qt= UX (task 4) — see
sibling tasks on the parent story.

* Acceptance

- A Python script (checked in, re-runnable, not a one-off) generates
  a JSON file (or small set of JSON files) describing all four Acme
  entities' parties/LEIs, contacts, business units, portfolios,
  books, and staff accounts.
- LEIs are checksum-valid (ISO 17442 / mod-97) on GLEIF's reserved
  =9695= pre-LOU test prefix.
- The four-entity/office shape (which desks, headcounts) is defined
  as data/config in the script, not hardcoded per-row — adding a new
  office is a config change.
- Employee names/emails/addresses are generated per office locale:
  Faker for GB/US, a curated name pool for Hong Kong.
- All generated accounts reference one fixed, real password hash
  (scrypt hash of "Secure-Password-123", matching the
  =barclays_system_provision.ores= convention) computed via the
  actual =ores::security::crypto::password_hasher=, not a
  placeholder.
- Each employee account is linked to the specific legal entity
  (party) they work for in the JSON (not just the top holding
  company), so a later single publish pass can assign every account
  to its correct party.
- Output follows the =ores.seeder= dataset convention: =manifest.json=,
  =methodology.txt=, the data JSON, under
  =projects/ores.seeder/datasets/acme_bank/= — registered/discoverable
  the same way as =slovaris=/=external/facestudio=.
- No SQL, no server-side/service code, no shell/Qt changes in this
  task — those are tasks 2-4.
