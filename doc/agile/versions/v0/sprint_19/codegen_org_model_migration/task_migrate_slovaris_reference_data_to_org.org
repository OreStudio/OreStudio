* Result

ABANDONED — superseded by [[id:5104ADAF-390D-4ADB-B005-CE726B553123][Introduce ores.seeder component for
database test-data generation]].

The original scope was to migrate the 6 slovaris JSONs to
org-mode for uniformity with the other codegen-model migrations
in the parent story. A brainstorm before implementation
concluded:

1. The slovaris files are *data*, not codegen *models*. The
   prior migrations (entity, junction, table, lookup_entity,
   field_group, component, service_registry) were all
   source-code descriptors where literate org unlocked real
   value (generator expressions, FK rationale, prose context).
2. A 100-row org table for =country_currency.json= is just an
   uglier JSON list — no value added.
3. The right move is to keep JSON for bulk data but extract the
   machinery into a dedicated component (=ores.seeder=) with
   proper documentation, compass plumbing, and a roadmap toward
   a synthesised (Faker-driven) flavour that *will* benefit from
   the entity-model linkage the other migrations have built up
   (=generator_expr= column hints).

See [[id:5104ADAF-390D-4ADB-B005-CE726B553123][the seeder story]] for the new direction.
