* Goal

Prove that one org-mode file can be the *unified source* for both C++ and
SQL codegen for an entity, by migrating one entity — =party= — and
validating that codegen produces *byte-identical output* compared with the
current JSON-driven path (which today uses *two* separate JSON files: a
=_domain_entity.json= for C++ and a =_table.json= for SQL).

The pilot validates four claims simultaneously:

1. C++ JSON + SQL JSON structures both map cleanly onto a single org-mode
   file via headings, property drawers, prose, and babel blocks.
2. A small hand-written Python parser (regex + state machine, ~150 LOC)
   handles the restricted org subset we need without any external
   dependency.
3. Custom methods (=read_system_party=, paginated =read_latest=,
   =get_total_party_count=, =read_descendants=) become a natural part of
   the same literate file, eliminating the need for separate extension
   files or "restore from HEAD" workarounds.
4. Basic Python validation (~50 LOC) is sufficient — this is internal
   tooling, we are the only users, no need for a heavy schema framework.

If the POC succeeds, a follow-up story migrates the remaining ~100 models.
If it surfaces problems we hadn't anticipated, the JSON path stays and the
extension idea retreats to a simpler sidecar-fragment design.

** Why party

- It is the most complex refdata model: UUID PK with by-code lookup,
  multiple flags (=has_tenant_id=, =service_find_by_uuid=,
  =service_find_by_code=, =service_pagination=), missing-in-current-model
  fields (=codename=, =transliterated_name=, =party_category=).
- It has 4 custom repository methods that exercise the extension half of
  the design (=read_system_party=, paginated =read_latest=,
  =get_total_party_count=, =read_descendants=).
- It is heavily depended on by refdata.api callers, so build verification
  is a real test.

* Plan

** 1. Write a small Python org parser

No external dependency required. The org subset we need is small:

- =:PROPERTIES: ... :END:= drawers, with =:KEY: VALUE= entries.
- =#+keyword: value= frontmatter lines.
- Headings (=*=, =**=, =***=) with hierarchy.
- Plain prose body between headings.
- Named babel source blocks: =#+begin_src LANG :name NAME= ...
  =#+end_src=.

~150 LOC of regex + line-walking state machine. If we hit a real corner
case during implementation (e.g., escaped =#+= sequences inside source
blocks), we add =orgparse= then. Not before.

** 2. Design: unified org file structure for an entity

The org file replaces *both* the current C++ JSON
(=*_domain_entity.json=) and SQL JSON (=*_entity.json= /
=*_table.json=).

Frontmatter:

#+begin_example
,:PROPERTIES:
,:ID: <uuid>
,:END:
,#+title: party (refdata)
,#+description: ...
,#+type: entity
,#+component: refdata
,#+filetags: :model:entity:refdata:
#+end_example

Top-level sections:

- =* Flags= — drawer holds schema, has_tenant_id, service_find_by_*,
  has_batch_*, etc. (C++ codegen flags + SQL codegen flags).
- =* Primary key= — drawer holds =:column:= =:type:= =:cpp_type:=
  (SQL uses :type:, C++ uses both); body is the description.
- =* Natural keys= — each natural key as a =** <column>= sub-heading
  with drawer + body + babel block for =generator=.
- =* Columns= — same pattern as natural keys; one =**= sub-heading per
  column.
- =* SQL= — drawer holds =:tablename:=, optional =:triggers:=,
  optional =:indices:=. Sub-headings (=** Trigger XYZ=) hold any
  custom SQL fragments as babel src blocks.
- =* Repository conventions= — drawer holds =entity_singular_short=,
  =entity_plural_short=, =entity_singular_words=, =entity_plural_words=.
- =* C++ includes= — sub-headings =** Domain= and =** Entity= with the
  include lists as plain item lists.
- =* Qt= — large block of properties.
- =* Custom repository methods= — each method as a =** <name>=
  sub-heading with its own =:ID:= property, prose body explaining why
  it exists, and =declaration= + =implementation= babel blocks.

Each section is consumed by one or both paths:
- C++ codegen consumes: Flags (C++ subset), Primary key, Natural keys,
  Columns, Repository conventions, C++ includes, Qt, Custom repository
  methods.
- SQL codegen consumes: Flags (SQL subset), Primary key, Natural keys,
  Columns, SQL.

* Notes

** Prior art: Dogen's org-model codec

Marco's Dogen project already migrated to org-mode for modeling. Key takeaways
relevant to this POC ([[file:/home/marco/Development/mcraveiro.github.io/dogen/docs/literate_modeling_with_org_model.org][Dogen literate modeling chapter]]):

- *Tag-based dispatch* rather than title-matching. Headlines carry semantic
  tags like =:column:= or =:method:=. The parser dispatches on tags, not
  heading titles. Renaming a heading does not break codegen as long as tags
  stay. Adopt in *Stage 1.5* (after the basic POC works).
- *Natural-language headlines.* =* Customer's full name :column:= reads
  better than =* full_name=. The machine identifier lives in
  =:PROPERTIES: :name: full_name :END:=. Pairs with tag-based dispatch.
- *Stay close to native org-mode idioms.* Files should not surprise regular
  org-mode users.
- *Org-roam IDs for cross-document references.* Lines up with the
  Stage 2 template→block linkage by UUID.
- *Property drawers carry structured metadata*, headline body is the
  human-readable comment. (Already doing this.)

** Staged approach to template-block linkage

The POC will land in two stages:

*Stage 1 (this task)* — codegen reads org files, parses each section into a
dict shaped like the JSON model, and feeds it to the existing mustache
templates. For things that have a code body (includes, custom methods),
the parser does language-specific stripping (e.g. removing =#include= from
C++ include lines). This is *acceptable but inelegant* because the parser
has to understand the syntax of the language inside each block.

*Stage 2 (follow-up)* — once all JSON models have been migrated to org,
refactor the linkage so each named block carries its own =:ID:= UUID, and
mustache templates reference them by ID with a marker like =\<\<UUID\>\>= or
=\{\{ext:UUID\}\}=. Codegen then substitutes the block content verbatim, no
syntax parsing. This is the cleaner endgame; doing it now would require
either (a) coordinated updates to all templates and all models, or
(b) supporting both schemes during a long transition. Better to land
Stage 1 first.

** =orgparse= capability sanity check (to do before starting)

Confirm:
1. Reads =:ID:= from =:PROPERTIES:= drawers cleanly.
2. Captures =#+keyword:= frontmatter as accessible metadata.
3. Walks nested headings with body text intact.
4. Doesn't choke on Mustache-like syntax in babel blocks if any.

If any of these fail, fall back to a custom regex parser (~200 LOC) for the
restricted org subset we need.

** Mustache marker syntax (decide during implementation)

Two approaches:

| Approach | How | Tradeoff |
|----------+-----+----------|
| Post-render substitution | Output template has =<<EXT:NAME>>= literals; codegen replaces them with extension content after rendering. | Simple, no pystache changes. Markers visible during template review. |
| Custom Mustache partial | =\{\{> ext:NAME \}\}= with a custom partial loader. | More native to Mustache. Requires pystache configuration. |

Post-render is simpler; prefer unless it produces awkward template files.

** Org as model: pros and cons summary

| Pro | Con |
|-----+-----|
| Single source of "what + why" per entity. | More flexible structure than JSON; needs a validator. |
| Custom methods become inline, not sidecar. | ~100 models to migrate if POC succeeds. |
| Each section has =:ID:= for cross-referencing. | Org is more verbose than JSON for purely structured data. |
| Same literate pattern as the rest of ORE Studio. | Tooling (jsonschema, lsp completion) doesn't apply. |
| Generator expressions get babel-block treatment. | Org parsing is more brittle than JSON if the format drifts. |
