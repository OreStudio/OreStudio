* Steps

Preconditions: environment provisioned with a fresh DB, Barclays
tenant/party provisioned (=barclays_system_provision.ores=), all
services running, =ores.iam.service= left running throughout — do not
restart it at any point during this scenario.

** Log in as tenant_admin and confirm no stale-cache warning

Log in as =tenant_admin@barclays_plc= / =Secure-Password-123= via the
Qt client's =LoginDialog=. Confirm login succeeds with no "Party setup
completed, but the party is still marked Inactive" warning — the root
party (BARCLAYS PLC) was activated during provisioning while
=ores.iam.service= was already up, so this checks the cache picked up
that activation via the =ores.refdata.party_changed= notification
(not a restart).

** Deactivate the party and confirm the login warning appears

Using an admin path that sets the party's =status= back to =Inactive=
(e.g. re-running the relevant provisioning step, or a DB-level status
change through the normal save path — not a raw SQL edit bypassing
the app), then log out and log back in as =tenant_admin@barclays_plc=.
Confirm the login response now surfaces the "still marked Inactive"
warning, without restarting =ores.iam.service= — this confirms the
cache reloads on *every* change, not just the initial activation.

** Reactivate the party and confirm the warning clears

Reactivate the party via the same admin path. Log out and log back in
again. Confirm the warning is gone once more, still without
restarting =ores.iam.service= at any point across all three logins.
