* Goal

Walk every page of the three provisioner wizards —
=SystemProvisionerWizard= (7 pages), =TenantProvisioningWizard= (6
pages) and =PartyProvisioningWizard= (5 pages) in
=projects/ores.qt/application= — and produce a mapping from each page
to its shell equivalent. For every page record: the configuration it
gathers (fields, defaults, validation), the backend NATS messages its
execute step issues (e.g. =iam.v1.bootstrap.create-admin=,
=iam.v1.bootstrap.provision-tenant=, =iam.v1.tenants.complete-provisioning=,
=dq.v1.publish-bundle=, =synthetic.v1.generate-organisation=,
=reporting.v1.report-definitions.create=), the existing =ores.shell=
command that covers it (e.g. =bootstrap=, =add tenant=) or the gap
where none exists. The mapping feeds the interface brainstorm and the
decision task, and ultimately specifies the follow-up implementation
story.

Per the story's no-1-1-mapping decision, the wizard pages define
/capability coverage/, not the command surface: alongside the
coverage table, propose the idiomatic ores-shell interaction —
where a single command with flags should span several wizard pages,
where defaults remove inputs entirely, and where the shell medium
suggests a smoother flow than the GUI's.

** Cross-cutting shell gaps

1. *Workflow waiting.* Bundle publishes are async workflows; the
   wizards poll =workflow.v1.instances.steps= until terminal state.
   The shell has no equivalent — scripted provisioning needs a
   blocking wait (e.g. publish that waits, or =workflow wait
   <instance>=).
2. *Stop-on-error in scripts.* =load= continues after a failed
   command; a provisioning script must abort on failure or the run
   produces a half-provisioned system that "completed".
3. *Party context.* The shell session has tenant context from login
   but no notion of a current party, which the party flow requires
   (=currentPartyId=). Needs a party selector or explicit
   =--party= arguments.
4. *No parties/account-parties/bundles/reports commands* at all —
   these are new command groups, not extensions.
5. *Bootstrap subject discrepancy.* The shell =bootstrap= command and
   the wizard both use =create_initial_admin_request=; one report
   claimed the shell also receives tenant_id/tenant_name in the
   response. Verify during implementation whether =bootstrap= already
   provisions a default tenant or only creates the admin.

** Initial command-surface sketch (input to the brainstorm task)

Strawman only — the brainstorm task owns the real design:

- =provision system <user> <password> <email> [--multi-tenant]
  [--tenant-code ... --tenant-name ... --tenant-type ...
  --tenant-hostname ... --tenant-description ...]
  [--tenant-admin <user> --tenant-admin-password <pw>
  --tenant-admin-email <email>]= — spans wizard pages 2-6; defaults
  reproduce single-tenant mode exactly.
- =provision tenant [--bundle <code>] [--source gleif|synthetic]
  [--root-lei <lei>] [--dataset-size large|small] [--seed N]
  [synthetic knobs with wizard defaults]= — spans pages 2-5 including
  the workflow wait, association and finalize.
- =provision party [--dataset-size small|large] [--reports all|none]
  [--party <id>]= — spans pages 2-4.
- Supporting commands: =bundles list=, =reports templates=,
  =lei countries= / =lei entities <country>=, =workflow wait
  <instance>=, =parties list=.
- Script semantics: stop-on-error for =load= (flag or default).
