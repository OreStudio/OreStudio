* Goal

Audit profiles.json and components.json: understand what each models (profile → template/output mapping vs component → discovery roots), whether any data belongs elsewhere, and whether a merge makes sense or not. Document the decision. Then migrate both to org-mode — either as org tables with a tangle to JSON consumed by the loaders, or as a purpose-built org reader — whichever is simpler and consistent with the library convention.

* Acceptance

- The role of each file is documented (one paragraph each, in the file or a companion org).
- Both files are authored as org-mode source; manifest.py and generator.py (or equivalent) load from the org representation.
- No behavioural change: codegen output for every existing component and profile is identical before and after.

* Plan

*Reconciliation decision.* =profiles.json= and =components.json= model different
concerns and remain separate:

- =profiles.json= (now =facet_catalogue.org=): maps profile names to facet
  activation rules — which templates run, which model types each binds to, and
  how composite groups compose leaf facets. The /what-to-generate/ config;
  consumed by =generator.py= at generation time.
- =components.json= (now =component_catalogue.org=): maps component names to
  model-discovery roots — legacy JSON model dir, entity-glob filter, and
  co-located org modeling dir. The /where-to-find-models/ config; consumed by
  =manifest.py= at startup.

No merge. Neither file's data belongs in the other.

*Org representation.*

- =facet_catalogue.org= — 23 profiles in three sections (Facets / Facet groups /
  Component archetypes), each as a level-2 heading with a =:PROPERTIES:= drawer
  (=:description:=, =:model_types:=, =:includes:=) and for leaf profiles a
  templates table with an optional =Model types= column for per-template overrides.
- =component_catalogue.org= — 16 components in a single org table.

=generate_profiles.py= and =generate_components.py= under =scripts/= parse the
org sources and emit the JSON artefacts. =library-config-drift.yml= CI re-runs
both and compares parsed JSON — fails if either JSON was hand-edited.

* Notes

** profiles.json analysis (folded in from BFBFB825, 2026-06-10)

This task supersedes [[id:BFBFB825-18EF-4274-A72E-5924E69FFBAC][Migrate profiles.json to a literate org-mode
source]] (now ABANDONED), which scoped only the =profiles.json= half.
Its analysis applies directly here; the full handoff (23-profile
inventory table, detailed stale-pattern audit, step-by-step plan) lives
in that task's =* Notes= — read it before starting the profiles.json
side. The load-bearing points:

- *generator.py reads =profiles.json= directly* (=load_profiles=,
  =generate.py:38=, =generator.py:415/2294=). It needs *no change*: keep
  =profiles.json= as the live artefact and tangle/generate it from the
  org source. (Same pattern likely applies to =components.json= via
  =manifest.py= — confirm as part of this task's wider audit.)
- *Formatting gotcha (the zero-diff trap).* =profiles.json= emits
  =model_types= /inline/ but =templates= /multi-line/; a naive
  =json.dump(indent=2)= will not reproduce it. Preferred fix: the drift
  check compares /parsed JSON/ (=json.load= both sides, assert equal),
  not bytes — which is what =generator.py= actually consumes.
- *Stale output patterns.* =profiles.json= still carries output paths
  from the pre-regroup layout (capture [[id:12450E70-EF24-49D9-B156-808B7529932A][12450E70]]:
  =ores.qt.{component}=, =ores.{component_include}= forms targeting
  directories that no longer exist). Recommendation: migrate
  /JSON-identical first/ (pure format change, meaningful zero-diff
  check), then fix the paths as a separate pass on the org source
  (closing 12450E70). Do not trust a regex for the stale set — it needs
  the per-pattern audit the capture describes.
- *Don't duplicate prose.* The [[id:0BD6F9EC-34DD-44D9-8AA1-917EE5966396][ORE Studio Variability Model]] already
  documents the profile catalogue (the PMM, model-types scope filter,
  activation). The org source should /link/ to it and to the per-facet
  template docs, not restate them.

* Result

Audited the two library config files and represented both as org-mode literate sources:

- =library/facet_catalogue.org= (23 profiles, three sections: Facets / Facet
  groups / Component archetypes). Each profile is a level-2 heading with a
  =:PROPERTIES:= drawer and a templates table. Consumed by
  =scripts/generate_profiles.py= → =library/profiles.json=.
- =library/component_catalogue.org= (16 components, single org table with 5
  columns). Consumed by =scripts/generate_components.py= → =library/components.json=.

Both generators verify clean parsed-JSON round-trips against the committed files
on first run. =generator.py= and =manifest.py= are unchanged — they continue to
load the JSON artefacts.

=.github/workflows/library-config-drift.yml= re-runs both generators on every
PR touching the org sources, scripts, or JSON files and fails if the parsed JSON
diverges.

*Reconciliation decision:* the two files model different concerns (what-to-generate
vs where-to-find-models) and should remain separate. No merge.
