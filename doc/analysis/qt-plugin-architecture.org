* Background

=ores.qt= is currently a single monolithic project: ~800 source/header files,
~145K lines of C++, all compiled into one static library and one executable.
=MainWindow= directly owns 50+ ~unique_ptr<XxxController>~ members and must
include every domain controller header at compile time.

This document analyses how to split the monolith into a plugin-based
architecture and recommends a concrete approach.

* Current State

** Structure

| Item            | Count |
|-----------------+-------|
| Header files    |   406 |
| Source files    |   394 |
| Lines of code   | ~145K |
| Controllers     |    50 |
| Flat directory? | Yes   |

Everything lives in two flat directories: =include/ores.qt/= and =src/=.
There is no sub-organisation by domain.

** Dependency problem

The static library declares almost every API lib in the system as a PUBLIC
dependency:

#+begin_src cmake
target_link_libraries(ores.qt.lib PUBLIC
    ores.ore.lib ores.ore.api.lib ores.storage.lib
    ores.marketdata.api.lib ores.http.api.lib ores.refdata.core.lib
    ores.analytics.api.lib ores.trading.api.lib ores.compute.api.lib
    ores.scheduler.api.lib ores.reporting.api.lib ores.nats.lib
    ores.shell.lib ores.dq.api.lib ores.synthetic.api.lib
    ores.assets.api.lib ores.iam.api.lib ores.workflow.lib
    ores.connections.lib ores.eventing.lib ores.variability.api.lib
    ores.telemetry.lib ores.controller.api.lib ores.security.lib
    ores.utility.lib ores.platform.lib Qt6::Core Qt6::Gui ...)
#+end_src

This means any change anywhere in the system potentially rebuilds the entire Qt
application.

** Pain points

1. *Compilation time*: changing one dialog header causes the whole lib to
   rebuild (AUTOMOC scans every header).
2. *Cognitive load*: navigating 800 files in a single directory is hard;
   nothing communicates which files belong together.
3. *Coupling*: =MainWindow= cannot be compiled or tested without pulling in
   every domain — trading, compute, market data, IAM, etc.
4. *Windows build size*: the debug static lib exceeds WiX's 2 GB per-file
   limit, requiring an explicit workaround in CMake.
5. *Cross-controller coupling*: controllers emit signals like
   ~showRoundingTypesRequested()~ which =MainWindow= wires to another
   controller — encoding domain knowledge inside the host.

* Qt Idioms for Large Applications

Three canonical Qt patterns address the large-application problem.

** 1. Qt Plugin System (~QPluginLoader~)

Plugins are built as shared libraries (.so / .dll).  The host loads them at
runtime via ~QPluginLoader~.  Each plugin implements a pure-virtual interface
annotated with ~Q_PLUGIN_METADATA~ / ~Q_INTERFACES~.

/Pros/: true runtime extensibility; lazy loading; each plugin is an independent
build artefact.

/Cons/: requires shared libraries, which are painful on Windows with Qt (symbol
visibility, DLL export macros, debug vs release runtimes).  The Qt documentation
explicitly warns against mixing debug and release DLLs.  Also requires
distributing .so/.dll files alongside the executable.

** 2. Qt Static Plugins (~Q_IMPORT_PLUGIN~)

The same interface/metadata machinery as above, but the plugin is a static
library.  ~Q_IMPORT_PLUGIN(PluginClassName)~ in =main.cpp= forces the linker to
include the plugin's registration TU, avoiding the dead-stripping problem.

/Pros/: single binary, no DLL distribution, same interface as dynamic plugins
so the door is open to go dynamic later.

/Cons/: slightly heavier boilerplate (plugin metadata JSON files); relies on
Qt's plugin machinery which adds ~QJsonObject~ serialisation overhead.

** 3. Explicit Factory / Registry (no ~QPluginLoader~)

Each domain module is a static library that exports a plain
~register_xxx_plugin()~ C++ function.  A central ~PluginRegistry~ owns a
~vector<unique_ptr<IPlugin>>~.  =main.cpp= calls every registration function in
sequence before constructing ~QApplication~.

/Pros/: zero magic; deterministic initialisation order; no dependency on Qt
plugin metadata; easiest to debug; integrates naturally with the project's
existing pattern.

/Cons/: adding a new plugin requires touching =main.cpp=; not dynamically
extensible at runtime (acceptable for an internal application).

** Recommendation

Use *Explicit Factory / Registry* (option 3).  ORE Studio is a single-binary
internal application; runtime extensibility is not a goal.  The explicit
registration pattern is transparent, easy to understand, and avoids the
complications of Qt's shared-library plugin infrastructure on Windows.

The ~IPlugin~ interface (see Section [[*Proposed Interface]]) is identical to what
you would write for options 1 or 2, so migrating to ~QPluginLoader~ later
requires only adding the metadata annotations and rebuilding as a shared library
— the interface stays stable.

* Proposed Interface

** ~PluginContext~

Passed to each plugin on login.  Carries everything a plugin needs to interact
with the application.

#+begin_src cpp
// ores.qt.api/include/ores.qt.api/plugin_context.hpp
namespace ores::qt {

struct plugin_context {
    QMainWindow*   main_window    = nullptr;
    QMdiArea*      mdi_area       = nullptr;
    ClientManager* client_manager = nullptr;
    eventing::service::event_bus* event_bus = nullptr;

    // Shared caches (owned by ores.qt.app, passed down)
    ImageCache*        image_cache         = nullptr;
    BadgeCache*        badge_cache         = nullptr;
    ChangeReasonCache* change_reason_cache = nullptr;

    QString username;
};

} // namespace ores::qt
#+end_src

** ~IPlugin~

#+begin_src cpp
// ores.qt.api/include/ores.qt.api/i_plugin.hpp
namespace ores::qt {

class IPlugin : public QObject {
    Q_OBJECT
public:
    ~IPlugin() override = default;

    // Human-readable name for diagnostics
    virtual QString name() const = 0;

    // Called once at startup to set up any pre-login UI
    virtual void initialize(QMainWindow* mw, QMdiArea* area) = 0;

    // Called after successful login
    virtual void on_login(const plugin_context& ctx) = 0;

    // Called on logout / disconnect; plugins should close their windows
    virtual void on_logout() = 0;

    // Menus to insert into the host menu bar (before Help)
    virtual QList<QMenu*> create_menus() = 0;

    // Optional toolbar actions
    virtual QList<QAction*> toolbar_actions() { return {}; }
};

} // namespace ores::qt
#+end_src

** ~PluginRegistry~

#+begin_src cpp
// ores.qt.api/include/ores.qt.api/plugin_registry.hpp
namespace ores::qt {

class PluginRegistry {
public:
    static PluginRegistry& instance();

    void register_plugin(std::unique_ptr<IPlugin> plugin);
    const std::vector<std::unique_ptr<IPlugin>>& plugins() const;

private:
    PluginRegistry() = default;
    std::vector<std::unique_ptr<IPlugin>> plugins_;
};

} // namespace ores::qt
#+end_src

** Registration function pattern

Each domain plugin exports a plain function and calls it from =main.cpp=:

#+begin_src cpp
// ores.qt.refdata/include/ores.qt.refdata/register_plugin.hpp
namespace ores::qt::refdata {
void register_plugin();
}

// ores.qt.refdata/src/register_plugin.cpp
void ores::qt::refdata::register_plugin() {
    PluginRegistry::instance().register_plugin(
        std::make_unique<RefdataPlugin>());
}
#+end_src

#+begin_src cpp
// ores.qt.app/src/main.cpp
#include "ores.qt.refdata/register_plugin.hpp"
#include "ores.qt.party/register_plugin.hpp"
#include "ores.qt.trading/register_plugin.hpp"
#include "ores.qt.mktdata/register_plugin.hpp"
#include "ores.qt.admin/register_plugin.hpp"
#include "ores.qt.compute/register_plugin.hpp"

int main(int argc, char* argv[]) {
    ores::qt::refdata::register_plugin();
    ores::qt::party::register_plugin();
    ores::qt::trading::register_plugin();
    ores::qt::mktdata::register_plugin();
    ores::qt::admin::register_plugin();
    ores::qt::compute::register_plugin();

    QApplication app(argc, argv);
    // ...
}
#+end_src

** Simplified ~MainWindow~

Login and logout no longer require touching 50 controller pointers:

#+begin_src cpp
void MainWindow::createControllers(const plugin_context& ctx) {
    for (auto& plugin : PluginRegistry::instance().plugins()) {
        plugin->initialize(this, mdiArea_);
        plugin->on_login(ctx);
        for (auto* menu : plugin->create_menus())
            menuBar()->insertMenu(helpMenu_->menuAction(), menu);
    }
}

void MainWindow::performDisconnectCleanup() {
    for (auto& plugin : PluginRegistry::instance().plugins())
        plugin->on_logout();
    // close shared windows, clear caches, etc.
}
#+end_src

* Cross-Cutting Concerns

** Inter-plugin navigation

Some controllers signal navigation to another domain.  For example,
~CurrencyController::showRoundingTypesRequested()~ asks the app to open the
rounding-types list (which lives in refdata).

With the plugin model, cross-plugin navigation should go through the shared
event bus (=eventing::service::event_bus= already present in =ClientManager=
and threaded through =PluginContext=):

#+begin_src cpp
// currency controller wants to navigate to rounding types
plugin_context_.event_bus->publish("ui.show_rounding_types", {});

// refdata plugin subscribes in on_login():
ctx.event_bus->subscribe("ui.show_rounding_types",
    [this](auto&) { roundingTypeController_->showListWindow(); });
#+end_src

This eliminates hard cross-controller references in =MainWindow= and is
consistent with the rest of ORE Studio's event-driven design.

** Shared caches (~ChangeReasonCache~, ~ImageCache~, ~BadgeCache~)

These caches are consumed by many controllers across multiple plugins.  They are
owned and populated by =ores.qt.app= (after login) and passed into every plugin
via ~PluginContext~.  Placing them in =ores.qt.api= ensures every plugin can
depend on them without creating circular dependencies.

** ~ChangeReasonDialog~

~ChangeReasonDialog~ (the pick-a-reason modal) must be accessible to every
entity dialog.  It belongs in =ores.qt.api=, even though the change-reason
/entity management/ UI (list/edit/history) lives in =ores.qt.refdata=.  This
mirrors the existing pattern where =ChangeReasonCache= is already a shared
infrastructure type.

** ~PartyPickerDialog~

Used by trade detail dialogs in =ores.qt.trading= to select a counterparty.
Rather than placing it in =ores.qt.api= (which would pull party-domain headers
into the framework), keep it in =ores.qt.party= and let =ores.qt.trading= take
a compile-time dependency on =ores.qt.party=.  This is an explicit, intentional
domain coupling.

* Migration Path

A full cut-over at once is impractical given the size.  The recommended sequence
is:

** Phase 0 — Restructure in place (no functional change)

Introduce =ores.qt.api= as a new project containing only the framework files
extracted from =ores.qt=.  Move headers and sources to the new project.
=ores.qt= becomes a transitional name that will eventually become =ores.qt.app=.

At this stage the final binary is identical; the goal is to establish the new
dependency boundary and confirm it compiles cleanly.

** Phase 1 — Introduce plugin interface (no functional change)

Add =IPlugin=, =PluginContext=, =PluginRegistry= to =ores.qt.api=.  Create a
single =LegacyPlugin= in =ores.qt= (= the old =ores.qt.app=) that wraps all
existing controllers and implements the interface.  =MainWindow= drives the
plugin registry instead of the controller members directly.

The binary is again identical; all controllers still live in one place.

** Phase 2 — Extract domain plugins one at a time

Move one domain at a time (e.g. admin first as it is the most self-contained)
into its own project, implementing =IPlugin=.  The =LegacyPlugin= shrinks with
each extraction.  CI must remain green after each step.

Suggested order:
1. =ores.qt.admin= (fewest cross-domain signals)
2. =ores.qt.compute= (mainly compute API, no shared pickers)
3. =ores.qt.refdata= (large but self-contained)
4. =ores.qt.party= (depends on refdata for flag picker only)
5. =ores.qt.mktdata= (depends on refdata)
6. =ores.qt.trading= (depends on party)

** Phase 3 — Rename and clean up

Rename the remaining shell to =ores.qt.app=.  Delete the =LegacyPlugin= wrapper
if it is empty.  Update CMakeLists, documentation, and any CI configuration.

* Trade-offs and Open Questions

** Static vs shared libraries

Static plugins give a single binary and avoid DLL distribution complexity.  If
"hot-reload" of domain modules ever becomes a requirement, the interface is
already designed for it — just add ~Q_PLUGIN_METADATA~ and switch the CMake
target to ~SHARED~.  This is a valid future path, not wasted design.

** Plugin ordering

The explicit registration in =main.cpp= makes ordering deterministic.  If
plugin A's menus should appear before plugin B's, simply register A first.  With
static initialisers this would be platform-dependent.

** Testing

Each domain plugin is now a static lib with no dependency on =MainWindow=.  Unit
tests for, e.g., a refdata controller only need to link =ores.qt.refdata= and a
mock ~ClientManager~, not the entire application.  This was impossible in the
monolith.

** Build times

Splitting into 7+ sub-projects means CMake/Ninja can build them in parallel.
AUTOMOC only re-runs on files that actually changed in a given project, not the
entire 800-file set.  The expected build-time improvement for incremental builds
is substantial.

** =ores.qt.app= still depends on all domain APIs transitively

Via the plugin libs, the executable still transitively links every API lib.
This is unavoidable without runtime .so plugins.  The advantage is that
individual projects no longer need to be aware of unrelated domains.
