* Plan

Root cause, found by live investigation against Barclays Plc (not the
system tenant), then confirmed at the SQL level:

1. Reproduced live: the tree showed only =All Datasets= and an empty
   =Domains= folder; =Catalogs=/=Bundles=/=Dimensions= were missing
   entirely. Compared against a reference screenshot of the intended
   layout (business_units/portfolios/currencies-style domain tree,
   plus a =Catalogs= top-level branch).
2. Client logs showed =Domains=/=Catalogs= fetch requests succeeding
   with 0 rows (no error), while =Bundles=/=Dimensions=/=Subject
   Areas= fetches on the SAME session succeeded with real data.
3. DB inspection: =data_domain=/=catalog= rows exist only under the
   system tenant (never copied to Barclays), exactly like
   =subject_area= and the dimension tables that DO work. Postgres RLS
   policies for all of these tables are identical: SELECT allows own
   tenant OR system tenant (per =dq_rls_policies_create.sql='s
   documented "DQ governance data ... shared data pattern").
4. Found the actual bug: =data_domain_repository.cpp=/
   =catalog_repository.cpp= (and, latently, =change_reason=/
   =change_reason_category=/=badge_severity=/=badge_definition=) added
   an explicit =where("tenant_id"_c == tid ...)= filter on every read
   query -- strictly narrower than the RLS policy, silently dropping
   the system-tenant fallback rows. =subject_area=/=dataset_bundle=/
   the dimension repositories have no such filter and rely on RLS
   alone, which is why they worked.
5. Fixed properly in codegen, not as a one-off patch (per discussion
   with Marco): added a new =tenant_read_scope: shared= domain_entity
   model property (=** Repository= section), precomputed as
   =read_tenant_filtered= in =core.py=, and updated
   =cpp_domain_type_repository.cpp.mustache= (edited via its literate
   =.org= source, re-tangled) so READ queries omit the tenant filter
   for entities that opt in, while WRITE/DELETE queries keep it
   unconditionally. Applied the new property to all 8 affected DQ
   governance entities' =.org= models; regenerated via
   =compass codegen regenerate --component dq --address
   ores.cpp.repository=.
6. Two entities (=subject_area=, =dataset_bundle=) turned out to be
   far more architecturally stale than expected -- regenerating them
   changed public interfaces (removed a =ctx_=-member constructor in
   favour of per-call =ctx=), a much bigger blast radius than this fix
   warranted. Reverted their regeneration; kept the =.org= model flag
   (documents already-correct intended behaviour) but left their
   modernisation for a separate task.
7. Kept the other 6 entities' regenerated repositories
   (=catalog=, =data_domain=, =change_reason=,
   =change_reason_category=, =badge_severity=, =badge_definition=) --
   clean, minimal diffs (query bodies only, no signature changes).
   Reverted an unrelated =artefact_type= rewrite that got swept in by
   regenerating the whole component.
8. Independent second bug in the Qt client:
   =DataLibrarianWindow::buildNavigationTree()= had
   =if (dataDomainModel_->rowCount() == 0 && catalogModel_->rowCount()
   == 0) return;= -- since both were empty (bug #4), the entire tree
   build aborted, hiding =Bundles=/=Dimensions= too even though their
   data had loaded fine. Removed the guard; each section already
   rebuilds independently and safely from its own model.
9. Verified with a temporary raw-SQL diagnostic query (added, tested,
   then removed) executed via the exact same connection/tenant context
   as the real query, to rule out an RLS/connection-pooling
   explanation before concluding it was the =where= clause itself.
10. Verified end-to-end via a QA Validation Runner scenario against
    Barclays Plc on a freshly recreated database and rebuilt services:
    Domains, Catalogs, Bundles, and Dimensions all populate correctly;
    scenario recorded PASSED.

* Result

Root cause found and fixed properly in codegen. The Data Librarian
tree's =Domains=/=Catalogs= branches were empty for any tenant other
than =system= because =data_domain_repository=/=catalog_repository=
(and, latently, =change_reason=/=change_reason_category=/
=badge_severity=/=badge_definition=) added an app-level
=tenant_id = ctx.tenant_id()= filter to every read query -- narrower
than the Postgres RLS policy's documented "own tenant OR system
tenant" shared-reference-data pattern, silently hiding the
system-tenant-seeded rows every tenant is meant to see (exactly like
=subject_area=/=dataset_bundle=/the dimension tables, which have no
such filter and worked correctly).

Added a new =tenant_read_scope: shared= codegen model property
(precomputed as =read_tenant_filtered= in =core.py=) and updated
=cpp_domain_type_repository.cpp.mustache= (via its literate =.org=
source) so read queries omit the tenant filter for entities that opt
in, relying on RLS alone; write/delete queries are unaffected.
Regenerated 6 of the 8 affected entities cleanly; deferred
=subject_area=/=dataset_bundle= to a separate modernisation task since
regenerating them changes their public interface. Also fixed an
independent Qt bug: =buildNavigationTree()='s early-return guard hid
Bundles/Dimensions whenever Domains+Catalogs were both empty.

Verified end-to-end via a QA Validation Runner scenario against a
freshly recreated database and rebuilt services: Domains, Catalogs,
Bundles, and Dimensions all populate correctly for Barclays Plc.
Scenario recorded PASSED. Acceptance met in full.
