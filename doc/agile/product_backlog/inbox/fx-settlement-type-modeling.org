#+title: Model FX settlement type per currency pair (physical vs. NDF, business-centre dependent)
#+description: Currency pair deliverability isn't a single boolean: a pair can have multiple settlement types (physical/cash-NDF), sometimes business-centre dependent, requiring a proper settlement-convention junction model instead of a currency_pair.non_deliverable flag.

This page is a [[id:671F18E4-E09C-4B3B-BD24-D33DF8AE38A6][capture]] in the *inbox* bucket of the product backlog — a pre-sprint idea, not yet pulled into a sprint as a story.

* What

Model currency pair deliverability properly: a =Settlement Market
Convention= entity, many-to-one from =Currency Pair= and from
=Business Center= (a pair can have different settlement types
depending on where the trade is booked), carrying settlement type
(physical/cash-NDF), whether it's the default for the pair globally
vs. for a specific location, and optional settlement/fixing
currencies. See [[id:32F89EFA-6CF9-4A14-A2C9-6A7B59802123][FX Settlement Market Convention Data Model]] for the
full structural analysis (entities, relationships, holiday-calendar
handling).

* Why

A =currency_pair.deliverable=/=non_deliverable= boolean (with an
optional =settlement_currency=) was tried and reverted twice this
sprint — first as a straight rename, then as a drop — once it became
clear a single boolean per pair can't express reality: some pairs
settle differently depending on the trading location/business centre,
and in principle a pair could support more than one settlement type.
Needs a real junction model, not another boolean, before
reintroducing deliverability/settlement to =currency_pair=.
