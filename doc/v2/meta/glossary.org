#+title: Glossary
#+description: Definitions of every load-bearing term in the ORE Studio information architecture.
#+type: meta
#+level: cross
#+filetags: :meta:glossary:terminology:
#+created: 2026-05-18
#+updated: 2026-05-18

* Task

The smallest unit of executable work. One task = one PR. Bounded by a
concrete size limit (see [[id:8b29a8d8-e004-4e87-b81d-abeaf68f2b98][document types]]). Owned by System 1.

* Story

A coherent user-visible outcome, composed of one or more tasks. The unit of
sprint planning. Owned by System 2.

* Sprint

A bounded period of work delivering a set of stories aligned to a sprint
mission. Owned by System 3.

* Sprint mission

A one-sentence statement of what the sprint exists to achieve. Used to
filter candidate stories. Owned by System 3.

* Version

A major release target (e.g. 1.0). A coherent set of sprints. Owned by
System 4.

* Version identity

The mini-mission of a version: what it exists to deliver, what makes it
distinct. Owned by System 4.

* Product identity

The durable statement of what ORE Studio is and is not. Used to filter
candidate versions, far stories, and feature requests. Owned by System 5.

* Function

A named role that operates the system. Each function is one document
describing its purpose, inputs, outputs, and the recipes it uses. Spans
levels. Examples: PR Reviewer, Sprint Planner, Identity Steward.

* Recipe

A how-to document answering one operational question (e.g. "how do I run
the tests"). Has executable scripts where applicable. Tested by CI.

* Knowledge document

A durable factual document about the domain, architecture, or components.
Not state, not process — just facts that outlive any sprint.

* Plan

A transient implementation strategy for a single task or story. Lives next
to the work, dies with it. Key decisions distil into a =decisions.org= on
the story before the plan is archived.

* Discovered task

Work spotted in passing while executing another task. Filed without triage
into =tasks/discovered/=; processed later by System 2 or System 3.

* Backlog

A TODO state, not a folder. Tasks and stories in =BACKLOG= have been
triaged but not started. The product backlog (work not yet pulled into
a sprint) is the union of =BACKLOG= stories outside the current sprint,
plus the broader product wishlist; it is split into =near/= and =far/=
collections at the product-identity level.

* TODO state

The TODO keyword on a stateful document's =* Status= headline. The full
vocabulary is =DISCOVERED BACKLOG STARTED BLOCKED | DONE ABANDONED=
(active states left of the bar, terminal states right). Documents that
are not stateful (knowledge, recipes, functions, meta) have no Status
headline.

* Status headline

The =* Status= headline inside a stateful document. Carries the TODO
keyword (state) and four subheadings (=** Now=, =** Waiting on=,
=** Next=, =** Last touched=) that describe the live operational
detail. Updated every time an orchestrator or agent acts on the
document.

* Predecessor story

A story in a prior sprint that the current story continues. Captured
as =#+predecessor: <id>= frontmatter plus a =Continued from: [[id:...][...]]=
line in the body. See cross-sprint stories in [[id:8810e0f9-c5b9-4cfa-a710-774ad6c771dd][lifecycle]].

* Successor story

A story in a later sprint that picks up where this one left off.
Captured as =#+successor: <id>= frontmatter plus a =Continued in:
[[id:...][...]]= note in =* Decisions=. Audit verifies bidirectional
linkage.

* Agent

A subprocess forked by a System 2 orchestrator LLM to execute one task.
Operates at System 1. See [[id:f58ca9e7-c65f-4d17-a9a7-95fde25b9ebd][execution model]].

* Orchestrator

The System 2 LLM session the user has assigned to a story. Forks agents,
reviews their output, decides what to do with discovered work.
