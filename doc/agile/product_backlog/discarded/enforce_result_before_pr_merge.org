*Resolved (deep backlog refinement, 2026-07-11)*: projects/ores.compass/src/compass_pr.py lines ~355-381 already implement a guard: 'Guard: * Result section must have substantive prose' and 'task * Result section is empty — write the result before ...' warning logic, matching the capture's exact ask.

* What

Add a guard to =compass pr merge= that checks whether the active
task's =* Result= section is non-empty before proceeding. If the
section is missing or still contains the boilerplate placeholder, the
command should warn (or optionally block with =--require-result=) and
remind the user to write the result, commit, and push first. As a
lighter-weight complement, add a one-line note to the
=pr-address-review= skill and the =Handle a PR review round= runbook
(=doc/llm/runbooks/handle_pr_review_round/runbook.org=) calling out
the ordering constraint explicitly: "write =* Result=, commit, push,
/then/ =compass pr merge=".

* Why

During the PR #1156 post-mortem the result was written after the
force-merge rather than before. Because the remote branch had already
been deleted by the merge, the result commit landed on a
zombie branch and required a cleanup PR (#1158) to reach main. The
runbook already states the correct order but nothing enforces or
prominently flags it, so it is easy to skip under time pressure.
