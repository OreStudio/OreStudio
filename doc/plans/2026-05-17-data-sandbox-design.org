* Note on naming

The concept is called "Data Sandbox" in this document as a working name.  The
UI component that manages sandboxes needs a better name.  Candidates:

- /Data Workshop/ — implies active exploration; not overloaded in finance
- /Sandbox Explorer/ — clear and descriptive
- /Study Manager/ — academic connotation; good for the educational use case

The existing product backlog used "workspace" for a client-side caching concept
(later renamed to ~ores.caching~), so "workspace" is available but carries some
baggage.  Final naming decision deferred.

* Problem Statement

** Two import tracks with incompatible data isolation requirements

The [[file:2026-05-16-trade-import-failed-state.org][Data Import Session]] design identified two fundamentally different import
use cases:

*Track 1 — Educational / ORE sample import*: A complete ORE example is
imported so that a user can explore what it does — run it, see the results,
understand the trade types and market configuration.  Each ORE sample is a
self-contained universe: its own market data, conventions, curve configs,
pricing engine, and trades.  Importing multiple samples into the same data
space causes them to collide: curve names clash, trade IDs overlap, market
data overwrites, and results become meaningless.

*Track 2 — Production trade import*: Trades from an external system (FpML,
CSV, another TMS) are imported into the live book for review and eventual
booking.  Market data comes from live curated sources.  This track does not
need isolation — it targets the shared live data space.

The ~is_live~ flag and Pending Book design handles Track 2.  Track 1 has no
solution without data isolation.

** No way to reproduce an ORE sample run faithfully

Even if isolation is achieved via flags, there is no mechanism to re-run an
ORE sample and get the same results, because the ORE orchestration XML
(~ore.xml~) — which defines exactly which analytics to run against which data
— is not persisted in the system.  Without storing the orchestration
definition, OreStudio cannot drive the compute grid to reproduce the original
run.

** Live data space is polluted by sample data

Without isolation, importing 30 ORE samples creates hundreds of synthetic
trades, thousands of curve config rows, and dozens of conventions in the live
books — all mixed with real production data.  Users cannot distinguish sample
artefacts from curated data.

* Core Concept: Data Sandbox

A *Data Sandbox* is a named, isolated data context within OreStudio.  It
contains a complete and self-contained copy of all data types — trades, market
data, conventions, curve configs, pricing engine configs, today's market
definitions, and any report/analysis definitions.

Key properties:

- *Isolated*: data in sandbox A is never visible to sandbox B.  Queries within
  a sandbox see only that sandbox's data.
- *Self-contained*: a sandbox has everything needed to reproduce the ORE run(s)
  it was created from, without depending on data outside it.
- *Durable*: sandboxes are long-lived; they are not deleted when the session
  ends.
- *Readonly-by-default*: sandbox data is not editable in the same way live data
  is.  Edits are possible but explicit.
- *Non-promotable*: data cannot be directly "promoted" from a sandbox to live.
  Instead, a deliberate /copy/ operation is used (see below).

** Sandbox 0 — the live sandbox

Sandbox with ~id = 0~ is the live data space.  It always exists, cannot be
deleted, and is the default context for all UI windows.  All existing data in
OreStudio implicitly belongs to sandbox 0.

Using ~id = 0~ rather than ~NULL~ means:
- Uniform FK semantics across all tables (no nullable FK special-casing).
- Existing queries need only a ~WHERE workspace_id = 0~ clause added, not a
  ~WHERE workspace_id IS NULL~ pattern.
- The live sandbox participates in copy operations like any other sandbox.

* Data Model

** Sandbox table

#+begin_src sql
create table ores_workspaces_tbl (
    id              integer      not null generated always as identity primary key,
    name            text         not null,
    description     text         not null default '',
    source_path     text         not null default '',   -- original import path, if any
    created_at      timestamptz  not null default now(),
    created_by      text         not null,
    status          text         not null default 'active',  -- active | archived
    constraint ores_workspaces_tbl_name_uq unique (name)
);

-- Seed the live sandbox (id must be 0; override the identity sequence)
insert into ores_workspaces_tbl (id, name, description, created_by)
overriding system value
values (0, 'Live', 'Live production data space', 'system');
#+end_src

** workspace_id column on all major tables

Every table that holds user or import data gains a ~workspace_id~ column:

#+begin_src sql
-- Pattern applied to each affected table:
alter table <table_name>
    add column workspace_id integer not null default 0
    references ores_workspaces_tbl(id);

create index on <table_name> (workspace_id);
#+end_src

Affected tables (non-exhaustive; full list in implementation phase):

| Category | Tables |
|----------+--------|
| Trading | ~ores_trading_trades_tbl~, ~ores_trading_books_tbl~, ~ores_trading_instruments_tbl~ |
| Reference data | ~ores_refdata_conventions_tbl~, ~ores_refdata_currencies_tbl~, etc. |
| Market data | ~ores_marketdata_*_tbl~ |
| Curve config | ~ores_curveconfig_*_tbl~ |
| Pricing engine | ~ores_pricingengine_*_tbl~ |
| Today's market | ~ores_todaysmarket_*_tbl~ |
| Instruments | ~ores_instrument_*_tbl~ |

Tables that are genuinely global (users, IAM, parties, DQ FSM definitions,
workflow definitions) do *not* get ~workspace_id~.  These are system-level and
belong to no specific sandbox.

** Analysis definitions (ore.xml storage)

Each ~ore.xml~ imported into a sandbox is stored as an *analysis definition*:

#+begin_src sql
create table ores_workspace_analyses_tbl (
    id              uuid         not null default gen_random_uuid() primary key,
    workspace_id    integer      not null references ores_workspaces_tbl(id),
    name            text         not null,       -- display name (heuristic from path)
    source_path     text         not null,       -- relative path of ore.xml in import
    ore_xml_content text         not null,       -- full ore.xml content, verbatim
    created_at      timestamptz  not null default now(),
    last_run_at     timestamptz,
    constraint ores_workspace_analyses_name_wks_uq unique (workspace_id, name)
);
#+end_src

This allows OreStudio to re-run the original ORE analytics against the sandbox
data at any time, using the preserved orchestration XML.  The compute job
generator reads this record to reconstruct the ~ore.xml~, replacing file
references with generated exports from the sandbox's data.

* Import Track Mapping

| Track | Source | Target sandbox | Mechanism |
|-------+--------+----------------+-----------|
| Educational | ORE sample directory | New named sandbox | Importer creates sandbox per batch; all data lands there |
| Production | External trades (FpML, CSV) | Sandbox 0 (live) | Data Import Session; Pending Book; ~is_live~ flags |

For the educational track, the batch/unit hierarchy from the import analysis
maps to sandboxes as follows:

- *One sandbox per top-level group* (e.g. one "MarketRisk" sandbox).  Data
  shared across analyses in the same group (common conventions, curveconfig,
  shared market data) is stored once in the sandbox and shared.
- *One analysis definition per ~ore*.xml~* within the sandbox.  The analysis
  definition stores the orchestration XML and references the sandbox's shared
  data.
- Trades are tagged with both ~workspace_id~ (the sandbox) and ~analysis_id~
  (the specific orchestration that imported them), so that NPV results and
  errors can be scoped per analysis without the analyses trampling each other.

For groups where all ~ore_*.xml~ files reference the *same* portfolio (e.g.
~MarketRisk~ running different analytics on the same trades), the trades are
stored once and each analysis definition references them.  For groups where
each ~ore_*.xml~ uses a *different* portfolio (e.g. ~AmericanMonteCarlo~), the
trades are separate rows sharing the same ~workspace_id~ but different
~analysis_id~.

* No Inheritance — Copy Instead

Sandbox inheritance (overlay semantics) adds substantial complexity for limited
gain in this context.  The chosen approach is explicit copy:

- *Copy sandbox → sandbox*: duplicates all rows from source sandbox into target
  sandbox.  Both sandboxes then evolve independently.
- *Copy sandbox → live (sandbox 0)*: all copied rows enter the Pending Book /
  ~is_live = false~ state and require user approval before becoming live.  This
  is the deliberate "promote to live" pathway.
- *Copy selection*: users can copy individual trades, specific data categories
  (just conventions, just market data), or the entire sandbox.

Shared data duplication is acceptable for the ORE samples use case given the
data volumes involved.  If performance becomes a concern, copy-on-write can be
introduced later without changing the external interface.

* UI Design

** Per-window sandbox context

Each MDI subwindow operates in a sandbox context.  The context is:

- Shown in the window's title bar or header: e.g. ~[MarketRisk]~ in brackets
  after the window title for non-live sandboxes.  Live sandbox (0) shows no
  badge — it is the default.
- Selectable per window: a small sandbox selector appears in the window's
  toolbar or header area, allowing the user to switch a window between
  sandboxes without closing and reopening it.
- *Not global*: different windows can be in different sandboxes simultaneously.
  A user might have the Trades window showing ~MarketRisk~ data while the
  Market Data window shows ~Live~ data.

#+begin_example
+-------------------------------------------------------+
|  TRADES  [MarketRisk ▼]              [Import] [Filter] |
+-------------------------------------------------------+
|  ...trade grid scoped to MarketRisk sandbox...         |
+-------------------------------------------------------+
#+end_example

** Data Sandbox UI component (working name: Data Workshop)

A dedicated window accessible from the main menu (Data Transfer → Data
Workshop, or similar).  Shows:

#+begin_example
+----------------------------------------------------------+
|  DATA WORKSHOP                        [New] [Import ORE] |
+----------------------------------------------------------+
| ▶ Live                        0 trades  —  live          |
| ▼ MarketRisk                 27 analyses · 312 trades    |
|     ore_sensi                 12 trades  last run 10:15  |
|     ore_hist                  12 trades  last run 10:17  |
|     ore_correlation           12 trades  not yet run     |
|     ...                                                  |
| ▶ Legacy · Example_1          1 analysis · 12 trades     |
| ▶ Legacy · Example_2          1 analysis · 3 trades      |
| ...                                                      |
+----------------------------------------------------------+
| [ Open ] [ Run Analysis ] [ Copy to... ] [ Archive ]     |
+----------------------------------------------------------+
#+end_example

Actions:
- *Open*: opens the selected sandbox in the Data Import Session window (scoped
  to that sandbox), or focuses an already-open window for it.
- *Run Analysis*: submits the selected analysis definition to the compute grid
  for NPV computation.  Prompts for compute configuration if needed.
- *Copy to...*: copies selected sandbox (or selection within it) to another
  sandbox or to Live.  If copying to Live, shows a warning that data will enter
  the pending/approval workflow.
- *Archive*: marks sandbox as archived; data is retained but hidden from normal
  views.

** Sandbox indicator in other windows

All windows that display sandbox-scoped data show a sandbox chip/badge in
their header.  Clicking the chip opens the sandbox selector.  Windows default
to Live (sandbox 0) on first open.

* Compute Integration

Each analysis definition contains the original ~ore.xml~.  To run it:

1. Export the sandbox's trades (scoped to that analysis's ~analysis_id~) as a
   ~portfolio.xml~ file.
2. Export the sandbox's market data, conventions, curveconfig, pricingengine,
   and todaysmarket as the corresponding ORE input files.
3. Substitute file paths in the stored ~ore.xml~ with the generated file paths.
4. Override the ~<Analytics>~ section to request only ~npv~ (for valuation
   runs; full analytics for reproduce runs).
5. Submit to the compute grid as a normal ORE job.
6. Receive results; store NPVs back on trade rows tagged with this ~analysis_id~.

For educational runs: step 4 can be skipped — the original analytics are run
verbatim to reproduce the sample's original output.

For production valuation: step 4 strips expensive analytics so the job is fast.

* Relationship to Existing Plans

** Data Import Session plan

The [[file:2026-05-16-trade-import-failed-state.org][Data Import Session]] plan remains valid for Track 2 (production import).
Track 1 import replaces the "batch session" concept with sandbox creation:
instead of creating ~ores_data_import_sessions_tbl~ rows, the importer creates
an ~ores_workspaces_tbl~ row and populates it with all data from the sample.
The Data Import Session window is then used to browse the sandbox's data in
exactly the same way, since it is already scoped by sandbox context.

The ~ores_data_import_sessions_tbl~ and its batch/unit hierarchy may be
retained as audit/lineage metadata (recording what was imported when), even
though the primary isolation mechanism is the sandbox.

** is_live flags and Pending Book

These remain Track 2 concerns only.  Within a sandbox, all data is
unconditionally the sandbox's data — there is no ~is_live~ distinction needed,
since the sandbox boundary provides the isolation.  The ~is_live~ flag only
becomes relevant when data is copied from a sandbox into the live workspace (0).

* Implementation Phases

** Phase 1: Core schema

1. Create ~ores_workspaces_tbl~; seed Live sandbox (id = 0).
2. Create ~ores_workspace_analyses_tbl~.
3. Add ~workspace_id integer not null default 0~ to all affected tables.
4. Add indexes on ~workspace_id~.
5. Run ~recreate_database.sh~; validate schemas.

** Phase 2: Service layer

1. All service queries gain a ~workspace_id~ filter parameter.
2. NATS message protocol: add optional ~workspace_id~ field (defaults to 0) to
   all list/get/create requests.
3. New NATS endpoints: ~workspace.v1.list~, ~workspace.v1.create~,
   ~workspace.v1.archive~, ~workspace.v1.copy~.
4. Analysis definition endpoints: ~workspace.v1.analysis.list~,
   ~workspace.v1.analysis.create~, ~workspace.v1.analysis.run~.

** Phase 3: UI — sandbox context propagation

1. Add ~workspace_id~ to application session state.
2. Propagate sandbox context to all MDI windows via a ~WorkspaceContext~ object.
3. Add sandbox chip/badge to window headers.
4. Add per-window sandbox selector (combo box in toolbar).

** Phase 4: Data Workshop UI

1. Sandbox tree view (grouped by top-level import path).
2. Analysis list within sandbox.
3. Open / Run Analysis / Copy to / Archive actions.

** Phase 5: ORE sample importer

1. Implement recursive orchestration XML discoverer (per analysis section of
   [[file:2026-05-16-trade-import-failed-state.org][Data Import Session]] plan).
2. Create sandbox per top-level group; create analysis per ~ore*.xml~.
3. Import all data types into the sandbox.
4. Store orchestration XML verbatim in ~ores_workspace_analyses_tbl~.

* Open Questions

1. *Sandbox selector granularity*: should the per-window selector show a flat
   list of all sandboxes, or a grouped tree (Live / sample groups / other)?
   For 30+ sandboxes a tree is necessary.

2. *Analysis_id on trade rows*: is a separate ~analysis_id~ UUID column on
   ~ores_trading_trades_tbl~ the right mechanism, or should it be part of
   ~source_position~?  Affects how NPV results are partitioned per analysis.

3. *Shared data within a sandbox*: conventions and curveconfig are shared across
   analyses in the same sandbox.  If two analyses in MarketRisk use different
   conventions for the same curve name, which wins?  Needs a conflict resolution
   policy.

4. *Archive semantics*: does archiving a sandbox immediately hide it from
   selectors, or after a grace period?  Can archived sandboxes be restored?

5. *Sandbox storage quota*: large samples (e.g. full ORE examples = hundreds of
   thousands of rows) may need per-sandbox storage reporting.  The product
   backlog mentions a Baobab-style map for workspace visualisation — relevant
   here.

6. *UI name*: "Data Workshop" vs "Sandbox Explorer" vs other.  Decide before
   implementation begins.
