* Goal

The [[id:6D953558-A61C-461F-859D-CBD28AD5D8F5][CRM DQ publishing task]] created three new tables
(=ores_marketdata_crm_topology_configs_tbl=,
=ores_marketdata_crm_driver_pairs_tbl=,
=ores_marketdata_crm_enabled_derived_pairs_tbl=) with no entity layer
above raw SQL -- no domain type, repository, service, HTTP endpoint,
shell command, or Qt UI exists for any of them (confirmed: zero Qt
references to =crm_topology= in the codebase today). Use the
=codegen-add-entity= orchestrator to generate the full stack for all
three entities (domain JSON/table I/O, repository, service,
=ores.http.server= REST endpoints, =ores.shell= CLI commands, Qt list
and detail views), then wire the new list view(s) into the
reference-data menu, so the DQ-published "majors"/"exotics" CRM
bundles are inspectable and testable through the app UI ahead of --
and independent of -- the still-blocked cross-rates matrix UI panel.

- Full-stack entities exist for =crm_topology_config=,
  =crm_driver_pair=, and =crm_enabled_derived_pair=: domain type with
  JSON/table I/O, repository, service, messaging, Qt list window +
  detail/history dialogs -- following the same codegen conventions as
  other refdata entities. (HTTP CRUD endpoints and shell list/history
  commands deferred to a backlog capture, matching the =contact_type=
  precedent -- see Result.)
- The =crm_topology_config= list view (and its driver/derived pair
  detail) is reachable from the reference-data menu in the Qt client,
  not just creatable via raw SQL or DQ publish.
- After a synthetic party publish, the UI shows both "majors" and
  "exotics" configs for the party with correct driver/derived pair
  counts (8+3 and 3+0 respectively), matching what was already
  confirmed at the SQL layer.
- Full local build clean, =ctest= green.

* Plan

Generated Qt UI (list/detail/history/controller) via the =ores.cpp.qt=
codegen address for all three entities, initially under
=ores.marketdata= (matching where their domain/repository/service
layers already lived from the earlier, separately-merged CRM DQ
publishing task). Manual QA then surfaced a classification question:
by the project's own definition of refdata ("configuration data that
changes infrequently"), these three entities -- created once per
party, edited rarely -- belong in =ores.refdata= alongside
=book=/=portfolio=, not in =ores.marketdata= (genuinely live/streamed
data). Reclassified across every layer: moved the codegen models,
regenerated domain/repository/service/protocol/generator/eventing/
messaging/presentation/Qt under =ores.refdata= /
=ores.qt.refdata=, renamed the SQL tables, the publish-from-dq
function, the NATS subject, and the DQ dataset code from
=marketdata.*= to =refdata.*=, and repointed =crm_ingest_bridge=
(which legitimately stays in =ores.marketdata.service= as a
cross-service consumer of the rate-engine data) at the new
tables/channels.

Follow-on manual-QA fixes: moved the Qt menu entry from Market Data to
a new Reference Data > "Cross Rates Matrix" submenu (documenting the
placement as a considered decision, not an improvisation); removed
=party_id= from the UI entirely (it's implicit from the active-party
session context everywhere else in the codebase -- =book= and
=party_identifier= both hide it, this broke that convention);
kept =config_id= list-only (hidden by default, not in the editable
Detail form, since there's no Explorer/parent-relationship wiring yet
to make it fully implicit); made
=base_currency_code=/=quote_currency_code=/=pivot_currency_code=
flagged combo boxes with per-row flag icons (matching
=CurrencyPair=/=Book='s pattern) instead of plain text entry; and
expanded the DQ-seeded "majors" derived-pair sample from 3 rows to the
complete C(8,2)=28 combinatorial set of crosses among the 8 majors
driver currencies, a genuinely representative sample.

Also fixed, as a byproduct of investigating a missing Qt menu entry: a
codegen bug where Qt Detail-field UUID detection
(=is_uuid=/=is_optional_uuid=, driving
=boost::uuids::to_string()=/conversion code generation) only looked at
an entity's =columns=, never its =natural_keys= -- any UUID-typed
natural key (not the primary key, which had its own knob) silently
fell through as a plain =std::string= field and produced
non-compiling generated code.

* Result

rows to the complete 28-row combinatorial set.
