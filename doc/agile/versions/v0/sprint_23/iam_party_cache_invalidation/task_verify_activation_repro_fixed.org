* Acceptance

- Environment recreated/started with the fix applied.
- A party is activated via the normal =save_party_request= path
  (=barclays_system_provision.ores= or equivalent), on an IAM
  instance that already cached that party pre-activation.
- A subsequent IAM login shows the party as Active — no "still
  marked Inactive" warning — without restarting the IAM service.
- Result recorded in this task's =* Result= section.

* Plan

1. Recreate the DB (=compass db recreate -y -k=) and start the full
   service fleet.
2. Run =barclays_system_provision.ores= via =compass shell -f= — its
   last step activates the root party (=BARCLAYS PLC=) via the real
   messaging path while =ores.iam.service= is already running.
3. Confirm via SQL that the party's =status = 'Active'= in the DB.
4. Log in as =tenant_admin@barclays_plc= via =compass shell= and
   confirm the login succeeds with no stale-cache warning.
5. Cross-check =ores.iam.service.0.log= for =party_cache= reload
   activity around the activation step, and confirm no "still marked
   Inactive" warning around the login.

* Notes

Ran against a freshly recreated DB with the full service fleet up
(services never restarted across the session):

- =barclays_system_provision.ores= completed all 10 commands; its
  final step logged =✓ Party 'BARCLAYS PLC' provisioned and active.=
- SQL directly confirms: =BARCLAYS PLC= (id
  =1965b8e9-49c3-41f4-8080-10a70b5fff60=) has =status = 'Active'= in
  =ores_refdata_parties_tbl= (current version, =valid_to =
  9999-12-31 23:59:59=).
- =ores.iam.service.0.log= shows a burst of =party_cache= "Loaded 82
  parties for tenant daecbe07-...=" entries during provisioning
  (the reload-on-notification path firing repeatedly as the tenant's
  parties/counterparties were imported and the root party activated),
  followed later by a =tenant_admin@barclays_plc= login that
  completed (=auth_handler= "Completed ... auth.login") with **no**
  "still marked Inactive" warning logged anywhere in the file.
- =ores.iam.service= was never restarted at any point in this
  session — the DB recreate, provisioning, and login all happened
  against the one continuously-running instance.

This reproduces the original capture's scenario and confirms the fix:
activation via the normal message path is picked up by IAM's cache
without a restart. Companion manual scenario
