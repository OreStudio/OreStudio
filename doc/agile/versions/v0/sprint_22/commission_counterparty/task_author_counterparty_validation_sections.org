* Plan

Populated all three =.org= models with the facet content identified
in the appraisal: =* Insert trigger ** Validations= rows,
=* SQL ** Soft FK validations= entries, a =* SQL ** Indexes= entry
for the non-unique =full_name= search index, a corrected =:default:=
on =status=, a corrected =:nullable:= on =business_center_code=, a
new =transliterated_name= column, and =id_value= promoted from
=* Columns= into =* Natural keys= on =counterparty_identifier.org= to
restore the 3-column composite uniqueness a naive regen had collapsed.

Regenerated with =./compass.sh codegen entity generate <entity>
--profile all= (per [[id:20D0A77D-E0C0-492E-B0EC-45C7638DD220][How do I run codegen?]]'s entity-name entry point) for
each of the three entities individually — this is properly scoped to
just the touched models, unlike =codegen regenerate --component=
which touches the whole component and requires manually discarding
unrelated drift afterward. First pass used the component-wide command
by mistake and picked up ~20 unrelated entities' pre-existing drift;
reverted in full and redid it scoped.

*Found and fixed a real codegen template bug along the way*: the
compiled =sql_schema_domain_entity_create.mustache= (tangled from
=projects/ores.codegen/library/templates/ores.sql.schema.domain_entity_create.org=)
referenced =\{\{#domain_entity.validations\}\}= (flat), but the
org-model loader (=org_loader.py=) actually stores parsed
=* Insert trigger ** Validations= rows nested at
=domain_entity.insert_trigger.validations=. This meant the
Validations facet — despite looking correct in every =.org= source —
had *never actually worked* for any entity that lacks a legacy
=_table.org= companion, because every existing user of the facet
(e.g. =currency=) still has one, and the SQL generation path for a
=_table.org= file goes through a different loader
(=load_org_table_model=) with its own, correctly-flat parsing.
=counterparty= — having no =_table.org= companion — is the first
entity to actually exercise the unified entity→SQL path for this
facet, and hit the latent bug. Fixed the =.org= template source
(=\{\{#domain_entity.insert_trigger.validations\}\}=, matching the
loader) and re-tangled via =./compass.sh build --direct
tangle_codegen_templates=. Verified the fix doesn't regress =currency=
(zero diff on regen) before scoping down to counterparty-only output.

Also found and fixed one pre-existing, unrelated compile bug while
verifying: =counterparty.org='s =full_name= generator block called
=faker::company::name()=, which doesn't exist in the vendored
faker-cxx version (only =faker::company::companyName()= does) — this
had presumably never been compiled before. Fixed in the model.

Verified: all 12 touched =.cpp= files across =ores.refdata.api= and
=ores.refdata.core= compile cleanly (built individually via the
component Makefiles, since the full library target also drags in
unrelated, pre-existing broken generators for =business_unit= and
=cds_convention= — out of scope here, not touched).
