#+title: How do I add a domain service?
#+description: Add a NATS domain service by editing the single service registry (a codegen model) and regenerating; the registry is the one source for service_vars.sh, SQL users/roles/grants, and the per-service .env credentials.
#+type: recipe
#+level: cross
#+filetags: :codegen:services:registry:database:recipe:
#+created: 2026-06-02
#+updated: 2026-06-02

The list of NATS domain services has a *single source of truth*: the
service registry, a [[id:08FBD248-257A-A474-DD43-DB08949978F1][ores.codegen]] model. Everything else is derived
from it — never hand-maintain a service list anywhere else, and never
infer the service set from the =projects/= directory layout.

* Question

How do I add (or remove) a NATS domain service so that its database
user, roles, grants, generated shell vars, and =.env= credentials are
all created consistently?

* Answer

1. /Edit the registry/ — the one authored source:

   #+begin_src sh :results verbatim
   $EDITOR projects/ores.codegen/models/services/ores_services_service_registry.json
   #+end_src

   Add (or remove) a service object. Fields per service: =name=,
   =psql_var=, =env_key=, =iam_role=, =description=, =email=,
   =dml_prefixes= (the =ores_<name>_*= tables it owns), and the optional
   =select_tables= / =select_prefixes= for cross-service reads.

2. /Regenerate/ the artefacts the registry drives via the
   =service-registry= codegen profile. Run from the repository root —
   =run_generator.sh= changes to its own directory internally, and its
   model path is relative to =projects/ores.codegen/=:

   #+begin_src sh :results verbatim
   ./projects/ores.codegen/run_generator.sh \
       models/services/ores_services_service_registry.json \
       --profile service-registry
   #+end_src

   This (re)writes =projects/ores.sql/service_vars.sh= (=SERVICE_NAMES=)
   and the SQL for service users, accounts, roles, and DB grants. Staying
   at the repository root keeps the relative paths in steps 3–4 valid.

3. /Regenerate credentials/ so the new service gets its =.env= entries
   (=ORES_<NAME>_SERVICE_DB_USER= / =_PASSWORD= / =_DATABASE=):

   #+begin_src sh :results verbatim
   ./build/scripts/init-environment.sh --preset <preset> -y
   #+end_src

4. /Recreate the database/ to provision the new role and grants:

   #+begin_src sh :results verbatim
   ./projects/ores.sql/recreate_database.sh -y
   #+end_src

5. /Commit/ the registry edit and all regenerated files together.

* Who reads the registry

| Consumer | How |
|----------+-----|
| =projects/ores.sql/service_vars.sh= (=SERVICE_NAMES=) | generated by the =service-registry= profile |
| SQL service users / accounts / roles / grants | generated by the same profile (4 templates) |
| =build/scripts/init-environment.sh= | reads the registry to emit per-service =.env= credentials |
| =projects/ores.lisp/src/ores-prodigy.el= | derives the launch list from =ORES_*_SERVICE_DB_USER= in =.env= (downstream of init-environment) |

=wt= is intentionally /not/ in the registry — it uses =rw_role= /
=ORES_WT_DB_*= naming rather than the service-role naming.

* Script

Generation: =projects/ores.codegen/run_generator.sh ... --profile
service-registry=. Credentials: =build/scripts/init-environment.sh=.
Provisioning: =projects/ores.sql/recreate_database.sh=.

* Tested by

Manual: after step 2, =git diff projects/ores.sql/service_vars.sh= shows
the new =SERVICE_NAMES= entry; after steps 3–4 the new
=ORES_<NAME>_SERVICE_DB_*= vars exist and =recreate_database.sh= reports
no missing-password errors. CI exercises the same path in the canary
"Setup Database" step.
