Conduct a System 2 analysis of the operational script surface and settle
the operational pillar taxonomy that all per-pillar migration tasks build
from — the same design-first approach the [[id:A596D1E7-6A74-4E55-996D-6F1F2E48F261][Compass CLI UX redesign]] used
before touching code. The analysis must answer the open questions below
and map every operational script to a compass pillar + verb.

The design lives *inline* — the working mapping and analysis in this
task's =* Plan= / =* Notes=, and the durable decisions distilled into the
parent story's =* Decisions=. It is /not/ written to =doc/plans/= (that
directory is the historical archive of older one-off plans, not where
live design belongs under the v2 architecture).

This is design, not migration. No scripts are folded in or deleted in
this task; it produces the plan and scaffolds the migration tasks.

* Open questions

** Q1. What are the operational pillars?

The five existing pillars (Orient, Search, Scaffold, Capture, Journal)
serve the information architecture. Operational work (running/building
the system) is a distinct concern. Candidate set from the story:
=Provision= (env/db/nats/certs setup), =Operate= (services lifecycle,
logs, nats CLI), =Build= (build, site), =Generate= (codegen wrappers).
Decide the final set, names, and boundaries. Are these new top-level
pillars, or one "Operate" pillar with sub-areas?

** Q2. Per-script mapping

Map each of the 61 inventoried scripts to a pillar + verb (noun-first to
match the existing entity-command style, e.g. =compass db recreate=,
=compass env init=, =compass services start=). Flag any script that
should /not/ move (e.g. =install_debian_packages.sh= is a one-time
bootstrap; =compass.sh= is the wrapper itself).

** Q3. Backward compatibility

For each migrated script: thin shim that execs the compass verb (keeps
muscle memory + existing callers working), or hard delete with all
callers updated? Likely a mix — shims for dev-facing scripts during a
deprecation window, deletion for internal ones.

** Q4. CI migration

CI workflows (=.github/workflows/*=) call scripts directly
(=recreate_database.sh=, =setup_postgres_extensions.sh=,
=init-environment.sh=, ...). Decide whether CI calls =compass <verb>= or
keeps calling shims. Account for the CI Python/venv bootstrap that
compass needs.

** Q5. Codegen — fold in or delegate?

=ores.codegen= is already a Python tool, and =compass add= already
delegates to it. Do the =generate_*_schema.sh= / =run_generator.sh=
wrappers become =compass gen ...= verbs that delegate, or stay as
codegen's own CLI? (18 of the 61 scripts are codegen.)

** Q6. Shared plumbing

Most =.sh= scripts re-implement =.env= sourcing + =ORES_PRESET=
resolution + =build/output/<preset>= path construction. Define the
single compass-side helper(s) that replace this idiom, and the single
service-registry source (=projects/services.json= per the absorbed task).

** Q7. Naming convention

Confirm the verb grammar: noun-first (=compass db recreate=) vs
verb-first (=compass recreate db=). The existing entity commands
(=compass sprint status=, =compass story new=) are noun-first; follow
that.

** Q8. Documentation surface

Where the operational pillars get documented: =compass --help= /
=_EPILOG=, the compass recipes (=doc/recipes/compass/=), a knowledge doc,
and the ores.lisp dashboard (which currently launches the scripts
directly — it should call the compass verbs instead).

* Plan

2. *Decide.* Resolve Q1–Q8 with reasoning — captured inline here, the

4. *Scaffold.* Create per-pillar migration tasks under the story, each

   pilot and gets its full plan written.
5. *Commit + PR.* Docs only (this task's filled-in design, the story's
