* Emacs

- load babel and test all recipe types (SQL, cli etc).
- add a capture command which gets latest capture buffer (e.g. latest version,
  latest sprint, locate capture.org).
- instead of having a DB buffer for all envs, we should just have a ORES env
  buffer. This will have all the common commands for that env (init env etc).
  Actually what we need is a ORES Studio console in emacs which wraps all of the
  available scripts and also provides some blurb as to what the scripts do.
  Functionality is in groups: for example, all of the start/stop scripts in a
  group etc. Each console is relative to an environment. The console reads .env
  and makes it clear what the key settings are (env version, ports, etc). Should
  also have bookmarks for log directory, bin directory, etc.
- when running commands (babel) we should use the .env file (preset, etc).
  However the issue is we need to reload as we move envs. Or ideally the babel
  functions should query the .env file dynamically to determine paths.
- integrate captures with emacs capture: [[https://www.gnu.org/software/emacs/manual/html_mono/org.html#Setting-up-capture][Setting up capture]].
- we could use emacs dashboard: https://github.com/emacs-dashboard/dashboard. We
  could provide a way to edit the environment manually if required.
- we could even provide plots for some ORE Studio functionality.
  https://francismurillo.github.io/2017-04-15-Exploring-Emacs-chart-Library/
- the dashboard should link up to recipes and user manual. We should have a
  manual for the dashboard itself.
- in the future we could extend the dashboard to provide some of the screens we
  have in the UI via HTTP: list users, etc. This would just be another facet via
  codegen. We could make a "standard" table like proced: [[https://laurencewarne.github.io/emacs/programming/2022/12/26/exploring-proced.html][Exploring Proced]].
  Dashboard should have a login screen integrated with emacs. However, we should
  be careful to distinguish the three modes of operation:
  - running local scripts.
  - remote login operations via HTTP. A true client.
  - integration with CLI. Direct database access.
