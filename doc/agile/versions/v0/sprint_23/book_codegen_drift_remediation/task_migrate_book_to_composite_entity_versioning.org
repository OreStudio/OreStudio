* Goal

Register Book as a composite child of Portfolio (not of book_status/
regulatory_book_type) so that adding/editing/deleting a book bumps its
parent portfolio's version, and "portfolio as of version N" can
correctly compose the books it contained at that point in time.

* Plan

Added =:bump_parent_version: true= and =:list_by_as_of: true= to
book.org's =parent_portfolio_id= FK entry, and
=:generate_touch_function: true= to portfolio.org's Flags (the same
shape as the counterparty/counterparty_identifier pilot).
Diff-reviewed before generating: portfolio's regen was restricted to
=--address ores.sql.schema= only (its Qt layer is still fully stale
and out of scope -- a full regen would have pulled in a 3700+ line
unrelated diff). book's full regen only touched
C++/SQL (handler/registrar/protocol/repository/service/SQL) -- its Qt
files came back byte-identical (already current from the earlier
regen task).

Built clean, =db recreate= applied both modified SQL files with no
errors. Verified the mechanism directly via SQL, not just trusting the
generated code:

- Created a portfolio (v1). Inserted a book referencing it -- portfolio
  correctly bumped to v2, v1's window closed at the exact insert
  instant.
- Deleted the book -- portfolio correctly bumped to v3, symmetric with
  the insert-side bump.
- Composed "portfolio as of v2" via the interval-overlap join
  (=book.valid_from < v2.valid_to and book.valid_to > v2.valid_from=)
  -- correctly returned exactly the book that existed during that
  window.

=ores.refdata.api/core/service.tests= 3/3 passed after the change.
