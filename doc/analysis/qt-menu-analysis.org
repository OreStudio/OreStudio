* Overview

This document captures the current state of the ORE Studio Qt application
menu bar and toolbar, followed by a UX, discoverability, consistency, and
idiomaticness analysis. The goal is to inform a future menu restructuring
effort.

* Current Menu Structure

Menu bar order:
=File | Reference Data | Analytics | Entities | Trading | Market Data | Compute | Reporting | System | Window | Help=

** Architecture note

=MainWindow.cpp= pre-creates the shared =&Reference Data= menu and inserts it
before =&System=. Plugins then contribute in two phases:

1. =setup_menus()= — populate shared menus (System, Reference Data, Telemetry)
2. =create_menus()= — return standalone menus inserted before =&System=

Plugin registration order determines the final menu bar order between
=Reference Data= and =System=.

* Analysis

** Toolbar — critically overcrowded

28 slots, icon-only, no labels, no grouping headers. Most financial desktop
applications (Bloomberg Terminal, Reuters Eikon) limit toolbars to 8–12 slots
and rely on menus for everything else. At 22 buttons the toolbar is
essentially a visual duplicate of the menus that adds noise rather than speed.

Particular problems:
- Refdata trio (Currencies, Countries, Data Librarian) are low-frequency
  setup items that do not justify permanent toolbar real estate.
- Shell is a developer/admin tool visible to all users at all times.
- A user unfamiliar with the icons cannot locate =Org Explorer= or
  =Data Librarian= without hovering every button.
- No plugin (AnalyticsPlugin, MktdataPlugin) contributes zero toolbar
  actions, creating an implicit two-tier system with no documented rationale.

** Structural inconsistencies within menus

| Menu          | Problem                                                          |
|---------------+------------------------------------------------------------------|
| Trading       | =Book Statuses= is a top-level item alongside primary entities.  |
|               | Every other domain hides type/status enumerations in a           |
|               | submenu (e.g. =Entities > Entity Types=). No submenu equivalent  |
|               | exists for =Trading=.                                            |
| Reference Data| Mixes leaf items (=Monetary Natures=, =Rounding Types=,          |
|               | =Purpose Types=, =Change Reason Categories=, =Change Reasons=)   |
|               | at the top level alongside three submenus. The bottom half       |
|               | appears structurally inconsistent with the top.                  |
| Compute       | =Apps= and =App Versions= live under Compute, but               |
|               | =Job Definitions= (which schedules app executions) lives under   |
|               | Reporting. The split is arbitrary.                               |
| Analytics     | Only 4 leaf items total, split into two submenus of 1–3 items   |
|               | each. Both submenus are shallow enough to be flat.               |
| Market Data   | Thinnest top-level menu: 3 items, 1 separator, no submenus.      |
|               | Could be a submenu of another domain.                            |

** Conceptual boundary problems

- *Analytics vs Compute*: Pricing models, configurations, and products
  (Analytics menu) are the inputs to computation jobs (Compute menu).
  Separating them at the top level forces users to navigate two menus to
  complete a single workflow.

- *Reporting vs Compute*: =Job Definitions= lives in Reporting but is a
  scheduling concern closer to Compute. Meanwhile Compute has =Apps= and
  =App Versions= — the execution units for reports. The boundary is
  arbitrary from a user perspective.

- *Data Librarian placement*: Data Librarian has a toolbar button, marking
  it as a primary workflow tool. Yet its menu location is 4 levels deep:
  =Reference Data > Data Catalogue > [separator] > Data Librarian=. Depth
  contradicts importance.

- *Shell in Telemetry*: Shell is a developer/admin tool. Its presence in
  =System > Telemetry= alongside Log Viewer and Event Viewer is reasonable
  from an implementation perspective, but it also has a permanent toolbar
  button, making it visible and one-click accessible for all users including
  non-technical ones.

- *Service Dashboard vs Compute Dashboard*: =Compute > Dashboard= shows
  the compute workload; =System > Telemetry > Service Dashboard= shows
  service health. Both are dashboards, both are compute-adjacent, separated
  across two menus. The label distinction is insufficient for a new user.

** File menu mixing unrelated concerns

=File= contains connection management (Connections, Connect, Disconnect),
user account management (My Account, My Sessions), and application exit. In
standard desktop UX, =File= is for document/session-level operations.

- Connect/Disconnect are borderline acceptable in File.
- =My Account= and =My Sessions= are user-profile concerns and belong in a
  dedicated =Account= or =Profile= menu, or under a logged-in username
  widget in the toolbar or status bar.

** Naming inconsistencies

| Issue                          | Details                                      |
|--------------------------------+----------------------------------------------|
| Exit vs Quit                   | File menu says "Exit"; system tray says      |
|                                | "Quit". Should be consistent.                |
| Console vs Shell               | =Compute > Console= and                      |
|                                | =System > Telemetry > Shell= are both        |
|                                | interactive terminals. Distinction unclear.  |
| Service Dashboard vs Dashboard | =Compute > Dashboard= vs                     |
|                                | =System > Telemetry > Service Dashboard=.    |
|                                | Both are dashboards, both compute-related,   |
|                                | separated across two menus.                  |
| Catalog vs Catalogue           | Submenu is "Data Ca&talogue" (British) but   |
|                                | its child item is "&Catalogs" (American).    |

** No keyboard shortcuts

Zero =Ctrl+= shortcuts are defined anywhere in the application. For a
financial back-office tool where users switch between Portfolios, Trades,
Market Data, and Reports many times per day, this is a significant
productivity gap. Standard desktop idiom expects at minimum:
- =Ctrl+W= or =Ctrl+F4= to close an MDI window
- =Ctrl+Q= to quit
- Domain-specific shortcuts for the highest-frequency windows

* Scheduler: missing top-level domain

The scheduler backend (=ores.scheduler.*=) is fully implemented but has no
dedicated menu. Currently =Job Definitions= is the only exposed item and it
lives under =Reporting=. This is wrong: scheduling is a cross-domain concern
that will drive report runs, trade expiry, housekeeping, data archival, and
future scheduled workflows. It should have its own top-level =&Scheduler= menu
between =Reporting= and =System=:

#+begin_example
&Scheduler
├── &Job Definitions        ← move from Reporting > Job Definitions
├── &Job Instances          ← missing Qt controller/window; backend exists
├── ─────
└── &Monitor                ← missing: live view of running jobs,
                                   next-fire times, queue depth
#+end_example

See sprint backlog 16 story "Scheduler: complete Qt UI and promote to
top-level menu" for implementation tasks.

* Proposed Menu Structure

** Naming rationale

*** The "auxiliary types" problem

Every domain in the system contains two distinct layers:

1. *Primary objects* — the things users create and manage daily: Parties,
   Portfolios, Trades, Report Definitions, Job Definitions, etc.

2. *Controlled vocabularies* — pre-seeded code tables that classify the primary
   objects: Party Types, Party Statuses, Book Statuses, Pricing Engine Types,
   Report Types, Concurrency Policies, Monetary Natures, Rounding Types, etc.

These are a well-recognised category in financial systems. The naming
options — with pros and cons — are:

| Candidate      | Used by                        | Pros                                    | Cons                                              |
|----------------+--------------------------------+-----------------------------------------+---------------------------------------------------|
| *Setup*        | SAP, some SaaS apps            | Clear intent; separates from daily work | Feels transient; implies one-time not ongoing     |
| *Codes*        | FpML, ISO 20022, SWIFT         | Short; aligns with codelist standards   | Sounds like programming to non-technical users    |
| *Maintenance*  | Murex, Calypso, Temenos T24,   | Historically accurate; signals          | Old-fashioned; suggests IT rather than user work  |
|                | Finastra Fusion, most legacy   | "edit occasionally"                     |                                                   |
|                | banking systems                |                                         |                                                   |
| *Attributes*   | Bloomberg Terminal             | Short; professional; Bloomberg idiom    | "Party Attributes" sounds odd to some             |
| *Configuration*| Many enterprise apps           | Accurate                                | Already used in System > Configuration            |
| *Reference*    | Generic                        | Short; professional                     | Already the top-level menu name                   |
| *Parameters*   | Some trading systems           | Used in domain (Trading Parameters)     | "Organisation Parameters" feels unnatural         |
| *Static Data*  | Goldman, most IB systems       | Most precise financial term             | Jargon; implies data never changes                |
| *Types*        | —                              | Short                                   | Severely overloaded — every item already has      |
|                |                                |                                         | "Types" in its name                               |

The strongest candidates in this codebase's context:
- *Codes* — FpML-aligned, short, professional, scales consistently across all
  domains: "Organisation Codes", "Trading Codes", "Pricing Codes", "Reporting Codes".
- *Maintenance* — the most historically accurate financial term; every major
  legacy platform uses it for exactly this category of tables.
- *Attributes* — Bloomberg idiom; short and unambiguous.

The current tree uses =Setup= as a placeholder pending a decision. The chosen
name should be applied uniformly across all domain submenus.

*** Entities → Organisation

"Entities" is too abstract and from a data-modelling perspective, not a user
concept. The menu contains parties, counterparties, business centres, and
business units — all organisational structures. "Organisation" is the
idiomatic term in FpML and ISDA CDM for this category.

*** Identity top-level menu

=System > Identity= (Accounts, Roles, Tenants, Tenant Types, Onboard Tenant)
plus =File > My Account= / =My Sessions= are all IAM concerns. Elevating them
to a top-level =&Identity= menu removes account management from =File= (where
it does not belong) and surfaces tenant management outside of =System=
(where it is buried from tenant admins). "Identity" is preferred over
"Accounts" because the menu also contains Roles and Tenants — not just accounts.

*** External data menu

=Import ORE Data= does not belong in =File= (which should be app operations
only) or in =Trading= (import is not a trading operation). A top-level
=&External= menu groups all inbound connectivity concerns together and provides
a natural home for future Bloomberg/Reuters/SWIFT feeds.

The Data Catalogue (Data Domains, Subject Areas, Catalogues, Dataset Bundles,
Methodologies, Dimensions) exists solely to support the DQ pipeline — it
defines the structural taxonomy within which the Data Librarian operates.
A financial user doing trades or running reports never touches it. It belongs
in =&Data Transfer= alongside the Librarian, not in =&Reference Data= which
is a domain term covering financial reference data (currencies, countries,
trading conventions). The Dimensions sub-submenu is also eliminated; Origin,
Nature, and Treatment Dimensions are peers of the other catalogue items and
sit at the same level.

*** Compute = grid only

Analytics/Pricing models are *inputs* to the compute grid, not part of grid
infrastructure. =&Compute= is reduced to grid execution concerns (Dashboard,
Console, Apps, App Versions). A new top-level =&Pricing= menu holds model
configuration.

*** Purpose Types relocation

=purpose_type= (Hedging, Trading, Investment) classifies portfolio purpose —
portfolios belong to the Trading domain. It currently lives in Reference Data
as a loose item, which is architecturally wrong. It is moved to
=Trading Codes=.

*** Shell → Console

=Compute > Console= and =System > Telemetry > Shell= both open interactive
terminals. "Console" is the more idiomatic term and is already used elsewhere;
"Shell" implies OS-level access which may alarm non-technical users. Rename
=Shell= to =Console= throughout.

** Open questions

1. =(*open*)= *Auxiliary types submenu name*: "Codes" is used as a placeholder
   throughout. To be revisited — other candidates: *Maintenance* (classic
   banking term; Murex/Calypso/Temenos), *Attributes* (Bloomberg idiom).
   The chosen name applies uniformly to all domain submenus: Currency Codes,
   Organisation Codes, Trading Codes, Pricing Codes, Reporting Codes.

2. =(*open*)= *Identity vs Accounts*: "Identity" accurately covers accounts,
   roles, and tenants. "Accounts" would be simpler and more approachable.
   Which is preferred?

3. =(*open*)= *Data Catalogue placement*: Currently proposed under =Data Transfer=
   alongside the Librarian, on the grounds that both serve the same DQ
   pipeline workflow. Alternative: keep it in =Reference Data= as it is
   structural taxonomy for the system, not a data movement operation.

4. =(*open*)= *Toolbar scope*: The proposed toolbar drops 12 buttons from the
   current 22. Confirm which (if any) of the removed items warrant
   reinstatement: Currencies, Countries, Accounts, Tenants, System Settings,
   Business Centres, Counterparties, Business Units, Books, Portfolio
   Explorer, Org Explorer, Data Librarian.

5. =(*open*)= *Shell vs Console*: Are =Compute > Console= and
   =System > Telemetry > Shell= the same tool (rename Shell → Console) or
   two distinct tools that should keep distinct names?
