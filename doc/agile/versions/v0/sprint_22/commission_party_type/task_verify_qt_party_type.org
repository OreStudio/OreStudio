* Notes

Attempted to start the service fleet twice (=compass services start=,
after =compass db recreate -y -k= against a freshly rebuilt
=linux-clang-debug-make= preset) to drive the Qt client against live
services. Both attempts: NATS starts fine, controller starts, launches
=ores.wt.service[0]=, then crashes moments later writing the phase
update — =ERROR: invalid byte sequence for encoding "UTF8": 0x99= on
the first attempt, =0x8f= on the second (different byte, same call
site) — see [[id:D48BEA76-F879-4C16-8310-933B7EAEBCDF][bug capture]] for
full analysis. Filed as a separate bug rather than debugged in-line
since it looks like memory corruption (ASAN/valgrind territory), not a
party_type-related regression. Blocking further progress on this task
until fixed.

*Update:* not memory corruption. A legacy, untracked =nats-server= from
the deleted =OreStudio.local2= checkout was squatting on solid_dirac's
NATS ports (42222/8222); solid_dirac's own NATS fatal-crashed on the
port conflict every time, but =compass services start='s readiness
probe was fooled because something was answering on 42222. Controller
and wt.service connected to the wrong broker, producing the corrupted
reads. Fixed in [[id:05243CFE-F838-4759-9956-9400D6C90FDC][a separate
infra story]] (PR #1415): killed the stray process and migrated NATS
ports to derive from =ORES_BASE_PORT=. Full 24-service fleet now starts
clean.

*Eventing was genuinely broken, separately:* provisioned a fresh
Barclays Plc tenant and found real eventing did not fire for
=party_type=. Root cause:
=projects/ores.refdata/service/src/app/application.cpp= (the
hand-maintained Postgres LISTEN/NOTIFY → NATS relay) never had
=party_type= registered — the SQL trigger and
=party_type_changed_event.hpp= were both correctly generated, but
nothing wired them into the relay. Confirmed this is a known, already
captured class of bug
([[id:1FEE0D30-0466-41F0-9137-97864DA6DA3E][capture]], originally found
for =ores.synthetic=); at least 14 other =ores.refdata= entities have
the same gap and are presumably still broken. Fixed =party_type='s
registration inline (include + =register_mapping<>= +
=event_bus.subscribe<>=), verified it builds and the service restarts
clean. Added a line item to the
[[id:897A7156-504D-4581-AD4C-026D643C0838][domain entity evaluation
checklist]] naming this exact mechanism so future commissioning work
catches it without needing two live Qt clients open.

*Second, unrelated bug found on the Qt side:* even after the backend
fix, the Qt client itself never subscribed to party_type's NATS event.
=PartyTypeController= existed as two full duplicates:
=ores.qt/party/src/PartyTypeController.cpp= (hand-written, still
wired up by =PartyPlugin=, but constructed with an empty event name —
so it silently never subscribed to anything) and
=ores.qt/refdata/src/PartyTypeController.cpp= (correctly
codegen-generated after party_type's earlier Qt sync task, but never
actually compiled into any running plugin). The old copy also lacked
the =ChangeReasonCache= parameter present on its sibling
=PartyStatusController= — a real feature regression from that earlier,
incomplete migration.

Root-caused further: every entity in =ores.qt.party= has backend
component=refdata; there is no =ores.party= C++ component. The plugin
is a UI-only grouping with zero backend justification — this mismatch
is what let the duplication go unnoticed. Fixed by regenerating
party_type's Qt profile (lands under =ores.qt/refdata=, matching its
component, with no codegen change needed), deleting the stale
=ores.qt/party= copy, and moving its instantiation from =PartyPlugin=
to =RefdataPlugin=. Added =shared_menus_context::organisation_codes_menu=
(host-owned in =MainWindow.cpp=) so both plugins can contribute to the
same physical "Organisation Codes" submenu without a cross-plugin
library dependency, since the remaining =ores.qt.party= entities
migrate one at a time as each is (re-)commissioned — see
[[id:98D6A549-01D5-4E99-A530-9AFB9CFFD858][capture]]. Verified: full
=ores.qt.exe= builds clean; live end-to-end retest (fresh Barclays Plc
tenant, full 24-service fleet, Qt client login) confirms party_type
list/detail/save/eventing all work correctly.
