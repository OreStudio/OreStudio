#+title: Task: Fix change_reason_code never assigned before save in ~9 Qt detail dialogs
#+description: Found while commissioning catalog (819B050A/PR #1644 review): CatalogDetailDialog called promptChangeReason() but never assigned the result's reason_code to the entity before saving, which would fail the insert trigger's change-reason validation. Fixed there, and ChangeReasonCategoryDetailDialog/ChangeReasonDetailDialog were fixed too once those entities went live on the generated stack (819B050A commissioning). ~8 more dialogs still have the same bug: CodingSchemeAuthorityTypeDetailDialog, CodingSchemeDetailDialog, DataDomainDetailDialog, MethodologyDetailDialog, NatureDimensionDetailDialog, OriginDimensionDetailDialog, SubjectAreaDetailDialog, TreatmentDimensionDetailDialog -- each calls promptChangeReason() but never assigns crSel->reason_code to the entity (only DatasetBundleDetailDialog and the now-fixed dialogs do it correctly). Each will fail change-reason validation once/if its entity's save path is actually exercised against a real generated backend requiring change_reason_code. Audit and fix all remaining ones, verifying the same one-line pattern used in CatalogDetailDialog (entity_.change_reason_code = crSel->reason_code;).
