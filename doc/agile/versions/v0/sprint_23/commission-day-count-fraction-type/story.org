* Goal

Discovered while modelling [[id:04A121FA-00D6-43EB-9B21-04EDC1FA493D][fx_convention]]:
=day_count_fraction_type= lives in =ores.trading=
(=ores.trading.day_count_fraction_type=, since removed — see
[[id:D639B3AB-7421-4A7C-A17A-02000709E22F][ores.refdata.day_count_fraction_type]]), with exactly one consumer
(=trading_swap_legs=), while every =*_convention= entity that would
also need it (=ibor_index_convention=, =zero_convention=,
=fx_convention=, etc.) lives in =ores.refdata=. Move it to
=ores.refdata=, commission it fully — SQL, codegen model, domain,
repository, service, messaging, Qt UI — following the
[[id:897A7156-504D-4581-AD4C-026D643C0838][Domain entity evaluation checklist]], update =trading_swap_legs='s one
soft-FK to point at the new location, and delete the =ores.trading=
original. This unblocks =currency= soft-FK'ing =day_basis= to it
same-component, matching every other refdata soft-FK.

* Decisions

- Followed the =business_day_convention_type= move precedent exactly
  (same aux-type shape, same removal checklist for the =ores.trading=
  original) rather than inventing a new pattern — consistency across
  sibling migrations matters more than any per-entity optimisation.
- Verified before repointing: none of the =ores.refdata= =*_convention=
  tables validated =day_count_fraction= against a function prior to
  this change (plain unvalidated text columns), so only
  =trading_swap_legs='s existing soft-FK needed repointing.
- =ores.qt/refdata/RefdataPlugin= already carried speculative
  forward-declarations and menu wiring for a
  =DayCountFractionTypeController= with no backend — a latent bug
  (missing =change_reason_cache= constructor argument) surfaced only
  once the backend was generated and wired.
