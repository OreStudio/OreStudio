* What

*Fixed inline* while capturing manual-chapter screenshots for the
Party Types section, rather than left as an open capture: root cause
was that =ores.refdata.party_type.org= never had a =*** Detail
fields= table (unlike =party_status=/=party_id_scheme=, which both
declare one), so codegen fell back to its default field set
(Code/Name/Description only) and silently dropped =display_order=
from the generated dialog, even though the column exists on the
domain model, the list-window columns table, and the SQL
=order_by=. Added the missing =*** Detail fields= table (mirroring
=party_status='s) and regenerated — zero-diff except the intended
=displayOrderEdit= widget appearing.

* Why

The party_type/party_status/party_id_scheme trio is meant to behave
identically as tenant-configurable lookups (this is the whole premise
of the Parties chapter's "Party types and statuses" section), so a
missing field on one of the three was a real, user-visible gap
rather than a documentation nuance.
