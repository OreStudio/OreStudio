#+TITLE: Strict Service Table Isolation
#+SUBTITLE: Mandatory role drops + zero cross-component DB access
#+DATE: 2026-05-12
#+AUTHOR: ORE Studio Team
#+STATUS: DRAFT
#+OPTIONS: toc:t num:nil

* Context

Two findings in
[[file:../analysis/bootstrap-and-architecture-violations.org][Bootstrap and
Architecture Violations]] motivate this plan:

1. ~recreate_database.sh~ does not drop cluster-level roles, so historical
   group memberships (notably the old broad ~rw_role~ on domain service
   users) silently mask missing per-service GRANTs. The system is therefore
   not reproducible across machines.
2. The IAM, compute, scheduler, reporting, trading, workflow and ORE
   service users today hold cross-component table grants. The Service
   Account RBAC Design
   ([[file:2026-03-28-service-rbac-design.org][2026-03-28-service-rbac-design.org]])
   stipulated that cross-component access should be the exception; in
   practice it has become the norm.

This plan establishes a strict architectural invariant — *a domain service
user holds DB privileges on its own component's tables only* — and lays
out an incremental, low-risk migration to reach it. It also makes role
drops mandatory in ~recreate_database.sh~ so that the invariant cannot
silently regress.

* Goal and Invariant

** Invariant (target state)

For every domain service ~S~ with DB user ~ores_<env>_<S>_service~:

1. ~S~ has ~SELECT/INSERT/UPDATE/DELETE~ only on tables matching
   ~ores_<S>_%~ in the ~public~ schema (its own component).
2. ~S~ has ~USAGE/SELECT~ on sequences only for its own tables (default
   privileges).
3. ~S~ has *no* grants on any table prefixed with another component.
4. ~S~ has *no* membership of the broad ~rw_role~ or ~owner_role~. The
   only group membership is ~service_role~ (CONNECT on DB, USAGE on
   schema).
5. *FK columns referencing other components' tables* are removed from
   ~S~'s tables; cross-component referential integrity is enforced by
   the producing service via NATS write APIs and confirmed by IAM
   tenant-id resolution / cached lookups.

Equivalently: removing every line in
~create/iam/iam_service_db_grants_create.sql~ except its own
~_ores_grant_dml_fn('ores_<S>_', ...)~ call must not break any service.

** Non-goals

- Splitting ~public~ into per-component PostgreSQL schemas. Today every
  table lives in ~public~ with a ~ores_<component>_~ prefix; moving to
  one schema per component is desirable but orthogonal to this plan and
  carries its own migration cost.
- Removing the C++ link between ~ores.iam.core~ and ~ores.variability.core~
  for *types and configuration shapes*. Only the DB-reading code paths
  must move to NATS or to IAM-owned tables.
- Touching the in-process unit tests that talk directly to a shared
  ~public~ schema as the DDL user.

* Phase 0 — Mandatory Role Drops (prerequisite)

Goal: make ~recreate_database.sh~ produce the same cluster state regardless
of the cluster's history. Lands first because every later phase needs it
to demonstrate that the new GRANT matrix is actually being used.

** Step 0.1 — Generate ~drop_roles.sql~ from the service registry

The current file ~projects/ores.sql/drop_roles.sql~ hard-codes the user
list (lines 59-79). Replace it with a generated file produced from
~projects/ores.codegen/models/services/ores_services_service_registry.json~
via a new mustache template ~sql_drop_roles.mustache~.

Output path is unchanged; only the contents become generated.

** Step 0.2 — Wire ~drop_roles.sql~ into ~recreate_database.sh~

Add a phase between "drop database" and "create roles":

#+begin_src bash
# Phase 0b: Drop environment roles (mandatory)
if [[ -z "${KEEP_ROLES:-}" ]]; then
    PGPASSWORD="${PGPASSWORD}" psql \
        -h "${ORES_DB_HOST}" -U postgres \
        -v env_label="${ORES_ENV_LABEL}" \
        -f ./drop_roles.sql
fi
#+end_src

Add a ~--keep-roles~ flag (off by default) for the narrow case where the
operator deliberately wants to preserve role state (e.g. CI parallel runs
sharing a cluster). The default behaviour drops; opting out is explicit.

** Step 0.3 — Derive ~env_label~ inside the script

~drop_roles.sql~ takes an ~env_label~ and reconstructs role names by
concatenation (~ores_<label>_owner~). ~recreate_database.sh~ already
receives the *full role names* via env vars. Either:

(a) Have the wrapper extract the label from ~ORES_DB_OWNER_ROLE~
    (strip ~ores_~ prefix, ~_owner~ suffix), or
(b) Add ~ORES_ENV_LABEL~ as a required env var.

Option (b) is cleaner and matches the rest of the env contract. Update
~init-environment.sh~ to write it.

** Step 0.4 — Idempotency smoke test

New script ~ci/scripts/test_recreate_idempotent.sh~:

1. Run ~recreate_database.sh -y~.
2. Snapshot ~pg_roles~, ~pg_auth_members~, and
   ~has_table_privilege(user, table, priv)~ across all combinations of
   (service user × representative tables).
3. Run ~recreate_database.sh -y~ again.
4. Snapshot again and diff. Empty diff is the pass criterion.

Wire into CI as a fast (<60s) job on every PR that touches
~projects/ores.sql/~ or the service registry model.

** Step 0.5 — Move static infrastructure-user populate entries

~populate/iam/iam_service_accounts_populate.sql~ and
~iam_service_account_roles_populate.sql~ contain rows for ~http_user~,
~wt_user~, ~compute_wrapper_user~ that are not in the service registry.
These survive only by being added back after every codegen run.

Resolution: add a ~category: "infrastructure"~ discriminator to the
service registry, with these three users entered under it. Update the
mustache template that produces the populate files to emit
infrastructure rows in a separate section but in the same file. Then
~iam_service_db_grants_create.sql~ continues to receive only domain
users.

** Step 0.6 — Smoke-test on a clean cluster

Run on a Postgres container with no pre-existing ~ores_*~ roles:

#+begin_src bash
docker run -d --name pg-smoke -e POSTGRES_PASSWORD=... postgres:16
ORES_DB_HOST=localhost ... ./recreate_database.sh -y
# Manually verify each service can start and complete provisioning.
#+end_src

Acceptance: the wizard ("Create initial admin" + "Provision tenant")
completes end-to-end without any ~permission denied~ in the logs of any
service.

* Phase 1 — Establish the Invariant and CI Enforcement

The invariant is worthless if a future codegen change can re-introduce a
cross-component grant. Lock it down before doing any of the user-visible
refactors.

** Step 1.1 — Codegen-side validation

The service registry has three relevant fields per service:
~dml_prefixes~, ~select_tables~, ~dml_tables~. Add a validator to the
service-registry codegen profile:

- For each service ~S~, every entry in ~dml_prefixes~ must start with
  ~ores_<S>_~.
- ~select_tables~ and ~dml_tables~ must be empty *except* during the
  migration windows of phases 2-5, controlled by a per-service
  ~migration_exception: { until: "phase-N", reason: "..." }~ field that
  must be removed before merge.

Validator runs at codegen time and at CI. Adding a new cross-component
grant without a migration_exception fails the build.

** Step 1.2 — Snapshot tests on the generated GRANTs file

In ~projects/ores.sql/tests/~ add a snapshot test that asserts the SHA-256
of ~create/iam/iam_service_db_grants_create.sql~ matches a checked-in
hash for the "final state" of the invariant. The snapshot is updated by
hand when a phase legitimately changes it. This catches accidental
regenerations that drift away from the invariant.

** Step 1.3 — Runtime assertion (optional)

Each service, at startup, can run:

#+begin_src sql
select count(*) from information_schema.table_privileges
 where grantee = current_user
   and table_schema = 'public'
   and table_name not like 'ores_<my_component>_%';
#+end_src

If non-zero in a non-bootstrap environment, log a startup warning and emit
a metric. Optional because Step 1.1 + 1.2 should be enough at compile
time.

* Phase 2 — Tenant Lookup (Universal Cross-Read)

Today 16 of 17 services have ~GRANT SELECT ON ores_iam_tenants_tbl~. Tenant
lookup is the most pervasive cross-component access and the hardest to
remove. It must go first.

** Usage analysis

Services use ~iam.tenants~ for:

1. *RLS / FK validation*: every table in every component has a
   ~tenant_id uuid~ column and (in many cases) an FK to
   ~ores_iam_tenants_tbl~. PostgreSQL requires the ~REFERENCES~ privilege
   on the parent, *not* SELECT — but many places also do a SELECT-based
   lookup.
2. *Hostname → tenant_id*: ~auth_handler~ uses ~read_latest_by_hostname~
   on login. This is IAM-internal; not a cross-read.
3. *Tenant name in audit trails*: low-volume cosmetic lookups during
   reporting and admin UIs.
4. *Service startup*: a service may resolve its own tenant_id range when
   subscribing to NATS subjects.

** Strategy: per-service tenant cache table

Each service owns a small, denormalised cache table:

#+begin_src sql
create table ores_<S>_tenant_cache_tbl (
    tenant_id      uuid primary key,
    code           text not null,
    name           text not null,
    hostname       text,
    status         text not null,
    last_synced_at timestamptz not null default now()
);
#+end_src

- Populated at service startup from a one-shot NATS request
  ~ores.<env>.<inst>.iam.v1.tenants.list-active~.
- Kept warm via JetStream subscription to
  ~ores.<env>.<inst>.iam.v1.tenants.changed~ (IAM publishes on create /
  rename / status change).
- All FKs that today reference ~ores_iam_tenants_tbl~ change to reference
  the local cache table.

This keeps the strict invariant (the service owns its cache table) and
keeps FK enforcement (the FK is now intra-component). The cost is one
NATS subscription per service and ~O(n)~ rows of duplicated data, where
n is the number of tenants (small, hundreds at most).

** Step 2.1 — IAM tenant publication API

In ~ores.iam~:
- New request/reply subject ~iam.v1.tenants.list-active~ →
  array of ~{tenant_id, code, name, hostname, status}~.
- New JetStream stream ~iam.tenants~ with subjects
  ~iam.v1.tenants.created~, ~iam.v1.tenants.updated~,
  ~iam.v1.tenants.deleted~. Publish from
  ~tenant_repository~ post-commit hooks (existing LISTEN/NOTIFY
  infrastructure can drive this).

** Step 2.2 — Cache table + sync client in ~ores.database~

Promote the cache table pattern to a reusable template under
~ores.database~: a header-only ~tenant_cache<S>~ that:
- creates and queries the service-local cache table,
- exposes a synchronous "ensure-warm" call run at service startup,
- exposes a sync function called by the JetStream subscriber on every
  ~tenants.*~ event.

Codegen the per-service ~ores_<S>_tenant_cache_tbl~ from the registry so
each service gets a matching SQL artefact.

** Step 2.3 — Migrate FKs

For each service component, in a single SQL migration:

1. Add the new ~tenant_cache~ table.
2. Add a column ~tenant_cache_id~ defaulting to ~tenant_id~ (same uuid).
3. Drop the old FK to ~ores_iam_tenants_tbl~.
4. Add the new FK to ~ores_<S>_tenant_cache_tbl~.
5. Drop the helper column (no-op if reusing ~tenant_id~).

Done component-by-component. Components migrate independently because
each FK is private.

** Step 2.4 — Remove the GRANT

Once a service is migrated, remove its line
~grant select on ores_iam_tenants_tbl to :<S>_service_user~ from the
registry. CI enforces (Step 1.1).

* Phase 3 — Change-Reason Validation (Universal Cross-Read)

Today 4 services hold ~SELECT~ on ~ores_dq_change_reasons_tbl~ (and 3 on
~ores_dq_change_reason_categories_tbl~). Every audit-trail INSERT
references the change-reason code via FK.

** Strategy

Change reasons are a low-cardinality, very-stable enum-like dataset
(dozens of rows, changes at human pace). Two viable approaches:

(a) *Local cache table* (same pattern as tenants). Each service holds a
    local ~ores_<S>_change_reason_cache_tbl~ keyed by code, kept in sync
    via JetStream. FKs move to the local cache. Costs: one more
    subscription, one more table per service.

(b) *Drop the FK, enforce via service-level validation*. Audit-trail
    inserts validate ~change_reason_code~ against an in-memory set
    refreshed periodically. Costs: lose DB-level referential integrity;
    a stale cache could let through a "tombstoned" reason.

Recommendation: *(a)* for consistency with Phase 2. Build the cache once
in ~ores.database~ as a generic ~lookup_cache<table, key>~ template; use
it for both tenants and change reasons. Code reuse pays back the
infrastructure cost.

** Steps (mirror Phase 2)

3.1 — DQ publishes ~dq.v1.change-reasons.changed~ events.
3.2 — Generic cache infrastructure absorbs change-reason cache.
3.3 — Migrate FKs in scheduler, reporting, trading, compute.
3.4 — Remove the four ~grant select on ores_dq_change_reasons_tbl~ lines
       and the three ~..._categories_tbl~ lines.

* Phase 4 — IAM-Specific Cross-Reads

IAM is the most entangled. It reaches into *variability* and *refdata*.

** 4.1 — IAM-owned settings table (replaces ~ores_variability_system_settings_tbl~)

The IAM service reads four kinds of flag from
~ores_variability_system_settings_tbl~:

- ~system.bootstrap_mode~
- ~system.user_signups~, ~system.signup_requires_authorization~ (signup
  policy)
- JWT token settings (~token.*~)
- (a few tenant-scoped extensions)

These are all IAM concerns disguised as generic "variability". Move them
into IAM's own tables:

- ~ores_iam_bootstrap_state_tbl~ (one row per tenant, ~is_in_bootstrap~,
  ~entered_at~, ~exited_at~, ~exited_by~).
- ~ores_iam_token_settings_tbl~ (one row per tenant — most rows are the
  defaults).
- ~ores_iam_signup_policy_tbl~ (one row per tenant).

Migration:

1. Create the three new tables under ~ores_iam_~.
2. Backfill from the existing rows in
   ~ores_variability_system_settings_tbl~ where the key matches.
3. Update ~bootstrap_mode_service.cpp~, ~auth_handler.hpp~,
   ~account_handler.hpp~ to use repositories on the new tables. Remove
   the include of ~ores.variability.core/service/system_settings_service.hpp~
   from the IAM handlers.
4. Drop the corresponding rows from
   ~ores_variability_system_settings_tbl~ (or leave them, marked
   ~deprecated~, depending on whether variability needs to keep them for
   the UI).
5. Remove ~grant select on ores_variability_system_settings_tbl to
   :iam_service_user~.
6. Remove the CMake link from ~ores.iam.core/src/CMakeLists.txt~ to
   ~ores.variability.core~ once no IAM source references it.

Variability remains the home of *generic* tenant feature flags
(non-IAM-specific). Nothing else moves.

** 4.2 — Provisioning returns the system_party_id

Modify ~ores_iam_provision_tenant_fn~ to ~RETURNS RECORD~ with
~(tenant_id uuid, system_party_id uuid)~. Update
~bootstrap_handler.hpp:153~ to consume both, and remove the
~party_repository.read_system_party~ call (lines 203-204).

The subsequent ~save_account_party~ still inserts into IAM's own
~ores_iam_account_parties_tbl~. Examine that table's FK to
~ores_refdata_parties_tbl~ — it must be replaced by:

(a) Removing the FK and trusting the value from provisioning, *or*
(b) Replicating the relevant parties into an IAM-local cache (see 4.3).

(a) is simpler and acceptable because the row was just created in the
same transaction.

** 4.3 — Refdata party NATS read API + IAM party cache

IAM's hot-path party reads are:

- ~auth_compute_visible_party_ids~: on every login, compute the set of
  party UUIDs visible to the authenticating account (= descendants of
  the account's home party in the party hierarchy).
- ~auth_lookup_party~: cosmetic lookup of a single party for JWT
  enrichment.

Strategy: cache, do not call NATS on the hot path. Add
~ores_iam_party_cache_tbl~ owned by IAM, populated and kept warm by:

- One-shot ~refdata.v1.parties.snapshot~ at IAM startup for each
  tenant IAM knows about.
- JetStream subscription to ~refdata.v1.parties.changed~ events.

The visible-party set is then computed *locally* by walking the cache
table.

Steps:

1. Refdata exposes ~parties.snapshot~ (request/reply) and publishes to
   the ~refdata.parties~ JetStream stream on every party CRUD.
2. IAM creates ~ores_iam_party_cache_tbl~ with columns
   ~(party_id, tenant_id, parent_party_id, code, name, status,
    last_synced_at)~. Indexed on ~(tenant_id, parent_party_id)~ for
   descendant walks.
3. Replace ~refdata::repository::party_repository~ usages in
   ~auth_handler.hpp~, ~account_handler.hpp~, and
   ~bootstrap_handler.hpp~ with calls into the local cache.
4. Remove
   ~select _ores_grant_dml_fn('ores_refdata_parties', :'iam_service_user')~
   from the GRANTs file.
5. Remove the ~ores.refdata.core~ link from
   ~ores.iam.core/src/CMakeLists.txt~ once no IAM source references it.

Note: there *is* a tail latency cost during the first login of a tenant
that IAM hasn't snapshotted yet. Acceptable because the warm path is
the dominant case; cold-path can wait for the NATS round-trip.

* Phase 5 — Other Service-Specific Cross-Reads

The remaining cross-component grants, with the smallest blast radius.

** 5.1 — compute reads ~refdata.parties~ and ~dq.change_reasons~

- ~refdata.parties~: same pattern as Phase 4.3 — add
  ~ores_compute_party_cache_tbl~, consume the refdata JetStream stream.
- ~dq.change_reasons~: handled by Phase 3.

** 5.2 — trading reads ~refdata.*~ (select prefix)

Today: ~select _ores_grant_select_fn('ores_refdata_', :'trading_service_user')~.

Audit the trading codebase for which refdata tables it actually reads
(likely currencies, business centres, parties, conventions). For each:
add a service-owned cache table populated via a NATS snapshot API in
refdata. Remove the prefix grant.

Estimated count: 4-6 refdata tables. Each takes one cache + one
subscription, but the refdata-side publication can be batched into one
~refdata.v1.<entity>.snapshot~ surface.

** 5.3 — scheduler / reporting reads ~ores_dq_change_reason_categories_tbl~

Categories are even more static than reasons. Bundle them into the
change-reason cache built in Phase 3 (categories are typically denormalised
into the reason row anyway).

** 5.4 — reporting reads scheduler tables

Today: ~select _ores_grant_select_fn('ores_scheduler_', :'reporting_service_user')~.

This is reporting reading job execution history to produce report
artefacts. Options:

(a) Move the read into scheduler via a NATS query subject
    ~scheduler.v1.executions.query~. Reporting calls it on demand.
(b) Scheduler publishes execution-completed events; reporting maintains
    its own ~ores_reporting_execution_history_tbl~ projection.

(b) is the better fit since reporting already builds projections. Adopt
the same JetStream + cache pattern.

** 5.5 — workflow has DML on ~ores_iam_~ and ~ores_refdata_parties~

This is the biggest residual entanglement. Workflow today orchestrates
tenant onboarding flows that touch IAM accounts and refdata parties
directly. Refactor pattern:

- Workflow steps invoke IAM's existing NATS write APIs
  (~iam.v1.accounts.create~, ~iam.v1.account-parties.save~) instead of
  doing the INSERTs as the workflow DB user.
- Workflow does *not* hold DML on IAM or refdata tables.
- Workflow continues to own its own state tables
  (~ores_workflow_run_tbl~, ~ores_workflow_step_tbl~ etc.) which record
  the orchestration journal.

This is potentially a multi-PR effort. Bundle with the workflow service's
own milestone rather than fitting under this plan.

** 5.6 — ORE Import service has DML on ~ores_workflow_~

~ore_service_user~ has ~_ores_grant_dml_fn('ores_workflow_', ...)~. The
ORE import flow uses workflow tables as a job queue. Migration:

- Add a NATS write API on workflow:
  ~workflow.v1.runs.create~, ~workflow.v1.steps.append~.
- ORE service calls those instead of direct INSERTs.

** 5.7 — synthetic reads everything

The synthetic data generator reads from every domain to produce
realistic seed data. It is a developer-only tool, not a production
service. Two options:

(a) Run it as the DDL user (it does need broad read access to do its
    job; treating it like a service is the wrong model).
(b) Keep it as a service but accept that its grant matrix is the one
    documented exception. Mark it in the registry as
    ~role: "tooling"~ and exclude it from the Phase 1 invariant check.

Option (a) is cleaner. Either is acceptable.

* Phase 6 — Lock the Invariant

After Phases 0-5:

1. ~create/iam/iam_service_db_grants_create.sql~ contains only one line
   per service: the ~_ores_grant_dml_fn('ores_<S>_', :'<S>_service_user')~
   call plus the bulk sequence grants.
2. The codegen validator in Step 1.1 has no remaining
   ~migration_exception~ entries.
3. The snapshot hash in Step 1.2 is bumped to the new file's SHA-256.
4. The "DB user can only touch its own tables" rule is documented in
   ~CLAUDE.md~ and in the system model
   (~projects/modeling/system_model.org~).
5. The cross-service GRANT matrix from the original RBAC design
   ([[file:2026-03-28-service-rbac-design.org][2026-03-28]]) is updated
   to record that "shared SELECT" is now the empty set everywhere except
   bootstrap.

* Bootstrap Concerns Across the Plan

Bootstrap remains the one place where IAM legitimately crosses domain
boundaries. The mechanisms that survive this plan:

1. ~SECURITY DEFINER~ provisioning function continues to handle
   tenant + system-party creation in a single transaction.
2. After Phase 4.2, the function returns the new ~system_party_id~, so
   IAM never reads ~ores_refdata_parties_tbl~ even during bootstrap.
3. The initial IAM tenant-cache population is done by an in-process
   SQL helper run as the *DDL user* during ~setup_schema.sql~, not by
   the IAM service user. Same for the initial IAM-side party cache (a
   single row for the system party of the system tenant).
4. Bootstrap-mode gating remains in IAM's own
   ~ores_iam_bootstrap_state_tbl~ (Phase 4.1), so no cross-service read
   is needed even to ask "are we still in bootstrap?".

The chicken-and-egg ("IAM cannot get tenants over NATS at startup because
NATS-auth requires an IAM tenant context") is avoided because each
service's *first* tenant-cache population can run as the DDL user during
~setup_schema.sql~. The runtime NATS sync only handles deltas.

* Open Questions

1. *Schema-per-component.* Should each component move to its own Postgres
   schema (~iam.~, ~refdata.~, ~dq.~, …) as part of this plan? It would
   make the GRANT story trivial (~GRANT ALL ON SCHEMA <S> TO <S>_user~).
   Argument against: large mechanical refactor of every SQL file and
   every C++ repository. Recommended: out of scope for this plan, but
   list it as the natural next step once the invariant holds.
2. *FK cascade semantics.* When tenant deletion happens, cascading from
   ~ores_iam_tenants_tbl~ no longer reaches other components' tables
   (the FK now points to the local cache). Replace with explicit NATS
   ~tenant.deleted~ handlers per service, and decide whether the local
   cache row should be deleted or soft-deleted.
3. *Bootstrap-time NATS availability.* Phase 2 assumes NATS is up before
   any service does its first cache population. Verify this against the
   actual systemd / service start order. If NATS lags, fall back to
   a DDL-user-driven seed in ~setup_schema.sql~.
4. *Synthetic tooling.* Worth a brief design review with whoever
   maintains it before committing to "re-home as DDL user".
