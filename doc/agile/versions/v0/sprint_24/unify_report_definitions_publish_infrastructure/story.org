#+title: Story: Make party-scoped bundle provisioning data-driven, not one hand-written phase per bundle
#+description: Party provisioning (both the Qt wizard and the ores.shell CLI script) creates report definitions via a bespoke list-templates-then-create-individually RPC path, bypassing the standard DQ bundle-publish/publish-from-dq infrastructure that report_definitions is already otherwise wired into (dataset, artefact table, ores_reporting_publish_report_definitions_from_dq_fn). More broadly, every party-scoped bundle a provisioning run needs to publish (organisation, ore_analytics, and now marketdata's reference-vintage FX driver rates) is its own hand-written phase duplicated in both the Qt wizard and the shell CLI — proven a real, recurring cost by the FX driver-rates dataset needing a brand-new phase added to both places just to be provisioned at all. Replace the fixed phase list in both call sites with a data-driven list of bundle codes to publish, so a new dataset/bundle never requires new phase code in either provisioner.

This page documents a [[id:699A8D45-B67E-4D3E-9206-24FA17C51ADA][story]] in [[id:D269862E-2035-4857-A0D0-A6E93B0E24C9][Sprint 24]]. It captures the goal, current status, acceptance criteria, and the tasks that compose it.

* Goal

Discovered while implementing
[[id:E5EC80BB-789B-4BD6-B6E5-DC7AA790692A][Synthetic data librarian support: FX foundation]]: adding a new
dataset (=synthetic.fx_spot_configs=) to the =ore_analytics= bundle
had *no effect* on either party-provisioning path, because neither
actually publishes that bundle for reports. Both
=projects/ores.qt/application/src/PartyProvisioningWizard.cpp=
(=PartyExecutePage::startReportInstall=) and
=projects/ores.shell/src/app/commands/provision_commands.cpp=
(=process_party=, the =[3/N]= "Creating report definitions" step)
call =dq::messaging::list_dq_report_definition_templates_request= to
list templates, then send one =reporting::messaging::save_report_definition_request=
per row — a bespoke, per-record RPC path that bypasses
=dq.v1.bundles.publish= and =ores_reporting_publish_report_definitions_from_dq_fn=
entirely, even though that publish-from-dq function already exists
and is correctly wired (per
[[id:F05B779F-769B-49D5-B369-D2AAC029B36D][DQ Bundle Publication Pipeline]]).

This is a stale halfway-migration: the backlog capture
[[id:B347A156-5AC1-485B-A637-4E9055C1B698][Source report definitions from DQ instead of hardcoded C++]] already
flagged the hardcoded-array problem and the DQ *storage* side (dataset,
artefact table, publish function) was subsequently built — but neither
consumption call site was ever switched over to use it. Replace both
with a generic =dq.v1.bundles.publish= call for =bundle_code =
"ore_analytics"=, mirroring the =organisation= bundle publish that
already works this way in both places, so report definitions (and any
future =ore_analytics= member, e.g. synthetic FX configs) provision
identically with zero new code per dataset.

Confirmed as a live, recurring cost while implementing
[[id:C236C0DC-4ED1-4EEE-8DDD-CD841CED3AB2][Publish a curated FX driver-rate dataset]]: publishing
=marketdata.reference_vintage_2016_02_05= for a party required adding
a *sixth* near-identical hand-written phase (=startFxDriverRatesPublish=/
=onFxDriverRatesWorkflowComplete= in the Qt wizard,
=[5/6]= in the shell CLI) — on top of the pre-existing gap where
=synthetic.fx_spot_configs= itself had never been wired into either
provisioner despite already being an =ore_analytics= bundle member
(fixed in the same pass, adding a *fifth* near-identical phase). Every
party-scoped bundle (=organisation=, =ore_analytics=,
=marketdata.reference_vintage_2016_02_05=, and any future one) is
currently its own bespoke phase — a =start*Publish=/=on*WorkflowComplete=
callback pair in the Qt wizard plus a =[N/M]= block in the shell CLI —
duplicated in two languages/call-sites per bundle. The fix scope should
widen from "report definitions only" to: replace the fixed phase
sequence in both provisioners with iteration over a data-driven list of
party-scoped bundle codes to publish (e.g. a static list today, ideally
sourced from DQ itself later), so a new bundle is a data change, not a
code change, in either provisioner.

* Acceptance

- Both =PartyExecutePage::startReportInstall= (Qt) and
  =provision_commands::process_party='s reports phase (CLI) call
  =dq.v1.bundles.publish= with =bundle_code = "ore_analytics"=,
  replacing =list_dq_report_definition_templates_request= +
  per-record =save_report_definition_request= calls.
- The =--reports all|none|<name,...>= CLI flag's behaviour is
  preserved or deliberately simplified with rationale recorded here —
  bundle-level publish operates per-dataset, not per-row, so exact
  per-report-name selection may need to become an =opted_in_datasets=-
  style filter, a post-publish prune step, or be dropped with a
  documented reason.
- =reporting::messaging::save_report_definition_request= and
  =dq::messaging::list_dq_report_definition_templates_request= are
  either retired or clearly re-scoped to a use they're still needed
  for (e.g. read-only preview UI), not left as unused dead code.
- Adding =synthetic.fx_spot_configs= (already a member of
  =ore_analytics=, see the FX story) to a provisioning run requires no
  further code changes — proving the fix is genuinely data-driven.
- The =organisation= and =marketdata.reference_vintage_2016_02_05=
  bundle publishes (currently their own hand-written phases,
  =startOrgPublish=/=startFxDriverRatesPublish= in Qt,
  =[2/6]=/=[5/6]= in the shell CLI) are folded into the same
  data-driven bundle-list iteration, not left as further bespoke
  phases — a new party-scoped bundle added in future requires a data
  change (adding a bundle code to the list) in both provisioners, not
  a new phase method/callback pair.
