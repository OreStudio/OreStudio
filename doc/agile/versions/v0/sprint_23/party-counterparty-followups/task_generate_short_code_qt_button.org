* Goal

Today, manually creating a party or counterparty requires typing the
short code by hand -- =ores::utility::string::generate_short_code()=
(and its SQL mirror, =ores_utility_generate_short_code_fn=) already
exists and is used server-side by the GLEIF/synthetic bulk-import
paths (=refdata_publish_from_dq_create.sql=), but nothing in the Qt
Create dialog calls it. Add a small "generate" affordance (button or
icon) next to the short code field that calls it against the current
Full Name value and fills the field, for entities whose codegen model
marks a natural (non-UUID) =is_key= field -- i.e. exactly the shape
party/counterparty have (=short_code= is user-entered/editable, unlike
an auto-generated UUID =id=).

Scope: extend the =ores.cpp.qt= detail-dialog codegen templates
(=cpp_qt_detail_dialog.*.mustache=) so this is a generic, declarative
facet (e.g. a =:short_code_generator: true= flag on the Detail field,
mirroring how =is_auto_generated_key= is already derived), not a
one-off hand-wired button on party/counterparty alone -- other
entities with the same natural-key shape should get it for free on
next regen.

* Acceptance

- A "generate" button/icon appears next to the short code field on
  party and counterparty's Create dialogs (edit mode: field stays
  locked as today, generator hidden/disabled).
- Clicking it calls =generate_short_code(fullName)= client-side (Qt
  links =ores.utility=, no server round-trip needed) and populates the
  short code field, still editable afterwards.
- The generator affordance is declarative in the =.org= model (a Qt
  Detail-field facet), so it round-trips cleanly through
  =codegen entity generate <entity> --address ores.cpp.qt --diff= with
  zero unrelated diff, and is available to any future entity with the
  same natural-key shape without hand-authored code.
- Builds cleanly; manually verified live for both party and
  counterparty Create dialogs.
