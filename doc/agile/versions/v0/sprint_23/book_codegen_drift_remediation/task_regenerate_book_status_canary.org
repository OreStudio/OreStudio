* Plan

Diff-reviewed =compass codegen entity generate book_status --diff=
before running for real: all changes were safe template evolution
(new NATS event registrar, a real bug fix in ClientBookStatusModel.cpp
missing a =result->success= check, a real bug fix in the insert
trigger — =current_timestamp= -> =clock_timestamp()= for
same-transaction multi-write safety, a new unique index on
(tenant_id, name), and cosmetic formatting). Ran the real generation,
wired the new event registrar into event_registrar.cpp (previously
missing there for book_status), and built cleanly.

=db recreate= applied the modified SQL with no errors; a manual SQL
insert confirmed the version-bump/bitemporal-close trigger path.

Manual QA via the scenario below surfaced a real regression:
=book_status.org='s Qt section was missing =:has_change_reason_cache:
true= (present on both =currency.org= and =book.org=) — the exact bug
class PR #1449 found for =book= itself. Save silently no-op'd with no
error (client-side: the change-reason prompt/cache was never wired, so
the save request likely never fired). Fixed the model, regenerated the
Qt layer, and found a second issue: =BookStatusController='s
constructor gained a =ChangeReasonCache*= parameter but its only call
site — in =ores.qt/trading/src/TradingPlugin.cpp=, a leftover from the
book Qt-layer relocation in PR #1449 — wasn't updated. Fixed the call
site to pass =ctx_.change_reason_cache= (mirroring how =BookController=
already does). Rebuilt cleanly; re-ran the scenario — all steps
passed, zero ERROR-level log lines.

That TradingPlugin.cpp call site is itself a symptom of a wider
architectural issue — =BookController=/=BookStatusController=/
=RegulatoryBookTypeController= live in =ores.qt/refdata= but are still
constructed and menu-wired from =ores.qt/trading=. Rather than patch
it inline again for the next two entities, filed a dedicated story to
fix it once for all three and unblock this one:
[[id:0E63013B-3B02-4968-A9EA-ED85C00EC7AF][Move book to refdata]].
