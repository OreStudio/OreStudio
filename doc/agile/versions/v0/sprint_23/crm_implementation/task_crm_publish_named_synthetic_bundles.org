* Goal

Once [[id:BA82072F-0B2B-48F4-9C22-8D6D39337099][multi-named-config engine support]] lands, seed demo/test parties
with realistic, concurrently-enabled CRMs via the DQ synthetic data
pipeline (=ores.dq= dataset bundles, following the pattern used by
[[id:D1234E9D-D79A-4C59-991C-988D1D8F515F][the Basic/Realistic synthetic data collections story]]) so the
cross-rates matrix UI panel can actually be exercised, and so any user
who runs a synthetic publish gets a usable CRM out of the box -- not
hand-seeded rows that only exist in one developer's local database.

Two named CRMs per party, matching real FX desk practice:
- *majors* -- USD-pivot, dense direct-quote mesh for G10 pairs
  (EUR/USD, USD/JPY, GBP/USD, EUR/GBP, EUR/JPY, ... direct crosses where
  liquidity justifies a direct quote, not just a triangulated one).
- *exotics* -- USD-pivot star only (no direct EM/exotic crosses --
  matches real liquidity: nobody quotes TRY/ZAR directly).

* Acceptance

- A DQ dataset bundle (or equivalent synthetic-generation config) that,
  on publish, creates two enabled =crm_topology_config= rows per demo
  party ("majors", "exotics"), each with its own =crm_driver_pair= set
  (spanning-tree edges) and, for majors, a curated
  =crm_enabled_derived_pair= list of commonly-quoted crosses.
- Driver pairs are currencies/pairs the synthetic FX generator already
  publishes ticks for (no CRM edge with no underlying feed).
- Re-running the publish is idempotent (matches the existing
  synthetic-data-collection conventions -- no duplicate configs).
- Manually verified: after a publish, the CRM cross-rates matrix UI
  panel ([[id:C685A8BE-0DF9-4D23-8AAB-E9DF9069E482][the blocked UI task]]) shows a non-trivial NxN grid for both
  CRMs.
