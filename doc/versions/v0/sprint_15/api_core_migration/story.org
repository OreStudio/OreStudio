* Goal

Split all ten service libraries into *.api (contracts) and *.core
(implementation) layers — a deliberate three-layer architecture
pattern.

* Acceptance

- Phase 1: ores.iam + ores.compute split.
- Phase 2: ores.refdata split.
- Phase 3: scheduler + assets + variability + synthetic + trading + reporting split.
- All 10 service libraries follow the three-layer api / core / service pattern.

* Decisions

- /Three-layer api/core/service pattern/ :: api = contracts + messaging + eventing; core = handlers + repositories + services; service = process boundary.
- /Iterative migration/ :: iam + compute first (smaller), then refdata, then the remaining six — minimises blast radius per PR.

* Out of scope

- Splitting test harnesses (separate).
