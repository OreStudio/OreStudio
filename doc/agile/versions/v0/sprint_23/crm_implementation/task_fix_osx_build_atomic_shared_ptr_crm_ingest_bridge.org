* Goal

macOS CI (Xcode 16.2, libc++) fails 4 errors deep on
=crm_ingest_bridge.hpp='s =std::atomic<std::shared_ptr<const engines_map>>
engines_;= member:

#+begin_example
error: _Atomic cannot be applied to type
'std::shared_ptr<const std::map<...>>' which is not trivially copyable
error: static assertion failed ... std::atomic<T> requires that 'T' be
a trivially copyable type
#+end_example

=std::atomic<std::shared_ptr<T>>= has a dedicated, non-lock-free-required
specialization since C++20 (P0718) precisely so it doesn't need to be
trivially copyable -- Linux/clang's libstdc++ implements it and accepts
this code. The macOS toolchain's libc++ apparently falls through to the
generic =atomic<T>= primary template instead, which does require
trivial copyability, hence the failure. Need to either confirm/fix the
libc++ version-gating (a newer libc++ atomic<shared_ptr> may just not be
enabled for this deployment target) or use a portable equivalent (e.g.
wrap in =std::atomic<std::shared_ptr<...>>= only where verified
supported, otherwise use a small hand-rolled RCU-style pointer swap with
a mutex, or reach for =immer::atom= as used elsewhere in
=ores.analytics.quant= for exactly this "lock-free single-producer
snapshot swap" need).

* Result

Replaced =std::atomic<std::shared_ptr<const engines_map>> engines_;=
with a =mutable std::mutex engines_mutex_;= guarding a plain
=std::shared_ptr<const engines_map> engines_;=. =snapshot()= now takes
the lock only to copy the =shared_ptr= (a refcount bump) and release;
=refresh()='s single map-swap does the same. Read/write semantics and
thread-safety guarantees are unchanged -- this is a portable expression
of the identical "atomic snapshot swap, many concurrent readers" pattern,
not a behavioural change. Filed
[[id:2CA8864F-336E-4632-A510-FC503213165B][a capture]] to switch back to
=std::atomic<std::shared_ptr<T>>= once macOS's libc++ implements the
C++20 =atomic<shared_ptr>= specialization for this case.
