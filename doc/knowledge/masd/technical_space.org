A /technical space/ (TS) is a programming language or platform ecosystem
that defines a coherent set of conventions — file extensions, directory
layout, toolchain, build system, and idioms. It is the broadest grouping
in [[id:787A7429-71D5-4295-829B-A9B56275B1F8][physical space]]: everything the generator produces for a given language
lives under its TS.

This concept originates in Model-Driven Engineering. Kurtev et al. define
it as: /a working context with a set of associated concepts, body of
knowledge, tools, required skills, and possibilities./ Mens and Van Gorp
refine it as: /determined by the meta-meta-model used at the M3 level/.
In MASD the definition is kept concrete — a TS is identified by what the
generator emits, not by a formal M3 specification.

See [[https://mcraveiro.github.io/dogen/docs/from_problem_space_to_solution_space.html][From Problem Space to Solution Space ↗]] for the original derivation.

* Position in physical space

Within the [[id:787A7429-71D5-4295-829B-A9B56275B1F8][physical space]] address:

: [technical space].[part].[facet].[archetype]

the TS is the outermost segment. Every [[id:B079DA67-8408-4BD0-A93C-D108AC780872][facet]] and every archetype
belongs to exactly one TS. The [[id:787A7429-71D5-4295-829B-A9B56275B1F8][Physical Metamodel (PMM)]]
encodes which TSs exist and which facets are defined within each.

A single [[id:4F24EEB9-58F1-42FD-BF4F-121D890F2D85][logical entity]] typically projects into several TSs simultaneously.
For example, the =currency= entity in ORE Studio projects into:

- the *C++ TS* — domain struct, JSON I/O, repository, service, NATS protocol, Qt UI
- the *SQL TS* — DDL create script, notify trigger, drop script
- the *CMake TS* — component build target

Each TS projection is independent: disabling the Qt TS for a component
suppresses those artefacts without affecting the C++ domain or SQL outputs.
