** Service gap detail

The generated =country_service= is insufficient because the service template lacks:

1. *Pagination* — generates =list_countries()= with no args; hand-written takes =offset= and =limit=.
2. *Count query* — no =count_countries()= in template.
3. *Naming convention* — template emits =remove_country/remove_countries=; hand-written uses =delete_country/delete_countries=.
4. *Party-scoped access* — =list_countries_for_party= and =count_countries_for_party= require a =party_country_repository junction_repo_= member; template has no mechanism to inject junction repositories.
5. *No paste injection point* — unlike =country_repository.hpp=, the service template has no =<<paste:...>>= marker; custom methods cannot be injected.

The hand-written service at =projects/ores.refdata/core/include/ores.refdata.core/service/country_service.hpp= is retained unchanged. A future task should extend the service template with pagination and paste injection.

** Generator gap detail

Country uses =generate_fictional_countries= which returns 50 hardcoded fictional alpha-2/alpha-3 codes (e.g., =ZX=, =QX=) that do not match any real ISO 3166-1 entry. This is intentional: test data must not use real country codes. The faker-based template output (=generate_synthetic_country/countries=) would produce real or realistic codes, defeating the purpose. The hand-written generator at =projects/ores.refdata/api/include/ores.refdata.api/generators/country_generator.hpp= is retained.
