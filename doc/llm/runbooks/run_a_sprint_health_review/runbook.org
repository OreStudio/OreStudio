This page documents a [[id:B555D2A6-10CA-4A93-973E-F2AF4E1D7E55][runbook]] — a named, repeatable composition of [[id:0C57932D-D13A-480F-B426-49525AA8BA3D][recipes]] and [[id:6054CC20-5F41-482F-A587-759512577B1D][skills]] for a complete multi-step procedure. Each step references a recipe or skill by id-link.

* Goal

Produce a structured System 2 health report for a sprint at close, with
up-to-date charts, and commit the findings into =sprint.org= under
=* Health Review=.

* Preconditions

- On the =main= branch, up to date (or on a fresh =feature/sprint-N-review= branch).
- Sprint is ending or recently closed — this is an end-of-sprint
  ritual. For mid-sprint check-ins use the same =sprint-reviewer=
  skill ad hoc (no chart regeneration required).
- Repository root is the current directory.
- =python3=, =gnuplot=, and =gh= are available on =PATH=.
- =SSH_AUTH_SOCK= is exported (see [[id:F7E4C41F-AF34-43E5-ADC3-1D837126B3DE][Set SSH_AUTH_SOCK for git operations]]).

* Steps

In execution order:

1. =compass goto= to create a review story and task, or use an
   existing sprint-health-review story from the sprint.

2. Regenerate the sprint charts so the review references fresh data:
   #+begin_src sh
   cmake --build build/output/linux-clang-debug-ninja \
     --target sprint_charts -DSPRINT=XX
   #+end_src
   Replace =XX= with the sprint number. See
   [[id:6F3D9B1A-5C7E-4A2D-8F1B-3C9D7E5F2A1B][How do I generate sprint health charts?]] for
   details.

3. Verify the charts rendered: check =doc/agile/versions/v0/sprint_XX/=
   for =prs_commits.png=, =line_churn.png=, =pr_cycle.png=, and
   =stories_done.png=. Open them to confirm they look sensible — a
   flat-line =stories_done.png= when the table says several stories
   are DONE is a data bug worth investigating.

4. Invoke the =sprint-reviewer= skill to produce the health review:
   load [[id:16851249-2FB0-443E-83E0-55D6B8051272][Sprint health review — System 2 analysis]] and
   follow its task instructions. The skill writes its findings into
   =sprint.org= under =* Health Review=.

5. Review the output. At sprint end this should include:
   - Goal alignment — every sprint goal has coverage; uncovered goals
     are flagged.
   - Sprint load — commit count and projected total against the
     ≤ 300 threshold.
   - PR velocity — merged per day, any lingering open PRs.
   - Story/task balance — orphan stories, decomposition completeness,
     DONE ratio.
   - Focus signal — simultaneously STARTED stories, theme dispersion.
   - Overall verdict — red/amber/green per dimension, narrative summary.

6. Commit the updated =sprint.org= (with new =* Health Review= section
   and chart PNGs if regenerated) and push. See
   [[id:F4B1A670-3B02-11F1-BCE2-4B06F4BBA5D9][How do I commit to git?]].

7. Raise a PR for the review. See
   [[id:CC928252-E0C5-4C3C-A7A8-D613593E4E37][How do I create a PR?]].

* Postconditions

- Sprint has a current =* Health Review= section in =sprint.org= with
  all seven dimensions assessed.
- Sprint chart PNGs (4 files) are regenerated and committed alongside
  the review, reflecting close-of-sprint data.
- PR is open for the health review, linked from the review task.
